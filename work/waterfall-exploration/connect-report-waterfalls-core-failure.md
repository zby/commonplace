## Connections Found

- [Agentic systems interpret underspecified instructions](kb/notes/agentic-systems-interpret-underspecified-instructions.md) -- **extends**: The waterfall note identifies the same core problem this note formalises. The waterfall note says "natural language lacks precise semantics" and that the same requirements document yields different interpretations in different readers; the underspecification note gives this a theoretical apparatus: spec-to-program projection, one-to-many mapping, and the distinction between semantic underspecification and execution indeterminism. The waterfall note is the historical case study; this note is the formal model. Crucially, the waterfall note's "LLM inversion" section -- where ambiguity moves from the input channel to the processing engine -- maps directly onto the underspecification note's framing of LLMs as projecting from a space of valid interpretations.

- [Legal drafting solves the same problem as context engineering](kb/notes/legal-drafting-solves-the-same-problem-as-context-engineering.md) -- **synthesizes**: The waterfall note argues that software engineering failed to solve the natural-language-specification problem; the legal drafting note shows that law has centuries of accumulated methodology for the same problem (natural language interpreted by a judgment-exercising processor). Together they suggest a historical irony: the waterfall community looked to formal methods and structured analysis for remedies, but the most mature tradition for managing specification ambiguity within natural language was always law, not mathematics or engineering. The legal note's observation that "both face the same impossibility: you cannot write a natural language specification that eliminates all ambiguity" directly validates the waterfall note's claim that more detail amplifies ambiguity surface area rather than reducing it.

- [Stabilisation](kb/notes/stabilisation.md) -- **grounds**: The waterfall note's catalogue of responses to the imprecise-semantics problem (formal methods, structured analysis, agile, TDD) maps onto the stabilisation spectrum. Formal methods and TDD are crystallisation -- crossing the medium boundary to precise semantics. Structured analysis is mid-spectrum stabilisation -- adding constraints (DFDs, ERDs) without leaving natural language. Agile is a different strategy entirely: instead of stabilising the spec, shorten the feedback loop so unstabilised specs can't compound errors far. The stabilisation framework provides the theoretical structure the waterfall note's response catalogue lacks.

- [Crystallisation](kb/notes/crystallisation.md) -- **exemplifies**: TDD, as described in the waterfall note ("specification-as-code, tests are requirements written in a language with deterministic evaluation"), is a textbook case of crystallisation -- requirements crossing the medium boundary from natural language to executable code. The waterfall note's formal methods response is also crystallisation, but one that demanded too much cognitive overhead. The crystallisation note's concept of "premature crystallisation" may also apply: waterfall attempted to crystallise requirements into exhaustive documents before the interpretation space had been explored through iteration.

- [Programming practices apply to prompting](kb/notes/programming-practices-apply-to-prompting.md) -- **extends**: The waterfall note describes TDD as "specification-as-code" -- a response to the natural language semantics problem. This note generalises the same insight: typing, testing, progressive compilation, and version control all apply to prompting because prompts face the same underspecification problem that plagued waterfall requirements documents. The waterfall note is the historical origin story; this note is the contemporary transfer.

- [Legal drafting solves the same problem as context engineering](kb/notes/legal-drafting-solves-the-same-problem-as-context-engineering.md) + [Stabilisation](kb/notes/stabilisation.md) -- **grounds (combined)**: The legal note's mapping of codification as "stronger stabilisation, not crystallisation" illuminates the waterfall note's structured analysis response. Structured analysis (DFDs, ERDs alongside natural language) is analogous to legal codification: it narrows the interpretation space while staying in a medium that judges (or engineers) must still interpret. Neither is crystallisation because neither leaves natural language entirely.

- [Writing styles are strategies for managing underspecification](kb/notes/writing-styles-are-strategies-for-managing-underspecification.md) -- **extends**: The waterfall community's response of demanding "longer, more elaborate documents" was an implicit bet on the descriptive and explanatory writing styles -- more natural language context to constrain interpretation. This note's framework shows why that bet fails: moving rightward on the style spectrum (toward descriptive and explanatory) increases coverage at the cost of reliability. What waterfall needed was to move leftward (prescriptive, prohibitive) or leave the medium entirely (crystallisation). The waterfall community's instinct -- more words for more precision -- corresponds to exactly the wrong direction on the constraint spectrum.

- [Methodology enforcement is stabilisation](kb/notes/methodology-enforcement-is-stabilisation.md) -- **exemplifies**: The waterfall model's document-driven phase gates are an early instance of methodology enforcement. Requirements documents, design documents, and test plans are methodology artifacts at different enforcement levels. The waterfall failure can be reframed using this note's gradient: waterfall kept all enforcement at the "instruction" level (natural language documents interpreted by humans) and never moved down to deterministic enforcement (executable tests, automated checks). Agile and TDD succeed partly because they push enforcement toward the deterministic end of the gradient.

- [Deploy-time learning: The Missing Middle](kb/notes/deploy-time-learning-the-missing-middle.md) -- **contradicts (partially)**: The waterfall note argues that agile works by shortening feedback loops rather than fixing the specification medium. The deploy-time learning note offers a more nuanced view: agile's iteration is not just a dodge but enables progressive stabilisation -- each iteration reveals which interpretations are correct, allowing commitments to accumulate. The waterfall note frames agile as avoiding the problem; the deploy-time learning framework suggests agile is actually solving it incrementally through the verifiability gradient.

- [Inspectable substrate, not supervision, defeats the blackbox problem](kb/notes/inspectable-substrate-not-supervision-defeats-the-blackbox-problem.md) -- **extends**: The waterfall note's TDD response ("the specification IS executable") anticipates this note's core claim: the way to defeat ambiguity is not supervision or review of natural-language artifacts, but choosing a substrate (executable code, repo artifacts) that is inherently inspectable and verifiable. Both notes identify the same move: escape the natural language medium for one with precise, checkable semantics.

## Synthesis Opportunities

1. **The specification problem is eternal, only the medium changes.** The waterfall note traces a historical arc: natural language specs failed because of ambiguity; formal methods, structured analysis, agile, and TDD were responses; LLMs relocate the ambiguity. The KB's underspecification framework, stabilisation spectrum, and legal drafting note complete this arc into a general theory. The claim: every specification medium sits somewhere on the stabilisation spectrum, and every engineering methodology is implicitly a strategy for managing where on that spectrum its specifications live. This could be a structured-claim note.

2. **Waterfall's "more detail" error is the descriptive-style trap.** Combining the waterfall note with the writing-styles note yields a precise diagnosis: waterfall's instinct to add more natural language detail is equivalent to doubling down on the descriptive/explanatory end of the style spectrum -- which increases coverage but decreases reliability. The successful responses (TDD, formal methods) moved to the prescriptive/crystallised end. This reframes waterfall's failure not as "requirements change" but as a systematic style mismatch between the reliability the process demanded and the constraint level the medium could deliver.

3. **Agile as progressive stabilisation, not just shorter loops.** The waterfall note frames agile as "a pragmatic dodge" -- shortening feedback loops instead of fixing the specification medium. But combining this with the deploy-time learning and stabilisation notes suggests agile is actually progressive stabilisation: each iteration reveals which interpretations of the underspecified requirements are correct, and the accumulating code base IS the stabilised spec. Working software is not a dodge; it is the spec, crystallised incrementally. This synthesis would strengthen the waterfall note's agile section and connect it to the deploy-time learning framework.
