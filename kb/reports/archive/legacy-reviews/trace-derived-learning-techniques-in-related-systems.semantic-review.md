=== SEMANTIC REVIEW: trace-derived-learning-techniques-in-related-systems.md ===

Claims identified: 18

WARN:
- [Completeness] The four-category Axis 1 taxonomy (single-session extension, cross-agent session aggregator, service-owned trace backend, trajectory-run pattern) is framed as the exhaustive decomposition of ingestion patterns across all seventeen systems. Boundary case: a system that **passively indexes existing artifacts** rather than mining traces — e.g. a tool that watches a repo for new markdown files and builds a retrieval index without extracting or transforming content. Such a system would have no extraction or promotion stage yet would still "learn from traces" in a loose sense (the artifacts are trace-derived upstream). The taxonomy's implicit scope boundary is "systems that perform extraction," but this is not stated, and the introduction's framing ("all learn from traces") could be read as covering passive indexers too. The note handles Autocontext's straddling explicitly (line 267), but the outer boundary of what counts as trace-derived learning remains implicit.

- [Completeness] The "What remains open" section claims "none of the fifteen systems demonstrates retirement grounded in explanatory reach, cross-domain abstraction, or open-ended judgment." This is now more nuanced than the prior version — the note explicitly acknowledges cass-memory's mechanical retirement and explains why it does not qualify ("mechanical rather than explanatory"). However, the claim still uses "none of the fifteen systems" when cass-memory's `invertToAntiPattern()` — converting a harmful pattern into an explicit anti-pattern rather than simply deleting it — goes beyond simple score-threshold deprecation. Inverting rather than discarding encodes a judgment that the pattern is actively harmful, not merely stale. Whether this constitutes "principled" retirement depends on an unstated definition of "principled" that distinguishes mechanical-with-semantic-encoding from purely threshold-driven removal.

- [Grounding] The concluding bold claim — "session- or trajectory-derived candidate generation is concrete enough in source code to adapt; oracle-backed evaluation is not" — is attributed as "the concrete update to automating KB learning is an open problem." The linked note's own backlink says "the extraction side of the loop is now concrete in source code across session miners and weight-learning systems; what remains open is evaluation of whether mined candidates deserve durable KB status." The linked note does not use the phrase "concrete enough to adapt" (which implies portability/borrowability), only "concrete in source code" (which states observability). The upgrade from "concrete" to "concrete enough to adapt" is the survey note's own inference. This is a reasonable editorial move — the per-system detail does support adaptability — but readers could mistake the bold sentence as a direct attribution to the linked note.

- [Completeness] The "Log formats matter more than the prompts" claim (section title, line 289) is a strong scope claim: the biggest differentiator across systems is source trace shape, not extraction prompt design. Boundary case: **ACE and ExpeL** both use similar trajectory source formats (benchmark task attempts with feedback) but differ substantially in extraction design — ACE's reflector-curator pipeline with bullet tagging versus ExpeL's explicit rule operations with strength counters produce markedly different artifact structures from comparable inputs. If trace shape were truly the biggest differentiator, systems with similar trace shapes should produce similar outputs. The note partially hedges this with "constrains what can be learned" (line 312), which is weaker than "biggest difference," but the section title makes the stronger claim.

INFO:
- [Completeness] The artifact structure spectrum lists six levels from "minimal verbal hints" to "executable code." A boundary case: **generated prompts or prompt templates** as promotion targets. Several systems produce artifacts that are directly injected into future prompts (cass-memory's playbook bullets, ACE's playbook sections), which are closer to "executable instructions" than to "structured records" — they are not general-purpose code like Voyager's JavaScript, but they are meant to be executed (interpreted by an LLM) rather than merely read. The spectrum implicitly treats LLM-interpreted text and human-readable records as the same category, which is reasonable but elides a functional distinction.

- [Completeness] The "What looks borrowable" list contains twelve items but does not include a pattern that several systems share: **incremental processing with cursor tracking**. ClawVault tracks per-session byte offsets in `.clawvault/observe-cursors.json`; cass-memory's `ProcessedLog` tracks which sessions have been reflected on; Pi Self-Learning checks session size growth. This pattern — tracking what has been processed to enable incremental rather than full re-processing — is arguably more universally borrowable than some items on the list (e.g. "bidirectional extraction from successes and failures" is benchmark-specific). Its omission may reflect editorial judgment about what merits highlighting, but it is a concrete, transferable engineering pattern visible across three of the fifteen systems.

- [Internal consistency] The introduction says "the first seven mine live sessions or service-owned event streams; the latter eight learn from repeated task trajectories or search trees." The count of seven maps to: Napkin, Pi Self-Learning, OpenViking, ClawVault, cass-memory, Autocontext, OpenClaw-RL. But Autocontext is later placed under the trajectory-run pattern in Axis 1 ("Autocontext straddles this boundary... it is placed here because episode-level iteration is its primary learning mechanism"). This means the introduction's grouping ("the first seven") differs from the Axis 1 classification, where only six of the first seven are session/service systems and Autocontext moves to trajectory-run. The note explains the reclassification, but a reader tracking the numbers from the introduction will find seven becomes six without an explicit reconciliation.

- [Grounding] The link to "a functioning knowledge base needs a workshop layer, not just a library" is annotated "grounds the artifact-promotion side of this survey: several systems operationalize workshop-to-library bridges from session traces or run trajectories, even though the weight-learning cases extend beyond that note's domain." The workshop note defines workshop documents by their consumed-value lifecycle — "a task becomes less useful as it progresses." But several surveyed systems' artifacts do not follow consumed-value trajectories: cass-memory's playbook bullets accumulate and mature (candidate to proven), Voyager's skills accumulate permanently, ExpeL's rules persist with growing strength counters. These are library-like in the workshop note's own terms. The annotation's qualifier "even though the weight-learning cases extend beyond that note's domain" acknowledges one mismatch but not this one — the artifact-learning systems themselves often produce library artifacts directly, without a workshop-to-library bridge.

- [Internal consistency] The note says Autocontext is "the clearest inspected system spanning both artifact learning and weight promotion" (line 105), then in Axis 2 says "Autocontext bridges both — symbolic artifacts first, then optionally weights" (line 273). The word "optionally" creates an asymmetry: if weight promotion is optional, then Autocontext is primarily a symbolic-artifact system that can also do weight learning, rather than a system that equally spans both. The Autocontext section's own description supports this — playbooks and session reports are the primary outputs, with JSONL export and model training as a separate offline step. The "clearest system spanning both" framing slightly overstates the bridging when the weight side is an optional add-on.

PASS:
- [Grounding] The link to "learning substrates, backends, and artifact forms" is accurately used. The survey note says "service memory is usually a backend choice within the symbolic artifact substrate, not a third top-level substrate alongside artifacts and weights," directly reflecting the linked note's argument that "tips, notes, rules, prompts, schemas, and playbooks belong to one symbolic artifact substrate even when stored in repos, databases, or memory services." The vocabulary (substrate class, backend, artifact form) is used consistently between the two notes.

- [Grounding] The link to "continuous learning requires durability, not weight updates" is accurately annotated: "the survey's non-weight cases count as continuous learning because their improvements persist, not because they imitate weight learning." The linked note's central claim — "continuous learning requires durable adaptive change, not specifically weight change" — directly supports the survey's treatment of all seventeen systems as learning systems regardless of promotion substrate.

- [Grounding] The link to "automating KB learning is an open problem" accurately represents the directional relationship. The survey note says extraction is now concrete while evaluation remains open; the linked note's "Open problems" section identifies evaluation as the primary gap, and the backlink confirms alignment. The one vocabulary mismatch ("concrete enough to adapt" vs "concrete in source code") is reported as WARN above but does not undermine the relationship.

- [Grounding] The link to "memory management policy is learnable but oracle-dependent" is consistent. The survey note says the remaining bottleneck across all systems is evaluation, not extraction. The linked note confirms from AgeMem's case: "RL can learn memory policy, but only because task completion provides a clear oracle; the KB's evaluation gap is the real bottleneck, not the learning mechanism." The survey generalizes this finding across fifteen systems; the linked note provides the single-system evidence.

- [Internal consistency] The two-axis framework (ingestion pattern x promotion target) is applied consistently across all seventeen systems. Each system maps to exactly one ingestion category and one or two promotion targets, with Autocontext as the explicitly noted bridge case and the source-only systems handled with appropriate confidence qualifiers.

- [Internal consistency] The per-system descriptions consistently follow the five-stage structure (Trigger, Source format, Extraction, Promotion, Scope), with cass-memory additionally including an explicit Reinjection section. The structural consistency supports cross-system comparison.

- [Internal consistency] The five-stage framework now explicitly addresses the deduplication/conflict-resolution concern: "This is common enough to note but varies too much in placement to warrant a sixth stage" (line 27). This is an honest acknowledgment rather than a silent gap, and the reasoning (placement variation) is coherent even if debatable.

- [Completeness] The boundary case of "the simplest possible trace-derived learning system" — one that merely logs and replays traces without extraction — is excluded by the note's framing ("all learn from traces"), where "learn" implies extraction and promotion. Reflexion, the simplest included system, still performs extraction (natural-language self-reflection) and promotion (rolling buffer injection). The lower bound is appropriately drawn.

Overall: 4 warnings, 5 info
