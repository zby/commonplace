WARN:
- [Completeness] The enumeration of frontloadable items (claim 5) lists four categories — variable resolution, file listings, aggregations, template expansion — but the note's own qualifying test ("can this be computed without the LLM's runtime state?") admits cases that do not map cleanly to any of the four. Boundary cases: (a) pre-computed validation or type-check results ("this file conforms to schema X") — not a listing, aggregation, variable, or template; (b) pre-resolved conditional branches (the indirection note's "conditional paths" example where one branch is always dead) — logically frontloadable but not represented; (c) pre-fetched tool outputs such as an embedded git log or API response — could be forced into "aggregations" but the fit is strained since they involve no aggregation. The note uses no exhaustiveness language ("all categories," "the complete set"), which softens this, but the section heading "What qualifies for frontloading" reads as definitional. A reader could treat the four-item list as the full taxonomy when it is illustrative.

- [Grounding alignment — domain coverage] The note claims "Frontloading addresses the complexity dimension of that scarcity" (claim 2), citing the context-efficiency note, which does classify "Frontloading and partial evaluation" as "(primarily complexity)" in its architectural-responses table. However, the note's own body describes savings that are clearly also volume savings: "A pre-computed listing costs only the bytes of the listing itself; the same listing produced at runtime costs the instruction, the tool call, the full output, and the LLM's interpretation of it." The tool-call output and full listing are volume costs. The discovery-avoidance paragraph reinforces this — "the agent never spends tokens determining them at runtime" is a volume argument (fewer tokens spent). The note's evidence straddles both dimensions, but the framing sentence claims only complexity. The context-efficiency note is more precise, saying "primarily complexity" — the qualifier "primarily" is load-bearing and is dropped in this note's claim. This is an overstatement of specificity: frontloading addresses both dimensions, with complexity as primary but not exclusive.

INFO:
- [Completeness — boundary case] The static/dynamic test admits a grey zone the note acknowledges ("some sub-procedures depend partially on known and partially on runtime information"), but a stronger boundary case probes the test itself: pre-computing an index structure (e.g., embeddings or a keyword index over a stable corpus) for use with a runtime query. The corpus is static; the query is dynamic; the index is a new representation optimized for future queries rather than a direct result insertion. This sits uncomfortably in the framework: the note's framing is "insert the result," but the most valuable frontloading here produces a different representation, not a result. "Aggregations" could stretch to cover this, but the mechanism differs (changing representation for future extraction vs. inserting a computed answer). This is worth noting as a case where the PE framing may apply more naturally than the note's own language — producing a specialized program (the index) is closer to PE residualization than to "inserting a result."

- [Grounding alignment — vocabulary] The note frames frontloading as partial evaluation: "you've specialised a program with respect to known inputs, producing a residual program in the same medium. That's partial evaluation, not just preprocessing." The homoiconicity note confirms instructions and data share the same token representation, which the note uses to ground the PE framing. However, the homoiconicity note never mentions partial evaluation — the PE framing is entirely this note's theoretical contribution. This is intellectually honest (the note builds on the observation rather than attributing the PE framing), but a reader following the link to the homoiconicity note will not find PE discussed there. The link relationship annotation says "enables," which is accurate — homoiconicity enables the PE framing — but the asymmetry could mislead a reader expecting bidirectional coverage.

- [Internal consistency — claim 7/8 vs examples] Claims 7 and 8 assert that frontloading does not require determinism and works "whether the result is deterministic or still underspecified." This is internally consistent with the codification distinction. However, every concrete example in the note (variable resolution, file listings, aggregations, template expansion) produces a deterministic result. The non-deterministic case — frontloading an approximate or underspecified result — is asserted but never illustrated. A concrete example (e.g., inserting a pre-generated summary, or a pre-computed relevance ranking that is approximate but sufficient) would ground the claim and demonstrate that the distinction from codification is practically operative, not just theoretically possible.

- [Grounding alignment — scheduling model framing difference] The note says frontloading is "the single-step case" of the scheduling model's separation. The bounded-context-orchestration-model note confirms this: "frontloading spares execution context — mechanism: the single-step mechanism this note extends to an iterative loop." The attribution is accurate and bidirectional. However, the scheduling model frames the single-step case as symbolic computation happening outside the LLM call (compute in code, reserve the call for judgment), while this note frames it as partial evaluation of the instruction itself (specialize the program, produce a residual). These are compatible perspectives but emphasize different things — one is an architectural separation, the other is a program-transformation view. Neither note acknowledges this framing difference, which could matter if a reader tries to reconcile the two accounts.

The underspecified-instructions note confirms: "Natural language specifications don't have precise denotations" and develops the spec-to-program projection model. The attribution is accurate and the disanalogy is well-supported.
- [Grounding alignment — build-time generation] The note cites generate-instructions-at-build-time as "a specialiser for skill templates." This is indeed textbook PE specialization — consistent with the frontloading note's PE mapping (row: "Specialisation / The build-time/setup-time step that produces concrete instructions").
- [Grounding alignment — context-efficiency note] The note's title claim — that frontloading spares execution context, which is the primary bottleneck — is grounded by the context-efficiency note, which establishes that "context is the single scarce resource in agent systems" and explicitly lists frontloading as an architectural response. The dependency is accurate and the context-efficiency note is the appropriate foundation.
- [Internal consistency — title vs body] The title claim ("frontloading spares execution context") is faithfully developed throughout. The body consistently frames the benefit as sparing execution context, the mechanism sections (PE, homoiconicity) support the title, and no section contradicts or drifts from this framing. "Frontloading" means the same thing in every section — pre-computing what is known before the LLM runs and inserting the result.
- [Internal consistency — codification distinction] The note carefully distinguishes frontloading from codification: "frontloading does not require determinism — the context saving comes from replacing derivation with insertion." This is consistent with the description field ("the mechanism is partial evaluation applied to instructions with underspecified semantics") and does not contradict the body's treatment of deterministic examples. The distinction is maintained cleanly throughout — no section collapses frontloading back into codification.
