## Mechanism test (reach test from `kb/notes/COLLECTION.md`)

**Do the four notes share one causal mechanism (something relocated, from where, to where, at what cost/benefit), or four separate observations under a vague shared label?**

Breaking down what each note actually contributes to the proposed "burden relocation" story:

- Context-efficiency = an economic/feasibility argument for *why* context is scarce. It says nothing about relocating cognitive burden between locations; it's about a resource-scarcity ranking (feasibility binds before cost).
- Deploy-time-learning = a *timing* taxonomy (training / in-context / deploy-time) for where durable adaptation lives. It already states the connection to context-efficiency directly in its own body: stored artifacts let effective context exceed the literal window. This is the strongest existing thread, and it's already written, unlabeled.
- Runtime-decomposition = a *structural* claim about which architectural component does what (scheduler / context engine / substrate). It doesn't argue that anything is being "relocated" for burden-reduction reasons — it's a taxonomy of responsibilities, and it is **already** integrated into `context-engineering.md` as "the operational core... within the broader architectural discipline," with no need for a new frame.
- Skills-derive = a specific *derivation operation* (distillation) between two artifact types (methodology, skill). It too already states the context-economics link explicitly in its own body ("distillation is a context-budget operation... driven by context economics").

So two of the four notes (deploy-time-learning, skills-derive) already assert, in their own words, exactly the connective tissue the proposed synthesis claims to newly supply — via ordinary reference to context-efficiency, not via a new "externalization" label. The third (runtime-decomposition) is already folded into the context-engineering definition note as a component view. The fourth (context-efficiency) is the common ancestor all three already cite. There is no missing connective mechanism here to discover; the graph already carries these edges in plain language.

**Is there a genuinely new predictive claim?** The one piece of the proposed frame that isn't already present anywhere in the KB is the *task-type transformation* claim from the source paper: recall → retrieval, generation → composition, ad hoc coordination → structured contract. This is the only part of "cognitive burden relocation" that would add reach beyond restating "artifacts save context" — a claim that moving reasoning into a durable artifact changes not just token volume but the *kind* of cognitive operation the bounded call performs (recognition/lookup being cheaper and more reliable than recall/generation, independent of token count).

But this claim is not actually derived from the four cited KB notes — none of them make a recall-vs-recognition argument. It is imported wholesale from the external survey paper. The connect report itself flags this precisely: "The paper is a broad survey/review, not an empirical study... do not treat its externalization thesis as experimentally established," and "Cognitive-artifact analogies are useful but easy to vary. Preserve the mechanism... rather than adopting the analogy as sufficient proof." The report is warning that this part is speculative, not grounded.

**Applying the KB's existing test for exactly this situation.** `psychology-to-agent-transfer-needs-per-principle-failure-mode-testing.md` gives a "simpler-mechanism test": if a proposed principle's engineering value reduces to a standard concept the KB already has, the imported framing is naming, not explaining — decorative. Applying it here: the engineering value of "externalization/cognitive burden relocation" reduces entirely to concepts already named and already linked — context efficiency (why), distillation (how), deploy-time learning (where), context engineering's routing/loading/scoping/maintenance (the umbrella). Four of five analogous psychology imports in that note's own worked example were judged decorative on the same test; this synthesis lands in the same bucket.

**Could someone say exactly how it's wrong?** Yes, cleanly: "these four notes don't share a causal mechanism, they share a topic (context scarcity and its architectural responses). Two of them already state the actual connective claim in their own prose without needing new vocabulary, and the definition note (`context-engineering.md`) already integrates all four as components/operations of one discipline. The only genuinely new content the frame would add — the recall/recognition task-type distinction — isn't backed by the four notes at all, only by an unverified external survey the connect report itself flagged as speculative." That is a specific, landing objection, not just "it's incomplete" — which is itself informative: it means the frame fails the reach test rather than passing it.

## Verdict: DISMISS

This is the relabeling trap the triage README flagged. The KB already has the pieces *and already has the connections between them*, stated directly in note prose (`deploy-time-learning`'s and `skills-derive`'s own context-economics arguments) and structurally (`context-engineering.md`'s existing integration of distillation and the runtime decomposition). Layering "externalization" or "cognitive burden relocation" on top adds a label, not a mechanism. The one part of the proposed frame that would be genuinely new — task-type transformation (recall→retrieval, generation→composition, coordination→contract) — isn't grounded in the four cited notes and rests entirely on an external paper the connect report already cautions against treating as established.

No KB note written. No compression-gate review run (not applicable to a DISMISS verdict).

## Note for future revisit

If a future source or note independently argues *why* recognition/lookup is more reliable than recall/generation for LLMs specifically (not just asserts it by analogy to Nonaka & Takeuchi), and ties that argument concretely to one of these four notes as a premise change with a testable prediction, the frame would be worth re-examining. Until then, treat `deploy-time-learning-is-the-missing-middle.md` and `skills-derive-from-methodology-through-distillation.md`'s existing context-economics arguments, plus `context-engineering.md`'s existing integration, as the complete and sufficient statement of this connection.
