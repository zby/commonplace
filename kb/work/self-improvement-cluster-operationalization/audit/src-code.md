# Phase-1 audit — `src/commonplace/` package (subsystem level)

Group: the shipped Python package, audited as five subsystems rather than per file. Date: 2026-07-23.

Theory applied: [self-improving-systems README](../../../notes/self-improving-systems-README.md), [behavioral authority](../../../notes/definitions/behavioral-authority.md), [the proposal-selection loop](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) (strict evaluation criterion), [false-positive generation is filtered before retention](../../../notes/false-positive-generation-is-filtered-before-retention.md) (automate-search-first), [warranted autonomy is bounded by oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md), [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md), [the declared Commonplace frame](../../../reference/commonplace-declared-frame.md), [the two-layer execution system](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md).

The parked [`warn` conditional](../warn-outcome-vs-loop-evaluation.md) governs subsystem 3; verdicts are recorded under reading A (gates are search over incumbents feeding the fix queue) and reading B (retain-with-obligation) wherever they differ. This clone has no `kb/reports/fixes/` and no populated store, so the empirical check cannot run here — but one half of it turns out to be code-decidable (see subsystem 3, finding 3.1).

Frame note: everything audited here is inside the [declared frame](../../../reference/commonplace-declared-frame.md) — "commands, validators, the review store" are named computational components. Attributions below are read against that frame.

## Validation subsystem (`lib/validation.py`, `lib/type_resolver.py`, `lib/frontmatter.py`, `cli/validate_notes.py`)

- **Authority:** Consumers: agents running `cp-skill-validate` / `cp-skill-write`, and maintainers invoking `commonplace-validate` directly. Channel: CLI execution, reached through skill loading and `AGENTS.md` instruction ("Tests: pytest — all tests must pass" plus the skills). Force: enforcement **at invocation** — exit 1 on any fail (`cli/validate_notes.py:224`), warns do not fail the run. There is no commit hook and no CI validation workflow (`.github/workflows/` contains only `pages.yml`), so the enforcement channel is instruction-mediated: an agent that never runs the validator is never blocked by it. `type_resolver` and `frontmatter` have no independent authority; they are the validator's (and review selector's) substrate.
- **Loop function:** Evaluation, and it passes the strict criterion cleanly: a fail verdict controls an operation distinct from producing the next candidate — per `cp-skill-write`, the note is not committed until validation passes, so rejection and revising-again are different events. The batch signals (orphan info, description-length warnings) are search — noticing surfaces feeding human attention, with no wired downstream operation.
- **Oracle warrant:** The best-scoped oracle in the repo. What it establishes is stated precisely and in the artifact itself: [validation-contract.md](../../../reference/validation-contract.md) says "a type declares what the document must contain; the framework checks that what it points at is really there," explicitly scopes itself to "the deterministic half only," and `commands.md` adds "description discrimination quality is handled by review gates, not deterministic validation." Structure, links, schema conformance, referential marks (`complete`/`covered_by` re-derived from the collection), and verbatim-quote identity — never quality. The severity design respects domain boundaries with stated reasons (a false verbatim copy fails, a dangling link warns — the derived-copy rationale). The `complete`-mark check is the enforced membership claim that [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md) calls the repaired form of the sharpest failure. No gate found whose acceptance exceeds what its check establishes. Consumer misread risk exists but is mitigated at the source: findings are labelled `[base]`/`[type:]`/`[schema]`, and the contract note says the pass is not the whole type contract (prose binds at review time).
- **Retention:** The validator retains commitments as code constants (`MAX_NOTE_TITLE_LENGTH`, tag-readme byte gates, `_NOTE_SLUG_LIMIT_EXEMPT_TYPES`) with in-code scope comments and ADR citations (ADR 024, 026, 046, 048, 050) — well-wired for a reader of the code. The marks system honors the Mark definition (recomputable, enforced-or-omitted) — but see finding 1.1 on when enforcement actually runs.
- **Findings:**
  - **1.1 — "Enforced-or-omitted" is really "enforced-when-invoked."** The `AGENTS.md` Mark definition sells `complete`/`covered_by` as safe to trust because "a stale trusted cache is a trap," and the validator does enforce them — but only inside a validation run that covers the collection. Between invocations, a note edit that adds a tag can leave a `complete`-marked README silently incomplete; the impact-expansion machinery (`impacted_marked_tag_readmes`) helps only when the *edited note* is validated. The trust story the mark makes to exhaustive consumers ("skip the sweep") is therefore only as strong as the invocation discipline, which is instruction-mediated ([behavioral authority](../../../notes/definitions/behavioral-authority.md): declared intent is not enough; the channel decides). This is a channel-strength gap, not an oracle gap.
  - **1.2 — No change suggested on scope statements.** The subsystem is the group's best worked example of [warranted autonomy bounded by oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md): every automated acceptance is matched to a mechanically decidable check, and everything judgment-bearing is explicitly routed to review gates.
- **Disposition:**
  - 1.1 → `proposal`. Decide the enforcement channel deliberately: either a CI/pre-commit validation gate for the collections carrying marked READMEs (making the mark's trust claim invocation-independent), or an explicit statement in `kb/types/tag-readme.md` that mark trust is conditional on the operating convention that edits go through the validating skills. Wiring is a design change (this repo has deliberately thin CI), so not an `apply`.
  - 1.2 → no change.

## Freshness store + transitions (`freshness/*`, `store.py`, `store-schema.sql`, `cli/freshness_*.py`)

- **Authority:** Consumers: the review subsystem (capture finalization, ack, selector), `commonplace-freshness-status` (maintainer-facing staleness report, exit 0/1/2), `store_healthcheck`. Channel: execution against the SQLite store (`kb/reports/commonplace-store.sqlite`). Force: the baseline CAS machinery is enforcement (a queued job against a moved baseline fails with `stale-baseline-revision`); the status output is audit trigger — it matters only when a maintainer or sweep instruction acts on it.
- **Loop function:** Retention bookkeeping, exactly as the workshop framing predicts ("freshness baselines as retention bookkeeping"). What is retained: exact input snapshots (`artifact_snapshots`), one current baseline per `(target_kind, target_key_json)` with a monotonic revision, accepted input roles, and the review evidence bridge. The staleness *selector* is search: it notices changed inputs and produces re-review candidates; nothing it emits controls a reject-capable operation. Note the retention is scoped correctly by its own vocabulary: `AGENTS.md` defines a baseline as "not endorsement or global approval," and `freshness-architecture.md` opens with "It does not adjudicate truth."
- **Oracle warrant:** Where the subsystem automates anything, the oracle is SHA-256 identity over file text — it establishes "these exact bytes were the accepted inputs" and is consumed only as that. The hash boundary is deliberately narrower than the full assay contract (prompt scaffolding is outside it), and the compensating rule is written at the point of change (`review/protocol/prompt.py` module docstring: "judgment-bearing criteria must live in criterion files, where the hash sees them"). That comment is the best piece of path-1-style change-time wiring in the whole package: the constraint is loaded by exactly the artifact an agent edits when it matters.
- **Retention (the freshness-module-review finding and the loop reading):** The [freshness-module-review finding 1](../../freshness-module-review/findings.md) — `commonplace-freshness-accept` can never succeed — was re-verified at the current revision: `V1_ACCEPT_TARGET_KINDS` is still `frozenset()` (`src/commonplace/freshness/transitions.py:20`), and `accept_target_observations` rejects `review-pair` ("must use review capture finalization") and everything else ("not supported for generic accept in v1"), so no input reaches the body. The console script is still shipped (`pyproject.toml:35`). **Effect on the loop reading:** the "general freshness substrate" has exactly one live retention path, and it is wholly review-owned (capture finalization + ack). The generic-target retention machinery is an approved-but-never-merged patch in the loop note's own terms — [retention without a channel is not operative](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) — and worse, it carries a false affordance: `commands.md` documents `commonplace-freshness-accept` as "Observation refresh or initial acceptance for non-review targets," which describes force the command does not have. A retained commitment whose documentation misstates its consumer/channel/force is a broken retrieval wire in the [retrieval-failure](../../../notes/retrieval-failure-is-reflection-failure.md) sense: an agent that finds the doc derives behavior from a capability that does not exist.
- **Findings:**
  - **2.1 — Endorse the freshness-module-review deletion, with a doc obligation the code review did not name.** Deleting the dead accept path (function, CLI, console script, pinning tests) is the theory-aligned move, but the deletion must also sweep the knowledge-artifact layer: `commands.md` (`commonplace-freshness-accept` section), `freshness-architecture.md` (transitions table row and command list), and `freshness-schemas.md` if it specifies the accept shape. Otherwise the false affordance survives the code fix.
  - **2.2 — Snapshot retention has no stated scope.** Nothing ever deletes `artifact_snapshots` rows (freshness-module-review finding 8); retire cascades clear inputs and evidence but orphan the snapshots. The store retains an unbounded evidence trail without a declared retention policy. Under [retention discipline](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) this is a commitment (keep all evidence forever) that was never stated as one — the code review already says "it should be a decision, not an oversight."
  - **2.3 — No change suggested on baseline semantics.** The "baseline is evidence-pinning, not endorsement" definition is stated in three places (`AGENTS.md` vocabulary, `freshness-architecture.md`, `commands.md` on ack) and the code matches it. This is the discipline the `warn` analysis (point 3 of the collision) requires, and it holds.
- **Disposition:**
  - 2.1 → `apply` (already specified in [freshness-module-review findings](../../freshness-module-review/findings.md) §1; extend the edit list with the three doc sweeps named above).
  - 2.2 → `proposal` (a `kb/reference/proposals/` entry deciding snapshot lifecycle: keep-forever-as-evidence declared explicitly, or GC on orphaning).
  - 2.3 → no change.

## Review execution (`review/*`, `cli/review/`) — audited under both `warn` readings

- **Authority:** Consumers: model workers (read rendered prompts, write job output), the finalizer (parses and persists), agents running the fix pipeline (consume `commonplace-warn-selector` output per `FIX-SYSTEM.md`), maintainers (ack commands). Channels: prompt assembly (criterion text into a worker's context), execution, store writes. Force: the criterion has instruction force on the worker for the duration of the assay; the recorded outcome has, as currently wired, **no enforcement force on the note** — a `warn` or `fail` note keeps full behavioral authority, exactly as the [warn analysis](../warn-outcome-vs-loop-evaluation.md) states.
- **Loop function:** Split by reading.
  - *Reading A (gates are search):* the whole execution path — selector, job creation, worker, finalization, warn_selector — is one mechanized search surface over incumbents, generating correction candidates for the fix queue. The evaluation event is downstream (human judging the fix diff); retention is the merge. Under A the system is on the right side of [automate-search-first](../../../notes/false-positive-generation-is-filtered-before-retention.md): automating this surface is the cheap, safe move, and model workers are appropriately placed.
  - *Reading B (retain-with-obligation):* verdict pairs are evaluation of incumbents whose `warn` outcome retains the note with a debt; the baseline plus the warn finding is the obligation record, and `warn_selector` is the discharge tracker.
  - Either way, `finalization.py`'s *own* gate — strict parse, coverage preflight, `ERROR` fails the whole job with no baseline writes — is genuine reject-capable evaluation over protocol conformance with a deterministic oracle, and it passes the strict criterion (the rejected job's baselines do not advance; rejection is distinct from re-running).
- **Oracle warrant:** The model worker is the oracle for semantic criteria; `warranted-autonomy` says this is bare autonomy unless the gate's acceptance stays inside what the model can establish. Two design choices keep the warrant honest: (i) no outcome is consumed as acceptance-with-force (under either reading, nothing operative is gated on `pass`), so a weak oracle cannot silently degrade the KB through this pipeline — the degradation path the theory fears requires an acceptance consumer, and there is none; (ii) the deterministic sub-oracles (partition normalization, CAS, parse) are scoped to what they establish. `ack_trivial_note_changes` is the subsystem's one *automated evaluation* in the strict sense — its verdict (ack) controls baseline advance without human judgment — and its oracle is warranted: a deterministic parsed-part comparison against the gate's declared `watches`, failing closed (no valid declaration → gate watches everything → nothing is trivial; conformance pairs land there by design).
- **Retention:** Baselines are written for `pass`, `warn`, and `fail` alike (`record_and_finalize_job` iterates all finalized pairs). Under A this is correct bookkeeping (evidence freshness is outcome-indifferent). Under B a `warn` baseline is a retain-with-obligation record — coherent — but a `fail` baseline is an obligation with no discharge machinery (see 3.1). `acknowledgement.py` advances baselines on human authority with no check content was reviewed; its scope is stated (`commands.md`: "it does not endorse or resolve the report"), so the human is the declared oracle — warranted exactly as far as the invoking human is.
- **Findings:**
  - **3.1 — `fail` has no computational consumer; this is code-decidable now and is half of the parked empirical check.** `warn_selector.py`'s `ACTIONABLE_FINDING_RE` matches only `- warn:` items, and the fallback chain (`extract_warns`) triggers only when `outcome == "warn"`. A `fail` review whose findings are labelled `- fail:` contributes nothing to the fix queue; grep confirms no other consumer of the `fail` outcome anywhere in `src/` (`protocol/outcomes.py:8` is the only occurrence of the literal outside tests). The [warn analysis](../warn-outcome-vs-loop-evaluation.md) empirical check, step 4, asks "does any consumer exist [for `fail`], or do `fail` notes simply persist untouched?" — code inspection answers the wiring half: **no consumer is wired**; only the data half (do `fail` rows exist and persist) still needs the store. Under reading A, `fail` is documented as "an escalation flag" but there is no wired escalation — the search surface silently drops its highest-severity signal, and [search's omission errors are the invisible, permanent ones](../../../notes/false-positive-generation-is-filtered-before-retention.md). Under reading B, `fail` is a retained obligation with no discharge or escalation consumer at all — strictly worse than `warn`. The verdicts differ only in vocabulary; the gap is real under both.
  - **3.2 — The fix pipeline has no rejection disposition (code side confirms the workshop's suspicion).** The warn-selector output schema has no field for "this finding is spurious," and `FIX-SYSTEM.md`'s fix-report contract names only `fixed` and `deferred`. Whether A is *true in practice* depends on rejections happening at the fix step; the pipeline currently cannot even record one. This is the "repair the pipeline to make A true" option the workshop doc itself flags as "arguably the better" move.
  - **3.3 — `watches` vocabulary drift (small, fails safe).** Code accepts `KNOWN_WATCHES = {body, title, description}` (`ack_trivial_note_changes.py:29`); `kb/types/review-gate.md:19` documents `watches: [body | frontmatter | ...]`; `review-gate.schema.yaml` constrains items only to non-empty strings. A gate declaring `frontmatter` (as the type doc invites) is silently never auto-acked — the safe direction, but a silent one: the author believes they enabled trivial-ack and did not. Retained commitment (the watch vocabulary) with an inconsistent scope statement across its three copies.
  - **3.4 — No change suggested on the freshness hash boundary.** The boundary and its compensating rule are stated in the code at the point of change and in `review-architecture.md`; this is the pattern the operationalization workshop should point to when it designs path-1 wiring.
- **Disposition:**
  - 3.1 → `conditional` on the parked `warn` placement for what the fix *is* (A: wire `fail` into an escalation consumer, e.g. the warn selector grows a severity field, or triage instruction gains a `fail` queue; B: give the obligation a discharge/escalation path) — but the *fact* (no wired consumer) should be recorded in [warn-outcome-vs-loop-evaluation](../warn-outcome-vs-loop-evaluation.md) now, since it partially discharges the empirical check's step 4 from code alone. Recording it is an `apply` for the coordinating session (append a code-verified note to the workshop doc); the wiring fix stays conditional.
  - 3.2 → `conditional` (same parked item; if A is adopted or the pipeline is repaired toward A, the exact edit is: add a `rejected`/`spurious` disposition to the fix-report contract in `FIX-SYSTEM.md` and the fix-strategy instructions).
  - 3.3 → `apply`: change `review-gate.md:19` to `watches: [body | title | description]`, and add the enum to `review-gate.schema.yaml`'s `watches.items`. Both edits are small, safe, and align three copies of one commitment.
  - 3.4 → no change.

## Init/scaffold (`cli/init_project.py`, `scaffold_manifest.py`)

- **Authority:** Consumer: the installer (`commonplace-init`), executed by an operator once per project (never in this repo, per `AGENTS.md`). Channel: execution — the manifest is data the installer walks. Force: configuration, with a consequential downstream effect: `promoted_skills` decides which instruction directories are copied into `.claude/skills/` / `.agents/skills/`, i.e. which prose acquires **harness-loaded instruction force** in consuming projects. This is authority promotion in the [behavioral-authority](../../../notes/definitions/behavioral-authority.md) sense: the same bytes at `kb/commonplace/instructions/<name>/` are advice; the copy under `.claude/skills/` is standing instruction.
- **Loop function:** Retention — it makes the "these instructions are wired" commitment operative in every installed project. The selection of which skills to promote is a human decision outside the code; no search or evaluation is implemented here.
- **Oracle warrant:** n/a — the installer evaluates nothing except file identity (`_record_existing` classifies preserved files as identical/different, never overwrites). That check is warranted and scoped.
- **Retention:** The scope statement is good where it exists: the module docstring says "Changing what ships — directories, scaffold trees, templates, promoted skills — is an edit here, not in installer code," which is a genuine writable-address declaration. The retrieval wire is the problem, in both directions:
  - The commitment is **retained twice**: `MANIFEST.promoted_skills` (9 names) and the Hatch `force-include` map in `pyproject.toml:87-95` (the same 9 paths). ADR 027 acknowledges "the Hatch include maps become maintained lists," but nothing checks they agree. A skill added to one list and not the other ships broken (in the manifest but not the wheel → `FileNotFoundError` at init; in the wheel but not the manifest → silently unpromoted).
  - Nowhere in the KB is the list enumerated. `AGENTS.md` names the family by example ("`cp-skill-write`, `cp-skill-validate`, `cp-skill-connect`, etc.") — "etc." is an unstated scope — and `architecture.md` says "selected skills" without saying which or where selection lives. An author writing a new `cp-skill-*` has no artifact that loads the manifest obligation at authoring time; they must remember `scaffold_manifest.py` exists ([retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md): a commitment that depends on someone remembering it is a best-effort wire). The naming convention partially compensates — per `AGENTS.md`, the `cp-skill-*` prefix *means* promoted framework skill, and repo-local skills use other names — which makes the membership rule enumerable and therefore checkable.
- **Findings:**
  - **4.1 — Add the mechanical consistency check the naming convention already warrants.** Since `cp-skill-*` is the declared promoted-family prefix, a test can enforce the three-way agreement: every `kb/instructions/cp-skill-*` directory appears in `MANIFEST.promoted_skills`, and every `promoted_skills` entry has a matching `pyproject.toml` force-include and source directory. Today all three agree (verified: 9 = 9 = 9); the test pins the invariant instead of the memory.
  - **4.2 — Name the retained commitment where authors will find it.** `kb/instructions/write-instruction.md` (or the instructions COLLECTION.md) should state: a skill intended for the promoted family takes the `cp-skill-` prefix and must be registered in `src/commonplace/scaffold_manifest.py` and `pyproject.toml`. One sentence converts the wire from link-mediated discovery to authoring-time instruction.
- **Disposition:**
  - 4.1 → `apply`: one test in `tests/` asserting `{d.name for d in Path("kb/instructions").glob("cp-skill-*")} == set(MANIFEST.promoted_skills)` and that each name appears in the `pyproject.toml` force-include map.
  - 4.2 → `apply`: the one-sentence registration note in the instruction-authoring artifact (exact placement is the coordinating session's call; `write-instruction.md` is the natural consumer-at-the-time-it-matters).

## Supporting commands (`lib/relocation.py`, snapshot CLIs, `promotion_candidates.py`/`lib/promotion.py`, `lib/full_pass.py` + `cli/guard_full_pass_report.py`, `lib/quote_verification.py` + `cli/verify_quotes.py`, `cli/store_healthcheck.py`)

- **Authority:** All are operator/agent-invoked CLIs. Relocation mutates the repository (enforcement over link rewrites when `--apply`); the snapshot commands and `promotion_candidates` write derived artifacts consumed as evidence (advice force); `guard-full-pass-report` and `verify-quotes` return exit codes consumed by instructions as gates; `store-healthcheck` is an audit trigger.
- **Loop function and oracle warrant, per command:**
  - **Relocation** — retention maintenance: it preserves the retrieval wire itself (rewrites every inbound link so retained references stay causal after a move). Review/freshness state is deliberately not relocated; the compensating rule is documented (`review-architecture.md`: "relocating notes or criteria requires a fresh assay under the new path") and the staleness is *visible* downstream (`freshness-status` reports the missing input) rather than silent, so the wire holds. No warrant issue; it evaluates nothing.
  - **Snapshots (`github_snapshot`, `x_snapshot`, `lib/snapshot.py`)** — pure search-side capture. No evaluation, no oracle claims. No change suggested.
  - **`promotion_candidates`** — search, and exemplary about staying there: it *ranks* text files by inbound links "so operators can decide which captures are worth structuring" (`commands.md`). The mechanical signal (link count) is never treated as a verdict; the automation stops exactly where the warranted oracle ends. This is the [automate-search-first](../../../notes/false-positive-generation-is-filtered-before-retention.md) rule followed to the letter.
  - **`guard_full_pass_report`** — genuine evaluation with a warranted deterministic oracle. Its verdict (exit 1, "Full-pass transition refused") blocks an operation distinct from producing the next candidate — the full-pass disposition transition — so it passes the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md). The oracle establishes exactly textual identity of guarded captures against live artifacts, and that is all the refusal claims. It never mutates. Alongside pre-promotion verification in the two-layer note, this is the package's cleanest reject-capable gate.
  - **`quote_verification` / `verify_quotes`** — evaluation within a mechanically decidable domain (normalized substring containment), with the strictness choices documented in the module docstring: mismatches fail (a false `verbatim` claim is a false copy), unresolved candidates warn only in notes that demonstrably use the convention (deliberate anti-cry-wolf narrowing). The oracle establishes "the normalized span occurs in the linked file," consumed as exactly that. No warrant issue.
  - **`store_healthcheck`** — audit trigger; establishes hash and invariant consistency of the store, not correctness of any review content. Scoped correctly (`commands.md` distinguishes it from routine structure-only checks).
- **Retention:** None of these retains commitments beyond their outputs; the one retention concern (relocation's non-migration of path-keyed baselines) is documented and detectable, per above.
- **Findings:**
  - **5.1 — No automation crosses from search into evaluation without a warranted oracle in this group.** The brief's key question comes back negative, and two commands (`promotion_candidates`, `guard_full_pass_report`) are positive exemplars worth citing when the change-time digest is distilled — one for stopping automation at the oracle boundary, one for a deterministic reject-capable gate.
- **Disposition:** 5.1 → no change (recorded as exemplar material for the later digest phase).

## Group summary

Most worth the maintainer's attention:

1. **`fail` has no wired consumer — and this is now partly settled from code, not just data** (3.1). `warn_selector` extracts only `- warn:` findings; nothing in `src/` consumes a `fail` outcome. Step 4 of the parked empirical check's wiring half is answered: `fail` rows can only persist untouched. Recommend the coordinating session append this code-verified fact to [warn-outcome-vs-loop-evaluation](../warn-outcome-vs-loop-evaluation.md) now; the wiring fix stays conditional on the A/B placement.
2. **The dead `commonplace-freshness-accept` is a broken retrieval wire, not just dead code** (2.1). The endorsed deletion must sweep `commands.md`, `freshness-architecture.md`, and `freshness-schemas.md`, or the docs keep advertising force the system does not have.
3. **`promoted_skills` is a high-force retained commitment with a memory-dependent wire** (4.1, 4.2). It decides which prose gets harness-loaded instruction authority in every installed project, is retained in two unchecked parallel lists, and is enumerated nowhere the authoring path loads. Both fixes are small `apply` items.
4. **The validator's marks trust story is invocation-conditional** (1.1). The one enforcement oracle strong enough to license "skip the sweep" runs only when invoked; whether to wire it (CI/hook) or scope the claim is a deliberate design decision → proposal.
5. **The fix pipeline cannot represent rejection** (3.2), which is exactly what the parked check needs to observe; repairing it is also the move that would make reading A true rather than merely adopted.

Phase-1 scheduled-check observations:

- **(a) Cumulativity's environment-mediated exclusion:** no case surfaced. The store — the package's retention substrate that later selection actually consumes (baselines feed the stale selector; retained snapshots feed diffs) — is *inside* the declared frame's boundary ("the review store" is a named computational component), so the informational dependence it provides counts without invoking environment mediation. The stated exclusion survived contact with this group.
- **(b) Reflection vs self-improvement routing:** `src/` code carries no links into the theory cluster at all — expected for code — but the freshness-hash-boundary comment in `review/protocol/prompt.py` ("judgment-bearing criteria must live in criterion files, where the hash sees them") is a working model of path-1 wiring: the constraint loads in exactly the artifact an agent opens when the constraint matters. The axis-factoring decision gets no signal from this group; nothing here routes cluster traffic.
- **(c) "Promotion" vocabulary:** two of the ledger's three senses appear in this group and both fit the "move into a more binding or durable layer" family. `scaffold_manifest.promoted_skills` / ADR 037 use the *authority* sense (advice → harness-loaded instruction force); `promotion_candidates` / `lib/promotion.py` use the *durability/structure* sense (unstructured text → typed note). Contexts fully disambiguate; no write-time collision observed, and neither artifact needs a sense registration.
