# Phase-1 audit — review/fix pipeline instructions

**Group:** the review and fix pipeline — the instruction and operating artifacts the parked `warn` conditional applies to. Per the [warn analysis](../warn-outcome-vs-loop-evaluation.md), every artifact here is audited under **both readings**: **A** (gates are search over incumbents feeding the fix queue; the human judging the fix diff is the evaluation event) and **B** (gates are evaluation extended with retain-with-obligation; `warn` names a debt something downstream must discharge or escalate). Verdicts are recorded under each wherever they differ.

**Date:** 2026-07-23

**Theory applied:** [self-improving-systems README](../../../notes/self-improving-systems-README.md), [behavioral authority](../../../notes/definitions/behavioral-authority.md), [proposal-selection loop](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) (strict evaluation criterion), [false-positive generation is filtered before retention](../../../notes/false-positive-generation-is-filtered-before-retention.md) (automate-search-first), [warranted autonomy is bounded by oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md), [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md), [declared Commonplace frame](../../../reference/commonplace-declared-frame.md), [two-layer execution system](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md).

**Environment note:** this clone has no `kb/reports/fixes/` directory and no populated commonplace store, confirming the warn analysis's premise that the empirical check cannot run here. Two of its textual sub-questions *can* be answered from instruction text and are answered below (fix-report disposition vocabulary; `fail` consumers).

## kb/instructions/FIX-SYSTEM.md

- **Authority:** Consumer: maintainer or orchestrating agent doing fix work. Channel: routed from standing instructions — `AGENTS.md` closes with "For fixing review warnings, read `kb/instructions/FIX-SYSTEM.md`", so the pointer is loaded every session and the doc is loaded on demand. Force: instruction when loaded (it defines the pipeline, components, and constraints the fix agent follows). Under the [declared frame](../../../reference/commonplace-declared-frame.md), the consuming agents and `commonplace-warn-selector` are inside-frame computational components.
- **Loop function:** The pipeline spec, so its classification is exactly the parked question. **Under A:** it coordinates *search* — `warn` findings are correction candidates, the fix agent's edit is the next candidate, and FIX-SYSTEM wires generation throughput (`review gates → review pairs (DB) → warn_selector → fix instruction → edited note + fix report`). The evaluation event (human diff review) is *not specified here* — see Findings. **Under B:** it is the obligation-discharge machinery — the downstream component that must discharge or escalate the debt a `warn` records. Retention it implements directly: fix reports in `kb/reports/fixes/` (audit trail) and taxonomy growth (see the taxonomy section).
- **Oracle warrant:** n/a directly (it delegates judgment to the fix instruction and the review DB), but it inherits the pipeline-level warrant question: "After fixes land, the next review sweep re-evaluates the note against the same gates" — a second pass against an unchanged oracle, which [false-positive-generation](../../../notes/false-positive-generation-is-filtered-before-retention.md) says "is not a second chance." Acceptable under A (the re-run is more search); under B it means the evaluation function never gets a stronger oracle on retry.
- **Retention:** Fix reports have a declared scope ("maps each warning to the fix applied, the strategy used, and the status") and a named consumer for one purpose: taxonomy evolution ("New strategies are added when `new-pattern` reports from fix sweeps show recurring patterns (3+ instances)"). Retrieval-wire soft spot: *cross-sweep* recurrence counting has no named reader — no step in any instruction reads old fix reports to accumulate the 3+ count, so recurrence across sweeps depends on someone remembering ([retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md)).
- **Findings:**
  1. **The fix pipeline has no representable "warning is spurious / reject" disposition.** Checked directly for the group summary: FIX-SYSTEM names the status set as "the status (fixed or deferred)"; the fix-report table and Outputs section carry only `fixed` and `deferred`; a grep for reject/spurious/invalid vocabulary across `kb/instructions/fix-warnings/` finds nothing. A fix agent that concludes a warning is wrong (the constraint "Verify before assuming staleness" explicitly anticipates this) can only shoehorn that judgment into `deferred`. Under **A** this is the missing downstream filter the whole reading depends on — search's false positives are supposed to be "filtered before retention," but the filter's verdict has no vocabulary, so it cannot be observed and (per the warn analysis's own decision rule) cannot even be measured. Under **B** it means an obligation cannot be discharged by showing it is void — the debt protocol lacks a cancel operation. The two readings **converge** on the same repair, and the warn analysis itself endorses it ("repairing the pipeline... adds the missing filter instead of weakening the theory").
  2. Cross-sweep recurrence counting for taxonomy promotion has no retrieval wire (see Retention).
- **Disposition:**
  1. `apply` — add a third status to the fix-report contract. Exact edits: in FIX-SYSTEM §Concepts change "the status (fixed or deferred)" to "the status (fixed, deferred, or rejected — the warning judged spurious, with the reason)"; in §Outputs add "Rejected warnings with reasons" to the report bullet list; matching edits in `fix-review-warnings.md` (see its section). Note for the coordinator: this also unblocks the parked empirical check's step 2, which currently has nothing representable to count.
  2. `proposal` (small) — give taxonomy promotion a real wire: either the sweep report accumulates against prior sweep reports (a step "grep prior `kb/reports/fixes/` sweep summaries for this pattern name"), or recurrence bookkeeping moves into the taxonomy file itself (a candidates section). Design choice, so not `apply`.

## kb/instructions/fix-warnings/fix-review-warnings.md

- **Authority:** Consumer: the per-note fix agent (a sub-agent under the sweep, or directly invoked). Channel: instruction loading (`$ARGUMENTS`-parameterized). Force: instruction — it binds the edit constraints and report format.
- **Loop function:** **Under A:** search — it generates the correction candidate (the edit). Its step-3 decision ("can this be fixed by changing framing/accuracy without changing the note's argument? ... No → defer") is reject-capable *with respect to automated fixing*: defer and edit are distinct operations, so by the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) this is a genuine micro-evaluation of *fix-candidate admissibility*, gating what the automated path may touch. **Under B:** discharge procedure for the gate's obligation; same defer gate. The classification of the surrounding machinery differs, the artifact's own function does not — recorded as non-conditional.
- **Oracle warrant:** The instruction is a model example of respecting [oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md): the automated agent's warranted domain is framing/accuracy/attribution edits, and the Constraints section narrows the gate to exactly that ("Don't change arguments," "Minimal edits," "Only edit flagged passages"), deferring everything else to "human review with a reason." The one warrant leak is the same as FIX-SYSTEM's: the agent is *implicitly* trusted to re-judge the warning ("Verify before assuming staleness... check whether it actually exists before removing or updating it") without a disposition to record the negative result.
- **Retention:** The fix report retains the warning-to-fix mapping with declared purpose ("so the fix can be audited against the original finding") — scope stated, consumer (auditor, taxonomy evolution) named. The DB is correctly treated as authoritative ("The DB-backed review text in the output is the authoritative source") — a strong retrieval wire to the evidence.
- **Findings:** Same finding 1 as FIX-SYSTEM, located concretely: step 3 offers only Yes → fix and No → defer; the report format's Status column and "Deferred items" section admit no rejected row.
- **Disposition:** `apply` (joint with FIX-SYSTEM finding 1) — exact edits: in §Procedure step 3 add a third branch: "**Warning is wrong** (flagged item verified to exist / claim verified accurate) → reject. Record it as rejected with the verifying evidence; do not edit the note." In §Fix report format, extend Status values with `rejected` and add a "### Rejected warnings" section parallel to "### Deferred items".

## kb/instructions/fix-warnings/fix-review-warnings-sweep.md

- **Authority:** Consumer: orchestrating agent. Channel: instruction loading on invocation. Force: instruction (queue building, delegation protocol, reporting).
- **Loop function:** **Under A:** search-throughput scaling — parallel sub-agents multiply candidate generation, the pattern [false-positive-generation](../../../notes/false-positive-generation-is-filtered-before-retention.md) calls safe *provided* evaluation capacity scales with it ("Automating search costs evaluation throughput"). **Under B:** parallel obligation discharge; the evaluation already happened at the gate, so throughput is less loaded.
- **Oracle warrant:** The sweep's only verification step is "After each worker returns, verify its note changes and report" — one line, performed by the orchestrating *agent*, not the human. Under **A** this is where the warrant question bites: reading A's evaluation event is "the human judging the fix diff" (warn analysis §A, leaning on README-REVIEW-SYSTEM's "reviewing that diff is the judgment step"), but **no instruction in the sweep — or anywhere in the fix pipeline — assigns diff review to a human, sequences it before commit, or records its verdict.** The claimed evaluation function is textually homeless; as consumed, the closest thing to a filter is an agent self-check. Under **B** this is a minor gap (verification of discharge, not the loop's evaluation). This is the group's sharpest genuinely **conditional** finding.
- **Retention:** The sweep report (fixed-by-strategy counts, deferred, new patterns) is transient conversation output — nothing states where it persists, yet taxonomy evolution (step 4) consumes "3+ instances" of recurrence, which a transient report cannot feed across sweeps. Same wire gap as FIX-SYSTEM finding 2.
- **Findings:**
  1. **Conditional:** under **A**, the pipeline's evaluation step (human diff judgment) is unwired — no consumer, channel, or force in any instruction; by [behavioral authority](../../../notes/definitions/behavioral-authority.md)'s exclusion rule ("a formal policy may have no effective authority if no component loads it"), reading A's evaluation currently exists only as declared intent. Under **B**, no change needed here (the gate already evaluated; the sweep merely discharges).
  2. Step 4's "Present the proposed entry to the user before adding" is a clean human gate on taxonomy growth — positive observation, no change.
- **Disposition:**
  1. `conditional` — if the empirical check lands on A (or the pipeline is repaired toward A), the sweep needs an explicit step: "Present the per-note diffs to the maintainer before commit; record declined fixes as rejected in the fix report." If B is adopted, `reject` this change (redundant with gate evaluation) and instead record in the loop note's amendment that discharge verification is agent-side. Either way the `rejected` disposition from FIX-SYSTEM finding 1 should be applied now, since it is what makes the A/B check measurable at all.

## kb/instructions/review-gates/ (gate criteria, 7 lenses, 28 catalog gates)

- **Authority:** Consumer: the review worker model. Channel: the rendered job prompt — the criterion text is snapshotted at job creation and delivered verbatim, an enumerated (not discovery-mediated) wire. Force: instruction — the Test section directs the judgment and names the allowed outcomes. Secondary authority: the file text is the persisted `criterion_path` identity, giving each gate configuration force in freshness bookkeeping (editing one stales exactly its pairs as `criterion-changed`).
- **Loop function:** **The conditional's core.** **Under A:** each gate is a *search specification* — a codified description of a failure class to notice in incumbents; `warn` outcomes are correction candidates for the fix queue. **Under B:** each gate is an *evaluation criterion* — an oracle rubric whose `warn` verdict is retain-with-obligation.
- **Oracle warrant:** Flips with the reading, exactly as the warn analysis predicts. **Under A**, model-run gates are the cheap safe automation ([automate search first](../../../notes/false-positive-generation-is-filtered-before-retention.md)) *provided the downstream filter is real* — which is the unproven premise (see FIX-SYSTEM finding 1). **Under B**, the gates automate evaluation on "the cheapest adequate model," and [warranted autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) makes the semantic lens the exposed flank: `semantic/grounding-alignment` asks for judgment across "at most 5 links" of sources — a domain where a cheap model's discrimination is unestablished, and a false `pass` silently persists (the selector treats the fresh pair as needing nothing until an input changes). The gate texts themselves are well-disciplined about strictness where it is declared: `structural/general-before-specific` has an explicit WARN/INFO tiebreaker; `frontmatter/title-as-claim` states WARN and PASS conditions; `semantic/grounding-alignment` states "Report WARN for scope overreach or invalid inference. Report INFO for plausible but not airtight inferences."
  A corpus-level fact, verified by grep: **no catalog gate's Test body ever names a FAIL condition.** FAIL exists in the type-spec outcome vocabulary (`kb/types/review-gate.md`: "the procedure for choosing PASS, WARN, FAIL, or ERROR") but no criterion tells a reviewer when to choose it — and nothing downstream consumes it (warn-selector selects `warn` only; README-REVIEW-SYSTEM assigns `fail` no handler). `fail` is thus an outcome with neither an authored trigger nor a consumer: by [behavioral authority](../../../notes/definitions/behavioral-authority.md), zero effective force.
- **Retention:** The gates are retained quality commitments with exemplary discipline: scope is declared per gate (`lens`, `watches`, `requires_trait`/`requires_type`), and the type spec's self-containment rule ("Review freshness hashes only note text and criterion text, so a test that leans on prose living elsewhere... carries a dependency that never invalidates acceptances") is precisely the [retrieval-wire standard](../../../notes/retrieval-failure-is-reflection-failure.md) applied to criterion dependencies — a stale-dependency trap converted into hashed, staleness-firing text. No change.
- **Findings:**
  1. **`fail` is dead vocabulary in the catalog** (no authored trigger, no consumer). Under **A**: mildly untidy — `fail` was to be "an escalation flag" (warn analysis §A) but nothing escalates; the honest move is to say so. Under **B**: worse — the evaluation function's strongest verdict cannot fire by any authored rule, so the gates are not even reject-*capable* at the level B needs; retain-with-obligation would rest entirely on `warn`.
  2. Gate strictness declarations are inconsistent across the corpus — some gates state outcome-choice rules (the three quoted above), most leave the WARN threshold implicit. Under B this is an undeclared strictness setting on an evaluation oracle; under A it merely adds noise to the fix queue, which the (currently missing) downstream filter would absorb.
- **Disposition:**
  1. `conditional` in framing, but both branches want an edit: if **A**, `apply` a one-line honesty fix to the review-gate type spec Authoring Instructions — state that catalog gates in practice choose between PASS and WARN, and that FAIL requires an authored trigger plus a named consumer before a gate may use it; if **B**, `proposal` — author FAIL conditions for the gates that need a hard verdict and wire a consumer (triage-to-retirement or maintainer escalation). Do not silently keep the four-valued vocabulary with a two-valued practice.
  2. `reject` as a corpus-wide edit for now — normalizing strictness across 28 gates is exactly the "manufacture findings" trap; the three gates that declare thresholds show the pattern, and new gates copy existing shapes per the type spec. Revisit per-gate if fix-queue noise data ever exists.

## kb/instructions/review-triage.md

- **Authority:** Consumer: the triaging agent (invoked before a sweep). Channel: instruction loading. Force: instruction — it governs the ack decision.
- **Loop function:** A genuine evaluation micro-loop under **both** readings, but its object is *evidence applicability*, not note quality: the diff judgment's verdict controls an operation distinct from producing anything ("Ack all insignificant pairs" vs "the remaining stale pairs will be picked up by the next review sweep") — ack and rerun are different events in the mechanism, satisfying the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md). Non-conditional: A and B agree.
- **Oracle warrant:** The oracle is agent judgment over a diff, guided by per-gate significance heuristics, with the strictness lever set safe: "When in doubt, do not ack — rerun the assay." Acceptance scope matches what the check establishes — ack "advances the existing freshness baseline... It produces no new judgment," and the report-pair asymmetry is stated up front ("for open-ended report pairs, it only reuses the report as fresh evidence and does not imply that its findings were handled"). The failure mode of a wrong ack is bounded (stale evidence persists until the next input change), consistent with the baseline-is-not-endorsement semantics. Warranted domain respected.
- **Retention:** Ack writes baseline rows — retention *bookkeeping* in reading A's vocabulary (evidence freshness, not acceptance); the instruction keeps that straight. Scope of guidance stated per gate family.
- **Findings:** No change suggested. Positive observation: this artifact and its ack semantics are what make the warn analysis's point 3 ("a baseline is... evidence-pinning rather than endorsement") operationally true.
- **Disposition:** n/a.

## kb/instructions/run-review-batches.md

- **Authority:** Consumer: the orchestrating parent agent; workers receive only the rendered prompt and job task block. Channel: instruction loading; the embedded worker task template is a secondary channel with instruction force over workers ("Do not edit the reviewed note... Do not finalize"). Force: instruction, plus one configuration-like retained commitment in prose: "**Default review model policy**: gate reviews run on the cheapest adequate model — currently `claude-sonnet-5`."
- **Loop function:** Executes the assay machinery — under **A** it runs search; under **B** it runs evaluation. Its own contribution is integrity *enforcement* around the bookkeeping: all-or-nothing finalization ("malformed or incomplete output fails the job and writes no freshness baseline"), partition validation (`build_model_partition(reported_model, reported_effort)` must equal the job's partition), single-use workers. These are deterministic checks whose verdicts control distinct operations (fail the job, re-run under the correct partition) — genuine reject-capable evaluation of *pipeline execution*, warranted by construction.
- **Oracle warrant:** The execution-integrity oracles are deterministic and respected. The **model policy** is where the conditional bites: **under B** (gates = evaluation), "cheapest adequate" is the exact move [false-positive-generation](../../../notes/false-positive-generation-is-filtered-before-retention.md) warns about — automating evaluation with an oracle chosen for cost, whose degradation "nothing announces"; **under A** it is the correct economy for search. The instruction *does* declare an escalation path — "if evidence shows a gate under-catching on the default model, that gate's criterion file may declare a stronger required partition" — but nothing in the system can *produce* that evidence: no instrumentation compares partitions, no artifact records miscatches, so the adequacy claim in "cheapest adequate" is unfalsifiable as wired. That gap matters under both readings (under A it hides search-omission failures, which the same theory note says "are invisible and permanent").
- **Retention:** The policy commitment's scope is stated ("No per-gate exceptions are currently declared") and its mechanical anchor is a real wire (`MODEL_PARTITION_REGISTRY` in `src/commonplace/review/review_model.py`). Duplicated in README-REVIEW-SYSTEM with a cross-link — managed, acceptable.
- **Findings:**
  1. **Conditional (stakes only, same text):** the cheapest-adequate-model policy is justified under A and dangerous under B. No text edit can resolve this before the warn placement lands; record it as the concrete operational consequence of the parked item.
  2. The under-catch escalation clause has no evidence producer. A cheap standing fix exists: an occasional dual-partition sample (run one gate bundle under a stronger partition over the same notes and diff outcomes) would give "under-catching" its first possible observation.
- **Disposition:**
  1. `conditional` — resolves with the warn placement; if B is adopted, the model policy for the `semantic` lens should be re-examined first (its warranted domain is thinnest).
  2. `proposal` — a small design entry: periodic dual-partition calibration sample as the evidence source the escalation clause presupposes. Not `apply` (it costs review budget and needs a cadence decision).

## kb/instructions/verify-review-quote-grounding.md

Force: instruction with a hard block: "If any citation FAILs, the review is **not** verified... Resolution failures must be fixed before commit."
- **Loop function:** Genuine **evaluation** under both readings — and notably it evaluates a *candidate* (the review note before commit), not an incumbent, so the parked ambiguity does not touch it. The verdict controls an operation distinct from producing the next candidate (blocking commit), satisfying the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md). Non-conditional.

This is [warranted-autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md)'s "narrow the gate to cases it can establish" implemented in an instruction.

The instruction is honest about this ("this cannot be run later from the KB alone"). Whether the check ran depends on the invoking flow remembering it — a soft retrieval wire, but the wire exists (the review-writing skill's procedure).
- **Findings:** No change suggested. Flag it for the eventual review-system-to-loop mapping note as the group's model case of a warranted deterministic oracle with a declared domain and a real reject.
- **Disposition:** n/a.

## kb/instructions/migrate-semantics-preserving-gate-changes.md

- **Authority:** Consumer: the agent/maintainer editing a gate criterion. Channel: instruction loading at gate-edit time. Force: instruction — it conditions when ack may substitute for fresh assays.
- **Loop function:** Retention maintenance over the review store: it carries completed evidence across a criterion re-wording. The migrate/rerun decision is reject-capable (pairs failing the preconditions must be rerun, a distinct operation), so it is a small evaluation of *evidence portability*. It is also the group's live instance of the [two-layer note](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md)'s **correspondence check** — a revised criterion must still account for verdicts established under its old wording, or the migration is disallowed.
- **Oracle warrant:** The core judgment ("the criterion semantically equivalent") is human/agent semantic judgment — judgment-dependent, and the instruction respects the boundary with both a hedge ("prose cleanup alone is not evidence that a behavioral change is harmless") and deterministic preconditions (step 1: only currently-fresh pairs may migrate — "A fresh pre-edit pair proves that both current files match its baseline snapshots"; note-changed pairs must rerun rather than "silently accepting both changes together"). **Conditional on stakes, not procedure:** under **B** a wrong equivalence call corrupts the *evaluation record* (verdicts asserted under a criterion that never judged them); under **A** it only mis-carries search bookkeeping, a cheaper error. The procedure is identical either way.
- **Retention:** Ack semantics restated accurately ("reuses completed evidence and advances the existing snapshot baseline; it does not append an event"). Scope stated in the first line ("only when a verdict gate's wording changes without changing the judgment it asks for").
- **Findings:** No change suggested; record the stakes asymmetry for the mapping note.
- **Disposition:** n/a.

## kb/instructions/evaluate-scenarios/SKILL.md

- **Authority:** Consumer: an invoked agent (`user-invocable: true`, `context: fork`, `model: sonnet` — the harness loads it as a skill). Channel: skill loading. Force: instruction during the run; the emitted report has advice/audit-trigger force toward the maintainer.
