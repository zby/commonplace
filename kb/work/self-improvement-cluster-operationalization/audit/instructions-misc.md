# Phase-1 audit — miscellaneous kb/instructions/ artifacts and the collection contract surface

Group: `kb/instructions/COLLECTION.md`, `kb/instructions/README.md`, and the ten remaining instruction artifacts not covered by the review/fix-pipeline and cp-skill groups. Date: 2026-07-23.

Theory applied: [behavioral authority](../../../notes/definitions/behavioral-authority.md) (consumer/channel/force; "a formal policy may have no effective authority if no component loads it"), [the proposal-selection loop](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) (strict evaluation criterion), [false-positive generation is filtered before retention](../../../notes/false-positive-generation-is-filtered-before-retention.md) (automate-search-first; evaluation bought with an oracle), [warranted autonomy is bounded by oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md), [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md) (the retrieval-wire standard), [the declared Commonplace frame](../../../reference/commonplace-declared-frame.md), [theory and methodology form a two-layer execution system](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md).

Consumption evidence was established empirically: `.claude/skills/` and `.agents/skills/` symlink listings, `rg` sweeps for inbound references from `AGENTS.md`, skills, instructions, types, ADRs, and `src/`, and reads of the consuming code (`src/commonplace/lib/validation.py`, `src/commonplace/cli/init_project.py`, `src/commonplace/scaffold_manifest.py`).

## kb/instructions/COLLECTION.md

- **Authority:** Strong, enforced, multi-consumer. (1) `cp-skill-write` hard-fails without it ("Hard fail if the target collection has no COLLECTION.md") and reads its Types and outbound-linking sections as authoritative; (2) `cp-skill-connect` hard-fails likewise ("The COLLECTION.md is the only linking-rules surface"); (3) `AGENTS.md` carries a standing routing instruction loaded in every session: "Read the target collection's `COLLECTION.md` before writing or connecting artifacts there." Consumer: agents (via skill-directed file read and standing instruction). Channel: skill loading + prompt assembly. Force: instruction — a system-definition artifact in the repo's own vocabulary. For skill-mediated writes the wire is enumeration-like (hard fail, not best effort); for manual edits it rests on the AGENTS.md routing line, which is itself session-loaded, so the wire holds.
- **Loop function:** Retention. The text contract is what makes an accepted procedure *operative in consistent, executable form* — it governs the shape of the fast-path layer. Notably, its "Reasoning constraint" ("Cut explanations of *why* each step exists from the instruction body. If worth preserving, record in a theory note and link via `rationale`") is the retention-side codification of the [two-layer split](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md): kb/instructions/ is the derived fast path, theory stays in kb/notes/, and the contract enforces the separation at write time. It also shapes evaluation indirectly via its "Tests for precision," but see oracle warrant.
- **Oracle warrant:** n/a for automation — the contract automates nothing. One gap worth naming: the declared quality goal ("executability + precision") and its four precision tests have **write-time consumers only**. No review-gate criterion under `kb/instructions/review-gates/` references the prescriptive contract or executability (verified by `rg`; the semantic bundle is notes-oriented — reach, generality, consistency). Evaluation of instruction quality after write time is currently self-check "Verify" sections plus human judgment; the precision tests are a rubric no evaluator loads.
- **Retention:** Good discipline. Scope stated positively (routing table role) and negatively ("What does NOT belong here"). The link-label table's scope is explicit per destination. Minor imprecision: "Some subdirectories are promoted into runtime skill surfaces ... by `commonplace-init`" — true for installed projects, but in this source checkout the projections are committed symlinks and AGENTS.md forbids running `commonplace-init` here; README.md carries the correct local-skill account.
- **Findings:**
  1. Precision tests have no evaluation-time consumer — the contract declares an evaluation standard nothing loads at review time.
- **Disposition:** (1) `reject` — write-time consumption is real and enforced, no instruction-quality failure evidence demands a review bundle for the prescriptive collection, and per YAGNI a new gate bundle is a design change; if instruction-review traffic ever materializes this becomes a proposal. No other change suggested.

## kb/instructions/maintain-curated-indexes.md

- **Authority:** The strongest wire of any plain instruction in this group, and the group's model case. Consumers: (1) the deterministic validator — `src/commonplace/lib/validation.py` line 52 defines `_TAG_README_FIX_HINT = "see kb/instructions/maintain-curated-indexes.md"` and emits it in five warning messages (weight gate, exit planning, `complete` membership, `covered_by` fan-out and coverage), so every tag-readme warning *names this instruction as its fix route*; (2) `kb/types/tag-readme.md` ("For the audit workflow ... read `kb/instructions/maintain-curated-indexes.md`"); (3) ADR 026 records the design intent ("The validator's warning/failure message names the fixing instruction ... so the maintenance loop is self-routing"). Channel: validation output + type-spec routing. Force: audit trigger that reliably becomes instruction — an enforced retrieval wire, not a best-effort one.
- **Loop function:** Search over incumbents (the audit sweep generates correction candidates: stale groupings, orphans, split candidates) plus retention maintenance of the `complete`/`covered_by` marks. The evaluation function for the marks sits in the validator, not here — and that gate is genuinely reject-capable at write/validate time (a marked README that fails membership blocks clean validation until fixed or the mark is dropped).
- **Oracle warrant:** Respected. The mark checks are deterministic over recomputable ground truth (tag membership), exactly the domain a mechanical oracle warrants; the enforced-or-omitted mark discipline (AGENTS.md "Mark" definition: "a stale trusted cache is a trap") is the [stale-index/trusted-wire](../../../notes/retrieval-failure-is-reflection-failure.md) lesson implemented. Editorial-quality judgments (groupings, context phrases) are left to human/agent judgment — no automation claims beyond the oracle's domain.
- **Retention:** Scope stated ("When to use"). Step 5's hub commitment ("Ensure `kb/notes/tags-README.md` lists all tag READMEs") is not a bare assertion: the hub itself carries validated `complete: true` frontmatter, so the commitment is enforced, not remembered.
- **Warn-conditional adjacency:** This artifact consumes *deterministic validator* warnings, not review-system `warn` outcomes, so the parked conditional does not govern it. But it is evidence for the empirical check's spirit: its step 2 carries explicit rejection vocabulary — a flagged missing note may be "(a) not important enough to curate, which is fine" — i.e. this warning consumer can decline a warning without "fixing" it, the representable-rejection disposition the [warn analysis](../warn-outcome-vs-loop-evaluation.md) found missing from the fix-report contract.
- **Findings:** No change suggested. Positive pattern for the eventual digest: validator-messages-name-the-fixing-instruction is the cheapest enforced retrieval wire the repo has, and the theory says to prefer it over discovery wherever a commitment has a mechanical trigger.
- **Disposition:** n/a (no findings).

## kb/instructions/evaluate-log-entry-for-note-creation.md

- **Authority:** Effectively unwired. Inbound references: one workshop inventory file only. `kb/log.md`'s own header does not name it; AGENTS.md does not; no skill invokes it. Its consumer is an agent reviewing the log who happens to know it exists — and [where change candidates come from](../../../reference/where-change-candidates-come-from-in-commonplace.md) already records both that the log is "rarely used in practice" and that "promoting an entry from any of them into an actual candidate still needs an explicit maintenance or triage step nobody has automated." Channel: memory or description scan. Force: advice in practice, despite being written as a gate.
- **Loop function:** **Evaluation, genuinely, under the strict criterion.** The artifact frames itself in loop vocabulary: "Treat the log entry as a **proposal**, not as evidence that a new note should exist." Its five novelty tests feed a verdict that controls operations distinct from producing the next candidate: Outcome A ("Reject ... do not create a note; remove the log entry") is a discard event, Outcomes B–D are explicit smaller-retention routes, and Outcome E is the only acceptance. Rejecting and changing-again are different events here. This is the evaluation function of the log→note promotion loop — the terminal filter before library retention.
- **Oracle warrant:** The oracle is structured agent/human judgment (a rubric in [oracle-strength](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) terms). Its acceptance claim is scoped correctly: passing the tests establishes "worth a note" against the AGENTS.md quality bar, nothing more, and the created note still passes through cp-skill-write conventions and later review sweeps. When an agent applies it unattended that is bare autonomy with a rubric oracle — acceptable because acceptance here creates a knowledge artifact subject to downstream review, not an enforcement surface.
- **Retention:** Scope and outcomes explicit; red-flag list is a good negative-scope statement.
- **Findings:**
  1. Retrieval-wire failure at the moment of use: the gate exists precisely to filter [false-positive generation](../../../notes/false-positive-generation-is-filtered-before-retention.md) out of log-triage, but nothing loads it at log-review time. A well-built evaluator that nothing consults contributes nothing (the standing discount).
- **Disposition:** (1) `apply` — add one routing line to the `kb/log.md` header (the artifact loaded at the time that matters), e.g. after the format line: "When an entry looks promotable into a note, run `kb/instructions/evaluate-log-entry-for-note-creation.md` before drafting." The edit target is `kb/log.md`, not the audited artifact.

## kb/instructions/write-agent-memory-system-review/ (local skill)

- **Authority:** Live and strong. Committed symlinks into both skill surfaces; harness-loaded (present in the session's available-skills roster); referenced from AGENTS.md, `kb/reference/README.md`, the review-framework design doc, ADR 023, and the review type contract. Consumer: agent (parent + delegated drafting worker); channels: skill loading, sub-agent prompt assembly (the embedded worker brief), execution; force: instruction, with several hard constraints.
- **Loop function:** A complete production pipeline with the three functions kept causally distinguishable, exactly as the [loop note's](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md) co-location floor requires: search = checkout refresh + delegated code-grounded drafting; evaluation = `commonplace-validate` as a block-capable deterministic gate ("Fix any structural or description-quality issues it reports and re-run until clean"; constraint: "never leave the review unvalidated") plus advisory taxonomy and semantic QA; retention = the note under `reviews/`, the archive protocol for displaced incumbents, and bounded index updates.
- **Oracle warrant:** The group's best positive example — three explicit refusals keep every oracle inside its domain:
  1. The blocking gate is only the deterministic validator, and it blocks only on what it establishes (structure and description quality).
  2. "Do not add `user-verified`; drafting and semantic review cannot grant human attestation" — a model oracle is barred from minting a human-attestation mark.
  3. "Never update `last-checked` without actually re-reading the reviewed system" — a freshness claim requires the act that warrants it.
  No gate's acceptance exceeds its check.
- **Retention:** Scope, Always/Never constraints, and archive lifecycle are explicit. The archive protocol is disciplined addressable retention: the displaced incumbent is preserved (`.replaced.{date}.md`), de-tagged, banner-marked, and stripped of `user-verified` — supersession is represented, not implicit.
- **Warn conditional (this artifact is in the review pipeline via step 12's semantic QA):**
