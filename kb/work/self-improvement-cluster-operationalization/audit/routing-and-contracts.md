## AGENTS.md

- **Authority:** Consumer: every harness-loaded agent session in this repo — the Claude Code harness loads `CLAUDE.md` → `AGENTS.md` into the system prompt at session start — plus human maintainers, plus reader-mode agents in vendored installs (the file itself declares the reader stance). Channel: prompt assembly, standing, unconditional. Force: instruction. This is the strongest wire in the repository, and it is the [behavioral-authority](../../../notes/definitions/behavioral-authority.md) misuse case inverted: prose that must not be read as low-authority precisely because the harness loads it as standing instruction. No machine consumer: it sits at repo root outside every collection, carries no frontmatter, and `commonplace-validate` never reads it — its correctness is maintained entirely by judgment.
- **Loop function:** Retention, primarily — it is the operative-retention channel for accepted repo-level commitments (vocabulary, git discipline, collection routing, development rules): an accepted convention becomes operative exactly by landing here, because this file is the one artifact with a by-construction consumer every session. Secondarily search-side routing for the change loop: the YAGNI bullet ("a system feature or design gap becomes a design proposal in `kb/reference/proposals/` ...; a transferable insight becomes a note in `kb/notes/`") captures noticed candidates into the candidate store. It implements no evaluation: under the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md), nothing in it can reject anything — it directs, it does not gate.
- **Oracle warrant:** n/a — it automates no evaluation. Where it *speaks about* oracles it respects their domains: the marks paragraph extends trust only where a validator establishes it ("validated by code ... enforced-or-omitted, because a stale trusted cache is a trap"), which is [oracle-domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) discipline stated as a house rule; "`pytest` — all tests must pass" delegates to a symbolic oracle without over-claiming what passing establishes.
- **Retention:** The retrieval wire is the best the theory allows: enumeration-like (loaded by construction, not discovered), the strong end of the contrast [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md) draws. Scopes are mostly stated per rule. Two weaker spots: (i) the Vocabulary section is a judgment-dependent derived digest of library notes (each entry glosses a note and links it), and nothing flags the gloss when the source note changes — no lineage is recorded at the source, no correspondence check exists; under [the two-layer note](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md) this is derived fast-path prose running without its managed-staleness regime. (ii) One reflectivity attribution is unanchored — see F2.

- **Findings:**

  **F1 — the verified negative: AGENTS.md contains no change-time routing to the self-improving-systems cluster (workshop README path 1 confirmed).** Checked exhaustively against the current text. What routing exists:

  - *"Using the KB"* opens with "Search it when working on methodology, design decisions, or operational patterns" — generic, discovery-mediated advice (it supports authority path 3, nothing stronger).
  - The *Collection Routing* table routes by artifact kind ("Use when" = what you are writing), i.e. placement routing for outputs, not consultation routing for changes.
  - *Navigation* entry points reach the cluster head only by two discovery hops: `kb/notes/tags-README.md` → `self-improving-systems-README.md`.
  - Two *task-conditional loads* exist: "For review work ..., read `kb/reference/README-REVIEW-SYSTEM.md`. For fixing review warnings, read `kb/instructions/FIX-SYSTEM.md`." This matters: the wire *shape* path 1 needs — "for X-type work, read Y" — is already an established AGENTS.md pattern; the cluster simply has no line in it.
  - The *Vocabulary* section defines 20 terms; none is a cluster term (no operative change, behavioral authority, proposal-selection, reflective system). `rg "self-improv"` over AGENTS.md: zero hits.
  - The word "reflective" occurs once, incidentally (see F2).

  What is absent, per the trigger scenarios: modifying a **skill** routes only to placement (`kb/instructions/` row) and install machinery (Skills section); modifying a **validator** routes only to engineering rules (YAGNI, no-backcompat, tests); modifying a **type spec** has no route at all (`kb/types/` appears only as a lookup surface, plus the mark-maintenance pointer to `kb/types/tag-readme.md` — a spec, not theory); modifying a **collection contract** has no route (the "Read the target collection's `COLLECTION.md` before writing or connecting artifacts there" rule covers writing artifacts *in* a collection, not editing the contract itself); modifying **AGENTS.md itself** gets only the mechanical symlink note. The workshop README's claim — "nothing loads the theory when someone modifies a skill, validator, type spec, or collection contract" — is accurate against the current text, and slightly understated: nothing loads *any* theory at change time; the gap is not cluster-specific.

  **F2 — unanchored reflectivity attribution.** Line 96 area (native-Windows section): "deterministic verification is part of operating this reflective system." Under [the declared frame](../../../reference/commonplace-declared-frame.md) ("Applications cite it instead of re-deriving the boundary"), the repo's strongest-authority artifact asserts Commonplace's reflectivity with no citation to the frame or the classification. Small, but it is the only cluster-relevant sentence in the whole file and it floats free.

  **F3 — vocabulary digest has no staleness signal.** As noted under Retention: the Vocabulary entries are derived prose whose sources never flag them. A revision to, e.g., `kb/notes/definitions/codification.md` leaves the AGENTS.md gloss silently stale in every subsequent session's context.

- **Disposition:**
  - F1 → `proposal` — this is the workshop's core authority-path-mix decision, not a spot edit. The audit's contribution: the minimal wire is now precisely characterized — one conditional-load line in "Using the KB", parallel to the two existing review-system lines (e.g. "For changes to skills, validators, type specs, collection contracts, or this file, read <change-time digest>"), pointing at the future `kb/instructions/` digest, or interim at the cluster head. Do not apply before the digest/coverage decision; a line pointing at the raw cluster trades context cost for unshaped theory (contra [frontloading](../../../notes/frontloading-spares-execution-context.md), which the workshop README already cites for why the digest should carry answers).
  - F2 → `apply` — exact edit: link the phrase, "operating this [reflective system](./kb/reference/commonplace-as-a-reflective-system.md)" (repo-root-relative, matching the file's existing link style). Safe, one-line, and gives the file's lone attribution its declared-frame anchor.
  - F3 → `reject` (as new mechanism) — the gap is real but already named: [where change candidates come from](../../../reference/where-change-candidates-come-from-in-commonplace.md) flags general theory-to-implementation lineage as "mostly absent" and assigns it to a wider freshness substrate. Recording the AGENTS.md vocabulary as a concrete instance for that future substrate is the whole action.

## kb/notes/COLLECTION.md

- **Authority:** Consumers: agents writing, connecting, or revising in `kb/notes/` — loaded conditionally at write time via two wires: the AGENTS.md instruction ("Read the target collection's `COLLECTION.md` before writing or connecting artifacts there") and `cp-skill-write`, which reads the `## Types` and outbound-linking sections and **hard-fails** if the contract is missing. Channel: conditional context load during authoring/connecting; force: instruction (binding text contract). Machine consumption is structural only: `project_paths.py` uses `COLLECTION.md` presence as the collection boundary marker (`src/commonplace/lib/project_paths.py`), validation scopes tag indexes by it and rejects nested contracts (`src/commonplace/lib/validation.py`, `validate_collection_structure`); the prose conventions themselves are never machine-parsed.
- **Loop function:** Retention — it retains the collection's accepted conventions with a real consumer. It also *supplies criteria* to evaluation (reach tests, composability/strength tests, "Add the `title-as-claim` trait ... so review gates check the promise") without itself gating anything: by the strict criterion it is a criterion store, not an evaluator. Note the parked-conditional contact point: whether the gates that consume these criteria are evaluation (reading B) or search feeding the fix queue (reading A) does not change this artifact's classification — under both readings the contract is the retained criterion source, so no conditional verdict is needed.
- **Oracle warrant:** The contract is notably honest about oracle strength: "Reach is a direction, not a gate" declines to pretend the quality goal is a gateable criterion; deterministic validation covers only what validators establish (structure, links, marks), and quality judgment is explicitly left to review — strictness matched to oracle strength, per [warranted autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md). No gate here accepts more than its check establishes.
- **Retention:** Scope is stated (this collection, theoretical profile). The wire holds where it matters most: at write/connect time, through skill hard-fail — a real consumer, not memory. It is weaker for freehand edits that bypass the skills, where compliance rests on the AGENTS.md sentence plus post-hoc validation — acceptable managed risk. One theory-positive detail: the `operationalized-from` / "`Operationalized into:` footer" convention is the retention half of the [two-layer promotion edge](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md) already wired into the contract, with the managed-staleness recheck semantics carried by the link grammar.
- **Findings:** No change suggested. The one gap it participates in — nothing routes *modification of the contract itself* through theory — is F1's gap, an AGENTS.md/instructions-level fix, not a per-contract one; duplicating a wire into each of eight contracts would be the wrong layer.
- **Disposition:** n/a (no findings).

## kb/work/COLLECTION.md

- **Authority:** Agent-consumed at workshop-creation/-writing time via the standard wire; instruction force.
- **Loop function:** This contract *is* the loop's candidate-stage discipline written down. "Value is consumed, not accumulated" plus "Library collections do not link **into** `kb/work/` — workshops are sinks, not sources of durable references. If a workshop produces something the library should cite, extract it first" is force isolation: in-flight candidates are denied inbound authority paths, so nothing unaccepted acquires a consumer/channel/force by accident — the structural complement of [false-positive acceptance becomes operative](../../../notes/false-positive-generation-is-filtered-before-retention.md). Closing a workshop ("extract the durable conclusions into the right library collection ... delete the workshop directory") is the retention step of a proposal-selection instance; deleting without extraction is the visible rejection event, so rejection and next-candidate remain distinct operations, satisfying the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md).
- **Oracle warrant:** The contract names no pre-promotion verification of its own; at extraction, the gate is supplied by the destination machinery (`cp-skill-write`, destination contract, validation, review). That composition respects oracle domains — the workshop layer claims no acceptance authority it lacks. Observed, not a defect.
- **Retention:** The Active Workshops list in `kb/work/README.md` is a retained navigation commitment with stated maintenance semantics (add on start, remove on close) and a declared allowance for lag — an honestly scoped, best-effort wire rather than a trusted-stale one, which is the safe side of [stale trusted claims](../../../notes/retrieval-failure-is-reflection-failure.md).
- **Findings:** No change suggested. (Observation for scheduled check (a), recorded in the summary: workshop deletion + the no-inbound-links rule make later work's dependence on earlier work run through *explicit library retention*, never through workshop residue — a clean instance of informational dependence that is not environment-mediated.)
- **Disposition:** n/a.

## Routing evidence for the axis-factoring check

How the three routing artifacts currently route reflection-related vs. self-improvement-related traffic:

1. **AGENTS.md** — routes neither. No vocabulary entry, no navigation line, no tag mention for either axis; the single occurrence of "reflective" is the unanchored aside (F2). Both kinds of traffic reach the cluster only via `tags-README.md` or lexical `rg`. Minting a `reflective-systems` tag would change nothing here.
2. **kb/notes/tags-README.md** — one entry serves both axes: "Self-improving systems — evidence-responsive operative change to a system's own organization; **reflective versus non-reflective self-improvement is the central distinction**, and the stake is a conjectured sample-efficiency payoff." A reader scanning the hub for reflection finds it only inside the self-improvement line — findable, but subordinated: reflection is presented as a distinction *within* self-improvement, not as an axis. The hub is `complete: true` over tag pages, so a new tag head would appear here mechanically once created.
3. **kb/notes/self-improving-systems-README.md** — reflection lives as a subsection of the pathway profile ("What reflection adds", four bullets plus the reflective-system definition link), adequate for a reader already at the head. **One consumer-visible seam found:** the head's Related Tags claims "[computational-model](../../../notes/computational-model-README.md) — reflection and intercession as computational concepts generalized to socio-technical boundaries," but `rg "reflect|intercession"` over `kb/notes/computational-model-README.md` returns **zero hits** — the receiving head never mentions either concept. A reader routed there for reflection-as-computation lands on a head with no hook back. Tag data confirms the pattern: the reflection notes (`definitions/reflective-system.md`, `reflection-buys-addressability.md`, `reflective-coverage-is-graded-across-representational-forms.md`) all carry `computational-model` alongside `self-improving-systems`, so the by-tag sweep finds them but the curated head does not.

Evidence read (decision belongs to the coordinating session): the only observed routing behavior a separate `reflective-systems` tag would fix is the computational-model dead end, and that has a one-line cheaper fix (add a reflection line to the computational-model head's curated sections). No traffic was observed in this group that enters with a reflection-shaped query and is misrouted by the self-improvement head itself. On this group's evidence alone, the seam is real but thin.

- Disposition for the dead-end seam → `apply` (one line in `kb/notes/computational-model-README.md`'s curated body linking the reflection-as-computation notes), **unless** the coordinator decides to mint the tag, which supersedes it. `kb/notes/computational-model-README.md` is outside this group's targets; flagged here because the evidence surfaced during the routing check.
