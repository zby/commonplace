# Phase-1 audit — the global type surface (`kb/types/`)

Date: 2026-07-23. Group: the eight global type specs and their schemas — the dual-consumed artifacts read as prose by agents at write time and enforced as schema by the deterministic validator (`src/commonplace/lib/type_resolver.py` resolves `type:` → spec → `schema:`; `src/commonplace/lib/validation.py` runs base checks, per-type imperative rules registered via `@type_rule`, then schema validation, with findings labelled `[base]` / `[type: name]` / `[schema]`). A third consumption channel exists for every type spec: since [ADR 038](../../../reference/adr/038-type-conformance-reviews-use-the-type-spec-as-the-gate.md), the spec body is the criterion text of the type-conformance review pair for every note of its type, applied by an LLM reviewer. That third channel places every type spec inside the review pipeline as a criterion, so the parked [`warn` conditional](../warn-outcome-vs-loop-evaluation.md) applies to that channel throughout and is recorded once here rather than repeated per artifact: under reading A the conformance verdicts are mechanized search feeding the fix queue; under reading B they are evaluation whose `warn` is retain-with-obligation. The dispositions below do not flip on that placement except where marked `conditional`.

Theory applied: [behavioral authority](../../../notes/definitions/behavioral-authority.md), [the proposal-selection loop](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md), [false-positive acceptance becomes operative](../../../notes/false-positive-generation-is-filtered-before-retention.md), [warranted autonomy is bounded by oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md), [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md), [the declared Commonplace frame](../../../reference/commonplace-declared-frame.md), [the two-layer execution system](../../../notes/theory-and-methodology-form-a-two-layer-execution-system.md), and the cluster head [self-improving-systems-README](../../../notes/self-improving-systems-README.md).

A shared write-time wire, verified once: `cp-skill-write` loads the selected type spec before authoring ("Read the selected type-spec doc… Follow that body as the structural authoring contract") and in edit mode ("Open the type-spec doc named by `type:` before making structural edits"). So the prose half of every spec has a real consumer at the time it matters for skill-mediated writes; edits made outside the skill bypass it and are caught only by the validator (symbolic half) and the conformance pair (semantic half, at review time).

## kb/types/type-spec.md (+ type-spec.schema.yaml)

- **Authority:** Three live paths. (1) Authors of new type specs read it (channel: `cp-skill-write` / direct read; force: instruction). (2) The validator consumes it doubly: `resolve_type_definition` (type_resolver.py:291) enforces its frontmatter contract on every type spec imperatively (`type: kb/types/type-spec.md`, `name`, `description`, `schema` — raised as errors even before schema validation), and `type-spec.schema.yaml` validates the same fields declaratively (enforcement). (3) It is the criterion text for the conformance pair of every type spec, including itself — the meta-pair `(type-spec.md, type-spec.md)` is derivable (channel: review; force: verdict into the review store).
- **Loop function:** Retention infrastructure for the whole type system — it defines the format in which type commitments are retained — plus *evaluation configuration*: its Validation Contract section is the routing document that allocates checks between the deterministic validator (symbolic) and the LLM conformance reviewer (prose). It implements no loop function directly; it tells other artifacts which function they implement.
- **Oracle warrant:** Exemplary scoping prose: "It parses structure; it never judges meaning," and the explicit note that a schema "cannot dereference." The known weak spot is inherited from ADR 038 and stated there: without an authored `## Review` section "the wrapper-prompted reviewer sets its own bar" — an oracle whose strictness is undeclared. That is a warrant gap of the conformance channel generally, not of this file.
- **Retention:** The file's own unenforced commitments are all *stated with their enforcement status*, which is exactly what the theory asks: the frontmatter-table/schema sync is named "symbolic-layer bookkeeping… nothing automates the comparison today; that's a validator gap to note," and the `## Review` section is named "a hand-sync obligation… nothing else catches it drifting." The backstop wire for template/table drift is real: instance validation errors ("that is where alignment is really enforced"), so a stale table costs one failed validation rather than a silent miss. This is the [retrieval-wire](../../../notes/retrieval-failure-is-reflection-failure.md) standard met by *downstream enforcement* rather than by keeping the copy fresh.
- **Findings:** No change suggested. The file is the group's second-best worked example after tag-readme: every prose/schema split is declared, and each declared gap either has a downstream wire or is explicitly logged as a gap per the YAGNI write-it-down rule.
- **Disposition:** n/a.

## kb/types/note.md (+ note.schema.yaml, note-base.schema.yaml)

- **Authority:** (1) Authors via `cp-skill-write` (instruction; the skill explicitly says to take trait vocabulary "from there, not from a remembered list"). (2) Validator: `note-base.schema.yaml` enforces `description`/`type` presence, description length as `severity: warn` subschemas (ADR 024 per-constraint severity, read in `_schema_error_message`, validation.py:671), `user-verified: const true`; `note.schema.yaml` adds `status: false`. (3) Conformance pairs for every plain note. (4) `user-verified` has a further runtime consumer: `review_target_selector.py:87` filters review targets by it (`--user-verified`), so the field carries selection influence over where review capacity goes.
- **Loop function:** Retention format for the KB's primary knowledge layer. The traits table additionally *configures evaluation routing*: `requires_trait` matching in `resolve_criteria.py:115` keys six catalog gates (`title-as-claim` cohort) off trait strings.
- **Oracle warrant:** n/a for the spec itself; the schema's warn-severity length checks are honest about what they establish (advice-grade, not blocking).
- **Retention:** Two retained commitments with weak or absent wires:
  - `user-verified` prose commits: "A substantive edit must remove it." No consumer loads that rule at edit time (edit flows outside `cp-skill-write` never see the spec), and the attestation records no anchor — no date, no snapshot hash — so the rule is not even recomputable in principle: nothing can compute "changed since attestation." Under [behavioral authority](../../../notes/definitions/behavioral-authority.md) the removal rule has declared intent but no reliable consumer/channel; under the tag-readme mark discipline this is a *trusted cache that is neither enforced nor recomputable*. Its consequences are operative: a stale `user-verified` misdirects the review selector's targeting.
  - The trait vocabulary is a retained list enforced nowhere: `note-base.schema.yaml` types `traits` as any unique strings. A typoed trait (`title-as-clam`) validates clean and silently drops the note from every `requires_trait` gate — a retrieval-wire cut precisely where the routing is trusted, the [stale-index shape](../../../notes/retrieval-failure-is-reflection-failure.md) at the review-routing level. Actual usage (repo-wide sweep, 2026-07-23) is exactly the table's vocabulary — 6 of 7 listed traits in use — so enforcement is currently cheap.
  - Reverse-direction split (schema enforces what prose never states): `note.schema.yaml` forbids `status` (`status: false`) and note.md's prose does not mention it. Cost is low — the schema error is the wire — but it is an undeclared commitment.
- **Findings:**
  1. The `definition` trait row is dead vocabulary: zero instances carry it, no gate requires it, and its job passed to the `definition` *type* (which the explication-quality gate keys on via `requires_type`). A retained commitment with no consumer of any kind.
  2. Trait values should be validator-checked. Exact edit: add to `note-base.schema.yaml` under `traits.items` an `enum` of the seven registered traits with `severity: warn` (ADR 024 pattern), so an unknown trait surfaces as a warning instead of silently dropping gate routing, while leaving room for future vocabulary growth without a hard fail. Grounded in [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md) and the tag-readme "enforced-or-omitted" discipline this cluster's worked example already established.
  3. `user-verified` needs either an anchor or a stated downgrade. Design change: record the attested snapshot (hash or store snapshot id) at attestation time so "user-verified but changed since" becomes a deterministic check — the same move ADR 026 made for `complete` (convert an asserted claim into an enforced one). Alternative considered: leave as-is and state in the spec that the field's integrity is discipline-only; rejected because the field has a real runtime consumer (review targeting), so its staleness is operative, and [false-positive acceptance becomes operative](../../../notes/false-positive-generation-is-filtered-before-retention.md) says the unchecked trusted mark is the dangerous shape.
- **Disposition:** (1) `apply` — delete the `definition` row from the traits table in `kb/types/note.md`. (2) `apply` — the schema edit described above; safe because current usage is exactly the registered vocabulary and the severity is warn. (3) `proposal` — attestation anchoring touches store schema and CLI, a `kb/reference/proposals/` entry.

The split is covered by the meta-contract's general statement (the schema owns "whatever the named `.schema.yaml` can statically specify") and the conformance reviewer carries the rest; instances observed (e.g. `behavioral-authority.md`) follow the shape. Acceptable.

No spec change follows; the fix, if any, belongs in `resolve_criteria.py`.

## kb/types/tag-readme.md (+ tag-readme.schema.yaml) — the cluster's canonical worked example

- **Authority:** The densest, most deliberately engineered authority profile in the group. (1) *Maintenance path only* by declared contract: "load it when declaring or dropping a mark, fixing a validator warning, or executing a lifecycle exit" — and the wire making that real is the validator's own messages, which carry `see kb/instructions/maintain-curated-indexes.md` (`_TAG_README_FIX_HINT`, validation.py:52) — an audit-trigger force that names its consumer's next read. (2) Validator: the registered rule `validate_tag_readme` (validation.py:517) enforces weight gates (8 KB warn / 16 KB fail), the `complete` membership check, the `covered_by` coverage check and ~7-child fan-out warn; the schema enforces field shape and the `index_source: tag → index_key` conditional. (3) AGENTS.md frontloads the marks' read-side meaning to every session (instruction force; ADR 026 decision 6's knowledge split: meaning → AGENTS.md, mechanics → validator, maintenance → type spec). (4) Conformance pairs. (5) `cp-skill-connect` consumes `complete` to skip the by-tag `rg` sweep — the accelerator consumer the marks exist for.
- **Loop function:** The marks close a small proposal-selection loop for index maintenance, and the classification depends on which consumption path the verdict takes. On the *write path*, `cp-skill-write` validates before commit and a `fail` blocks the commit — the verdict controls an operation distinct from producing the next candidate, so this is genuine reject-capable evaluation by the [strict criterion](../../../notes/a-proposal-selection-loop-requires-search-evaluation-and-retention.md). On the *reactive path* (a new tagged note invalidates a marked README that gets validated later), the fail lands in a sweep and feeds the fix queue — under reading A that is search over incumbents; under reading B, retain-with-obligation. The same deterministic check is evaluation or search depending on where its verdict lands — a concrete illustration that loop function attaches to the consumption path, not the check, which is precisely the behavioral-authority point.
- **Oracle warrant:** The best case in the repo. The membership and coverage checks recompute *exactly the claim the mark makes* — membership(tag) vs linked set; membership(tag) ⊆ ∪ children — so acceptance never exceeds what the check establishes; the oracle domain and the mark's claim coincide by construction. The weight gate is honestly a proxy (bytes for "cheap read surface") and its failure message demands editorial remedies ("curate harder, split the tag, or narrow it") without pretending to judge them — the mechanical oracle stops exactly at its boundary and hands judgment over, the [warranted-autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) shape.
- **Retention:** The mark contract is the enforced-or-omitted discipline stated in full: "recomputable, therefore never load-bearing — dropping a mark costs one recomputation, never correctness; and enforced-or-omitted — a false mark tells exhaustive consumers to stop looking." The reactive-enforcement window (a stale marked README between a bypassing commit and the next validation, while AGENTS.md tells readers to trust the mark) is an accepted ADR 026 cost, and it is substantially closed by a wire worth naming: `ValidationRun.evaluate` expands every run over tagged notes to the impacted marked READMEs (`impacted_marked_tag_readmes`, validation.py:203, applied at :275), so the dominant write path surfaces the gap immediately even though "the write path carries no tag discipline."
- **Findings:**
  1. The schema types `complete` as `boolean`, so `complete: false` validates clean, triggers no check (`if fm.get("complete") is True`), and retains a token with no consumer and no meaning — the spec's own discipline is *declared-or-omitted*, and the neighboring precedent (`user-verified: const true` in note-base) already encodes absence-as-the-only-false. Exact edit: in `tag-readme.schema.yaml`, change `complete: { type: boolean }` to `complete: { const: true }`. No committed instance declares `complete: false` (both current marks are `true`).
  2. The reactive-window residue (commits that bypass validation leave a trusted stale mark until the next sweep) — considered and **rejected** as a finding: ADR 026 accepted the cost explicitly, the run-expansion wire closes the window on the skill-mediated path, and the marks' recomputable-never-load-bearing property caps the damage at a wasted trust, not a correctness loss. The theory is satisfied by the stated scope plus the existing wire.
- **Disposition:** (1) `apply`. (2) `reject`, reasons above.

## kb/types/review-gate.md (+ review-gate.schema.yaml)

This artifact is squarely in the review pipeline — its instances are the oracle texts — so the [`warn` conditional](../warn-outcome-vs-loop-evaluation.md) applies to how its instances' verdicts are read; the findings below do not flip under A vs B and are unconditional.

- **Authority:** (1) Gate authors at write time (instruction). (2) Validator: schema enforces frontmatter fields and the two required headings (`## Failure mode`, `## Test`). (3) Conformance pairs: every gate, being typed `review-gate`, is itself reviewed against this spec by the conformance wrapper — so the spec's discipline rules (notably self-containment) have a model-judged wire, not just authoring-time hope. (4) The runtime consumers of instance frontmatter are narrower than the spec implies — see findings.
- **Loop function:** Meta-evaluation infrastructure: it is the authoring contract for the oracles of the note-maintenance loop (under reading A: for the mechanized-search criteria). Its self-containment rule is retention discipline for oracle freshness — "a test that leans on prose living elsewhere carries a dependency that never invalidates acceptances. If the test needs contract language, quote it in the gate body — that converts the dependency into hashed criterion text." This is [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md) applied at the freshness-hash level, stated in the operative artifact — a genuine instance of the cluster's theory already wired into a system-definition artifact, and worth citing when the change-time digest is distilled.
- **Oracle warrant:** One positive and one negative case, both in the `watches` mechanism that keys ack-triviality:
  - Positive: `ack_trivial_note_changes.py` fails safe — an unknown or missing `watches` value means "the gate watches the whole note; no change is trivial against it" (`KNOWN_WATCHES` check, :96). A misdeclared value *narrows* unattended acceptance instead of expanding it, keeping the automated ack inside its warranted domain. This is the correct direction of failure for an automated evaluation component and deserves recording as a worked example of [warranted autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) respected in code.
  - Negative: the spec's prose example is out of sync with the recognized domain — it documents `watches: [body | frontmatter | ...]`, but `KNOWN_WATCHES = {"body", "title", "description"}`; `frontmatter` is not a recognized value, so a gate following the spec's own example silently loses ack-triviality (fail-safe, but the spec teaches a dead value while omitting two live ones).
- **Retention:** The schema *requires* four fields — `gate_id`, `name`, `lens`, `staleness` — of which **none has a code consumer**. Gate identity is derived from the file path (`resolve_to_criterion_ids` enumerates `{bundle}/{stem}`); no code reads `gate_id`, `name`, or `lens` frontmatter; and `staleness` is read by nothing at all — review staleness is decided by snapshot comparison (ADR 032, `freshness.py`), not by this field. The prose asserts behavior the system does not implement: "`staleness: changed | always | ...` — when an accepted review becomes stale." In [behavioral-authority](../../../notes/definitions/behavioral-authority.md) terms this is the textbook exclusion — "a formal policy may have no effective authority if no component loads it" — enforced *inward* (authors must supply values) with no force *outward*. The drift is already observable: one live gate declares `staleness: rewrite(0.5)`, a free-form value nothing parses, against 37 gates declaring `changed`. `gate_id` and `lens` are additionally unchecked derived copies of path facts, the exact shape [a derived copy of recomputable truth must be checked or absent] forbids — and the repo owns a registered-rule mechanism (`@type_rule`) that could check them in a few lines.
- **Findings:**
  1. Resolve the four unconsumed required fields, in the no-backcompat spirit: drop `staleness` from schema and prose entirely (or wire it to the freshness layer if per-gate staleness policy is genuinely wanted — nothing suggests it is); keep `name` only if a rendering consumer is added; for `gate_id` and `lens`, either drop them or add a `@type_rule("kb/types/review-gate.md")` enforcing `gate_id == {parent-dir}/{stem}` and `lens == parent-dir` — checked or absent, no third option. This is a 38-file sweep plus schema plus possibly `src/` changes.
  2. Fix the `watches` prose to the recognized domain: replace `watches: [body | frontmatter | ...]` with `watches: [body | title | description]` (subsets allowed), and state the fail-safe: an unrecognized value means the gate watches the whole note and is never auto-acked. Exact edit in the Frontmatter section of `review-gate.md`.
  3. The spec's `lens` enumerated list `(accessibility, semantic, structural, complexity, prose, frontmatter, sentence)` is documentation of catalog directory names, unenforced by schema — subsumed by finding 1 (whichever branch is taken).
- **Disposition:** (1) `proposal` — design change with a cohort-wide sweep and a `src/` question; note that editing 38 gate files stales every accepted review of those gates (ADR 038 cohort blast radius), which argues for batching with the next intentional gate revision. (2) `apply` — two-line prose correction. (3) folded into (1).

## Group summary

The five findings most worth the maintainer's attention:

1. **`review-gate.md` requires what nothing consumes** (`gate_id`, `name`, `lens`, `staleness` — `staleness` with prose asserting unimplemented behavior, and a free-form `rewrite(0.5)` already in the wild). The type surface's own oracle-authoring contract is the group's clearest behavioral-authority failure: enforcement force pointed at authors, zero force at runtime. Proposal-grade; batch the sweep with an intentional gate revision to contain the cohort staling.
2. **The trait vocabulary and `user-verified` are the two unenforced trusted marks** on the base note type. Traits: cheap warn-severity enum closes a silent review-routing drop (apply). `user-verified`: no attestation anchor makes the "remove on substantive edit" rule unenforceable in principle while the field steers review targeting (proposal — record the attested snapshot, the ADR 026 assert→enforce move).
3. **`tag-readme` verifies as the model the cluster should generalize from**, with one cosmetic tightening (`complete: const true`). Its notable theory contribution: the same deterministic check is reject-capable evaluation on the write path (fail blocks commit) and search/obligation on the reactive path — loop function attaches to the verdict's consumption path, not the check. The audit of other artifacts should ask "which path does the verdict take" rather than "what kind of check is this."
4. **`index.md` is a completed migration still holding the door open** — zero instances, live transition scaffolding. Apply after one hook check.
5. **Positive findings worth retaining for the digest:** the review-gate self-containment rule is the cluster's retrieval-wire theory already operating inside a system-definition artifact, and the `watches` fail-safe in ack-triviality is warranted-autonomy respected in code (unknown value narrows, never widens, unattended acceptance). The future change-time digest should cite both as in-repo precedents rather than deriving the pattern fresh.

Phase-1 scheduled-check observations:

- **(a) Cumulativity's environment-mediated exclusion:** no environment-mediated case in this group. The nearest contact is a deliberate *anti*-dependence design: the tag-readme marks are recomputable-never-load-bearing by contract, i.e. the system engineers later rounds *not* to informationally depend on the retained cache, while the curated entries and context phrases are the genuinely consumed retained information. The stated exclusion survived contact here by not being needed.
- **(b) Reflection vs self-improvement routing:** the type surface routes no cluster traffic at all. A type-spec edit is exactly the change-time event the workshop's trigger finding names (a change to behavior-determining organization), and the mechanical wire that fires on such an edit — cohort staling of conformance pairs — carries the *type spec* as criterion, never the cluster. If authority path 1 (wired) is built, the type-spec editing path is a natural attachment point; nothing in this group currently distinguishes reflection from self-improvement traffic, so this group contributes no evidence for or against minting `reflective-systems` as a separate tag head.
