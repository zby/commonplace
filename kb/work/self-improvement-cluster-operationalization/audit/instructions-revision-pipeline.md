## kb/instructions/revise-note.md

- **Authority:** Consumer: an agent, only when a user (or a calling workflow) explicitly points it at the instruction — nothing wires it. It is not a promoted skill, no command references it, and notably `run-full-improvement-pass-on-note.md` step 9 does **not** invoke it: the full pass uses its own inline prompt ("revise the note for flow, coherence, logic and readability"). The only inbound references found are workshop inventories (`kb/work/bulk-operations/directed-reading-inventory.md`, `kb/work/runner-execution-profiles/inventory.md`). Channel: manual prompt reference. Force: instruction when loaded; effectively dormant otherwise. Per [behavioral authority](../../../notes/definitions/behavioral-authority.md), declared intent ("Editorial revision of a single KB note") exceeds actual consumption.
- **Loop function:** Search fused with immediate retention. The instruction generates a revision candidate and applies it in place ("Edit the file in place") in one act — the working tree copy becomes the operative incumbent the moment the agent saves. Its "Assess" step (step 3) is not evaluation under the strict criterion: its only downstream effect is shaping the candidate. The reject-capable evaluation event is the maintainer's `git diff` review before commit — which is real in this repo (AGENTS.md: "Check `git diff` before committing") but lives entirely outside this artifact.
- **Oracle warrant:** n/a inside the artifact — it automates no acceptance. The de facto oracle for what it retains is the downstream human commit review, which the instruction never names.
- **Retention:** The "Report" block routes attention (changes + flagged items) to the user, which is the right shape, but the instruction retains its output directly into the live note with no statement that the edit is a candidate pending judgment. Scope of the edit ("Do NOT change the note into a substantively different note… add new claims…") is well stated. The retrieval wire for the *instruction itself* is weak (nothing loads it); the wire for its *output* is the repo's git discipline, held in a different artifact.
- **Findings:**
  1. The instruction performs search and retention with no named evaluation event. [False-positive generation is filtered before retention](../../../notes/false-positive-generation-is-filtered-before-retention.md) says automated search is the safe half *because* a downstream filter exists; `FIX-SYSTEM.md` makes that filter explicit for fixes ("when the fix produces a diff, reviewing that diff is the judgment step"), but this instruction leaves it implicit. Concrete edit: add one sentence to "What this is" or step 6, e.g. *"The in-place edit is a candidate: the maintainer's review of the resulting diff before commit is the judgment step; nothing in this procedure accepts the revision."*
  2. Observation, no change: the artifact is a near-orphan (no wired consumer). That is not a defect by itself — a manual instruction is authority path 2 — but it means any theory-suggested hardening here buys little until something consumes it.
- **Disposition:** (1) `apply` — one-sentence addition as above. (2) `reject` as a change; recorded as consumption data for the authority-path decision.

## kb/instructions/run-full-improvement-pass-on-note.md

- **Authority:** Consumer: an orchestrator agent, on manual invocation (not a promoted skill; not routed from `AGENTS.md`). Referenced descriptively from `kb/reference/full-improvement-pass-closure.md` and `README-REVIEW-SYSTEM.md`. Parts of its contract are codified with enforcement force: the typed packet schema (`kb/reports/types/full-pass-report.md`, checked by `commonplace-validate`) and the version guard (`commonplace-guard-full-pass-report`, backed by `src/commonplace/lib/full_pass.py` and `src/commonplace/cli/guard_full_pass_report.py` — statuses `matching`/`changed`/`missing`/`corrupt-capture`; code audited by another agent). So: mixed authority — instruction force for the procedure, enforcement force for the packet/guard subset.
- **Loop function:** This is the group's fullest proposal-selection instance, and its functions separate cleanly:
  - **Search:** steps 1–6 (compression bundle, critique, friction gate, catalog bundles, connect — all report-only) plus step 7's synthesis, which produces the candidate edit plan (the packet).
  - **Evaluation:** three distinct places. (i) Step 7 reconciliation is reject-capable *for findings*: it can decline a critique addition as a hedge (with the codified scope-check before the label), and it records both sides of a disagreement without action — rejection is representable, distinct from producing the next candidate. (ii) The guard is deterministic evaluation whose verdict controls a real operation (blocks edits / renders `superseded`), but its domain is input-version identity only. (iii) For `delete`/`merge` dispositions, evaluation is deferred to a human via `resolve-full-pass-disposition.md`. **For `keep` dispositions there is no reject-capable gate between the packet and the applied edits**: step 8 applies the body edits directly, and the closing cycle "never starts another edit-and-review round" — it routes residual findings to Open items and stops. Under the strict criterion the closing cycle is **not** evaluation: its results control no select/discard/block/roll-back operation over the applied edits. It is monitoring with audit-record force (correctly framed by `full-improvement-pass-closure.md`: "evidence about the final note, not certification").
  - **Retention:** step 8's in-place application (working tree), then the operator's commit — the latter outside the instruction.
- **Oracle warrant:** The pass is careful about one oracle boundary and silent about another.
  - Honored: `composition-friction-gate`'s output is carried "unresolved" into a dedicated Routed-attention section because "reconciling this gate's output the way the others are reconciled would recreate the self-graded verdict its own hard rule forbids" — an explicit refusal to let a model resolve what was designed for a human. This is [warranted-autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) discipline stated in operational terms.
  - Silent: for `keep` edits, the effective acceptance event is the same orchestrator model that synthesized the packet — search and evaluation co-located in one process with no fresh or blind element, and the instruction never names the human diff review at commit as the judgment step (contrast `FIX-SYSTEM.md`, which does). The oracles feeding the packet (compression criteria, catalog gates) warrant editorial and criterion-relative findings; but `critique-note` and `semantic` findings can motivate claim-affecting edits, and nothing in the pass escalates those to a human the way `cp-skill-revise-autoreason` does with its claim-revision sidecar. Whether this exceeds warrant is exactly the parked conditional:
    - **Under reading A** (gates are search; the human diff review is the evaluation event): the design is sound — model workers on the report-generating steps is the automate-search-first move, and the packet gives warns a representable downstream rejection (step 7 can record a finding without converting it to action). The gap is only that the actual evaluation event (commit-time diff review) is unstated in the artifact; naming it closes the loop's description.
    - **Under reading B** (gates are evaluation): step 8's unattended application of model-accepted claim-affecting edits is evaluation autonomy that has not been bought — [false-positive acceptance becomes operative](../../../notes/false-positive-generation-is-filtered-before-retention.md) — and a claim-revision escalation valve (sidecar-style) becomes warranted, not just nice.
- **Retention:** Strong where codified: `source.txt`/`merge-target.txt` captures with SHA-256, "Never rewrite the capture," report+captures as one unit, re-entrancy preflight that *consumes* prior pending and rejected reports ("do not repeat the rejected disposition without materially new evidence"). That last rule is a genuine retrieval wire — a later pass on the same note loads the earlier verdict at the time it matters. Weaker: `kb/reports/full-pass/*` is gitignored (`.gitignore` line 57), so a pending `delete`/`merge` packet — which the instruction declares "still in use" — is retained only on the machine that ran the pass; see the resolve-instruction entry below.
- **Findings:**
  1. Name the human evaluation event for `keep` edits. Concrete edit: add to step 8 (or the Output Contract's closing text) a sentence mirroring FIX-SYSTEM, e.g. *"Applied body edits are candidates until committed: the maintainer's review of the resulting diff is the pass's acceptance step for `keep` dispositions; this instruction applies edits but accepts nothing."* Grounded in the loop note's requirement that rejection be a real event and in reading A of the parked conditional.
  2. Claim-affecting edits lack an escalation valve. Under reading B, findings from `critique-note`/`semantic` that change claims, evidence, or qualifiers should route to a human-reviewed artifact (as `cp-skill-revise-autoreason`'s claim-revision sidecar does) instead of being applied in step 8. Under reading A this is unnecessary provided finding 1's sentence lands.
  3. Meta-reader wiring to the cluster is absent. The Relevant Notes footer cites [error-correction](../../../notes/error-correction-works-above-chance-oracles-with-decorrelated-checks.md) and [synthesis-is-not-error-correction](../../../notes/synthesis-is-not-error-correction.md) but nothing from the self-improving-systems cluster, although this artifact is the repo's most developed operational proposal-selection loop. A maintainer changing the pass (e.g., automating the keep-edit gate) would not retrieve the oracle-warrant theory — [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md), authority path 3. Concrete edit: one `rationale` footer line, e.g. `- [False-positive generation is filtered; false-positive acceptance becomes operative](../notes/false-positive-generation-is-filtered-before-retention.md) — rationale: why steps 1–6 may be freely automated while any strengthening of the acceptance side (step 8's gate) must be bought with a warranted oracle.` (Permitted by the COLLECTION.md link contract: `rationale`, meta-reader only.)
  4. Observation, no change: the closing cycle is correctly *not* claimed as evaluation anywhere — `full-improvement-pass-closure.md` already scopes it ("not certification"). The theory read confirms the shipped framing.
- **Disposition:** (1) `apply`. (2) `conditional` — under A: reject (covered by finding 1); under B: `proposal` (sidecar-style escalation valve for claim-affecting edits). (3) `apply`. (4) no change.

## kb/instructions/resolve-full-pass-disposition.md

- **Authority:** Consumer: an agent, reached through a real conditional wire — `run-full-improvement-pass-on-note.md`'s re-entrancy preflight *requires* it when a pending or guard-failed report exists, and the full-pass footer links it `applies-when`. Channel: conditional-deviation link. Force: instruction, with two enforcement-force elements — `commonplace-validate` on the report and `commonplace-guard-full-pass-report` gating every mutation.
- **Loop function:** Evaluation and retention, cleanly separated. Evaluation is genuinely reject-capable: `accepted | rejected | alternative-applied | superseded` are distinct terminal states, and a rejection controls a real operation (the recommendation is declined and recorded; the re-entrancy preflight then constrains future passes). Retention is the execution of accepted dispositions (delete source / semantic merge / backlink updates) with filesystem postconditions checked before terminal state is recorded ("Never record a terminal state before its filesystem postconditions succeed").
- **Oracle warrant:** Exemplary — the artifact allocates authority exactly along oracle domain. Human judgment: "Proceed only with an explicit user decision" for accept/reject/alternative (`resolution_authority: user`). Deterministic check: "Agents may mark `superseded` only from a deterministic changed-input guard result" (`resolution_authority: version-guard`), and the guard's domain is precisely what it establishes — input identity, nothing semantic. `missing`/`corrupt-capture` explicitly do *not* license conclusions beyond what the check establishes ("Absence cannot prove a delete or merge succeeded; capture corruption is not source drift"). This is [warranted autonomy bounded by oracle domain](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) implemented in an instruction, and worth citing as such in the digest.
- **Retention:** Scope well stated (report + captures one unit; "pending reports must never be pruned"; rejected reports retained while load-bearing). The weak point is the retrieval wire for **pending** dispositions: `kb/reports/full-pass/*` is gitignored, so a pending `delete`/`merge` recommendation exists only in one clone, and its only automatic consumer is a *future full pass on the same note* (the preflight). No command lists pending resolutions; no routine or triage surface loads them. A recommendation the pipeline produced and declared "still in use" can sit unread indefinitely — [retrieval failure is reflection failure](../../../notes/retrieval-failure-is-reflection-failure.md): the loop's evaluation function exists but depends on someone remembering to convene it.
- **Findings:**
  1. Pending dispositions need a consumer that fires without a same-note re-run: e.g., a `commonplace`-side listing of `resolution: pending` full-pass reports surfaced in review triage, or at minimum a documented operator check. This is a design change touching commands and possibly the triage instruction.
- **Disposition:** (1) `proposal` (`kb/reference/proposals/` entry: "pending full-pass dispositions need a scheduled consumer").

## kb/instructions/cp-skill-revise-autoreason/SKILL.md

- **Authority:** Consumer: the harness — this is a projected skill (present in the runtime skill list; `pyproject.toml` maps the directory), `user-invocable: true`, triggered by `/cp-skill-revise-autoreason`. Channel: skill loading. Force: instruction, on user trigger. Marked "Experimental," with routing away from itself ("prefer `cp-skill-revise-iterative` for the established revision path").
- **Loop function:** A complete miniature proposal-selection loop, and the group's best illustration of the three functions kept causally distinguishable:
  - **Search:** critic (problem-noticing), B author, synthesizer (candidate generation).
  - **Evaluation (layered, each reject-capable):** parent-side hard-constraint checks (can reject B/AB before synthesis or judging — a rejection distinct from a rerun, since "stop and finalize with the last good version" is an available outcome); the post-candidate auditor (`DECISION: REJECT` stops the pass; `CLAIM_REVISION_NEEDED` reclassifies the task); blind Borda judging (A is a first-class candidate, so "do nothing" can win — the verdict controls a select operation, and two consecutive A wins *stop the loop*, an operation distinct from producing the next candidate); the outer semantic fidelity check (reject-capable against the baseline); and the terminal human gate.
  - **Retention:** only on explicit user approval does `last_good` overwrite the original ("Never: Modify the original file until the user explicitly approves") — retention is the maintainer's act, with filename/backlink follow-through specified.
- **Oracle warrant:** The skill allocates each oracle exactly its domain, which is what question 3 asks for:
  - The **blind judges'** oracle establishes *decorrelated anonymized preference on editorial quality and apparent semantic safety* — three fresh judges, balanced label order (position-bias control), no provenance, and, critically, no baseline: "You do not have the original baseline, so do not try to infer which candidate is the incumbent." So the judging oracle *cannot* establish semantic fidelity to the incumbent, and the skill says so structurally: "Never: Let judge preference replace the outer semantic fidelity check." Blind judging is therefore not the acceptance gate — it is a ranking oracle inside a gate stack.
  - The **fidelity check** (orchestrator, with the baseline) covers what the judges cannot; claim-level changes are pushed *out of the automated path entirely* via the claim-revision sidecar, whose artifact is "outside the blind prose tournament" and "never auto-appl[ied]" — the exact move [warranted autonomy](../../../notes/warranted-autonomy-is-bounded-by-oracle-domain.md) prescribes when criteria outrun the oracle: narrow the unattended gate and retain a human evaluator.
  - The **human diff review at Step 3** is the actual acceptance gate for the file the KB retains. So: blind judging establishes preference; human diff review is the gate. Warranted domain respected throughout. One residual: the outer fidelity check is performed by the orchestrator that also built packets and aggregated scores — weaker decorrelation than the judges — but it holds the baseline the judges lack and sits upstream of the human gate; acceptable.
- **Retention:** One concrete hygiene defect. Step 1 writes run bundles to `kb/work/revise-autoreason/…`, but `.gitignore` ignores `kb/reports/revise-autoreason/` (line 72) — the path the skill actually uses is *not* ignored (`git check-ignore` confirms). Historical leakage is visible: `kb/reports/link-vocabulary.md` contains off-vocab rows citing `kb/work/revise-autoreason/*/current_a.md` files — stale candidate copies of notes were swept by a repo-wide link scan as if they were KB content. A run bundle contains near-duplicates of live notes; un-ignored, it pollutes lexical search (`rg` over `kb/`), link sweeps, and risks being committed — a retrieval-wire corruption in the [retrieval-failure](../../../notes/retrieval-failure-is-reflection-failure.md) sense: the self-representation's search surface picks up artifacts that are not commitments. Also good here: the skill revokes `user-verified: true` on any candidate and states that applying a winner "revokes the prior attestation" — a mark kept honest per the mark discipline.
- **Findings:**
  1. Run-bundle path vs. ignore rule mismatch. Concrete minimal edit: add `kb/work/revise-autoreason/` to `.gitignore` (next to line 72). Alternative (larger): change Step 1's `run_dir` to `kb/reports/revise-autoreason/…` to match the existing ignore entry and the "disposable inspection artifact" family — but that touches skill semantics (the skill calls it "a workshop directory"), so the ignore-line addition is the small safe form.
  2. Observation, no change: this skill is the group's worked model of oracle-domain allocation (blind ranking ≠ acceptance; claim changes escalate to human; human gate terminal). The eventual change-time digest should cite it as the pattern rather than re-deriving it.
- **Disposition:** (1) `apply` — one `.gitignore` line as described. (2) no change (digest input).

## kb/instructions/cp-skill-revise-iterative/SKILL.md

- **Authority:** Consumer: the harness — projected skill, `user-invocable: true`, `/cp-skill-revise-iterative`. Channel: skill loading. Force: instruction on user trigger. Named by the autoreason skill as "the established revision path."
- **Loop function:** Search: each `claude -p` pass generates a numbered revision candidate (`${source_file}.$i`). Evaluation: the outer agent's semantic fidelity check (2c) is reject-capable under the strict criterion — its verdict controls a discard-and-stop operation ("Do not apply this revision — the last good version is …; Skip to Step 3"), not merely the next pass; the significance check (2e) is a stopping rule (search-effort allocation, part of search). Retention: human-gated — "Never: Modify the original file until the user explicitly approves," with diff summary shown first.
- **Oracle warrant:** The automated oracle (the orchestrator comparing consecutive versions) establishes *absence of detected meaning-change between adjacent revisions* — a fidelity oracle, not a quality oracle, and the skill's acceptance matches: passing the check only allows the candidate to become the next baseline *inside the run*; nothing becomes operative in the KB until the human approves the cumulative diff. Decorrelation is moderate (the revision author is a separate non-interactive `claude -p` process; the checker is the orchestrator), weaker than autoreason's blind panel but adequate because the retained artifact passes through the human gate. One structural caveat worth knowing rather than fixing: the fidelity check is pairwise-adjacent, so slow semantic drift across five passes could evade each pairwise check — but Step 3's diff summary is against *the original*, so the human gate sees the total drift. The warrant chain closes at the human.
- **Retention:** Intermediate `.0`–`.5` copies live next to the note during a run and are deleted "regardless of outcome" — transient, and the non-`.md` suffixes keep them out of `*.md` globs; acceptable. `user-verified` revocation handled as in autoreason.
- **Findings:** No change suggested. The design already places the reject-capable check and the human retention gate where the theory wants them.
- **Disposition:** n/a.

## Group summary

The five findings most worth the maintainer's attention:

1. **`run-full-improvement-pass-on-note.md` never names its acceptance event for `keep` edits** (finding 1 there, `apply`). For delete/merge it built a genuinely exemplary human+guard gate; for body edits — the common case — the packet is applied by the same model that synthesized it, the closing cycle cannot reject, and the actual filter (maintainer diff review at commit) is stated in `FIX-SYSTEM.md` but not here. One sentence closes the description of the loop; under reading B of the parked `warn` question the deeper fix is a claim-revision escalation valve (`conditional`/`proposal`).
2. **Pending full-pass dispositions have no scheduled consumer** (`resolve-full-pass-disposition.md`, `proposal`). Packets are gitignored and their only automatic consumer is a future pass on the same note — a produced-but-possibly-never-convened evaluation function, the retrieval-failure pattern applied to the pipeline's own outputs.
3. **`cp-skill-revise-autoreason` run bundles are not gitignored** (`apply`: add `kb/work/revise-autoreason/` to `.gitignore`). Stale candidate copies of live notes have already leaked into a repo-wide link sweep — search-surface pollution of the self-representation.
4. **The pipeline's own theory wiring is absent** (full-pass finding 3, `apply`): the repo's most developed operational proposal-selection loop carries no `rationale` link into the cluster whose vocabulary describes it — a concrete path-3 instance of the workshop's trigger finding.
5. **Two artifacts are positive exemplars the digest should cite instead of re-deriving:** `critique-note.md` (search that refuses verdict authority) and `cp-skill-revise-autoreason` + `resolve-full-pass-disposition.md` (oracle-domain allocation: blind ranking ≠ acceptance; agent authority confined to deterministic guard results; human gate terminal).

**Phase-1 scheduled checks:**

- **(a) Cumulativity:** the full-pass re-entrancy preflight is a genuine cumulativity wire — a later pass *consumes* an earlier pass's rejected disposition ("do not repeat the rejected disposition without materially new evidence"), not merely replaces an incumbent. By contrast both revise skills deliberately delete their run bundles (judge rationales, critic reports) on completion, so successive runs on the same note retain nothing from each other beyond the note text itself — improvement retained only as the replaced incumbent, i.e., the profile note's non-cumulative case. Neither is an environment-mediated *exclusion* case (nothing here improves via untouched-environment effects); the group supplies contrast material, not a boundary test, for that exclusion.
