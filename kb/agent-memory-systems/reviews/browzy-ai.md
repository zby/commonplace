# browzy.ai

Browzy is a TypeScript terminal personal knowledge base published as the `browzy` npm package. It ingests URLs, PDFs, images, Markdown, and text into a local `~/.browzy/default` data directory, compiles sources into Markdown wiki articles with an LLM, indexes those articles in SQLite FTS5, and answers user questions by assembling relevance-ranked article context for a streaming LLM response.

**Repository:** https://github.com/VihariKanukollu/browzy.ai

**Reviewed commit:** [56c253042041ee2f483a5e9b824174d746891cf4](https://github.com/VihariKanukollu/browzy.ai/commit/56c253042041ee2f483a5e9b824174d746891cf4)

## Core Ideas

**The durable memory is a local compiled wiki, not a chat memory store.** Browzy's default data root is `~/.browzy/default`; `ensureDataDirs` creates `raw/`, `raw/images/`, `wiki/`, `output/`, and `.browzy/` beneath it ([config.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/config.ts)). Raw sources persist as Markdown files plus a manifest, while compiled wiki articles persist as Markdown with frontmatter for title, tags, source IDs, backlinks, timestamps, and summary ([filesystem.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/filesystem.ts), [types.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/types.ts)).

**Ingest keeps source material inspectable before compilation.** Web ingestion blocks private/internal targets, follows only same-host redirects, caps HTML and image sizes, strips common non-content HTML, converts to Markdown, downloads bounded image assets, and writes a raw source file with source frontmatter ([web.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/ingest/web.ts)). PDF, text, Markdown, and image ingestion follow separate processors; `ingest` updates the raw manifest, upserts source metadata into SQLite, logs activity, and invalidates the query cache ([ingest/index.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/ingest/index.ts)).

**Compilation is an LLM-mediated distillation step with a user-editable schema.** `WikiCompiler.compile` reads the raw manifest and existing articles, pre-indexes raw sources for temporary searchability, compiles new sources in batches of three LLM calls, creates template articles for small sources, updates backlinks, writes `_index.json`, and indexes final articles into SQLite ([compiler.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/compile/compiler.ts)). The optional `browzy.schema.md` file is loaded into both compiler and query system prompts, giving the user prose control over compilation preferences, query behavior, terminology, and emphasis ([schema.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/schema.ts), [prompts.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/prompts.ts)).

**Read-back is engineered context injection.** A query does not hand the entire wiki to the model. `QueryEngine.prepare` reads the schema, calculates a model-aware article budget, and calls `ContextBuilder.build`; the builder searches SQLite FTS5, adds index fallback candidates when needed, ranks articles by title/tag/content density, recency, and backlink authority, extracts relevant sections, enforces per-article and total token budgets, and returns confidence and gap signals ([engine.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/engine.ts), [contextBuilder.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/contextBuilder.ts), [relevanceRanker.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/relevanceRanker.ts)).

**Session features are useful, but only partly integrated with memory.** The Ink CLI saves session transcripts and metadata under `~/.browzy/sessions`, generates a short digest for the previous session, and optionally writes that digest as a wiki article ([useSession.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/hooks/useSession.ts), [app.tsx](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/app.tsx), [digest.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/digest.ts)). The query path also has an "insight crystallizer" that can draft a derived article from a multi-source answer, but it writes to `drafts/`, not directly into indexed wiki memory ([crystallizer.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/crystallizer.ts)).

**Some README claims are implemented as product-path affordances rather than hard guarantees.** The README's "compiled knowledge" architecture, local storage, FTS retrieval, schema, ingestion modes, streaming answers, and gap suggestions are represented in code ([README](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/README.md)). Stronger claims such as "every question you ask can surface gaps" and "insight crystallizer" are implemented through heuristic gap extraction and a once-per-session draft path, not through a reviewed promotion workflow or always-indexed learned memory ([engine.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/engine.ts), [crystallizer.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/crystallizer.ts)).

## Artifact analysis

**Raw sources and manifest.** The storage substrate is the local filesystem under `raw/`, with `_manifest.json` and source files produced by ingest processors; source metadata is also mirrored into SQLite `sources` rows ([filesystem.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/filesystem.ts), [sqlite.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/sqlite.ts)). The representational form is prose content with symbolic frontmatter and manifest fields. Lineage is imported source identity, type, origin, path, content hash for file sources, first/updated ingest timestamps, and stable source IDs; re-ingest preserves the original ID when updating an existing source ([ingest/index.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/ingest/index.ts), [deduplicator.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/deduplicator.ts)). Behavioral authority is knowledge-artifact authority: raw sources are evidence and source material for compilation, not direct instructions to the answering model.

**Compiled wiki articles.** The storage substrate is `wiki/*.md` plus `_index.json`, with SQLite `articles` and `articles_fts` as a derived retrieval index ([filesystem.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/filesystem.ts), [sqlite.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/sqlite.ts), [migrations.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/migrations.ts)). The representational form is mixed: Markdown prose with symbolic frontmatter, wiki links, source IDs, tags, backlinks, and FTS rows. Lineage is derived from source IDs in article frontmatter and from compilation time; there is no retained per-claim source offset, compiler model, prompt version, or review status. Behavioral authority is knowledge-artifact authority at rest, then advisory context when selected into a query prompt.

**Schema and prompts.** `browzy.schema.md` is stored as a local Markdown file, while built-in prompts are TypeScript strings ([schema.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/schema.ts), [prompts.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/prompts.ts)). The representational form is prose system-definition material. Lineage is authored: the default schema is generated on first run and user edits are read as-is, capped to 4,000 characters. Behavioral authority is system-definition authority because schema text is inserted above compiler and query prompts and can constrain how future sources are compiled and how future answers are framed.

**Retrieval indexes, caches, and budgets.** SQLite FTS5 and `_index.json` are derived symbolic/distributed retrieval surfaces; query cache entries are in-memory process state with a 30-minute TTL and generation invalidation on ingest ([sqlite.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/sqlite.ts), [compiler.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/compile/compiler.ts), [queryCache.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/queryCache.ts)). Their lineage is regenerated from raw and article stores. Their behavioral authority is system-definition authority on the read path: they rank, select, omit, budget, and cache which knowledge artifacts reach the LLM.

**Sessions, digests, activity logs, and drafts.** Session JSON, last-session metadata, digest text files, activity logs, session-digest wiki articles, and crystallized draft Markdown are local filesystem artifacts ([useSession.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/hooks/useSession.ts), [activityLog.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/activityLog.ts), [app.tsx](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/app.tsx), [crystallizer.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/crystallizer.ts)). Their representational form is prose plus JSON/Markdown metadata. Their lineage is trace-derived from local session messages, query outcomes, and activity events. Their behavioral authority is split: raw sessions, logs, and drafts are mostly human-facing knowledge artifacts, while a generated digest written into `wiki/` can become future advisory query context after the wiki/index surfaces catch up. The inspected code does not show trace-derived artifacts becoming automatically reviewed policy, validators, or binding instructions.

There is a limited promotion path: raw source -> compiled wiki article -> indexed retrieval candidate -> LLM context. Browzy does not have an internal promotion ladder from article or draft into validators, binding instructions, review gates, or typed lifecycle states.

## Comparison with Our System

| Dimension | Browzy | Commonplace |
|---|---|---|
| Primary purpose | Personal research Q&A over a local compiled wiki | Agent-operated methodology KB with typed artifacts and review/governance |
| Durable store | `~/.browzy/default` raw files, wiki Markdown, SQLite DB, sessions | Git-tracked Markdown collections, type specs, generated indexes, reports |
| Main derivation loop | LLM compiles imported sources into articles; retrieval injects selected article sections | Agents write and revise typed artifacts under collection contracts, validation, and review |
| Read-back | Relevance-gated context injection into query prompts | Agent navigation, explicit loading, skills, indexes, validation, and review workflows |
| Lineage | Source IDs and timestamps; weak per-claim derivation and review metadata | Source-pinned citations, frontmatter, lifecycle status, archives, authored links, git history |
| Authority model | Articles advise; schema and retrieval machinery instruct/select; health checks report | Distinguishes knowledge artifacts from system-definition artifacts with stronger enforcement surfaces |

Browzy is a tighter end-user application than Commonplace. It optimizes for "add sources, ask questions" rather than maintaining a durable methodology corpus. Its best comparison point is not Commonplace's review system but Commonplace's future runtime context assembly: Browzy shows a straightforward way to compile local knowledge into inspectable Markdown, use SQLite for recall, and keep the LLM's context bounded by relevance, section extraction, and token budgets.

The strongest alignment is the source-of-truth split. Both systems keep human-inspectable Markdown as durable memory and treat generated indexes as derived surfaces. Browzy's split is less governance-heavy: article frontmatter carries useful metadata, but there is no collection contract, type schema, status field, semantic review, or replacement archive. That makes Browzy more approachable and less suitable for methodology artifacts where authority and provenance must be audited.

The main divergence is behavioral authority. Browzy's compiled articles are knowledge artifacts until a query selects them; the retrieval stack then grants them advisory prompt context. Commonplace intentionally has more system-definition artifacts: instructions, skills, type specs, validation commands, and review gates can route, enforce, or block behavior, not merely advise.

**Read-back:** push — from the answering LLM's perspective, Browzy injects relevance-selected wiki context into the prompt in response to the user's query.

### Borrowable Ideas

**Schema-as-local prompt preface.** Browzy's editable `browzy.schema.md` is a simple, user-owned way to tune compilation and answering without changing code. Commonplace could borrow the pattern for small project-local context-pack preferences, but only where the schema remains visibly lower authority than collection/type contracts.

**Raw-to-compiled separation with immediate raw indexing.** Pre-indexing raw sources before LLM compilation gives the system a temporary recall surface while the richer artifact is still being produced. Commonplace could use an analogous workshop pattern for newly captured sources, making raw snapshots searchable before promoted notes exist.

**Section-level read-back budgets.** Browzy's context builder does not only rank files; it extracts sections and marks omitted material. This is ready to borrow for any Commonplace retrieval layer that needs to reduce both volume and complexity before handing artifacts to an agent.

**Confidence and gap signals as first-class query outputs.** Browzy's confidence and gap detection are heuristic, but the interface is right: every answer can expose whether the local corpus covered the question well. Commonplace should keep any future search/read-back layer explicit about coverage limits instead of silently padding with general model knowledge.

**Do not borrow unreviewed trace-to-wiki promotion.** Session digests and insight drafts are useful as a personal research affordance, but Commonplace should require review and lineage before session-derived material can enter durable methodology artifacts.

## Trace-derived learning placement

Browzy qualifies as trace-derived, but only at the symbolic artifact layer. It does not learn weights, train a ranker, or promote traces into hard rules. The qualifying mechanisms are session digest generation from prior user/assistant turns and insight crystallization from a question, an answer, and the source articles used to produce that answer.

**Trace source.** The raw traces are saved session message arrays under `~/.browzy/sessions`, last-session metadata, query answers with `sourcesUsed`, and activity-log events ([useSession.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/hooks/useSession.ts), [app.tsx](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/app.tsx), [activityLog.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/activityLog.ts)). Session digesting only fires for the previous saved session when there are at least three user messages and an LLM is available. Crystallization only fires after an answer uses at least two source articles and is gated to one saved insight per session unless no artifact is saved.

**Extraction.** `generateSessionDigest` truncates and summarizes non-system conversation turns with `SESSION_DIGEST_PROMPT`; the app writes the result as a digest text file, updates last-session metadata, and best-effort writes a `session-YYYY-MM-DD` wiki article tagged `session-digest` ([digest.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/digest.ts), [app.tsx](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/app.tsx)). `crystallize` asks an LLM whether a multi-source answer contains a novel cross-article insight; when the response parses as an article, it writes Markdown with `derived: true`, sources, and the triggering question under `drafts/`, not `wiki/` ([crystallizer.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/crystallizer.ts)).

**Scope and timing.** The scope is local to one user's Browzy data directory. Digesting is delayed to the next launch; crystallization runs asynchronously after an answer, so neither path affects the answer that produced it. The digest-as-wiki-article path is the only trace-derived route that can plausibly re-enter future query context, and even that is weakly wired: `writeArticle` alone does not immediately update SQLite FTS or `_index.json`, so later retrieval depends on subsequent index/compile maintenance rather than a dedicated trace-promotion workflow ([filesystem.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/storage/filesystem.ts), [compiler.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/compile/compiler.ts), [contextBuilder.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/contextBuilder.ts)).

**Survey placement.** On the [trace-derived survey](../trace-derived-learning-techniques-in-related-systems.md), Browzy belongs in symbolic artifact learning with a weak promotion path: traces can produce prose summaries and draft insights, but the system does not add review, deduplication, invalidation, tests, validators, or stronger behavioral authority around those artifacts.

## Read-back placement

**Direction.** Browzy is push for the answering model and pull for the human operator. The user pulls by asking a question or running `/search`; the model receives a prepared prompt where Browzy has already selected and formatted article context ([app.tsx](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/app.tsx), [engine.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/engine.ts)).

**Trigger and relevance signal.** The trigger is a user query. The relevance signal combines SQLite FTS5 candidate retrieval, fallback from `_index.json`, article scoring, section scoring, recency, backlink count, and follow-up exclusion for already-used slugs ([contextBuilder.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/contextBuilder.ts), [relevanceRanker.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/relevanceRanker.ts)).

**Timing relative to action.** Read-back occurs before the answer-generating LLM call. The prepared context, confidence note, and gap note can change the next generated answer; session digest generation and crystallized insight drafting happen after sessions or answers and therefore cannot affect the just-produced answer ([engine.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/engine.ts), [digest.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/digest.ts), [crystallizer.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/query/crystallizer.ts)).

**Selection, scope, and complexity.** The implemented policy considers up to 15 ranked articles, caps each article at about 8,000 tokens, includes at least two articles when possible, extracts up to five relevant sections when section matches are narrower than the article, and uses model-aware article budgets computed before prompt assembly ([contextBuilder.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/contextBuilder.ts), [tokenCounter.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/tokenCounter.ts)). Actual precision, recall, and context dilution are not verified from code.

**Authority at consumption.** Selected article sections are advisory context. The schema and built-in query prompt are stronger system-definition artifacts because they tell the model how to treat Browzy context, cite articles, distinguish local knowledge from general knowledge, and admit limitations ([prompts.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/prompts.ts), [schema.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/schema.ts)).

**Faithfulness.** Browzy has retrieval-quality tests for FTS behavior, stemming, ranking, token counting, and schema migration, but I did not find an ablation or post-answer audit that proves injected memories changed answer behavior as intended ([retrieval-quality.test.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/tests/e2e/retrieval-quality.test.ts)).

**Other consumers.** Humans consume the same memory through `/search`, `/health`, `/log`, the banner, exports, and direct Markdown/SQLite inspection. The health linter also consumes wiki articles to produce quality issues, mixing deterministic checks with an LLM consistency pass ([linter.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/lint/linter.ts)).

## Curiosity Pass

**The "compiled" metaphor is accurate but incomplete.** The code really does compile imported sources into article artifacts and derived indexes. What it does not yet compile is authority: there is no typed promotion from draft insight to reviewed article, from article claim to test, or from repeated correction to durable instruction.

**Trace-derived artifacts exist, but the trace-learning loop is weak.** Sessions, digests, activity logs, session-digest wiki articles, and crystallized drafts are derived from user/assistant interactions. They are durable files, but the inspected code does not show a strong promotion path that reliably indexes every trace-derived draft or turns traces into behavior-shaping rules.

**Conversation compaction is computed but not retained into the next prompt path.** `compactConversation` can summarize older messages, and `handleQuery` calls it after long sessions, but the returned compacted messages are not stored back into the displayed session or a future prompt state in the inspected path ([app.tsx](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/cli/app.tsx), [compactor.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/retrieval/compactor.ts)). The comments say it is "used internally for LLM context in future queries," but the code shown does not wire that result into later `buildLLMHistory` calls.

**The schema is powerful precisely because it is under-specified.** A user can put domain preferences, terminology, and answering rules into `browzy.schema.md`. That is a low-friction system-definition artifact, but without validation or review it can also silently distort compilation and answers.

**The local-only claim is mostly architectural, not privacy-complete.** Browzy stores data locally and has no Browzy server, but LLM calls send selected source content and query context to the configured provider unless the user runs a local Ollama model. The README notes provider use, and the code supports Claude, OpenAI, OpenRouter, and Ollama providers rather than an offline-only engine ([provider.ts](https://github.com/VihariKanukollu/browzy.ai/blob/56c253042041ee2f483a5e9b824174d746891cf4/src/core/llm/provider.ts)).

## What to Watch

- Whether crystallized drafts become an indexed, reviewed promotion path or remain side-channel notes under `drafts/`.
- Whether session digests and compaction summaries become actual read-back inputs to future queries, with clear authority and invalidation.
- Whether article lineage gains compiler prompt/model/version metadata, per-claim source offsets, or review status.
- Whether health checks move from report-only findings toward enforced gates before articles are used as high-confidence context.
- Whether retrieval tests grow answer-level WITH/WITHOUT checks for context faithfulness, not just search-quality checks.

- [Behavioral authority](../../notes/definitions/behavioral-authority.md) - sharpens: the same article is passive evidence at rest and advisory context when injected into an answer prompt.
