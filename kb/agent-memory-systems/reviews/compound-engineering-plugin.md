# compound-engineering-plugin

`EveryInc/compound-engineering-plugin` is a multi-host coding-agent plugin and converter for the Compound Engineering workflow: strategy, ideation, brainstorm, planning, work execution, review, debugging, product pulse reports, and learning capture. The reviewed checkout contains the `compound-engineering` plugin payload, a Bun/TypeScript CLI that converts Claude Code plugin artifacts into other agent-platform formats, and scripts that let the workflow retain project-local records such as `STRATEGY.md`, `docs/solutions/`, `CONCEPTS.md`, pulse reports, and prior agent sessions.

**Repository:** https://github.com/EveryInc/compound-engineering-plugin

**Reviewed commit:** [09391874b4be1a248bc7d627b0ebd5c29f0c886b](https://github.com/EveryInc/compound-engineering-plugin/commit/09391874b4be1a248bc7d627b0ebd5c29f0c886b)

**Last checked:** 2026-06-03

## Core Ideas

**The workflow is an installable skill-and-agent operating system.** The root README frames Compound Engineering as a loop from strategy and brainstorm through plan, work, review, and compounding, while `plugins/compound-engineering/` holds the actual plugin payload: 39 `SKILL.md` entry points and 43 agent markdown files in the inspected checkout ([README.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/README.md), [plugins/compound-engineering/README.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/README.md), [plugins/compound-engineering/skills](https://github.com/EveryInc/compound-engineering-plugin/tree/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills), [plugins/compound-engineering/agents](https://github.com/EveryInc/compound-engineering-plugin/tree/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/agents)). The retained behavior is mostly authored prose instructions plus symbolic frontmatter and manifests, not embeddings or a database.

**The compounding substrate is the user's repository.** Several skills create or maintain durable project files: `ce-strategy` writes `STRATEGY.md` as an upstream anchor, `ce-brainstorm` and `ce-plan` write requirements and plan artifacts under `docs/`, `ce-compound` writes structured learnings under `docs/solutions/`, `ce-compound-refresh` audits those learnings against the current codebase, `ce-product-pulse` writes a browseable timeline under `docs/pulse-reports/`, and `ce-compound`/`ce-compound-refresh` maintain `CONCEPTS.md` as shared vocabulary ([plugins/compound-engineering/skills/ce-strategy/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-strategy/SKILL.md), [plugins/compound-engineering/skills/ce-compound/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-compound/SKILL.md), [plugins/compound-engineering/skills/ce-compound-refresh/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-compound-refresh/SKILL.md), [plugins/compound-engineering/skills/ce-product-pulse/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-product-pulse/SKILL.md)). This makes the system closer to a project-local knowledge workflow than to a standalone memory service.

**Context efficiency comes from staged documents, subagent isolation, and extractor scripts.** The skills do not load a whole project memory by default. They read well-known anchors only at the phase that needs them, ask subagents to return distilled text instead of files, and use script-mediated extraction for large session histories. `ce-sessions` explicitly forbids reading whole session JSONL files into context and instead discovers metadata, ranks sessions, extracts skeletons to scratch files, and dispatches a synthesis-only historian agent ([plugins/compound-engineering/skills/ce-sessions/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/SKILL.md), [plugins/compound-engineering/skills/ce-sessions/scripts/extract-metadata.py](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/extract-metadata.py), [plugins/compound-engineering/skills/ce-sessions/scripts/extract-skeleton.py](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/extract-skeleton.py)). The bounds are procedural rather than enforced by a central context allocator.

**Learning is manual or semi-structured, not autonomous model memory.** `ce-compound` documents a recently solved problem while context is fresh, classifies it with a schema, searches related docs, optionally folds in session-history findings, validates frontmatter, and writes one solution doc. `ce-compound-refresh` later checks those docs against current code and can update, consolidate, mark stale, replace, or delete them ([plugins/compound-engineering/skills/ce-compound/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-compound/SKILL.md), [plugins/compound-engineering/skills/ce-compound-refresh/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-compound-refresh/SKILL.md)). The system preserves provenance labels such as "(session history)" and "(auto memory [claude])" when those sources inform a learning, but the durable artifact is an authored Markdown learning, not a learned embedding or hidden profile.

**The converter is part of the memory system's adoption layer.** The CLI parses Claude plugin manifests, skills, agents, commands, hooks, and MCP servers, then writes target-specific bundles for OpenCode, Codex, Pi, Gemini, and Kiro ([src/parsers/claude.ts](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/src/parsers/claude.ts), [src/targets/index.ts](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/src/targets/index.ts), [src/commands/install.ts](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/src/commands/install.ts)). For Codex, the converter intentionally defaults to agents-only because native Codex plugin install owns skills, while the Bun step fills the custom-agent gap ([src/converters/claude-to-codex.ts](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/src/converters/claude-to-codex.ts)). This matters architecturally: the same retained instruction library is designed to enter several host runtimes without rewriting the core skill corpus.

## Artifact analysis

- **Storage substrate:** `repo` — The primary retained artifacts are Git-tracked plugin files, skill/agent Markdown packages, TypeScript converter code, and project-local Markdown/YAML artifacts the skills create in the consuming repository.
- **Representational form:** `mixed` — Markdown prose instructions dominate, but frontmatter, JSON manifests, YAML config, Python/Bash scripts, TypeScript converter code, and structured solution-doc frontmatter all carry behavior-shaping authority.

**Plugin manifests and marketplace metadata.** Storage substrate: repository files under `.claude-plugin/`, `.cursor-plugin/`, `.agents/plugins/`, and plugin-local `.claude-plugin/` / `.codex-plugin/` directories. Representational form: symbolic JSON metadata plus prose descriptions. Lineage: authored and release-managed; `src/release/metadata.ts` counts component files, synchronizes descriptions/versions, and enforces Codex manifest parity. Behavioral authority: system-definition artifacts for installation, routing, platform presentation, and release consistency. They expose the retained skill corpus but do not themselves decide task relevance at runtime.

**Skill packages.** Storage substrate: `plugins/compound-engineering/skills/<name>/` directories with `SKILL.md`, references, scripts, and assets. Representational form: mixed prose, YAML frontmatter, embedded command templates, and occasional executable support scripts. Lineage: authored workflow instructions, with some skills reading or writing project-local retained artifacts during use. Behavioral authority: system-definition artifacts when loaded by a host agent: they instruct planning, review, debugging, research, work execution, and documentation behavior. Their reference files are knowledge artifacts until a loaded skill routes the agent to consume them as instructions or evidence.

**Agent packages.** Storage substrate: `plugins/compound-engineering/agents/*.md`. Representational form: mixed frontmatter and prose task contracts. Lineage: authored persona and worker definitions. Behavioral authority: system-definition artifacts when a skill dispatches them through the host's subagent mechanism. They shape behavior through isolated context and narrow role instructions rather than by storing new facts themselves.

**Project-local CE memory artifacts.** Storage substrate: files in the user's repository, including `STRATEGY.md`, `docs/brainstorms/`, `docs/plans/`, `docs/solutions/`, `docs/pulse-reports/`, `.compound-engineering/config.local.yaml`, and `CONCEPTS.md`. Representational form: mostly prose Markdown with structured frontmatter or flat YAML where needed. Lineage: authored through interactive skill runs; solution docs can incorporate codebase findings, related-doc search, auto-memory excerpts, and session-history synthesis with explicit source labels. Behavioral authority: mostly knowledge artifacts when read as context or evidence, but they become system-definition artifacts on specific paths: `STRATEGY.md` grounds ideation/brainstorm/planning, `docs/solutions/` informs future plans and reviews, config controls pulse routing, and stale/refresh decisions can govern whether a learning remains trusted.

**Session-history extraction artifacts.** Storage substrate: host session JSONL files under Claude Code, Codex, Cursor, and temporary scratch skeleton/error files created by `ce-sessions`. Representational form: mixed raw event JSONL, filtered text skeletons, collapsed tool summaries, and synthesized prose. Lineage: trace-extracted from prior agent sessions, then filtered by repo, branch, time window, and keywords. Behavioral authority: raw and skeleton traces are knowledge artifacts for recall and diagnosis; the `ce-session-historian` synthesis becomes advisory context for the caller. When `ce-compound` folds those findings into `docs/solutions/`, the distilled portions can later become behavior-shaping knowledge artifacts and sometimes soft system-definition guidance.

**Converter and target writers.** Storage substrate: TypeScript source under `src/`, plus install manifests written into target roots such as `.codex/<plugin>/install-manifest.json`. Representational form: symbolic executable code and JSON/TOML output. Lineage: authored conversion logic from Claude-format plugin source into host-specific bundles. Behavioral authority: system-definition and installation authority: the converter decides which skills, agents, commands, hooks, MCP servers, sidecars, prompts, and cleanup manifests are emitted for each host. Its path-safety, legacy-cleanup, and native-vs-converted decisions constrain what retained instructions survive installation.

**Promotion path.** The most important promotion path is from solved work to durable project knowledge: current conversation and code findings -> optional session-history synthesis -> `docs/solutions/` learning -> later plan/review/research context. Product-operation signals follow a weaker record path: `ce-product-pulse` writes `docs/pulse-reports/` for human-browseable history, but the inspected source does not show prior reports being read back into later agent context. A second promotion path is from plugin source to host-specific runtime artifacts: Claude-format skills and agents -> converter bundle -> installed target files. Neither path is fully automatic memory learning; each depends on explicit skill invocation and authored review/assembly rules.

## Comparison with Our System

| Dimension | compound-engineering-plugin | Commonplace |
|---|---|---|
| Primary purpose | Ship a multi-host engineering workflow plugin and converter | Maintain a typed methodology KB and framework for agent-operated knowledge bases |
| Main retained unit | Skills, agents, project-local docs, session digests, converter outputs | Typed Markdown artifacts with collection contracts, schemas, indexes, sources, and review gates |
| Memory substrate | User repo plus host session logs and plugin install roots | Commonplace repo collections plus generated indexes/reports |
| Read-back | Skill-triggered reads of known files, related-doc search, session-history synthesis, and project-artifact grounding | Mostly deliberate `rg`/index/link pull, skills, reports, and validation/review loops |
| Governance | Skill contracts, frontmatter schemas, validation scripts, refresh workflow, release metadata checks, converter tests | Collection/type contracts, schema validation, semantic review, link vocabulary, generated indexes |
| Context control | Phase-local reads, subagent isolation, filtered session extractors, per-skill references | Collection routing, progressive search, type specs, scoped source snapshots, validation and review bundles |

The closest match is that both systems treat Markdown-in-git as retained state with real behavioral consequences. Compound Engineering pushes this state into day-to-day software workflows: brainstorms, plans, code reviews, product pulses, and solution docs are meant to make the next engineering unit easier. Commonplace is more explicit about artifact taxonomy and long-term knowledge governance; it names collection register, type contracts, link semantics, and validation status as first-class design surfaces.

Compound Engineering is stronger on adoption and workflow packaging. The same plugin installs across Claude Code, Cursor, Codex, Copilot, Droid, Qwen, OpenCode, Pi, Gemini, and Kiro paths, with converter code preserving host-specific expectations ([README.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/README.md), [src/targets/index.ts](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/src/targets/index.ts)). Commonplace is stronger on separation between knowledge artifact and system-definition artifact: a Commonplace note, source snapshot, type spec, instruction, generated index, and review artifact have different schemas and authority. Compound Engineering often encodes those differences in prose workflow rules inside a skill.

**Read-back:** `both` — Project-retained memory can be pulled deliberately by invoking skills such as `ce-sessions` or by reading `docs/solutions/`, and it is also pushed into later workflow context when CE skills automatically read `STRATEGY.md`, search `docs/brainstorms/` or `docs/solutions/`, include session-history findings, or load pulse configuration during an invoked workflow.

### Borrowable Ideas

**Make project-local learning a first-class workflow output.** Commonplace already has durable notes and reviews, but Compound Engineering's `ce-compound` pattern is narrower and operational: after a fix, create one structured learning under a predictable path and check overlap before duplicating it. Ready now as a Commonplace workshop-to-library pattern for solved operational problems, with Commonplace schemas replacing CE's local YAML.

**Use session-history extraction as an optional evidence source, not a default memory flood.** `ce-sessions` shows a useful middle ground: discover metadata, cap deep dives, extract skeletons, and synthesize only relevant prior attempts. Ready as an optional Commonplace review aid; it needs strict privacy and provenance rules before becoming routine.

**Separate static shipped skills from project-retained memory.** CE's installable skills are baseline system-definition artifacts, while `docs/solutions/`, `STRATEGY.md`, session findings, and pulse reports accumulate per project. Commonplace should preserve that boundary when packaging skills: shipped instructions should not be counted as read-back of local memory.

**Convert once-authored agent assets into multiple host formats.** The converter's explicit mappings for Codex, Gemini, Pi, Kiro, and OpenCode suggest a portability layer for Commonplace operational skills. Useful, but only after Commonplace has enough host-specific users to justify the maintenance burden.

**Keep refresh as a lifecycle workflow, not just validation.** `ce-compound-refresh` evaluates whether learnings should be kept, updated, consolidated, replaced, or deleted. Commonplace review gates catch many local issues, but a dedicated "is this artifact still true against current code/source?" lifecycle pass is worth borrowing for volatile descriptive artifacts.

**Machine-readable review contracts for skill-to-skill calls.** `ce-code-review` has a `mode:agent` JSON output path for programmatic callers while preserving a Markdown human report ([plugins/compound-engineering/skills/ce-code-review/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-code-review/SKILL.md)). Commonplace skills that are called by other skills should similarly expose a stable machine-readable result, rather than forcing downstream automation to parse human prose.

## Trace-derived learning placement

**Trace source.** The qualifying trace source is prior coding-agent session history across Claude Code, Codex, and Cursor. `ce-sessions` discovers JSONL files under host-specific session directories, extracts metadata such as branch/cwd/timestamps, filters by repo, branch, keywords, and scan window, and then extracts skeletons rather than loading whole transcripts ([plugins/compound-engineering/skills/ce-sessions/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/SKILL.md), [plugins/compound-engineering/skills/ce-sessions/scripts/discover-sessions.sh](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/discover-sessions.sh), [plugins/compound-engineering/skills/ce-sessions/scripts/extract-metadata.py](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/extract-metadata.py), [plugins/compound-engineering/skills/ce-sessions/scripts/extract-skeleton.py](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/extract-skeleton.py)).

**Extraction.** The extraction is deliberately lossy. Metadata extraction reads only enough lines to classify platform and session identity, keyword matching scans user/assistant text while skipping metadata, tool calls, tool outputs, and thinking blocks, and skeleton extraction emits user/assistant text plus collapsed tool-call summaries. A synthesis agent then turns the scratch files into sections such as "What was tried before", "What didn't work", "Key decisions", and "Related context" ([plugins/compound-engineering/skills/ce-sessions/SKILL.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/SKILL.md), [plugins/compound-engineering/skills/ce-sessions/scripts/extract-metadata.py](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/extract-metadata.py), [plugins/compound-engineering/skills/ce-sessions/scripts/extract-skeleton.py](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills/ce-sessions/scripts/extract-skeleton.py)).

**Scope and timing.** Session-history use is opt-in inside `ce-compound` interactive Full mode, skipped in lightweight and headless modes, and explicit when a user directly invokes `ce-sessions`. The trace loop is per-repo and time-windowed rather than global. Raw session traces remain host-managed; scratch skeletons are temporary; durable learning happens only if the orchestrating skill folds findings into a solution doc or the user acts on the answer.

**Survey placement.** This belongs in the trace-to-prose-distillation family. The raw trace is not the behavior-shaping memory by itself. The important split is raw session JSONL -> filtered skeleton/synthesis as a knowledge artifact -> optional `docs/solutions/` learning as durable project knowledge that future skills can read. It strengthens the survey claim that trace-derived memory becomes maintainable only after a curation and provenance boundary; CE requires explicit labels for session-sourced content when it enters a learning.

## Read-back placement

**Direction.** The system uses both pull and push for project-retained memory. Pull appears when a user invokes `ce-sessions`, runs `ce-compound-refresh` on `docs/solutions/`, or asks a skill to inspect a named artifact. Push appears when a CE workflow automatically reads well-known project state after invocation: `ce-strategy` is read by ideation/brainstorm/planning, `ce-plan` searches requirements docs, `ce-code-review` dispatches `ce-learnings-researcher`, `ce-compound` searches related solution docs and optionally session history, and `ce-product-pulse` reads pulse config before writing a browseable pulse report.

**Targeting and signal.** Targeting is mixed. Well-known anchors such as `STRATEGY.md`, `.compound-engineering/config.local.yaml`, and `CONCEPTS.md` are coarse identifier-based paths. Requirements and learning retrieval are instance-targeted by directory, filename, frontmatter, keywords, branch, repo, and semantic judgment. Session retrieval uses identifier signals where available (`repo`, branch/cwd, time window) and inferred lexical signals through keyword counts before handing filtered traces to an LLM synthesizer.

**Timing relative to action.** Most read-back is pre-action context: strategy grounds planning before the plan is written, prior learnings inform a review before findings are finalized, and session-history findings can alter a learning before it is saved. `ce-product-pulse` is after-the-fact reporting, but its saved reports become future browseable project memory.

**Selection, scope, and complexity.** The system has local caps and filters rather than a global memory budget. `ce-sessions` caps deep dives at five sessions and extracts skeletons to files. `ce-compound` tells related-doc searchers to grep frontmatter first, read only candidates, and fully read only strong/moderate matches. `ce-plan` narrows source requirements by path, recency, and relevance. These are practical context controls, but effective precision/recall is not verified from code.

**Authority at consumption.** The same retained artifact can carry different authority depending on the read path. `docs/solutions/` are knowledge artifacts when cited as past evidence, but they become soft system-definition artifacts when a reviewer or planner treats their prevention rules as constraints. `STRATEGY.md` is a stronger system-definition artifact for product direction because downstream skills are told to use it as grounding. Session skeletons and synthesized prior attempts are knowledge artifacts unless distilled into a solution doc or used by a workflow decision.

**Faithfulness.** I found implemented selection, extraction, validation, and refresh machinery, but not an end-to-end WITH/WITHOUT test proving that pushed project memory changes downstream agent behavior correctly. The code and skill contracts make the read-back path observable; behavioral compliance remains a runtime quality question.

**Other consumers.** Humans consume the same project-local artifacts through docs, reports, and terminal output. Converter writers consume plugin source as system-definition input. Release validation consumes manifests and counts. CI and tests consume converter output fixtures, not project memory quality.

## Curiosity Pass

**The strongest memory design is outside the plugin repo at runtime.** The plugin ships the workflows, but the valuable retained state accumulates in each consuming repository. Reviewing only the plugin source can show the contracts and write paths, not the quality of any team's actual `docs/solutions/` or session summaries.

**The README overstates component counts relative to the inspected tree.** The root README says the plugin "currently ships 37 skills and 51 agents", and the plugin README says "38+" skills and "50+" agents, but the inspected checkout has 39 `SKILL.md` files and 43 agent markdown files under `plugins/compound-engineering/` ([README.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/README.md), [plugins/compound-engineering/README.md](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/README.md), [plugins/compound-engineering/skills](https://github.com/EveryInc/compound-engineering-plugin/tree/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/skills), [plugins/compound-engineering/agents](https://github.com/EveryInc/compound-engineering-plugin/tree/09391874b4be1a248bc7d627b0ebd5c29f0c886b/plugins/compound-engineering/agents)). That is a documentation freshness issue, not a memory-system flaw, but it matters because manifests and inventories are part of the adoption surface.

**Static skill activation should not be confused with memory read-back.** Installed CE skills entering context are baseline system instructions. The memory-system behavior begins when those skills read accumulated project artifacts or session traces.

**The session-history design is privacy-aware but still powerful.** The scripts avoid thinking blocks and tool outputs, collapse tool calls, and emphasize technical content over personal content. Even so, the source traces live in host session directories and may contain sensitive user context, so any borrowed design needs a hard provenance and retention policy.

**Converter cleanup is a governance mechanism.** The Codex writer uses managed install manifests, safe path checks, and legacy cleanup to remove stale generated artifacts without clobbering unrelated user files ([src/targets/codex.ts](https://github.com/EveryInc/compound-engineering-plugin/blob/09391874b4be1a248bc7d627b0ebd5c29f0c886b/src/targets/codex.ts)). That is not memory retrieval, but it is lifecycle control over behavior-shaping retained artifacts.

## What to Watch

- Whether `ce-compound` gains stronger automated provenance from session ids, trace timestamps, model versions, and source artifact paths when session-history findings enter `docs/solutions/`; that would make trace-derived learning auditable rather than merely labeled.
- Whether CE adds behavioral faithfulness tests for pushed project memory, such as plan/review outputs with and without `docs/solutions/` or session-history context.
- Whether component inventory counts become fully generated from the plugin tree and release metadata; drift here weakens trust in the install surface.
- Whether target converters keep pace with native plugin capabilities, especially Codex custom agents and skills; converter decisions currently shape which behavior-shaping artifacts are native, copied, or suppressed.
- Whether `ce-compound-refresh` develops a stable report artifact for refresh decisions; that would make learning lifecycle governance easier to audit across teams.
