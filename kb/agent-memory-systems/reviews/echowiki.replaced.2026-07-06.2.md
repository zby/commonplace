# EchoWiki

> Replaced 2026-07-06. See [echowiki](./echowiki.md) for the current review.

EchoWiki, from `mohammadmaso/echowiki`, is a local-first Obsidian plugin and optional TypeScript CLI that turns user-supplied notes and voice transcripts under `raw/` into a structured Markdown wiki under `wiki/`. At the reviewed commit it is a compilation and maintenance pipeline over vault files: summaries, concept pages, entity pages, an index, and an operations log. It is not a query layer or vector retrieval system over the compiled wiki ([README.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/README.md), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [obsidian-plugin/src/main.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/main.ts)).

## Core Ideas

**A single raw lane feeds all compilation.** Manual Markdown/text notes, copied active notes, and voice transcripts all land under `raw/`; the watcher and manual compile command then treat compilable `.md`, `.txt`, and `.markdown` files the same way ([README.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/README.md), [obsidian-plugin/src/main.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/main.ts), [obsidian-plugin/src/raw-watcher.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/raw-watcher.ts), [obsidian-plugin/src/utils.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/utils.ts)). Voice capture preserves the audio file and writes a transcript back into `raw/`, so the compiler does not carry a voice-specific branch after transcription ([obsidian-plugin/src/voice-modal.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/voice-modal.ts), [obsidian-plugin/src/stt-client.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/stt-client.ts)).

**The compiler maintains a wiki, not an embedding store.** `compileShortDoc` runs a Vercel AI SDK backed sequence: summarize the raw document, read existing concept/entity briefs, plan concept/entity creates and updates, generate or rewrite pages, strip invalid wikilinks, add backlinks, update `index.md`, and append to `log.md` ([src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [src/wiki/prompts.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/prompts.ts), [src/wiki/page-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/page-writer.ts), [src/wiki/index-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/index-writer.ts)). I found no vector store, search index, ranking model, or query/chat generator in the inspected source.

**Runtime instructions live in the wiki.** The compiler reads `wiki/AGENTS.md` at runtime and falls back to a built-in default schema when the file is absent; the template version explicitly says edits to that file change future compilation behavior without redeploying code ([src/wiki/schema.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/schema.ts), [templates/wiki/AGENTS.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/templates/wiki/AGENTS.md)). That makes the wiki carry both knowledge artifacts and a local system-definition artifact.

**Context efficiency comes from brief lists and whitelisted links.** Before planning updates, EchoWiki reads descriptions or previews for all existing concept and entity pages, not their full bodies. It then gives the model a complete whitelist of existing wiki targets plus planned targets and strips ghost wikilinks after generation ([src/wiki/briefs.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/briefs.ts), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [src/wiki/wikilink.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/wikilink.ts)). This manages complexity through progressive summaries and symbolic targets, but it still scales with the number of concept/entity pages because the brief list is not top-k ranked.

**Adoption is file-native; trust controls are partial.** The retained wiki is plain Markdown in an Obsidian vault, with settings and pending approvals stored by the plugin, so users can inspect and edit the artifacts locally ([obsidian-plugin/src/settings.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/settings.ts), [obsidian-plugin/src/vault-storage.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/vault-storage.ts)). The code enforces approval gates, in-flight suppression, path bounds, and generated frontmatter, but I did not find tests, an OKF schema validator, a faithfulness check, or the documented conflict detector for manually edited wiki pages ([package.json](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/package.json), [src/storage/node-storage.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/storage/node-storage.ts), [src/wiki/frontmatter.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/frontmatter.ts), [templates/wiki/AGENTS.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/templates/wiki/AGENTS.md)).

## Artifact analysis

- **Storage substrate:** `files` `repo` - The durable memory surface is vault-local files: `raw/` inputs, generated Markdown under `wiki/summaries/`, `wiki/concepts/`, `wiki/entities/`, `wiki/index.md`, `wiki/log.md`, and the editable `wiki/AGENTS.md`. The package repository carries the compiler code, default schema, template instructions, manifests, and build workflow; settings and pending approvals are Obsidian plugin data rather than a separate database ([README.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/README.md), [src/storage/node-storage.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/storage/node-storage.ts), [obsidian-plugin/src/vault-storage.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/vault-storage.ts), [obsidian-plugin/src/settings.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/settings.ts)).
- **Representational form:** `prose` `symbolic` - Raw notes, summaries, concept/entity pages, prompts, and runtime wiki instructions are prose. Frontmatter, source lists, page types, entity types, slugs, path whitelists, plugin settings, pending queue items, command ids, and JSON LLM plans are symbolic. The inspected code stores no embeddings, adapters, or model weights ([src/wiki/prompts.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/prompts.ts), [src/wiki/frontmatter.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/frontmatter.ts), [src/wiki/llm.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/llm.ts), [obsidian-plugin/src/main.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/main.ts)).
- **Lineage:** `authored` `imported` - The compiler, plugin, prompts, default schema, and template `AGENTS.md` are authored system-definition artifacts. Raw files and voice transcripts are user/imported source material; generated summaries, concepts, entities, backlinks, index entries, and log entries are derived views over those imported inputs. I did not find durable memory distilled from agent session logs, tool traces, trajectories, or model probes ([CLAUDE.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/CLAUDE.md), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [obsidian-plugin/src/main.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/main.ts)).
- **Behavioral authority:** `knowledge` `instruction` `routing` `enforcement` `validation` `learning` - Generated wiki pages are knowledge artifacts for humans and later compiler calls. `wiki/AGENTS.md`, prompts, and plugin commands instruct behavior; `index.md`, brief lists, source lists, slugs, and whitelisted wikilinks route attention; approval, watch mode, in-flight suppression, and path-bound storage enforce operational boundaries; JSON parsing, model/key checks, connection tests, frontmatter construction, and ghost-link stripping provide light validation; existing concept/entity briefs give the compiled wiki learning authority over future wiki growth ([src/wiki/schema.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/schema.ts), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [src/wiki/wikilink.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/wikilink.ts), [obsidian-plugin/src/main.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/main.ts), [obsidian-plugin/src/settings-tab.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/settings-tab.ts)).

**Raw inputs and transcripts.** These are imported source artifacts in vault files. They are not directly searched by a later retrieval layer; a compile run reads the chosen file in full and turns it into a summary plus wiki maintenance actions.

**Generated summaries, concepts, and entities.** These are prose knowledge artifacts with symbolic frontmatter, source lists, and wikilinks. Summaries represent one raw document, while concepts and entities can accumulate sources through repeated updates. Their quality is LLM-mediated and not verified by tests in the repository.

**Runtime compiler instructions.** `wiki/AGENTS.md` is an authored prose instruction artifact consumed by the compiler on each run. It has higher behavioral authority than ordinary wiki pages because it changes how new wiki pages are organized and written.

**Index, briefs, and link whitelist.** `index.md` is a human and agent navigation artifact. Brief extraction and the known-target whitelist are transient read-side structures built from retained pages; they are not themselves retained as a separate index.

**Plugin settings and pending queue.** The Obsidian plugin stores configuration, approval state, and pending raw paths. These are symbolic system-definition artifacts governing when compilation may run, not knowledge artifacts.

**Promotion path.** EchoWiki promotes imported raw material into summaries, then into concept/entity pages with backlinks and index entries. It does not promote compiled claims into validators, typed review states, route tables beyond filenames/index entries, embeddings, or model updates.

## Comparison with Our System

EchoWiki and Commonplace share a file-first orientation: durable knowledge is inspectable Markdown, local instructions can shape future agent behavior, and links are part of the memory architecture. EchoWiki is much more Obsidian-native and lower-friction for personal capture: a user can speak or drop a note, approve compilation, and get a graph-browsable wiki.

The main divergence is authority. Commonplace separates collection contracts, type specs, source snapshots, validation, review gates, and curated indexes. EchoWiki lets an LLM rewrite concept/entity pages directly from raw notes and brief lists, with only light symbolic checks around paths, frontmatter, and wikilinks. That makes EchoWiki useful as a personal compiler, but weaker as a trusted agent memory substrate.

EchoWiki's context strategy is also narrower. Commonplace uses typed routing and curated indexes to help agents decide what to load. EchoWiki pushes all concept/entity briefs into the compiler and lets the model judge which pages to update. This is simple and local, but it becomes increasingly dependent on brief quality and page count.

### Borrowable Ideas

**Runtime-editable compiler instructions in the target wiki.** Ready with guardrails. Commonplace could use a lower-authority, workshop-local instruction file for importers or compilers whose conventions should be editable without changing package code.

**Single raw lane for multiple capture modes.** Useful when Commonplace gets richer capture tooling. Voice, pasted text, and dropped Markdown should converge before durable note writing so source modality does not leak into library structure.

**Brief list plus wikilink whitelist.** Ready now as a small context-control pattern. For LLM-assisted note generation, passing short retained descriptions plus a strict link whitelist is a cheap way to reduce ghost links.

**Approval queue before automatic compilation.** Ready for any future watcher. EchoWiki's pending queue keeps automatic detection separate from permission to write durable artifacts.

**Do not borrow direct LLM rewrites as sufficient curation.** Commonplace should not let a model silently rewrite reviewed library artifacts without source-grounding, diff review, and validation.

## Write side

**Write agency:** `manual` `automatic` - Users manually create/drop raw notes, copy active notes into `raw/`, record audio, approve/reject pending compilations, run the CLI, edit settings, and can edit wiki files. Automatically, the plugin watches `raw/`, queues or compiles changed files, transcribes audio to Markdown, and the compiler writes summaries, concept/entity pages, backlinks, `index.md`, and `log.md` ([obsidian-plugin/src/main.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/main.ts), [obsidian-plugin/src/raw-watcher.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/raw-watcher.ts), [obsidian-plugin/src/approval-modal.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/approval-modal.ts), [src/cli/compile.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/cli/compile.ts), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts)).

**Curation operations:** `evolve` - The automatic curation operation over already-stored memory is rewriting existing concept/entity pages and repairing backlinks/index entries when a new raw document arrives. Creating a summary from a raw file is acquisition, not curation; concept/entity page creation is mostly acquisition into a structured wiki surface. I did not find deterministic deduplication, decay, invalidation, promotion-by-recurrence, or a verified synthesis step beyond LLM-authored concept/entity updates ([src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [src/wiki/page-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/page-writer.ts), [src/wiki/index-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/index-writer.ts)).

## Read-back

**Read-back:** `both` - The compiled wiki is pulled when a human or agent browses Obsidian, follows `index.md`, opens graph view, or reads wiki files directly. Retained memory is also pushed into future EchoWiki compiler calls: `wiki/AGENTS.md`, concept/entity brief lists, known wikilink targets, and existing page bodies for planned updates are assembled into the LLM prompt before generation ([README.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/README.md), [src/wiki/schema.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/schema.ts), [src/wiki/briefs.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/briefs.ts), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts)).

**Read-back signal:** `coarse` `identifier` `inferred / judgment` - The compiler always loads coarse schema/instruction context and all concept/entity briefs. It uses identifiers such as page paths, doc names, slugs, and known wikilink targets. It also relies on LLM judgment to decide which existing concepts or entities to update from the current document and brief list. There is no lexical search engine, embedding similarity, or top-k retrieval layer.

**Faithfulness tested:** `no` - The repository has a manual checklist and connection tests, but I found no test suite, ablation, or audit proving that pushed wiki memory changes compiler or future-agent behavior faithfully. The root `package.json` test script exits with an error placeholder ([README.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/README.md), [package.json](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/package.json), [obsidian-plugin/src/settings-tab.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/settings-tab.ts)).

The injection point is pre-generation inside the compile call. EchoWiki reads retained schema/brief/page state before asking the LLM to summarize, plan, generate, or rewrite; after generation it writes updates and appends logs, which are write-side maintenance rather than a second read-back phase.

Selection and complexity are mostly prompt-shaped. Brief extraction keeps page bodies out of the initial planning prompt, and the known-target whitelist limits link explosion. Once the model selects an update, EchoWiki reads that existing page body in full. There is no token budget, recency filter, source-confidence filter, or ranked retrieval policy beyond these symbolic conventions.

Authority at consumption varies. `wiki/AGENTS.md` is instruction authority for the compiler. Existing concept/entity briefs and pages are advisory context for future updates. `index.md` and wikilinks route attention for humans, Obsidian, and agents that inspect the vault. Approval settings and pending queue state control whether writes occur.

## Curiosity Pass

**The implementation has moved away from the `.ai` architecture docs.** The PRD and technical spec still describe a Mastra implementation, while `CLAUDE.md`, `package.json`, and the source code show a Vercel AI SDK compiler bundled into an Obsidian plugin ([.ai/01-PRD.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/.ai/01-PRD.md), [.ai/02-TECH-SPEC.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/.ai/02-TECH-SPEC.md), [CLAUDE.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/CLAUDE.md), [package.json](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/package.json)).

**The `sources/` claim is only partly wired.** Summary frontmatter points `full_text` at `sources/<doc>.md`, and the wiki schema reserves `sources/`, but `compileShortDoc` reads source content from the raw file and does not appear to write a copied source file under `wiki/sources/` for short documents ([src/wiki/page-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/page-writer.ts), [src/wiki/compiler.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/compiler.ts), [src/wiki/schema.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/schema.ts)).

**OKF is more a convention than an enforced schema.** The docs promise OKF-style frontmatter, and the code constructs `type`, `description`, `doc_type`, `full_text`, and `sources` fields, but I did not find an OKF validator or report generator in the source ([README.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/README.md), [src/wiki/frontmatter.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/frontmatter.ts), [src/wiki/page-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/page-writer.ts)).

**Conflict handling is promised but not evident.** The template says not to silently overwrite manually edited wiki pages, but the storage and page writers modify existing files directly after an LLM update without retained edit provenance, checksums, or conflict detection ([templates/wiki/AGENTS.md](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/templates/wiki/AGENTS.md), [src/wiki/page-writer.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/src/wiki/page-writer.ts), [obsidian-plugin/src/vault-storage.ts](https://github.com/mohammadmaso/echowiki/blob/11446997423e88177dba62d6a3d5c1e8b4886c62/obsidian-plugin/src/vault-storage.ts)).

## What to Watch

- Whether EchoWiki starts persisting source copies under `wiki/sources/`. That would make summary `full_text` pointers verifiable and strengthen source lineage.
- Whether OKF validation, link health checks, or reports under `wiki/reports/` are implemented. That would move the system from generated wiki pages toward a governed knowledge base.
- Whether manual-edit conflict handling is added. That is the difference between safe personal wiki maintenance and silent LLM overwrite.
- Whether a query/chat or ranked retrieval layer appears over the compiled wiki. That would change read-back from brief-list push plus human pull into a real memory retrieval system.
- Whether tests or faithfulness checks are added for concept/entity updates. EchoWiki's main quality risk is not file I/O, but whether the LLM rewrites preserve and improve knowledge.
