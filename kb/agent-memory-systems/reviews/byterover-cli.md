# byterover-cli

ByteRover CLI, by campfirein/ByteRover, is a TypeScript/Oclif CLI and daemon for giving coding agents a project-scoped memory substrate. The inspected repository ships the `brv` binary, a daemon-routed CLI/TUI/WebUI, MCP tools, agent connectors, a `.brv/context-tree/` storage model, context-tree version control, query and curate logs, human review, and a deterministic "dream" cleanup pass.

**Repository:** https://github.com/campfirein/byterover-cli

**Reviewed commit:** [c498b2f4f764a9da9cd7905fc7a3585d4f98469e](https://github.com/campfirein/byterover-cli/commit/c498b2f4f764a9da9cd7905fc7a3585d4f98469e)

**Last checked:** 2026-06-01

## Core Ideas

**Project memory is a context tree on disk.** ByteRover initializes `.brv/context-tree/` under the project and treats topic files, folder context files, generated summaries, manifests, archives, and index files as the local memory substrate ([file-context-tree-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/file-context-tree-service.ts), [constants.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/constants.ts)). Recent tool-mode curation writes validated `<bv-topic>` HTML files rather than free-form Markdown: the writer requires one `<bv-topic>` root, validates every closed-vocabulary `<bv-*>` element, rejects unsafe paths, preserves `createdat`, stamps `updatedat`, and writes atomically ([html-writer.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/render/writer/html-writer.ts)).

**Authoring is agent-driven, but the storage grammar is system-defined.** The `brv curate` CLI now runs a session protocol where ByteRover asks the calling agent to author a `{html, meta}` envelope, then validates and writes the topic; MCP collapses the same pattern into a single `brv-curate` tool call whose description embeds the element vocabulary ([docs/curate-protocol.md](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/docs/curate-protocol.md), [curate/index.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/oclif/commands/curate/index.ts), [brv-curate-tool.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/mcp/tools/brv-curate-tool.ts)). The older in-daemon curate executor still uses an internal LLM agent, but its final response is also routed through the HTML writer ([curate-executor.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/executor/curate-executor.ts)).

**Retrieval is mostly deterministic and tiered.** `SearchKnowledgeService` builds a MiniSearch BM25 index over local and linked context trees, parses HTML topic text plus selected attributes, supports symbolic path/scope/kind/maturity filters, propagates child scores to parent summaries, and blends BM25 with runtime importance/recency/maturity signals ([search-knowledge-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/agent/infra/tools/implementations/search-knowledge-service.ts)). Query execution first tries cache and direct retrieval, then may prefetch high-confidence context and manifest context before running an LLM answer synthesis; tool-mode query and MCP `brv-query` return deterministic matches for the calling agent to synthesize, with no ByteRover-side LLM provider required ([query-executor.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/executor/query-executor.ts), [brv-query-tool.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/mcp/tools/brv-query-tool.ts)).

**Derived context bounds volume and complexity.** Directory summaries `_index.md` are generated from leaf files or child summaries using a children hash, token counts, and condensation order; `_manifest.json` allocates summaries, contexts, and archive stubs into token-budgeted lanes for broad context injection ([file-context-tree-summary-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/file-context-tree-summary-service.ts), [file-context-tree-manifest-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/file-context-tree-manifest-service.ts)). Archive stubs preserve a ghost cue while moving full content aside, and `expand_knowledge` can drill back into full archived content or overview files ([file-context-tree-archive-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/file-context-tree-archive-service.ts), [expand-knowledge-tool.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/agent/infra/tools/implementations/expand-knowledge-tool.ts)).

**Runtime signals are deliberately outside the shared tree.** Importance, recency, maturity, access counts, and update counts live in a file-backed key-value sidecar under the project data directory, not in versioned topic frontmatter, so query-time ranking updates do not dirty `brv vc` state ([runtime-signals-schema.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/core/domain/knowledge/runtime-signals-schema.ts), [runtime-signal-store.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/runtime-signal-store.ts), [file-key-storage.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/agent/infra/storage/file-key-storage.ts)). Search hits and curate writes bump these signals; dream prune candidates then use them alongside file age ([memory-scoring.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/core/domain/knowledge/memory-scoring.ts), [prune-candidates.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/dream/tool-mode/prune-candidates.ts)).

**Adoption is through native agent surfaces.** The connector subsystem can install skill files, rules sections, MCP server configs, and hook configs for coding agents; a hidden `hook-prompt-submit` command emits ByteRover instructions for Claude Code `UserPromptSubmit`-style pre-prompt hooks ([skill-connector.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/connectors/skill/skill-connector.ts), [rules-connector.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/connectors/rules/rules-connector.ts), [mcp-connector.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/connectors/mcp/mcp-connector.ts), [hook-connector.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/connectors/hook/hook-connector.ts), [hook-prompt-submit.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/oclif/commands/hook-prompt-submit.ts)).

## Artifact analysis

- **Storage substrate:** `files` — Project-local files under `.brv/context-tree/`, currently primarily HTML `<bv-topic>` files written by `writeHtmlTopic`, with legacy Markdown still indexed
- **Representational form:** `prose` `symbolic` — Prose topic bodies, summaries, and log fields plus symbolic HTML elements, JSON envelopes, runtime signals, Git state, connector config, validation grammar, and indexes
- **Lineage:** `authored` `trace-extracted` — Topics and connector artifacts are authored or template-generated, while runtime signals, logs, review backups, and dream state are extracted from query, curate, review, and task events
- **Behavioral authority:** `knowledge` `instruction` `enforcement` `routing` `validation` `ranking` `learning` — Topics and logs provide knowledge; connector prompts instruct; writers and review paths enforce/validate; derived artifacts route and budget context; signals rank, prune, and carry learned operational state

**Context-tree topic files.** Storage substrate: project-local files under `.brv/context-tree/`, currently primarily HTML `<bv-topic>` files written by `writeHtmlTopic`, with legacy Markdown still indexed. Representational form: mixed prose and symbolic HTML elements; `<bv-rule>`, `<bv-decision>`, `<bv-fact>`, `path`, `related`, `tags`, and timestamps are symbolic enough for validation, routing, indexing, and rendering, while the bodies remain prose. Lineage: authored by a calling agent from a user intent, MCP input, or in-daemon curate prompt; `createdat`/`updatedat` are system-managed, but the topic body usually does not preserve a source transcript or proof object. Behavioral authority: knowledge artifact when returned by search/query as evidence or context; system-definition artifact when installed connector rules tell an agent to query or curate from it, when high-impact `meta` creates pending review, or when topic element types and attributes drive validation, navigation, ranking, and UI behavior.

**Derived summaries, manifests, indexes, abstracts, overviews, and archive stubs.** Storage substrate: derived files inside the context tree, with most excluded from sync and search by `isDerivedArtifact`; root `index.html` is intentionally tracked for peer navigation, while `_archived/*.stub.md` remains searchable and points to full archived content ([derived-artifact.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/derived-artifact.ts), [index-generator.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/index-generator.ts)). Representational form: prose summaries plus symbolic frontmatter/JSON/HTML navigation records. Lineage: generated from children hashes, source fingerprints, topic metadata, LLM compaction, or deterministic fallback; stale summaries and manifests are regenerated from source files. Behavioral authority: ranking, routing, scope, and context-budget authority, because these artifacts decide what broad context is visible before an agent reads full topics. They are derived system-definition artifacts, not source-of-truth knowledge artifacts.

**Runtime-signal sidecar.** Storage substrate: file-backed `FileKeyStorage` records under the project data store, keyed as `["signals", ...pathSegments]` and validated by `RuntimeSignalsSchema`; the daemon and agent process both create `RuntimeSignalStore` instances over that shared storage ([agent-process.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/daemon/agent-process.ts), [service-initializer.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/agent/infra/agent/service-initializer.ts)). Representational form: symbolic numeric state. Lineage: trace-derived from search access hits, curate add/update/merge/delete operations, and dream archive/undo flows; sidecar failures are explicitly fail-open in most paths. Behavioral authority: ranking and lifecycle authority. Signals feed compound retrieval scoring, maturity filters, manifest lane allocation, and prune candidate selection. They are system-definition artifacts because they change future retrieval and cleanup behavior without being shown as ordinary knowledge claims.

**Query, curate, review, task, and dream logs.** Storage substrate: JSON files under per-project data directories such as `query-log/`, `curate-log/`, `review-backups/`, `task-history/`, and dream log/state stores. Representational form: symbolic JSON envelopes plus prose fields. Lineage: captured from daemon task lifecycle events, query metadata, matched docs, write results, agent-supplied curate metadata, and pre-write file backups ([query-log-handler.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/process/query-log-handler.ts), [file-query-log-store.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/storage/file-query-log-store.ts), [curate-html-log.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/process/curate-html-log.ts)). Behavioral authority: mostly audit and governance. Query logs can summarize coverage, cache hit rate, top recalled docs, and knowledge gaps, which advises future curation but does not directly rewrite topics ([query-log-summary-use-case.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/usecase/query-log-summary-use-case.ts)). Curate logs and backups carry stronger review authority: high-impact writes can surface in `brv review pending`, and rejection restores prior file bytes.

**Connector artifacts and hook prompts.** Storage substrate: agent-specific config files, rule files, skill directories, MCP server config entries, and stdout from the hidden hook command. Representational form: prose instructions plus symbolic JSON/YAML/TOML config. Lineage: generated from ByteRover templates and installed into the host agent's native surfaces. Behavioral authority: system-definition artifact authority. The emitted `brv-instructions` section tells the agent to run `brv query` before code work and `brv curate` after code changes, so it changes future action through instruction rather than through ordinary evidence ([brv-instructions.md](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/templates/sections/brv-instructions.md)).

**Context-tree version-control state.** Storage substrate: a nested Git repository in `.brv/context-tree/` managed through isomorphic-git and protected from the host repository by project `.gitignore` entries ([isomorphic-git-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/git/isomorphic-git-service.ts), [vc-handler.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/transport/handlers/vc-handler.ts), [gitignore.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/utils/gitignore.ts)). Representational form: Git commits, branches, remotes, and blobs. Lineage: preserves file-level history of curated knowledge and generated trackable navigation files, but intentionally excludes most derived artifacts and runtime signals. Behavioral authority: collaboration, rollback, merge, and audit authority over the source memory corpus.

The promotion path is partial. A user or agent can move observations into `<bv-topic>` topics, add high-impact review metadata, have a human approve or reject, commit with `brv vc`, and let repeated usage promote runtime maturity. What is missing is a strong promotion boundary from "often retrieved" or "agent-authored" into "trusted rule": sidecar maturity changes ranking and pruning, not the evidential status of the claim.

## Comparison with Our System

| Dimension | ByteRover CLI | Commonplace |
|---|---|---|
| Primary purpose | Operational project memory for coding agents, with CLI/TUI/WebUI/MCP surfaces | Methodology KB for agent-operated knowledge-base design |
| Main substrate | `.brv/context-tree/` HTML/Markdown files, generated summaries, sidecar signals, logs, nested Git | Git-tracked Markdown notes, sources, instructions, reviews, ADRs, schemas, generated indexes |
| Retrieval | BM25, symbolic paths, scope filters, maturity filters, score propagation, manifest injection, optional LLM synthesis | `rg`, curated indexes, descriptions, links, skills, validation/review workflows |
| Trace use | Query/curate traces update sidecar ranking signals and logs; curate can turn work into topics | Trace use is mostly manual review, source snapshots, and generated review artifacts |
| Governance | HITL review queue, backups, VC, strict topic element validation | Type contracts, link vocabulary, semantic review, validation, archive/replacement lifecycle |
| Activation | Baseline agent instructions/hooks/MCP push the query/curate workflow; retained topic recall is pull/search | Mostly deliberate pull through search/index/links, plus instruction skills when invoked |

The strongest alignment is that both systems treat memory as a typed artifact landscape, not just a vector store. ByteRover's `<bv-topic>` grammar is more UI- and tool-oriented than Commonplace Markdown frontmatter, but both make storage inspectable, versionable, and partially validated.

The largest divergence is where authority lives. Commonplace keeps durable claims in reviewed Markdown and gives stronger authority to explicit instructions, schemas, and validation commands. ByteRover lets operational surfaces carry stronger implicit authority: a topic can affect search ranking through sidecar usage signals, MCP tools can render matched topics directly into the calling agent's context, and hook-installed instructions can require a query/curate loop. That makes activation stronger but also spreads authority across templates, daemon code, connector configs, sidecar scores, logs, and topic files.

ByteRover is ahead on productized adoption. It has connectors for existing agents, a web dashboard, context-tree version control, worktrees, shared sources, MCP tools, query logs, review UI, cancellation, settings, and a background cleanup path. Commonplace is ahead on methodology governance: review artifacts are source-pinned, type contracts are explicit, link semantics are a first-class vocabulary, and the knowledge layer is not optimized around silent runtime promotion.

**Read-back:** `pull` — Retained topic content returns through explicit `brv query`, MCP `brv-query`, `search_knowledge`, or `expand_knowledge` calls; installed rules, hook output, and MCP descriptions inject shipped workflow instructions, which are a baseline context surface rather than memory read-back

### Borrowable Ideas

**Sidecar volatile ranking signals.** Ready to borrow for Commonplace only if clearly scoped. ByteRover's move of access counts, importance, maturity, and recency out of versioned files is a good pattern for per-agent or per-machine activity signals. Commonplace could keep reviewable Markdown clean while maintaining local "recently useful" hints for navigation.

**Path-exists correction instead of silent overwrite.** Ready now as a design pattern. The tool-mode curate protocol returns a structured `path-exists` error with existing content and asks the agent to merge or consciously overwrite. Commonplace could use the same shape for agents writing notes or review artifacts.

**Generated navigation as a synced but derived artifact.** Worth borrowing where human/agent navigation needs a stable handoff file. ByteRover's root `index.html` is derived but intentionally synced so peers do not need to rebuild before browsing. Commonplace generated indexes are Markdown and regenerated locally; a tracked "reader index" could be useful when consumers do not run our tooling.

**Query-log summaries as curation guidance.** Needs a concrete use case. ByteRover's query-log summary turns unmatched queries and top recalled docs into actionable maintenance signals. Commonplace could generate "coverage pressure" reports from failed searches or repeated `rg` patterns, but should not automatically promote the result into notes.

**Keep hook instructions short and operational.** Borrow cautiously. ByteRover's hook prompt is blunt: code task means query first, code change means curate after. Commonplace agent instructions could use similarly concrete trigger tables, but the instruction should remain small enough not to crowd the user's actual task.

**Do not borrow usage-based maturity as trust.** ByteRover maturity is useful for ranking and pruning. Commonplace should not let repeated retrieval alone increase epistemic or governance status; review and source lineage should carry that promotion.

## Trace-derived learning placement

**Trace source:** `tool-traces` `event-streams` — Search hits, curate outcomes, query metadata, review/dream choices, and task lifecycle events are the trace inputs named by the review.

**Learning scope:** `per-project` `cross-task` — Signals are project-scoped, and the resulting ranking/lifecycle state plus curated topics can shape later tasks in that project.

**Learning timing:** `online` `offline` `staged` — Query and curate paths update signals online, query-log summaries are offline operator views, and dream is a staged maintenance loop.

**Distilled form:** `prose` `symbolic` — Trace-derived state becomes prose/symbolic topic files, symbolic runtime signals, JSON logs, and dream candidate/archive records.

**Trace source.** ByteRover consumes several trace classes: search result hits returned by `SearchKnowledgeService`, curate add/update/merge/delete outcomes, query task metadata and matched docs, high-impact curate metadata, review decisions/backups, dream scan/finalize choices, and task lifecycle events. The strongest behavior-shaping trace path is not full transcript mining; it is small operational traces converted into runtime signals and logs.

**Extraction.** Extraction is mostly deterministic. Search results are accumulated as access hits and later flushed into sidecar increments; curate writes seed, bump, merge, or delete sidecar records; query logs aggregate matched docs and gap keywords; dream scans turn topic structure, BM25 similarity, sidecar scores, and mtimes into link/merge/prune/synthesize candidates ([tool-mode-sidecar-updaters.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/context-tree/tool-mode-sidecar-updaters.ts), [search-knowledge-service.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/agent/infra/tools/implementations/search-knowledge-service.ts), [dream-session.ts](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/src/server/infra/dream/tool-mode/dream-session.ts)). The semantic extraction from "what happened in a coding session" into a new topic is delegated to the calling agent or the internal curate agent; ByteRover validates the shape and stores the result, but it does not expose an independent judge that verifies the factual claim.

**Storage substrate.** Raw-ish traces persist as JSON logs and backups in project data directories. Distilled trace-derived behavior state persists as runtime signals in `FileKeyStorage`, as curated topic files in `.brv/context-tree/`, and as dream archive state under `.brv/archive/` with captured prior text, mtime, and signals for undo.

**Representational form.** Query and curate logs are symbolic JSON plus prose fields. Runtime signals are symbolic numeric records. Curated topics are mixed prose/symbolic HTML. Dream candidates are transient symbolic envelopes containing topic HTML, paths, summaries, scores, ages, and signals.

**Lineage.** Sidecar signals have clear operational lineage to access hits and curate operations, but they are intentionally not human-readable evidence. Query logs keep the query, matched docs, timing, tier, and response when available. Curate logs can keep agent-supplied reason, impact, confidence, summaries, file paths, and backups. Topic claims themselves have weaker lineage unless the author included source paths or explanation inside the topic body.

**Behavioral authority.** Raw logs are knowledge artifacts for audit and maintenance. Runtime signals are system-definition artifacts because they alter ranking, maturity filtering, manifest allocation, archive candidacy, and dream pruning. Curated topic files are knowledge artifacts when retrieved as evidence; when represented as `<bv-rule>` and returned through query/MCP workflows, they can acquire instruction-channel influence over the next agent action.

**Scope and timing.** Signals are project-scoped and update online from query and curate paths. Query-log summaries are offline operator views. Dream is a staged maintenance loop: scan candidates, let the calling agent decide semantic actions, then finalize archives with undo metadata.

**Survey placement.** ByteRover belongs in the trace-to-operational-signal branch of the [trace-derived learning survey](../trace-derived-learning-techniques-in-related-systems.md). It strengthens the survey's split between raw trace retention and distilled behavior-shaping artifacts: the durable learned behavior is not a model weight or automatic rule, but sidecar ranking/lifecycle state plus agent-authored topic files.

## Curiosity Pass

The most interesting design choice is the split between inspectable topic files and non-versioned runtime signals. ByteRover avoids dirtying `brv vc` on every query, but that also means important behavioral state is outside the reviewable Git history.

The HTML topic grammar is more constrained than a Markdown note but less transparent than a full schema-backed document type. It gives enough symbolic structure for UI, validation, and search, while still allowing prose inside tags. That is a pragmatic middle point for agent-authored operational memory.

The README's LoCoMo and LongMemEval-S numbers are not contradicted by the implementation, but the checked-in repository mainly contains paper text and benchmark images, not the evaluation harness or data needed to reproduce the claims ([README.md](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/README.md), [paper/README.md](https://github.com/campfirein/byterover-cli/blob/c498b2f4f764a9da9cd7905fc7a3585d4f98469e/paper/README.md)). Treat those numbers as project claims, not as verified evidence from this review.

The "dream" system is less autonomous than the name suggests in current tool mode. It deterministically surfaces link/merge/prune/synthesize candidates; the calling agent still supplies semantic judgment and `brv curate` writes, and finalize only archives chosen files.

The connector layer may be ByteRover's strongest memory feature. It solves the hard adoption problem: memory that is not surfaced in the host agent's native prompt/tool flow is easy to forget.

## What to Watch

- Whether runtime signals gain a first-class audit/export view that explains why a topic became `validated` or `core`.
- Whether topic claims gain stronger lineage to source files, prompts, query logs, or review decisions instead of relying on agent-authored prose.
- Whether the structural element-axis search hint grows into a public selector grammar; that would move ByteRover closer to typed KB querying than plain BM25.
- Whether the benchmark harness and datasets become reproducible from the repository rather than paper-only claims.
- Whether dream starts producing governed promotion records, not just candidate lists and archive moves.
- Whether hook-driven query/curate compliance is measured with an ablation or post-action audit.
