# Basic Memory

Basic Memory, from Basic Machines' `basicmachines-co/basic-memory` repository, is a local-first Markdown knowledge graph for AI assistants and humans. At the reviewed commit, its core system keeps user-owned Markdown files as the durable source, materializes them into SQLite or Postgres entities, observations, relations, search rows, and optional vector chunks, and exposes that memory through a FastAPI API, Typer CLI, MCP tools, shared skills, and host integrations including a Claude Code plugin.

**Repository:** https://github.com/basicmachines-co/basic-memory

**Reviewed commit:** [fc2ee07076eb397b09db7b2681e5213002df0d70](https://github.com/basicmachines-co/basic-memory/commit/fc2ee07076eb397b09db7b2681e5213002df0d70)

**Last checked:** 2026-06-03

## Core Ideas

**Markdown files are the user-owned source of truth; the database is a projection.** The README presents Basic Memory as local-first plain Markdown that humans and AI clients can read, write, and search, and the note-format docs state that files update the knowledge graph in the database ([README.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/README.md), [docs/NOTE-FORMAT.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/docs/NOTE-FORMAT.md)). The sync and indexing code matches that claim: file scans find new, modified, deleted, and moved files; the batch indexer parses Markdown, normalizes frontmatter/permalinks, upserts entities, resolves relations, refreshes FTS rows, and optionally generates semantic embeddings ([src/basic_memory/sync/sync_service.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/sync/sync_service.py), [src/basic_memory/indexing/batch_indexer.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/indexing/batch_indexer.py)).

**The note format turns ordinary Markdown into a graph.** Frontmatter supplies title, type, tags, permalink, custom metadata, and optional schema attachments. Markdown list items with `[category]` become observations; explicit relation lines and wikilinks become graph edges; schemas map Picoschema fields back onto the same observation/relation syntax rather than inventing a separate authoring surface ([docs/NOTE-FORMAT.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/docs/NOTE-FORMAT.md), [src/basic_memory/markdown/entity_parser.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/markdown/entity_parser.py), [src/basic_memory/schema/validator.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/schema/validator.py)).

**Context efficiency is built around explicit retrieval budgets, not full-vault loading.** `search_notes` exposes text, title, permalink, metadata, tag, status, type, date, vector, and hybrid retrieval with pagination; `build_context` takes a `memory://` URI, depth, timeframe, page size, and related-result cap, then returns either structured JSON or compact Markdown ([src/basic_memory/mcp/tools/search.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools/search.py), [src/basic_memory/mcp/tools/build_context.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools/build_context.py), [src/basic_memory/services/context_service.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/services/context_service.py)). Volume is bounded by page size, graph depth, `max_related`, time filters, project routing, and hook output caps. Complexity is reduced by returning observations/relations alongside selected notes rather than dumping every connected file, but graph expansion can still produce dense context when relation neighborhoods are broad.

**Semantic retrieval is a derived index over graph rows, not a replacement for readable memory.** When semantic search is enabled, entities, observations, and relations are split into Markdown-aware chunks, embedded with FastEmbed or OpenAI, stored in SQLite `sqlite-vec` or Postgres/pgvector tables, and retrieved in vector or hybrid mode ([docs/semantic-search.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/docs/semantic-search.md), [src/basic_memory/repository/search_repository_base.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/repository/search_repository_base.py), [src/basic_memory/repository/sqlite_search_repository.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/repository/sqlite_search_repository.py), [src/basic_memory/repository/postgres_search_repository.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/repository/postgres_search_repository.py)). The vector layer is disposable and regenerated from search-index rows; source text remains Markdown and symbolic graph state.

**The MCP surface is agent-shaped.** Tools carry FastMCP annotations such as read-only, destructive, idempotent, and open-world hints; tool signatures accept aliases that LLMs are likely to use; error responses often return recovery instructions rather than raw exceptions ([src/basic_memory/mcp/tools/search.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools/search.py), [src/basic_memory/mcp/tools/write_note.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools/write_note.py), [src/basic_memory/mcp/tools/build_context.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools/build_context.py)). The architecture doc describes the tool -> typed client -> API -> service -> repository flow and per-project routing for local and cloud-backed projects ([docs/ARCHITECTURE.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/docs/ARCHITECTURE.md)).

**Schemas provide soft governance over memory shape.** Schema notes are themselves Basic Memory artifacts. `schema_validate`, `schema_infer`, and `schema_diff` expose validation, inference, and drift reports through MCP, while validation defaults to warnings rather than hard write rejection ([src/basic_memory/schema/validator.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/schema/validator.py), [src/basic_memory/mcp/tools/schema.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools/schema.py)). This gives Basic Memory stronger structure than an untyped note vault, but weaker authority than Commonplace's collection/type validation and review workflow.

**The Claude Code plugin closes a trace-to-brief loop.** The shipped plugin is not only documentation. Its `SessionStart` hook runs bounded structured Basic Memory queries and prints a session brief to Claude's context; its `PreCompact` hook reads Claude Code JSONL transcript payloads and writes extractive `type: session` checkpoint notes into Basic Memory; its output style instructs Claude to search before recall and capture material decisions as typed notes ([plugins/claude-code/hooks/session-start.sh](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/session-start.sh), [plugins/claude-code/hooks/pre-compact.sh](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/pre-compact.sh), [plugins/claude-code/output-styles/basic-memory.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/output-styles/basic-memory.md), [plugins/claude-code/hooks/hooks.json](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/hooks.json)). That makes Basic Memory both an explicit pull tool and a host-mediated push activation system when the plugin is installed.

## Artifact analysis

- **Storage substrate:** `files` — The central retained source is user-owned Markdown files under configured Basic Memory project directories, with SQLite/Postgres/search/vector state as derived projections.
- **Representational form:** `prose` `symbolic` `parametric` — Basic Memory combines prose Markdown, symbolic frontmatter/Picoschema/relations/tool schemas, and distributed-parametric embeddings for semantic retrieval.
- **Lineage:** `authored` `imported` `trace-extracted` — Notes can be authored by humans or agents, imported from conversation exports, or trace-extracted by Claude Code plugin hooks; indexes and vectors are derived projections.
- **Behavioral authority:** `knowledge` `instruction` `routing` `validation` `ranking` `learning` — Notes and imports advise as knowledge, output styles instruct, project/tool metadata routes, schemas validate, search/vector indexes rank, and trace/session capture feeds future read-back.

**Project Markdown notes.** Storage substrate: filesystem Markdown under configured project roots. Representational form: prose Markdown with symbolic frontmatter, observations, relations, wikilinks, tags, permalinks, and optional schema metadata. Lineage: authored by humans, agents, MCP tools, importers, shared skills, cloud sync, or plugin hooks; invalidated by file edits, file moves, permalink changes, or explicit delete/move/write operations. Behavioral authority: usually knowledge artifacts when read, searched, or used as evidence; stronger system-definition artifacts when notes are loaded as instructions, schemas, tasks, decisions, or session checkpoints.

**Materialized graph rows.** Storage substrate: SQLite by default and Postgres for cloud/server configurations, with ORM rows for projects, entities, observations, relations, and note content ([src/basic_memory/models/knowledge.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/models/knowledge.py), [src/basic_memory/config.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/config.py)). Representational form: symbolic relational state. Lineage: derived from Markdown parsing or API writes, with checksums, mtimes, project ids, external ids, and sync state. Behavioral authority: retrieval and routing authority because these rows decide what `read_note`, `search_notes`, `recent_activity`, `build_context`, relation resolution, and metadata filters can see.

**Search indexes and vector chunks.** Storage substrate: SQLite FTS5 or Postgres tsvector search indexes plus optional SQLite `sqlite-vec` or Postgres/pgvector vector tables ([src/basic_memory/models/search.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/models/search.py), [src/basic_memory/repository/search_repository_base.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/repository/search_repository_base.py)). Representational form: mixed symbolic index rows and parametric embedding vectors. Lineage: derived from current graph/search rows; vector chunks carry `source_hash`, `entity_fingerprint`, and `embedding_model`, so unchanged chunks skip re-embedding and model/dimension changes force regeneration. Behavioral authority: ranking and selection system-definition artifacts; they decide which memories enter agent context first.

**Schemas and validation reports.** Storage substrate: schema notes in the project plus transient validation/inference/drift outputs. Representational form: symbolic Picoschema with prose descriptions. Lineage: authored, seeded by plugin setup, inferred from existing notes, or edited by users; invalidated by schema edits or changing note practice. Behavioral authority: validation and routing advice. In the default warn mode, schemas do not block writes, but they make later structured recall more precise.

**MCP/API/CLI tool contracts.** Storage substrate: repository Python modules, FastMCP tool definitions, typed clients, Typer commands, and `server.json` package metadata ([server.json](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/server.json), [src/basic_memory/mcp/tools](https://github.com/basicmachines-co/basic-memory/tree/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/tools), [src/basic_memory/mcp/clients](https://github.com/basicmachines-co/basic-memory/tree/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/mcp/clients)). Representational form: symbolic API and tool schemas with prose descriptions. Lineage: authored package code. Behavioral authority: system-definition artifacts for agent access, because they constrain which actions are read-only or destructive, how projects route, how outputs are shaped, and what recovery instructions an agent sees after failures.

**Claude Code plugin hooks and settings.** Storage substrate: plugin shell scripts, `hooks.json`, output style Markdown, plugin schemas, and installed `.claude/settings*.json` configuration ([plugins/claude-code](https://github.com/basicmachines-co/basic-memory/tree/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code)). Representational form: mixed executable shell/Python, JSON hook config, and prose instructions. Lineage: authored plugin files copied or installed into a Claude Code environment; runtime session notes are trace-extracted from Claude Code transcript payloads. Behavioral authority: session-start context injection, pre-compaction memory capture, output-style instruction, and project-scoped recall routing.

**Conversation import artifacts.** Storage substrate: imported Markdown files under a destination folder. Representational form: prose transcripts with frontmatter such as `type: conversation`, created/modified timestamps, and permalinks. Lineage: imported from ChatGPT or Claude export JSON, preserving conversation turns more than distilling them ([src/basic_memory/importers/chatgpt_importer.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/importers/chatgpt_importer.py), [src/basic_memory/importers/claude_conversations_importer.py](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/src/basic_memory/importers/claude_conversations_importer.py)). Behavioral authority: source-trace knowledge artifacts until later searched, linked, summarized, or reflected into stronger notes.

**Promotion path.** Basic Memory has several partial promotion paths: Markdown -> parsed graph -> search/vector index -> retrieved context; raw conversation export -> conversation note -> later manual search/reflection; Claude Code transcript -> extractive session note -> next SessionStart brief; schema inference -> schema note -> validation/drift reports. The system does not implement a Commonplace-style review gate that promotes a claim from evidence to validated library artifact, but the plugin's typed session/decision/task scheme does promote selected trace or conversation outcomes into structured read-back surfaces.

## Comparison with Our System

| Dimension | Basic Memory | Commonplace |
|---|---|---|
| Primary purpose | Local/cloud Markdown memory graph for AI clients and humans | Git-native methodology KB with typed artifacts, validation, reviews, and indexes |
| Main substrate | User Markdown plus SQLite/Postgres/search/vector projections | Git-tracked Markdown collections, type specs, source snapshots, reports, generated indexes |
| Retrieval | MCP/CLI/API search, metadata filters, semantic/vector/hybrid retrieval, graph context expansion | `rg`, authored indexes, links, collection contracts, review reports, validation commands |
| Read-back | Explicit MCP pull plus Claude Code SessionStart push when plugin installed | Mostly explicit pull through search, links, skills, instructions, and review/validation workflows |
| Trace learning | Conversation importers and Claude Code pre-compaction session checkpoints | Deliberate source/review workflows; trace-derived learning is an analysis topic, not a default runtime loop |
| Governance | Picoschema warning validation, schema inference/diff, tool annotations, project routing | Collection contracts, type schemas, deterministic validation, semantic review gates, replacement archives |
| Adoption surface | PyPI/uvx, MCP server, cloud option, Obsidian-compatible files, host plugins/skills | Repo-local framework and installed `commonplace-*` commands for KB operators |

Basic Memory is closest to Commonplace in its preference for inspectable Markdown, explicit metadata, and agent-operable command surfaces. The major difference is product boundary. Basic Memory tries to be a general memory appliance for many AI hosts: it owns MCP packaging, cloud routing, semantic retrieval, note import, and host plugins. Commonplace is narrower and stricter: it is a methodology KB whose core value is artifact governance, type contracts, and reviewable design knowledge.

Basic Memory's search stack is more productized than Commonplace's. Metadata filters, `memory://` URLs, vector search, `build_context`, and MCP-friendly error guidance give agents a richer runtime API than raw `rg`. Commonplace's advantage is that its artifacts carry stronger collection-local semantics: a note, ADR, source snapshot, review, and instruction are not just note types in frontmatter but part of a maintained library with validation, replacement history, and curated indexes.

The Claude Code plugin is the most relevant activation contrast. Commonplace has many retained artifacts, but agents still mostly pull them by search or instructions. Basic Memory demonstrates a concrete hook path where retained memory reaches a future agent before the first prompt. That solves lookup omission for some project/session state, but it also raises the governance question Commonplace cares about: once memory is pushed into context, stale or low-quality notes have more chance to affect behavior.

**Read-back:** `both` — Core Basic Memory is pull through MCP/CLI/API search/read/context tools, while the shipped Claude Code plugin pushes project-scoped session context into Claude through `SessionStart` and writes future-readable session checkpoints through `PreCompact`.

### Borrowable Ideas

**Agent-facing tool annotations should be first-class.** Basic Memory uses MCP annotations and alias-tolerant schemas to tell agents which tools are read-only, destructive, idempotent, or bounded. A Commonplace analogue would expose validation, review, and note-operation commands with similarly explicit action metadata. Ready when Commonplace grows an MCP/tool surface.

**Make compact context assembly a real API.** `build_context(..., output_format="text")` is a useful middle layer between search hits and full file reads. Commonplace could borrow a command that takes a note/link/path and returns a bounded local context pack with primary artifact, selected outbound/inbound links, and metadata. Ready as a CLI design if there is a concrete consumer.

**Index observations and relations as first-class retrieval units.** Basic Memory's vector/search layer can surface observations and relations instead of only whole notes. Commonplace could use the same principle for extracting and searching claims, links, review findings, and type fields without loading entire files. Needs a derived-index design and invalidation plan.

**Treat host hooks as activation infrastructure.** The Claude Code SessionStart and PreCompact hooks show a practical push path: checkpoint before context loss, then brief the next session with structured queries. Commonplace could use a similar hook for active workshops or review runs, but only for bounded, high-confidence context. Needs a governance policy before automatic injection.

**Keep schema validation soft where users are capturing, hard where artifacts are authoritative.** Basic Memory's warning mode fits a personal memory graph. Commonplace should preserve stricter validation for library artifacts, but a softer workshop/capture lane could improve ergonomics.

**Do not borrow opaque authority from retrieval success.** Basic Memory makes notes easy to retrieve and inject. Commonplace should keep distinguishing knowledge artifacts from system-definition artifacts before they are loaded as instructions, validators, or gates.

## Trace-derived learning placement

- **Trace source:** `session-logs` — Claude Code JSONL transcripts and imported ChatGPT/Claude conversation exports are the trace sources described here.
- **Learning scope:** `per-project` `cross-task` — Captured sessions are mapped to configured Basic Memory projects and can brief later sessions or tasks in that project.
- **Learning timing:** `offline` `staged` — Conversation import is offline bulk ingestion, while PreCompact capture and later SessionStart recall form a staged lifecycle.
- **Distilled form:** `prose` `symbolic` — The implemented checkpoint output is Markdown prose with symbolic type, tags, observations, status, and project metadata.

**Trace source.** Basic Memory qualifies as trace-derived. The strongest implemented path is the Claude Code plugin's `PreCompact` hook: it receives a hook payload with `transcript_path`, parses Claude Code JSONL turns, filters meta/tool-result records, and extracts user/assistant prose before compaction ([plugins/claude-code/hooks/pre-compact.sh](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/pre-compact.sh)). Conversation importers provide a second, more archival trace source: ChatGPT and Claude export JSON can be converted into `type: conversation` Markdown files.

**Extraction.** The implemented `PreCompact` hook is extractive, not a learned summarizer. It selects the opening request and recent user messages, writes a `type: session` checkpoint with observations such as context and next step, and tags it as `session` and `auto-capture`. The plugin design document discusses a future summarized path, but the reviewed hook code still implements the fast extractive path ([plugins/claude-code/DESIGN.md](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/DESIGN.md), [plugins/claude-code/hooks/pre-compact.sh](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/pre-compact.sh)). The oracle is mostly deterministic filtering plus the host's compaction trigger; there is no implemented LLM judge or reviewer deciding whether a checkpoint is high quality.

**Scope and timing.** Scope is per Claude Code project mapped to a configured Basic Memory project. Timing is after useful context has accumulated but before compaction, then before the next session's first prompt through `SessionStart`. Imported ChatGPT/Claude conversations are offline bulk imports and remain raw-ish transcript notes until an agent or human reflects over them.

**Survey placement.** On the [trace-derived learning survey](../trace-derived-learning-techniques-in-related-systems.md), Basic Memory belongs in the lightweight session-checkpoint family rather than the stronger profile-learning or skill-patching families. It turns host traces into durable prose/symbolic checkpoint artifacts and then recalls them through structured filters. It strengthens the survey distinction between raw trace preservation and distilled behavior-shaping artifacts: imported conversations are mostly preserved traces, while plugin session notes are distilled enough to become next-session briefing material.

## Read-back placement

**Direction.** Both. Agents can explicitly pull memory through `search_notes`, `read_note`, `build_context`, `recent_activity`, and related MCP/CLI/API surfaces. With the Claude Code plugin installed, retained Basic Memory state is also pushed into Claude's context at `SessionStart` without the receiving agent first calling a tool.

**Read-back signal:** `identifier` — SessionStart push keys on cwd, project references, note type, status, and date windows rather than semantic inference over the first prompt.

**Read-back timing:** `pre-action` `post-action` — SessionStart briefs before the future Claude session acts, while PreCompact captures after accumulated work for future read-back.

**Faithfulness tested:** `no` — The review found no implemented ablation or post-action audit proving that the pushed session brief changes Claude behavior.

**Targeting and signal.** The plugin's push is `instance`-targeted at the project/session level. It reads `.claude/settings*.json` under the current `cwd`, resolves `primaryProject`, `secondaryProjects`, `teamProjects`, `recallTimeframe`, and placement conventions, then queries active tasks, open decisions, and recent sessions for the selected projects ([plugins/claude-code/hooks/session-start.sh](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/session-start.sh)). The signal is mainly `identifier`: cwd, project ref, note type, status, and date window. It is not semantic query inference over the user's first prompt.

**Timing relative to action.** SessionStart output is available before the future Claude session acts. PreCompact capture runs before context compaction but after the current session's work; it cannot change the action just taken, only future read-back.

**Selection, scope, and complexity.** Selection is bounded by structured filters, `--page-size 5`, a maximum of six shared projects, per-call timeouts, and Claude Code's documented 10,000-character hook-output cap enforced by the hook's formatting discipline ([plugins/claude-code/hooks/session-start.sh](https://github.com/basicmachines-co/basic-memory/blob/fc2ee07076eb397b09db7b2681e5213002df0d70/plugins/claude-code/hooks/session-start.sh)). Core pull tools also carry page size, depth, timeframe, `max_related`, `search_type`, and metadata filter knobs. Precision and context dilution are runtime properties; the code shows selection policy, not whether the selected brief always changes behavior correctly.

**Authority at consumption.** Search/read/build-context results are advisory knowledge artifacts unless the host treats them as instructions. SessionStart output has stronger prompt-context authority because Claude sees the brief before acting, and the output style has instruction-like authority when enabled. Neither path is a hard gate: Basic Memory does not enforce that Claude used a memory, nor does it block actions that ignore it.

**Faithfulness.** I found no implemented with/without ablation or post-action audit proving the session brief changes Claude behavior. The mechanism is structurally real; behavioral uptake is not verified from code.

**Other consumers.** Humans and agents also consume memory through Markdown files, Obsidian, CLI, the cloud app, MCP-compatible clients, shared `memory-*` skills, schema reports, import commands, and project/status tools. Those surfaces make the memory more inspectable, but they are not all activation paths.

## Curiosity Pass

**The README's "real knowledge graph" claim is grounded, but the graph is mostly Markdown-derived.** Observations, relations, unresolved links, recursive context expansion, and graph rows are implemented. It is not an autonomous fact-extraction graph like Graphiti; graph structure appears when notes use Basic Memory syntax or when imports/agents write in that form.

**The system has two different meanings of "memory".** Core Basic Memory is a durable graph and retrieval API. The Claude Code plugin is an activation bridge over that graph. Treating them as one system is fair because the plugin ships in the repo and is an official package, but architecture claims should keep the pull core and push integration separate.

**Semantic search is powerful but disposable.** The vector layer can rank observations and relations by meaning, but the source of truth remains Markdown plus graph rows. That is a good lineage property: embeddings can be rebuilt, while notes remain inspectable.

**Soft schemas are an adoption win and a governance limit.** Picoschema can make recall precise without blocking users. The same softness means schema conformance is not a quality guarantee unless a host or workflow adds stronger enforcement.

**The implemented pre-compaction capture is more conservative than the design doc.** The design discusses a summarized checkpoint path with enough timeout budget for an LLM pass. The script at this commit writes an extractive checkpoint. That is safer and cheaper, but weaker as trace distillation.

**Conversation importers preserve traces more than they learn from them.** Imported ChatGPT/Claude conversations become searchable Markdown. They qualify as trace-derived retained artifacts, but their value depends on later reflection, search, or manual transformation.

## What to Watch

- Whether the Claude Code `PreCompact` hook changes from extractive capture to LLM-summarized checkpoints. That would strengthen trace-derived distillation and require stronger provenance and quality controls.
- Whether SessionStart evolves from type/status/date filters to query- or prompt-inferred relevance. That would move push activation from project-scoped recall toward instance-semantic activation.
- Whether schema validation gains stricter modes in plugin or MCP workflows. That would move Basic Memory from soft capture conventions toward enforceable artifact governance.
- Whether semantic search becomes default in all environments despite dependency fallback. Retrieval behavior changes materially when vector/hybrid modes are available.
- Whether cloud/team sync adds review, conflict, or permission semantics visible in this repository. Shared push memory needs stronger authority boundaries than personal local memory.
- Whether Basic Memory measures hook brief faithfulness with ablations. Without that, context presence remains a structural claim, not evidence of behavioral use.
