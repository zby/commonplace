# sage

SAGE, from Dhillon Andrew Kannabhiran's `l33tdawg/sage` repository, is a Go-based local and multi-agent memory infrastructure for AI agents. It exposes MCP and REST tools for remembering, recalling, reflecting, task tracking, and agent-to-agent pipeline work; stores memory records behind a CometBFT/ABCI consensus path; uses confidence, decay, corroboration, validation votes, RBAC, and optional encryption; and ships Claude Code/Codex lifecycle hooks that can prefetch or write memory without relying only on the agent's voluntary tool use.

**Repository:** https://github.com/l33tdawg/sage

**Reviewed commit:** [8122a91bf561c9c5c99bea380871116c70785988](https://github.com/l33tdawg/sage/commit/8122a91bf561c9c5c99bea380871116c70785988)

**Last checked:** 2026-06-02

## Core Ideas

**Memory writes are governed transactions, not direct notes.** `sage_remember`, `sage_turn`, and `sage_reflect` ultimately submit memory content through signed REST calls to `/v1/memory/submit`; the REST handler builds a `TxTypeMemorySubmit`, stages embeddings and triples in a supplementary cache, broadcasts with `broadcast_tx_commit`, and attaches tags only after commit because tags are node-local metadata ([internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go), [api/rest/memory_handler.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/api/rest/memory_handler.go)). In the ABCI app, submit processing verifies agent identity, enforces domain write access, writes on-chain hash/status/classification keys, buffers off-chain content for Commit, and rejects terminal-ID resubmits after the v8.4 fork ([internal/abci/app.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/abci/app.go)).

**Consensus is used as memory governance.** The app-validator layer includes sentinel, dedup, quality, and consistency validators; the REST pre-validation endpoint returns 3-of-4 acceptance before a client submits, while committed/deprecated status is driven by validator votes and quorum in the ABCI state machine ([internal/appvalidator/validator.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/appvalidator/validator.go), [api/rest/memory_handler.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/api/rest/memory_handler.go), [internal/abci/app.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/abci/app.go)). This is the system's main trust story: a memory becomes behaviorally visible only after a committed status, with challenge/deprecation paths for invalidation.

**The central agent loop is recall plus trace-derived capture.** `sage_turn` takes a topic, recalls committed memories relevant to that topic, then stores the current turn observation as an observation memory when it is not low-value or duplicate-like ([internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go)). `sage_reflect` turns completed-task experience into `[Task Reflection]`, `[DO]`, and `[DON'T]` memories with different types and confidence defaults. SessionEnd hooks write session-lifecycle observations, while PreCompact/UserPrompt hooks nudge the agent to preserve fresh context before it disappears ([cmd/sage-gui/hook.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/hook.go), [cmd/sage-gui/mcp.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/mcp.go)).

**Context efficiency comes from bounded recall surfaces, but not from artifact compression.** Recall paths return top-k committed records, can apply domain/provider/min-confidence/status/tag filters, and compute current confidence at read time ([api/rest/memory_handler.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/api/rest/memory_handler.go)). MCP defaults are cached from dashboard settings and normally return five memories unless configured otherwise ([internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go)). This bounds volume, but the returned units are whole memory records; SAGE does not implement progressive disclosure inside a long memory, source-span reading, or note-level outline/read ladders.

**Read-back uses several ranking modes.** `sage_recall` chooses semantic vector search, hybrid BM25/vector search, or FTS5 text search depending on embedding mode and environment settings. The hybrid endpoint accepts a query and embedding, optionally expansion variants, then fuses results with weighted Reciprocal Rank Fusion; an optional reranker can refine the candidate pool in SQLite mode ([internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go), [api/rest/memory_handler.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/api/rest/memory_handler.go), [internal/store/hybrid.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/store/hybrid.go), [internal/store/sqlite.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/store/sqlite.go)). Retrieval quality is not statically verifiable from code, but the selection and fusion policy is visible.

**Authority is pushed into the integration layer.** The MCP initialize response and bundled `sage-memory` skill instruct agents to call `sage_inception` before responding and `sage_turn` every turn ([internal/mcp/server.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/server.go), [sage-memory/SKILL.md](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/sage-memory/SKILL.md)). The MCP server also tracks non-SAGE tool calls, emits escalating nudges, and can block non-SAGE tool calls after a threshold until `sage_turn` is called. Claude Code hooks add an engineered host-level surface: SessionStart can inject recent committed memories, UserPromptSubmit can nudge turn capture, and SessionEnd can direct-write a lifecycle observation ([cmd/sage-gui/hook.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/hook.go), [cmd/sage-gui/mcp.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/mcp.go)).

## Artifact analysis

- **Storage substrate:** `kv` — On-chain BadgerDB keys for content hash, status, classification, votes, domain ownership, and related consensus state; full content, embeddings, triples, corroborations, and query metadata live in SQLite or PostgreSQL as off-chain projections ([internal/abci/app.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/abci/app.go), [internal/store/store.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/store/store.go))
- **Representational form:** `mixed` — Mixed symbolic and prose, with typed fields for memory id, type, domain, confidence, status, parent hash, task status, classification, and optional vectors/triples

**Consensus memory record.** Storage substrate: on-chain BadgerDB keys for content hash, status, classification, votes, domain ownership, and related consensus state; full content, embeddings, triples, corroborations, and query metadata live in SQLite or PostgreSQL as off-chain projections ([internal/abci/app.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/abci/app.go), [internal/store/store.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/store/store.go)). Representational form: mixed symbolic and prose, with typed fields for memory id, type, domain, confidence, status, parent hash, task status, classification, and optional vectors/triples. Lineage: authored or trace-derived by agents through MCP/REST tools, then consensus-ordered and mirrored; embeddings and triples are derived side data staged before consensus commit. Behavioral authority: committed records are knowledge artifacts when recalled as context, but their status, classification, confidence, and query rank have system-definition authority over what the agent can see and how strongly it is presented.

**Validation votes and PoE validator statistics.** Storage substrate: BadgerDB vote keys and validator statistics, plus off-chain vote/score tables in store interfaces. Representational form: symbolic accept/reject/abstain votes, weights, verdict statistics, epoch scores, and per-domain stats. Lineage: derived from validator decisions over submitted memories and later terminal verdicts; v8.3/v8.4 code credits verdict-correctness and per-domain accuracy only on the first proposed-to-terminal transition ([internal/abci/app.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/abci/app.go), [internal/poe](https://github.com/l33tdawg/sage/tree/8122a91bf561c9c5c99bea380871116c70785988/internal/poe)). Behavioral authority: system-definition artifacts for validation and ranking, because they decide whether a memory commits and how validator weight affects future quorum decisions.

**MCP tool definitions and server instructions.** Storage substrate: Go MCP server code and the packaged `sage-memory/SKILL.md`. Representational form: symbolic tool schemas plus prose instructions. Lineage: authored system-definition material shipped with the binary and skill. Behavioral authority: strong system-definition authority because tool descriptions, initialize instructions, auto-inception, turn nudges, and blocking behavior instruct or constrain the agent's future behavior ([internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go), [internal/mcp/server.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/server.go), [sage-memory/SKILL.md](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/sage-memory/SKILL.md)).

**Trace-derived observations and reflections.** Storage substrate: the same consensus memory store as ordinary memories, plus hook-generated session-lifecycle records. Representational form: prose observations, `[Task Reflection]` summaries, `[DO]` facts, `[DON'T]` observations, and task records with symbolic type/status fields. Lineage: derived from conversation turns, task outcomes, and session lifecycle events by the agent or hook layer; SAGE stores the distilled record, not the raw transcript by default. Behavioral authority: knowledge-artifact authority when recalled as context; stronger system-definition authority when the memory text is phrased as a rule, when boot instructions tell the agent to follow it, or when `sage_turn`/hooks inject it before action.

**Search indexes, embeddings, hybrid fusion, and reranker state.** Storage substrate: SQLite/PostgreSQL memory stores, embedding columns, FTS indexes, optional encrypted vault state, and optional reranker configuration. Representational form: mixed symbolic query options, lexical indexes, distributed-parametric embeddings, and ranking code. Lineage: derived from memory content at write time and query parameters at read time; encrypted-vault mode disables text search and routes to semantic search ([internal/store/sqlite.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/store/sqlite.go), [internal/store/hybrid.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/store/hybrid.go), [internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go)). Behavioral authority: ranking and selection authority over which memories enter context. Effective precision, recall, and interference are runtime qualities, not verified from static code.

**RBAC, classification, domains, and identity.** Storage substrate: BadgerDB domain/access/classification keys, agent identity stores, REST auth middleware, and optional TLS/certificate material. Representational form: symbolic access grants, clearance levels, Ed25519 signatures, domain ownership, org membership, and classification bytes. Lineage: authored by agents/admins or auto-created on first domain submit; query paths re-check access and can report filter application ([api/rest/memory_handler.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/api/rest/memory_handler.go), [api/rest/middleware/auth.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/api/rest/middleware/auth.go), [internal/abci/app.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/abci/app.go)). Behavioral authority: enforcement authority over read/write visibility and multi-agent memory boundaries.

**Lifecycle hooks and memory mode flags.** Storage substrate: generated `.claude/hooks/*.sh`, `.claude/settings.json`, `~/.sage/memory_mode`, and `sage-gui hook` subcommands. Representational form: shell scripts, JSON hook configuration, symbolic mode flags, and prose context/nudge text. Lineage: authored templates installed or self-healed by `sage-gui mcp install` and MCP startup. Behavioral authority: system-definition authority for push activation and write scheduling: SessionStart injects recent committed memories or a boot nudge, SessionEnd writes a lifecycle observation, PreCompact prompts reflection, and UserPromptSubmit prompts early `sage_turn` ([cmd/sage-gui/mcp.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/mcp.go), [cmd/sage-gui/hook.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/hook.go), [docs/HOOKS.md](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/docs/HOOKS.md)).

The promotion path is explicit but mostly status-based: proposed memory can become committed or deprecated by quorum; committed memory can later be challenged/deprecated; corroboration can strengthen confidence; tasks can move through task statuses. There is no implemented path that promotes a prose observation into a separate validator, schema, command, or tested instruction artifact. SAGE governs acceptance and activation of memory records more than it transforms them into higher-verifiability artifacts.

## Comparison with Our System

| Dimension | SAGE | Commonplace |
|---|---|---|
| Primary substrate | Consensus-backed service state: BadgerDB, SQLite/PostgreSQL, embeddings, MCP/REST, hooks | Git-tracked Markdown collections, type specs, validation, reviews, indexes, reports |
| Main memory unit | Typed memory record with status, domain, confidence, classification, embedding | Typed knowledge artifacts: notes, instructions, ADRs, reviews, sources, generated indexes |
| Write governance | App validators, BFT quorum, challenge/deprecation, RBAC | Collection contracts, schema validation, semantic review, git history, human/agent edits |
| Trace learning | Agent summarizes turns/tasks/session events into durable memory records | Trace-derived learning is explicit but usually mediated by workshop artifacts and reviews |
| Read-back | MCP pull tools plus SessionStart/UserPrompt/PreCompact hooks and turn-discipline enforcement | Mostly deliberate `rg`/index/link/skill navigation; less host-level push activation |
| Context strategy | Top-k recall over committed records; hybrid/vector/FTS ranking; recent-memory warm prefetch | Progressive search and authored navigation over smaller, typed artifacts |

SAGE is much stronger than Commonplace at runtime integration. It has concrete answers for first-session boot, per-turn memory discipline, session-start prefetch, session-end bookends, compaction reminders, MCP tool schemas, local dashboards, and multi-agent access control. Commonplace has richer artifact contracts and review practices, but a weaker default answer to "what happens automatically when an agent starts, acts, compacts, or exits?"

Commonplace is stronger at artifact interpretability and source-grounded lifecycle. A SAGE memory stores content, type, confidence, domain, and status, but does not require citations, source spans, review rationale, replacement archives, outbound link labels, or collection-specific quality contracts. Consensus can reject duplicates, short/noisy entries, and low-confidence facts, but it does not establish that a memory is well-evidenced in the Commonplace sense.

The deepest design difference is the trust boundary. SAGE treats consensus and access control as the governance layer around a memory service. Commonplace treats artifact type, source lineage, semantic review, and explicit authority as the governance layer around a knowledge library. SAGE is better at making a memory operationally available to agents; Commonplace is better at explaining what a retained artifact means and why a later agent should trust it.

**Read-back:** `both` — Agents can pull memories through `sage_recall`, `sage_turn`, `sage_list`, backlog, timeline, and REST search/query/hybrid endpoints; hooks and MCP server behavior can push recent memories, boot instructions, recall mandates, turn reminders, and pre-compaction reflection prompts into the agent's context or tool loop

### Borrowable Ideas

**Lifecycle hooks as memory activation infrastructure.** Ready as a design target, but not as a broad always-on import. Commonplace could use narrow start/end/compact hooks for workshop memory capture and validation reminders, with explicit mode flags and opt-out behavior.

**Turn discipline with a hard threshold.** Worth borrowing for workflows where lost state is expensive. SAGE's MCP server escalates from reminder to block after several non-SAGE tool calls. Commonplace could adapt this for review or migration sessions, but should avoid generic blocking until the required capture content is precisely specified.

**Separate memory mode from memory capability.** Ready now. SAGE's `full`, `bookend`, and `on-demand` modes separate available tools from activation policy. Commonplace could expose similar modes for future automatic context injection or trace capture.

**Consensus-style validation as a metaphor, not the mechanism.** The useful Commonplace analogue is not BFT replication; it is independent validators with explicit votes and reasons before high-authority memory commits. Ready as a review-bundle framing, but not as distributed infrastructure.

**Confidence decay and corroboration for operational memory.** Needs a use case. Decay is helpful for operational observations and open tasks, but Commonplace methodology claims should usually expire by source invalidation, contradiction, or review status rather than time alone.

**Do not borrow governed-service opacity.** SAGE's service architecture is powerful, but Commonplace should preserve inspectable file artifacts as the durable unit. A future service index or hook layer should remain downstream of source-readable artifacts rather than replacing them.

## Trace-derived learning placement

**Trace source.** SAGE qualifies as trace-derived learning. The main traces are conversation turns summarized by the agent in `sage_turn`, task outcomes summarized in `sage_reflect`, session lifecycle events written by SessionEnd hooks, and pre-compaction state that hooks ask the agent to distill before context is lost ([internal/mcp/tools.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/tools.go), [cmd/sage-gui/hook.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/hook.go), [cmd/sage-gui/mcp.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/mcp.go)). The code does not automatically ingest full raw chat transcripts by default; the ordinary path stores agent-authored distillates.

**Extraction.** Extraction is mostly agent-mediated rather than parser-mediated. `sage_turn` receives the agent's observation text and stores it as an observation after low-value and duplicate checks. `sage_reflect` maps task summary, dos, and don'ts into separate memory records with fixed prefixes and confidence values. SessionEnd direct-write records a lifecycle event but explicitly says per-turn content is captured by the agent's `sage_turn` calls. Validators then judge quality, consistency, and duplicate status before or during the commit path.

**Scope and timing.** Scope is local-node or multi-agent-network memory, organized by dynamic domains and filtered by provider, domain, access grants, classification, status, confidence, and tags. Timing spans online per-turn capture, end-of-task reflection, pre-compaction prompting, and session-end bookends. The learning artifact becomes available only after consensus commit.

**Survey placement.** On the trace-derived survey axes, SAGE belongs in the trace-to-operational-memory family: raw interaction is not the durable object; agent-written observations, reflections, tasks, and lifecycle records are. It strengthens the survey claim that activation and governance are separate from extraction: SAGE has strong activation/governance around relatively simple prose distillates.

**Curation policy.** The curation oracle is mixed. The agent decides what to summarize, local heuristics suppress low-value or duplicate observations, app validators reject obvious noise or inconsistent records, and consensus decides status. This is stronger than raw transcript accumulation, but weaker than a source-grounded review workflow because the stored observation usually lacks source spans or replayable evidence.

## Read-back placement

**Direction.** Both. Pull paths are `sage_recall`, `sage_turn`, `sage_list`, REST `/query`, `/search`, `/hybrid`, task/backlog, and timeline. Push paths are SessionStart hook injection, UserPromptSubmit/PreCompact nudges, MCP initialize instructions, auto-inception messages, turn nudges, and hard turn checkpoints.

**Trigger and relevance signal.** The strongest relevance-gated pull path is topic/query recall through vector, FTS5, or hybrid rank fusion. The push path is less semantic: SessionStart fetches the newest committed memories with a fixed limit, while UserPromptSubmit and PreCompact are event-keyed nudges rather than content relevance matches. MCP turn discipline is call-count/time based rather than memory relevance based ([internal/mcp/server.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/mcp/server.go), [cmd/sage-gui/hook.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/cmd/sage-gui/hook.go)).

**Timing relative to action.** SessionStart and MCP initialize instructions fire before the agent answers, so they can shape the next move. UserPromptSubmit nudges fire at prompt arrival. PreCompact fires before context compression, so it can only preserve memory for future action, not help the immediate one unless the agent reflects before continuing. SessionEnd writes after action and helps later sessions.

**Selection, scope, and complexity.** Query recall uses top-k and min-confidence settings, provider/domain/status filters, RBAC/classification gates, and current-confidence recomputation. SessionStart prefetch is capped to ten recent committed memories and truncates each line to 200 characters. Complexity is bounded by count and truncation, not by structural decomposition of long artifacts. Precision, recall, and actual context dilution are runtime properties not proven by the code.

**Authority at consumption.** Recalled memories are presented as advisory context, but surrounding instructions often give them stronger force: the MCP server says only committed memories are returned, instructs use every turn, and can block unrelated tools until `sage_turn` records state. Hook-injected memory therefore sits between knowledge artifact and system-definition artifact: the records advise, while the activation and turn-discipline wrapper instructs and enforces.

**Faithfulness.** The code proves retrieval and injection paths exist; it does not prove that the model actually uses a retrieved memory. The repository includes benchmark directories and result files, but this review did not find a code-level with/without hook faithfulness test for whether injected SessionStart or `sage_turn` memories change downstream behavior.

**Other consumers.** Humans consume CEREBRUM dashboard views, memory lists, settings, import suggestions, and governance screens. Validators consume memory proposals. REST/MCP clients consume tool schemas and returned memories. The chain and stores consume votes, scores, access grants, and status updates as governance state.

## Curiosity Pass

**The phrase "consensus-validated memory" is accurate operationally but easy to overread semantically.** Consensus can prove that configured validators accepted a memory and that replicas agree on status. It does not prove the memory is true, source-grounded, or useful in future context. The quality validators are intentionally lightweight: length/noise checks, duplicate checks, confidence/domain consistency, and sentinel acceptance.

**SAGE stores distilled turn records, not raw experience by default.** That is a good privacy and context-efficiency choice, but it changes the review posture. Many errors will be in the agent's summary before SAGE sees the memory. Without raw trace pointers, later agents cannot audit what was omitted or distorted.

**SessionStart push is recent-first, not relevance-first.** The direct-write hook usefully gets some memory into context before the agent acts, but it surfaces newest committed records rather than memories matched to the incoming user prompt. The agent is told to use targeted recall after that.

**The governance surface is split across chain and node-local state.** Status, classification, votes, and domains are consensus-sensitive; tags are explicitly node-local; embeddings and triples are supplementary off-chain data staged by the receiving node. Reviewers should not treat every visible field as equally replicated or authoritative.

**Open tasks not decaying is a small but important policy exception.** Confidence decay normally reduces stale memory influence, but open task records keep their initial confidence while planned or in progress ([internal/memory/confidence.go](https://github.com/l33tdawg/sage/blob/8122a91bf561c9c5c99bea380871116c70785988/internal/memory/confidence.go)). That is sensible for backlog continuity, but it means task cleanup becomes a governance responsibility.

## What to Watch

- Whether future hook work adds batched `PostToolUse` direct-write capture; that would shift SAGE from agent-mediated turn summaries toward richer automatic trace capture.
- Whether SessionStart or prompt-time hooks gain semantic relevance matching against the current prompt; that would strengthen the `push-activation` story beyond recent-memory warm prefetch.
- Whether memory records gain source pointers, raw trace ids, or replayable evidence; that would make trace-derived observations more reviewable and closer to Commonplace's lineage standard.
- Whether validation evolves beyond lightweight heuristics into source-aware or task-effect-aware review; that would make consensus status a stronger trust signal.
- Whether node-local tags, embeddings, triples, and encrypted-vault behavior receive clearer lineage and replication contracts; that matters for multi-agent deployments that treat query results as governed memory.

## Bottom Line

SAGE is a serious runtime memory system: it wires persistent memory into MCP, REST, consensus, hooks, identity, RBAC, confidence decay, and agent turn discipline. Its strongest lesson for Commonplace is activation infrastructure, especially lifecycle hooks and mode-controlled memory policy. Its main limitation for Commonplace's purposes is that the durable memory artifact is usually a short agent-authored record with weak source lineage. SAGE governs whether memory is accepted and activated; Commonplace still has the stronger model for explaining and reviewing what a knowledge artifact is allowed to mean.
