# xMemory

xMemory is research code for the paper "Beyond RAG for Agent Memory: Retrieval by Decoupling and Aggregation" by Zhanghao Hu, Qinglin Zhu, Hanqi Yan, Yulan He, and Lin Gui. The repository is a Python/Hugging Face evaluation harness for building agent-memory indexes over dialogue datasets, especially LoCoMo and PerLTQA. Its core pipeline converts dialogue streams into episodes, extracts persistent semantic facts, groups those facts into theme summaries, and uses the resulting hierarchy to retrieve a compact context for memory QA.

**Repository:** https://github.com/HU-xiaobai/xMemory
**Paper:** https://arxiv.org/abs/2602.02007
**Status:** Research code, latest inspected commit `375ae1495095aa14a39eb169f83737f4779391c6` from 2026-02-25; no package manifest beyond `environment.yml`.

## Core Ideas

**Agent memory as a bounded dialogue stream rather than a generic RAG corpus.** The paper and README argue that agent memory has different structure from normal RAG: memories come from a coherent dialogue stream, contain many correlated or duplicate spans, and often require temporally linked prerequisites. The implementation commits to that framing by building per-user memories from ordered dataset sessions in `evaluation/locomo/add.py`, preserving timestamps and original messages, and storing user-scoped artifacts under the configured storage path.

**Three-stage memory construction: episodes -> semantic facts -> themes.** `MemorySystem.add_messages()` buffers messages and uses an LLM boundary detector to decide when to end an episode. `EpisodeGenerator` then asks an LLM to produce a titled third-person episode narrative with an ISO timestamp, while preserving the raw `original_messages`. `SemanticGenerator` extracts long-term semantic memories from episodes. After semantic generation completes, `xMemory.fetch_incremental_batches()`, `update_themes()`, and `update_hierarchy_graph()` promote the new batch into a theme layer and a GEXF hierarchy graph.

**Prediction-correction semantic extraction is the learning mechanism.** When no semantic memories exist, `PredictionCorrectionEngine` directly extracts high-value persistent facts from the new episode. Once prior semantic memories exist, it retrieves relevant statements with Chroma vector search, uses them to predict the episode from its title, then asks an LLM to compare the original messages against the prediction and extract facts missing or misrepresented in the prediction. The "free energy" framing is mostly vocabulary; the implemented mechanism is LLM prediction plus gap extraction.

**Theme maintenance combines vector thresholds, LLM summaries, and a sparsity-semantics score.** `ThemeManager` attaches a new semantic memory to the nearest theme centroid if similarity passes `DEFAULT_THEME_ATTACH_THRESHOLD = 0.62`, with a lenient floor at 0.52 when the target theme is not full. Themes above `MAX_THEME_SIZE = 12`, or themes with low intra-similarity, are split by local similarity clustering or forced PCA partitioning. Candidate split and merge choices are scored with `f = SparsityScore + SemScore`, where sparsity rewards balanced theme sizes and semantic score combines cohesion with a kNN neighbor term. The final theme summaries are generated by an LLM over member semantic facts.

**Retrieval is top-down over themes, semantics, episodes, then raw messages.** The baseline search path still supports BM25 + Chroma vector retrieval with reciprocal rank fusion. The distinctive `adaptive_hier` strategy in `evaluation/locomo/xMemory_search_framework.py` first embeds the question, selects the top semantic pool, maps candidate semantics to themes, chooses representative themes with greedy coverage over the theme kNN graph, chooses representative semantic facts with the same coverage-plus-relevance method, induces candidate episodes from the selected semantic facts plus top vector episodes, then estimates answer entropy with the local HF model to decide which episodes to include and which raw messages to expand.

**Storage is files as source artifacts plus derived indexes.** Episodes and semantic memories are stored as per-user JSONL files. Themes are stored as JSONL plus `.npy` vector arrays and ID lists. Semantic kNN is stored as per-user JSON. The hierarchy graph is stored as GEXF via NetworkX. ChromaDB holds vector indexes; BM25 indexes are built in memory. This is not an opaque database-only memory system, but the human-facing artifacts are still machine-shaped records rather than authored notes.

**The hierarchy graph is mostly an export/inspection artifact.** `HierarchicalMemoryGraph` defines message, episode, semantic, and theme levels with `belongs_to`, `supports`, and `abstracted_by` edges. In the inspected code, `update_hierarchy_graph()` passes `new_messages = []`, so the message level is not populated through that path, and adaptive retrieval loads JSONL/theme/kNN files rather than traversing the GEXF graph. The graph is useful for visualization (`evaluation/gexf_view.py`), but it is not the primary retrieval engine.

## Comparison with Our System

| Dimension | xMemory | Commonplace |
|---|---|---|
| Primary use case | Dialogue-memory QA over benchmark conversations | Agent-operated KB methodology and shipped-system documentation |
| Knowledge unit | Episode JSONL record, semantic fact string, theme summary | Typed markdown note with frontmatter, prose, and explicit links |
| Storage | JSONL artifacts plus Chroma/BM25/theme vectors/GEXF | Markdown files in git, with generated indexes and scoped operational SQLite |
| Learning loop | LLM boundary detection -> LLM episode summary -> LLM fact extraction -> vector/LLM theme grouping | Human+agent writing, linking, validation, review, and maturation |
| Retrieval | Top-down theme/semantic/episode selection plus entropy-gated raw expansion | Agent navigation through descriptions, links, indexes, and search |
| Governance | Duplicate check by embedding similarity; no durable review or validation layer | Type system, structural validation, semantic gates, explicit status fields |
| Inspectability | Machine-readable JSONL and graph exports | Human-authored, browsable, diffable notes |

**xMemory is strongest as a retrieval architecture, not as a knowledge medium.** Its real contribution is the resolution ladder: high-level themes orient retrieval, semantic facts carry compact content, episodes provide grounded summaries, and raw messages are expanded only when useful. That directly matches the idea that a KB should support moving between abstraction levels. But xMemory's artifacts are not written to be read, revised, or linked as arguments. They are retrieval substrates for an answer-generation loop.

**Commonplace keeps the curation oracle explicit where xMemory automates it.** xMemory asks LLMs to decide boundaries, summarize episodes, extract persistent facts, summarize themes, and estimate whether episodes reduce answer uncertainty. That is appropriate for high-volume benchmark dialogue memory. Commonplace deliberately keeps durable note promotion human/agent-curated because [automating KB learning is an open problem](../automating-kb-learning-is-an-open-problem.md): extraction is easy compared with deciding what deserves durable trust.

**The storage model is a useful middle case for files-first thinking.** xMemory stores its durable records as files, but the files are not the interface. They are build artifacts loaded by the retrieval script and indexed by Chroma/BM25. This fits the derived-index boundary in [files-not-database](../files-not-database.md): vector and graph operations are reasonable derived layers when the access pattern needs them. It does not challenge our markdown-as-source model for authored methodology knowledge, because xMemory's source data is a conversation trace, not a curated library.

**The hierarchy is derived, while ours is authored.** xMemory's theme graph is generated from semantic fact embeddings and LLM summaries. Commonplace's hierarchy comes from directories, indexes, link semantics, and frontmatter descriptions chosen to support future reading. The derived hierarchy is cheaper at scale and more objective for benchmark retrieval, but it does not carry the editorial context phrases and relationship semantics that make a library navigable as reasoning.

## Borrowable Ideas

**1. A four-resolution retrieval ladder (ready as a design reference).** Theme -> semantic fact -> episode -> raw trace is a clean pattern for context budgeting. A commonplace analogue would be index orientation -> note summary/description -> note body -> source snapshot/log. The important part is not xMemory's exact data model; it is making expansion conditional on the next resolution level's expected value.

**2. Coverage-plus-relevance representative selection (needs embeddings first).** `_pick_representatives()` combines query similarity with graph coverage, so the system selects diverse representatives rather than the nearest neighbors only. If commonplace ever gains a derived semantic graph over notes or source snapshots, this is a better default than raw top-k for orientation packets.

**3. Entropy-gated raw expansion (needs a narrow use case).** xMemory estimates answer entropy before and after adding an episode, then expands raw messages only when the candidate appears to reduce uncertainty. This is expensive but conceptually strong for long trace corpora. It would be overkill for normal note navigation, but useful for a future trace-analysis workshop where raw logs dwarf the distilled artifacts.

**4. Keep trace layers separately addressable (ready as a workshop convention).** xMemory's separate `episodes/`, `semantic/`, `themes/`, and `semantic_knn/` outputs make the promotion path inspectable even if the artifacts are not polished notes. A workshop that mines agent traces could copy this layering: raw run, episode/segment summary, extracted candidate lessons, clustered theme.

**5. Split/merge scoring as a reference, not a direct borrow.** The sparsity-plus-cohesion objective is a concrete way to manage derived clusters. It is useful if we build automated source clustering, but it should not replace authored indexes: the score can say whether vectors form balanced cohesive clusters, not whether the resulting grouping explains the domain.

## Curiosity Pass

**What does "decoupling" actually transform?** The implemented decoupling is not a learned latent factor model. It is a sequence of LLM and embedding operations: summarize raw messages into episodes, extract fact strings, embed facts, cluster facts into theme summaries, then retrieve across those layers. The transformation is real, but it is symbolic/semantic distillation plus vector clustering, not a deeper decomposition of memory into independent causal components.

**The sparsity-semantics objective is local maintenance, not a global retrieval optimizer.** The score appears in split/merge evaluation for themes. It helps choose between candidate clusterings and merges, but the system is still governed by hand-tuned thresholds (`0.62`, `0.52`, `12`, `0.66`, `0.78`) and LLM summaries. That is a valid engineering design, but the paper-level phrase can sound more mathematically central than the code makes it.

**The entropy gate is lenient.** In `adaptive_hier`, `ig = H_before - H_after`, but the selection thresholds are negative (`tau_ep_stop = -0.4`, `tau_ep = -0.45`). That means an episode can be selected even when it slightly increases estimated answer uncertainty. In practice this makes the gate an early-stop and severe-harm filter rather than a strict positive information-gain requirement. The raw-message expansion threshold is positive (`0.25`), so expansion is stricter than episode inclusion.

**Duplicate prevention is weak by default.** `semantic_similarity_threshold` defaults to `1`, and duplicate checking skips only when embedding similarity is greater than that threshold. Since cosine similarity rarely exceeds 1, this mostly disables semantic duplicate filtering unless config overrides it. The theme layer can merge clusters later, but semantic memories themselves remain append-oriented.

**The benchmark target is conversational memory, not full agent competence.** xMemory tests whether a system can answer questions from long dialogue streams. That is a legitimate memory benchmark, but it is narrower than the commonplace concern that memory must improve action capacity: classification, planning, writing, debugging, and maintenance. The review should treat xMemory as strong evidence about context-efficient dialogue recall, not as a complete agent-learning architecture.

## Trace-derived learning placement

xMemory belongs in the trace-derived survey queue as a **conversation-trace artifact-learning system**.

**Axis 1 - ingestion pattern.** It ingests ordered conversation transcripts from LoCoMo/PerLTQA-style datasets through `evaluation/locomo/add.py`. The service does not merely index raw messages: it segments them into episodes, extracts semantic facts, and builds theme summaries.

**Axis 2 - promotion target.** It promotes traces into symbolic artifacts: episode summaries, semantic fact strings, theme summaries, semantic kNN adjacency, and a GEXF hierarchy export. No weight updates. The learned artifacts are mostly append-only and retrieval-facing; only the theme layer has active split/merge maintenance.

**What it adds to the survey.** Compared with REM, xMemory has a lighter storage stack and a more explicit resolution ladder. Compared with Hindsight, it has less graph semantics and governance but a clearer top-down retrieval budget. Compared with OpenViking, it derives hierarchy from conversation-memory facts rather than file/directory structure.

## What to Watch

- Whether future code releases package the system as a library or keep it as benchmark research code.
- Whether ablations isolate the marginal gain from the theme hierarchy versus LLM extraction, vector retrieval, and entropy-gated episode expansion.
- Whether the information-gain thresholds become calibrated to positive uncertainty reduction rather than permissive negative thresholds.
- Whether semantic memories gain lifecycle management: deduplication below exact similarity, contradiction handling, revision, or confidence decay.
- Whether the GEXF hierarchy becomes part of retrieval or remains a visualization/export layer.
- Whether evaluation expands beyond LoCoMo/PerLTQA QA into agent tasks where memory must improve action capacity rather than answer recall.
