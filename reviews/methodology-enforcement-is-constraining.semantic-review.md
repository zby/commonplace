=== SEMANTIC REVIEW: methodology-enforcement-is-constraining.md ===

Claims identified: 14

1. [Table] The enforcement layers form a six-tier gradient: ad hoc prompt, instruction, skill, hook (warn), hook (block), script — ordered by reliability from lowest to highest.
2. [Opening paragraph] The enforcement layers parallel the codification verifiability gradient.
3. [Opening paragraph] Each layer trades flexibility for reliability by reducing two things: semantic underspecification and execution indeterminism.
4. [Ad hoc prompt paragraph] Ad hoc prompts sit below instructions because they add a third source of unreliability: the prompt itself is ephemeral.
5. [Instructions paragraph] Instructions have the lowest *persistent* reliability because both phenomena compound.
6. [Key insight paragraph] The three-tier model (instruction -> skill -> hook) that arscontexta uses oversimplifies — the real picture is a gradient.
7. [Maturation trajectory] New best practices should start as underspecified natural-language guidance and constrain toward deterministic enforcement as they prove out.
8. [When to move down] The strongest signal for automation is when the agent consistently proposes the same correct next step.
9. [Not everything] Operations requiring semantic judgment belong permanently at the skill level — their oracle strength is too low to support deterministic verification.
10. [Not everything] Attempting to automate judgment produces confident systematic errors.
11. [Active observation] The context engineering study found that 50% of AGENTS.md files were never changed after creation.
12. [Active observation] Among the files that do evolve, additions (78 commits) and modifications (59) vastly outnumber removals (23) and section deletions (2).
13. [Active observation] Pruning is a discipline, not an emergent behavior.
14. [Maturation trajectory parallels] Document type maturation follows the same gradual-typing pattern as methodology maturation.

WARN:
- [Completeness] The six-tier gradient table presents "ad hoc prompt" and "instruction" as having different trigger reliability ("indeterministic (caller writes one)" vs "indeterministic (LLM remembers)"), but both are indeterministic triggers — the distinction is persistence, not trigger mechanism. A boundary case exposes the gap: a persistent instruction that lives in a file the agent does not always load (e.g., a `kb/instructions/` file that requires manual invocation or search to find). This has the persistence of an instruction but the trigger unreliability closer to an ad hoc prompt. The gradient collapses the trigger dimension into a single reliability axis, but persistence and trigger mechanism are separable properties. The note partially acknowledges this for hooks (deterministic trigger, indeterministic response) but does not apply the same decomposition to the instruction/ad-hoc boundary.
- [Completeness] The gradient omits a plausible enforcement layer between "skill" and "hook (warn)": **automated skill invocation** (e.g., a CI step that runs `/validate` on every commit). This would have a deterministic trigger (CI event) and an underspecified + indeterministic response (LLM interprets the skill), placing it squarely between manual skill invocation and hook warnings. The note's own ADR-001 example (sync_topic_links.py) jumped straight from skill to script, but many operations could sit at this intermediate level without being fully scripted. The framework's six tiers may not be exhaustive.
- [Grounding alignment] The note claims the enforcement layers "parallel the codification verifiability gradient" from deploy-time-learning-the-missing-middle.md. The source's gradient is: restructured prompts -> structured output schemas -> prompt tests/evals -> deterministic modules. The note's gradient is: ad hoc prompt -> instruction -> skill -> hook (warn) -> hook (block) -> script. These are not parallel structures — the source's gradient is about *verifiability of artifacts* (how cheaply you can check correctness), while the note's gradient is about *enforcement reliability* (how consistently a practice fires and executes). The note uses "parallel" but the mapping is loose: instructions do not correspond to restructured prompts in any structural way; skills do not correspond to structured output schemas. The note's gradient is better described as an *application of the constraining concept* to methodology, not a parallel to the specific verifiability gradient. The link to constraining.md is accurate; the link to deploy-time-learning is a stretch in the "parallel" framing.

INFO:
- [Completeness] The "when to move down" criterion ("the agent consistently proposes the same correct next step") is a sufficient condition but the note presents it as if it were the primary or only signal. A boundary case: an operation where the agent's output is always correct but varies in form across runs (e.g., different valid orderings of the same check). The output is correct but not convergent — does this trigger codification? The note's criterion conflates convergence on a single interpretation with correctness, but these are distinct. An operation could be reliably correct without being convergent (if multiple valid implementations exist), or convergent but incorrect (if the agent consistently makes the same mistake).
- [Internal consistency] The note says "not everything should complete the trajectory" and gives oracle strength as the criterion, but the maturation trajectory section (numbered 1-2-3) presents the trajectory as a linear path without marking the exit points. The body text clarifies this, but a reader scanning the numbered list could reasonably infer that all practices should progress through all three stages. The compressed representation (the list) elides the tension that the body acknowledges.
- [Grounding alignment] The note cites oracle-strength-spectrum.md for the claim that operations with low oracle strength "belong permanently at the skill level." The oracle strength note does discuss the relationship between verification cost and codification, but it frames this as speculative ("The framework is speculative; the individual hypotheses need testing") and does not make the specific claim that low-oracle-strength operations "belong permanently" anywhere. The note's use of "permanently" is stronger than the source supports — the oracle strength note is a seedling exploring hypotheses, not a settled framework that can ground permanent architectural prescriptions.
- [Completeness] The table's "Response" column marks skills, instructions, and ad hoc prompts all as "underspecified + indeterministic (LLM interprets/executes)." But skills vary enormously in their degree of underspecification — a skill with a 200-line procedure and output format constraints (like `/validate`) is far more specified than a one-line instruction in CLAUDE.md. The table treats the skill layer as uniform when it contains its own internal gradient.
- [Grounding alignment] The note cites the context engineering study for the claim that "50% of AGENTS.md files were never changed after creation." The ingest confirms this figure (77/155). However, the note uses this to support the claim that "the codification trigger only fires if someone is watching" — which is a reasonable inference but not what the source says. The source treats stagnation as an empirical observation about adoption patterns, not as evidence about observation requirements. The note's causal interpretation (stagnation = lack of observation) is plausible but is the note's own inference, not an attribution.

PASS:

The table, the body text, and the hook discussion all use the same framework without drift. The key insight about hooks (deterministic trigger, indeterministic response) is internally consistent with the two-axis model.
- [Internal consistency] The maturation trajectory (instruction -> skill -> hook/script) is consistent with the constraining gradient and with the ADR-001 example. The example genuinely illustrates the claimed pattern — a skill-level operation that was recognized as fully mechanical and moved to a script.
- [Grounding alignment] The link to constraining.md is accurate and well-scoped. The relationship descriptor "applies" is correct.
- [Grounding alignment] The link to programming-practices-apply-to-prompting.md is accurate. That note explicitly discusses "progressive compilation applied to methodology" and references the methodology enforcement note. The cross-reference is mutual and the claims align.
- [Grounding alignment] The statistics from the context engineering study (Add 78, Modify 59, Remove 23, Remove-section 2) match the ingest document exactly. The attribution of these specific figures is accurate.
