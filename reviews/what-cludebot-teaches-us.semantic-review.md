=== SEMANTIC REVIEW: what-cludebot-teaches-us.md ===

Claims identified: 16

WARN:
- [Completeness] The "What we already have" section enumerates 4 patterns (typed link semantics, progressive disclosure, hybrid retrieval, evidence linkage) as the set of cludebot ideas "we've already adopted." But cludebot also implements explicit memory typing/categorization (episodic, semantic, self-model) which maps closely to the KB's own type system (`note`, `structured-claim`, `adr`, etc.) and to the three-space model the note itself cites later. This is arguably a fifth pattern the KB already has that the note doesn't list. The omission is notable because the note references the three-space model in a later section but doesn't connect it back to the "already have" enumeration.

- [Grounding alignment] The note states: "We use `extends`, `foundation`, `contradicts`, `enables`, `example`." But CLAUDE.md — the KB's own authority on link semantics — lists the vocabulary as `extends`, `grounds`, `contradicts`, `enables`, `exemplifies`. The note uses `foundation` where the KB uses `grounds`, and `example` where the KB uses `exemplifies`. This is a vocabulary mismatch within the KB itself: the note's attribution of the KB's own link semantics is inaccurate. Readers trusting this note as documentation of the link vocabulary will get the wrong terms.

- [Completeness] The "Worth adopting now" section enumerates 3 techniques (active contradiction surfacing, explicit staleness decay, consolidation passes). A boundary case: cludebot's explicit memory-type-dependent behavior — where different memory types trigger different retrieval strategies and lifecycle policies — could be adopted now as type-dependent /connect behavior or type-dependent /review scheduling. The note mentions type-dependent decay rates in the staleness section, but the broader principle (type-dependent behavior across all operations, not just decay) sits between the enumerated items without being explicitly captured.

INFO:
- [Completeness] The "What to skip" section lists automated importance scoring as unnecessary because "the human decides what's worth writing down." However, the KB's own note [notes-need-quality-scores-to-scale-curation](../kb/notes/notes-need-quality-scores-to-scale-curation.md) proposes composite quality scores with automated components (inbound link count, recency decay, status weighting). The "skip" recommendation and the quality-scores note are in tension: one dismisses automated scoring, the other designs it. The quality-scores note is a `seedling` and explicitly defers implementation, so the tension is not a contradiction — but the note doesn't acknowledge it.

- [Grounding alignment] The note claims cludebot's `recallSummaries()` -> `hydrate()` pattern maps to the KB's `description` frontmatter scan-then-read. The analogy is reasonable but loose: cludebot's pattern is a two-stage API call within a single retrieval operation, while the KB's pattern involves separate tool invocations (rg for scanning, Read for loading) with human or agent decision-making between them. The "same cognitive benefit, zero infrastructure" conclusion is directionally correct but the mechanisms differ more than the prose suggests.

- [Internal consistency] The note says "At our scale, the human decides what's worth writing down. That implicit filter is more reliable than automated scoring." But the "Worth adopting now" section proposes consolidation passes — a form of automated curation. If the human filter is sufficient, why do we need automated consolidation? The note implicitly distinguishes between scoring (assessing individual memory importance) and consolidation (merging related clusters), but this distinction is not made explicit. A reader could see these as contradictory.

- [Grounding alignment] The note states the three-space model "already predicts that operational notes should churn faster than knowledge notes." Checking the source: the three-space model note describes different "metabolic rates" — "steady growth" for knowledge, "slow evolution" for self, "high churn" for operational — which does support the claim. However, the three-space note is marked `status: speculative` and explicitly notes the mapping "may be decorative." The cludebot note presents its prediction as established fact rather than reflecting the speculative status of its source.

- [Completeness] The note's "Watch for as the KB grows" section treats graph-based retrieval as premature, citing the KB's ~200 notes. But the note itself proposes consolidation passes (in "Worth adopting now") that would benefit from exactly the cluster-discovery that graph-based retrieval enables. There is a gap between these two sections: consolidation needs to find "these 5 notes about the same topic," which is a graph-neighborhood operation. The note doesn't acknowledge that adopting consolidation might pull graph-based retrieval forward from "watch" to "adopt."

Attribution is faithful.

The source note includes a table of content types with different recency decay rates (fast for sources and session artifacts, none for design notes and ADRs).

Every adopt/watch/skip decision is calibrated against this scale difference. No section silently drops the scale qualifier.
- [Grounding alignment] The claim that the three-space model's "graduation from operational to knowledge space" maps to compaction is a reasonable inference. The source note describes operational artifacts that "arrive raw, consolidate, and either graduate to knowledge or get archived," which is structurally similar to cludebot's compaction (summarize and consolidate old memories).
