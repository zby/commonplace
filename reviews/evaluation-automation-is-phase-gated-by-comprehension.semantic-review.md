WARN:
- [Completeness] The three-phase framework (comprehension, specification, generalization) claims to describe the "characteristic sequence" of evaluation automation. A boundary case probes this: **iterative re-comprehension after generalization reveals new failure modes.** When optimization runs against calibrated evaluators and produces outputs that fail in new, previously unobserved ways, the practitioner must loop back to comprehension. The note's framing is strictly sequential ("first, second, third") and does not acknowledge re-entry. The "Scope limits" section partially addresses this ("Mature systems may partially automate parts of comprehension, but only after prior manual cycles have stabilized the taxonomy"), but this concerns automating comprehension, not the need to revisit comprehension after generalization exposes novel failures. The sequential framing may mislead readers into treating the phases as a single pass rather than an iterable cycle.

- [Completeness] The note's central claim is that comprehension must be "human-led" in soft-oracle domains (Scope limits). A boundary case: **teams that use structured error taxonomies from adjacent, already-comprehended domains as starting points.** If a team has built a failure taxonomy for one writing-quality task and transfers it (with adaptation) to a related task, the comprehension phase is partially bootstrapped rather than built from scratch observation. The note's framing — "someone must inspect real outputs, identify concrete failure modes" — does not acknowledge that prior taxonomies from related domains can shorten or partially replace the manual observation step. This is not a refutation of the gate claim, but the note's silence on taxonomy transfer overstates how much comprehension must always start from zero.

- [Grounding — scope mismatch] The note claims (link annotation) that it "narrows" the-boundary-of-automation-is-the-boundary-of-verification by identifying "an intra-loop boundary where optimization depends on prior verifier construction." However, the automation-boundary note already contains a "refines" backlink to this note saying it adds intra-loop staging. The relationship semantics are consistent, but the note's claim that skipping comprehension "amplifies proxy quality rather than task quality" (claim 7) is a stronger assertion than the source ingest supports. The ingest explicitly notes in its Limitations section: "Multiple variables changed at once between takes... so causal attribution to any single change is weak." The note presents the causal chain (skip comprehension -> proxy amplification) as a general mechanism, but the sole practitioner evidence is a single-team narrative where confounds were not controlled. The note does not hedge this causal claim the way the ingest hedges the source.

INFO:
- [Completeness] The three practical gates in the "Practical implication" section (output-read pass, failure taxonomy, judge calibration) are presented as sufficient preconditions. A boundary case worth considering: **judge calibration may pass on a hand-scored mini set but fail on the distribution the optimization loop actually encounters.** The note does not distinguish between calibration on a representative sample and calibration on a convenience sample. If the mini set is not diverse enough, the gate is formally satisfied but substantively hollow. This is an edge case rather than a framework failure, but it could be worth noting since the note is prescriptive about pipeline design.

- [Grounding — vocabulary mismatch] The note links to oracle-strength-spectrum with the annotation "frames (provisional — target is speculative): the three phases can be read as a local oracle-hardening sequence." The oracle-strength-spectrum note describes a manufacture/amplify/monitor pipeline. Mapping the note's three phases onto this: comprehension ~ manufacture (observing to create oracles), specification ~ manufacture/amplify (building and calibrating judges), generalization ~ amplify (running optimization). The mapping is reasonable but not exact — the oracle-strength note's "monitor" step (detecting when hardened oracles encode vision features) has no clear counterpart in this note's framework. The note's "(provisional)" qualifier appropriately flags this, so this is informational rather than a warning.

- [Grounding] The note links to spec-mining-as-codification with "grounds: converting observed failures into reusable evaluators is spec mining." The spec-mining note describes extracting deterministic verifiers from observed stochastic behavior. The note's specification phase — "Convert observations into a failure taxonomy and evaluators" — is a reasonable instance of this pattern. However, spec mining in the source note emphasizes deterministic artifacts (functions, schema rules, unit tests), while the note's evaluators/judges in soft-oracle domains are inherently non-deterministic (rubric-based LLM judges). The note is extending the concept to softer verification targets than the source note primarily discusses. The inference is plausible but the fit is looser than "grounds" implies.

- [Internal consistency] The note says "In hard-oracle domains (compilers, strict schemas, deterministic tests), comprehension can be shorter because failure is already legible" (Scope limits). This is internally consistent with the gate claim — the note does not say comprehension can be skipped, only shortened. However, in pure hard-oracle domains (e.g., a compiler where the test suite already exists), it is unclear what comprehension adds beyond confirming the tests work. The note could be read as implying comprehension is always load-bearing, when in some hard-oracle cases it may be ceremonial. This is a minor ambiguity rather than a contradiction.
