=== SEMANTIC REVIEW: interpretation-errors-are-failures-of-the-interpreter.md ===

Claims identified: 12

1. [Opening] "Real LLMs produce outputs outside the space of valid interpretations."
2. [Opening] Interpretation error "is not underspecification (where multiple outputs are valid) and not indeterminism (sampling noise across runs)."
3. [Opening] Interpretation error is "a property of the interpreter itself: the gap between what a perfect interpreter would do and what a real LLM does."
4. [Examples] Five examples enumerated: constraint violation, hallucination, bookkeeping failure, content bias, emotional prompt sensitivity.
5. [Examples] "In each case, a perfect interpreter given the same spec would not make the error."
6. [Why this matters] "The idealised two-phenomena model implicitly assumes a perfect interpreter."
7. [Why this matters] "Narrowing the spec (underspecification remedy) can make things worse by overloading context."
8. [Why this matters] "A deterministic LLM still makes interpretation errors" — sampling control is irrelevant.
9. [Why this matters] "The correct remedies are error detection and correction: validation, oracles, voting, guardrails, and architectural separation."
10. [Why this matters] Discrimination — "knowing per-instance whether the output is wrong" — is the binding constraint on automation.
11. [Why this matters] "If LLMs were perfect interpreters, the only question would be which valid interpretation they chose."
12. [Implicit scope claim] The five examples together constitute a representative sample of interpreter failures.

---

WARN:
- [Completeness] The enumeration of five example types omits a significant class: **instruction-following failures that aren't constraint violations** — cases where the LLM follows the instruction but applies it to the wrong scope or wrong part of the input. For instance, "summarise section 3" and the LLM summarises section 4. This is not hallucination (the content is in the document), not a constraint violation (the output format is correct), not a bookkeeping failure (no compositional depth), not content bias (not about semantic content), and not emotional prompt sensitivity. It is a referential/grounding failure — the LLM misidentifies what the instruction applies to. The note's framing ("the spec is sufficient; the interpreter is not") clearly covers this, but no enumerated example maps to it cleanly. Since the note uses examples to illustrate the space, a reader might fail to recognize this common failure mode as an interpretation error.

- [Completeness] The five examples blur the line between **systematic bias** and **stochastic capacity failure**. Bookkeeping failure (F1 collapse at depth 100) is deterministic and predictable given the input structure. Content bias (reasoning accuracy varies with semantic content) is systematic but less predictable. Emotional prompt sensitivity is systematic and measured as bias. Constraint violation and hallucination are presented without specifying whether they are systematic or stochastic for a given input. The note groups all five under "interpretation error" without distinguishing whether the error is a consistent wrong answer (bias) or a probabilistic wrong answer (capacity limitation). The index note's parenthetical — "for a theoretical deterministic LLM: simply the wrong output" — handles this cleanly, but the reviewed note does not make the distinction, which matters for remedy selection: voting corrects stochastic errors but not systematic bias (as the scheduler-separation note explicitly argues).

- [Grounding alignment] The note claims "Narrowing the spec (underspecification remedy) can make things worse by overloading context." This causal claim — that spec-narrowing causes interpretation errors via context overload — is not grounded by any cited source. The parent note (agentic-systems-interpret-underspecified-instructions) discusses spec-narrowing as a remedy for underspecification but does not claim it worsens interpretation errors. The ConvexBench source shows F1 collapse at depth 100 with only 5,331 tokens (well under context limits), which is evidence about compositional depth, not context overload from spec-narrowing. The claim is plausible — longer system prompts could push the model into worse operating regimes — but it is presented as established fact rather than inference. A reader following the links would not find direct support for this specific causal chain.

INFO:
- [Completeness] The "perfect interpreter" framing is doing heavy lifting. Claim 5 states: "In each case, a perfect interpreter given the same spec would not make the error." For the hallucination example ("summarise this document" → LLM includes facts not in the document), the notion of "perfect interpreter" requires specifying what the valid interpretation space of "summarise" is. If the spec says "summarise this document" and the model includes a widely-known contextual fact that helps the reader understand the document, is that hallucination or a valid interpretation of "summarise"? The boundary between interpretation error and underspecification depends on how tightly you define the valid interpretation space — and that definition is itself underspecified for many of these examples. The note's clean trichotomy (underspecification vs. indeterminism vs. interpretation error) may be less clean in practice than the framing suggests.

- [Internal consistency] The note's title claim is "interpretation errors are failures of the interpreter," positioning interpretation error as a property of the interpreter. The emotional prompt sensitivity example, however, describes a case where "semantically equivalent prompts with different emotional framing produce systematic performance degradation." If the prompts are semantically equivalent (same spec), this is cleanly an interpreter failure. But the Ma et al. ingest notes that emotional framing "activates different training distributions" — which suggests the interpreter treats the prompts as functionally different. This creates a tension: from the user's perspective the spec is unchanged, but from the model's perspective the input is materially different. The note handles this by defining the phenomenon from the user's perspective ("the functional spec is unchanged"), which is a reasonable choice, but the mechanism (distribution activation) means the interpreter is responding to real input differences that the user considers irrelevant. This is not a contradiction but an acknowledged ambiguity in what counts as "the same spec."

- [Grounding alignment] The ConvexBench citation says "F1 collapses from 1.0 to 0.2 at depth 100 despite short context." The source confirms these numbers (F1=1.0 at depth 2, approximately 0.2 at depth 100) and confirms the short-context observation (5,331 tokens at depth 100). However, the note labels this "bookkeeping failure" and describes it as "tracking compositional depth (fully specified, one correct answer)." The ConvexBench task is actually recognizing convexity in composed functions — a mathematical reasoning task, not mere bookkeeping in the sense of state tracking or counting. The scheduler-separation note uses ConvexBench more precisely, describing it as compositional depth tracking that "can in principle be fully specified." The reviewed note's label is defensible (there is a unique correct answer, so it fits the "fully specified" criterion), but calling convexity recognition "bookkeeping" may mislead readers about the task's cognitive demands.

- [Grounding alignment] The Ma et al. citation is to the arXiv PDF URL (https://arxiv.org/pdf/2509.13680) rather than to the KB's own source or ingest file. The linked notes section at the bottom correctly links to the ingest file. The inline citation and the relevant-notes citation point to different targets for the same source. This is a structural issue (normally /validate territory), but it has semantic consequences: a reader following the inline link gets a raw PDF, while the ingest file contains the KB's analysis including important caveats (e.g., narrow evaluation domain, underpowered statistical test for the headline claim). The grounding alignment is accurate — the source does show systematic performance degradation under emotional framing — but the note does not surface any of the ingest file's caveats about the strength of that evidence.

PASS:
- [Completeness] The trichotomy (underspecification / indeterminism / interpretation error) is tested against its own defining criterion: "the gap between what a perfect interpreter would do and what a real LLM does." All five examples pass this test — in each case, a deterministic perfect interpreter given the same input and spec would produce the correct output. The trichotomy is exhaustive by construction: any deviation from intended output is either (a) a valid interpretation the user didn't want (underspecification), (b) a different execution of the same interpretation (indeterminism), or (c) an invalid output (interpretation error). These three are mutually exclusive and jointly exhaustive for the defined space.

- [Grounding alignment] The parent note (agentic-systems-interpret-underspecified-instructions) does describe a "two-phenomena model" covering underspecification and indeterminism, and does assume a perfect interpreter (the spec-to-program projection model presupposes the LLM picks a valid interpretation). The reviewed note's claim that the parent model "implicitly assumes a perfect interpreter" is accurate — the parent explicitly states "the spec-to-program mapping is one-to-many" without considering cases where the LLM lands outside the valid space entirely.

- [Grounding alignment] The claim that sampling control is irrelevant to interpretation errors ("a deterministic LLM still makes interpretation errors") is well-grounded. The ConvexBench evidence directly supports this: the F1 collapse occurs in a deterministic evaluation setting, not due to sampling variation. The sibling note (execution-indeterminism) explicitly states that its remedies (sampling control) leave "interpretation error untouched."

- [Internal consistency] The note's internal logic is consistent. The opening definition (outputs outside the valid space), the examples, and the remedies section all use "interpretation error" with the same meaning throughout. The distinction from underspecification and indeterminism is maintained without drift. The claim that remedies differ (error detection/correction rather than spec-narrowing or sampling control) follows from the definition.

- [Internal consistency] The link from interpretation error to discrimination as the "binding constraint on automation" is consistent with the discrimination note's argument. The discrimination note defines the boundary as "per-instance discrimination — the ability to know, per instance, whether this particular output is likely wrong," which is exactly the capability needed to detect when the LLM has "left the valid space entirely" (this note's phrasing).

Overall: 3 warnings, 4 info
