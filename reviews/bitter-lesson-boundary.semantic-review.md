=== SEMANTIC REVIEW: bitter-lesson-boundary.md ===

Claims identified: 15

WARN:
- [Completeness] The confidence signals table enumerates three signals (specifiability, definition-vs-proxy, local-vs-compositional failures), but the note itself deploys a fourth signal — the **chess hybrid analysis** — that does not map cleanly to any single row. The chess example distinguishes "rules" from "strategy" within one system by asking whether a component is *algorithmically determined* (move generation from rules) vs. *judgment-dependent* (evaluation of positions). This "algorithmically determined vs. judgment-dependent" distinction is related to but not identical with the three table rows. A reader trying to apply the table to classify a new system would lack guidance on how to handle components that are fully specifiable (row 1 says arithmetic) yet computationally intractable (like NP-hard optimization, which the note itself acknowledges has fully specified objectives but learned solution methods). The table's three signals are presented as the confidence framework, but the note's own examples exceed it.

- [Completeness] The note's central dichotomy is "spec IS the problem" vs. "spec is a theory about the problem." But the note does not address cases where the **spec is correct but incomplete** — where there is no gap between spec and problem for the specified portion, but the spec only covers part of the problem. Database schema validation is an example: the schema perfectly specifies structural constraints (arithmetic regime), but "is this data correct?" requires judgment beyond the schema (vision-feature regime). The note's chess example approaches this (rules are complete for legality, incomplete for quality of play), but the confidence signals table doesn't capture the "correct but incomplete spec" case as distinct from "spec is a theory." A spec that is genuinely the problem for its scope but insufficient in scope is different from a spec that is a theory about the whole problem.

- [Grounding] The note claims Deep Blue's victory illustrates the bitter lesson eating chess strategy: "Deep Blue's massive search beat programs that relied on selective, knowledge-heavy play." However, the Wikipedia bitter lesson source characterizes Deep Blue as using "a relatively simple alpha-beta search algorithm that scaled up by applying large amounts of specialized hardware." Sutton's actual bitter lesson argument here is that *search* (a general method) beat *chess-specific knowledge*. But the note then says "AlphaZero later replaced hand-crafted evaluation with learned functions entirely" — conflating two different bitter lesson mechanisms (search-based scaling in Deep Blue, learning-based scaling in AlphaZero) as a single narrative. The note's framing ("the bitter lesson ate the strategy") elides that Deep Blue's approach was not learned at all — it was brute-force search, which is itself closer to the arithmetic regime (exhaustive enumeration). The bitter lesson source supports the conclusion but through a different mechanism than the note implies.

INFO:
- [Completeness] The note asserts "it's irrational to bet on emergent reliability via scale when deterministic code gives you perfect correctness at near-zero cost." This frames the arithmetic regime as cost-dominant, but does not address cases where the deterministic solution exists but is *expensive to implement or maintain*. Fiscal period normalization (cited in the Fintool link) requires maintaining 10,000+ company calendars — the spec IS the problem, but the implementation cost is non-trivial. The claim holds in principle (deterministic correctness at finite cost always beats probabilistic correctness at scale-dependent cost for fully specified problems), but "near-zero cost" overstates the case for complex arithmetic-regime implementations.

- [Completeness] The NP-hard optimization passage ("vehicle routing, job-shop scheduling") acknowledges that "finding good solutions is where learned methods increasingly beat hand-crafted heuristics." This sits awkwardly with the binary framing. The objective and constraints are fully specified (arithmetic regime by the note's criterion), yet the note concedes learned methods win for the search procedure. The note handles this by calling it a hybrid, but the confidence signals table's first row ("Spec IS the problem") would classify NP-hard optimization as arithmetic, which the note itself says is wrong for the search component. The table may need a caveat about computational tractability.

- [Grounding] The Ebrahimi et al. link description says "kappa metric shows transformers learn length-isolated solutions (kappa approximately 1) while RNNs share mechanisms across variations — destructive interference (kappa < 1) is concrete evidence that more data can hurt without the right architectural constraint." The source does support this claim. However, the link description frames kappa < 1 as showing "more data can hurt," while the paper's finding is specifically about *mixed-length training* causing destructive interference in transformers. The note's framing is a reasonable but slightly broadened interpretation — the paper shows more *diverse* data can hurt, not more data in general.

- [Internal consistency] The note says "The difference isn't scope — it's whether the specification fully captures the problem" but also says chess rules (fully specified, narrow scope for legality) coexist with chess strategy (underspecified, broad scope for play quality) in the same system. The hybrid framing resolves this, but the initial "the difference isn't scope" claim could mislead a reader into thinking scope is irrelevant, when in fact scope determines *which parts* of a system fall on which side of the boundary. The note does address this with "the boundary running through the architecture," but the emphatic "the difference isn't scope" in the earlier section creates a tension with the hybrid analysis that follows.

- [Grounding] The multi-agent source (voooooogel) link says the source "applies: bitter-lesson reasoning to agent infrastructure — fixed role hierarchies dissolve, filesystem and spawning survive because they are structural." The ingest file confirms this is a fair characterization of the source's argument. However, the ingest file's own Limitations section notes the source reasons "by analogy without testing whether the analogy holds" and that the dissolution predictions are "unfalsifiable." The bitter-lesson-boundary note cites this source without hedging, presenting it as straightforward exemplification rather than noting it is itself an analogical application of the boundary concept.

PASS:
- [Grounding] The note's paraphrase of the bitter lesson ("general methods leveraging computation beat hand-crafted domain knowledge") accurately reflects the Wikipedia source, which states: "approaches that scale with available computational power... tend to outperform ones based on domain-specific understanding." The attribution is faithful.

- [Grounding] The vision features example (SIFT, Haar cascades, Canny edge detection losing to learned features) is well-supported by the Wikipedia bitter lesson source, which explicitly names "explicitly encoded edge detection or detecting high-level features with SIFT" as examples that "were outperformed by convolutional neural networks."

- [Internal consistency] The note's core framework — spec-is-the-problem vs. spec-is-a-theory — is used consistently throughout. The arithmetic examples (multiplication, sorting, cryptographic algorithms, schema validation) all satisfy the "spec IS the problem" criterion. The vision-feature examples (edge detection, corner detection, chess strategy) all satisfy the "spec is a theory" criterion. No definition drift was found in how these terms are used across sections.

- [Internal consistency] The chess hybrid example is internally consistent with the note's framework — it correctly classifies rules as arithmetic and strategy as vision feature, and the "boundary running through the architecture" conclusion follows from the preceding analysis.

- [Grounding] The MAKER source link ("million-step zero errors") accurately describes the source as exemplifying "the arithmetic regime at scale: decomposition to minimal subtasks with hard per-step oracles achieves zero errors over a million steps." The MAKER paper does decompose into minimal subtasks with per-step voting (error correction), and the Towers of Hanoi domain has fully specified rules — consistent with the arithmetic regime classification.

- [Grounding] The AgeMem link accurately characterizes the system as a hybrid with "arithmetic-regime operations (Add, Delete, Retrieve) composed by a learned vision-feature policy (RL-trained when-to-use)." The AgeMem note confirms: "the operations are the calculator part (specifiable, deterministic), while the composition policy is the vision-feature part (learnable, benefits from scale)."

- [Completeness] The "composition failure is the tell" diagnostic claim is well-grounded in the vision-features example and the chess hybrid. The note does not overclaim it as a sufficient test — the surrounding text ("you often can't tell which side of the boundary you're on until scale tests the distinction") appropriately hedges the diagnostic's reliability.

Overall: 3 warnings, 4 info
