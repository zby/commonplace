=== SEMANTIC REVIEW: bounded-context-orchestration-model.md ===

Claims identified: 18

WARN:
- [Completeness] The "What makes selection hard" section enumerates three difficulties: sequential dependence, dual cost dimensions, and framing effects. This omits a plausible fourth: **partial observability of LLM capability**. The scheduler does not know how well the sub-agent will handle a given prompt until it tries — selection difficulty is not just about what to include and how to frame it, but about uncertainty regarding the sub-agent's reception. The note's own "Scope and open questions" section gestures at this ("the value of including item X depends on the sub-agent's stochastic interpretation"), but that acknowledgment sits outside the enumeration rather than inside it. A reader treating the three-item list as exhaustive would miss this dimension.

- [Completeness] The two-component model ("a symbolic scheduler over unbounded exact state" and "bounded clean context windows") omits a third element that the note itself relies on: the **state store** that persists K between iterations. The pseudocode writes `K = K + r`, the prose says "accumulated state lives there, not in conversation history" and discusses files, databases, and in-memory structures. Yet the formal model names only two components. The state store is architecturally load-bearing (it is what makes the scheduler's state "unbounded" and "exact") but is folded into the scheduler rather than named. This is a presentation gap more than a conceptual gap, but boundary cases like distributed or shared state stores would stress the two-component framing.

- [Grounding — domain coverage] The note links to "information value is observer-relative" to ground the claim that "the same material under different framing yields different extractable structure" (select/call loop section) and that "framing matters, not just selection" (What makes selection hard). The source note's concept of observer-relativity is about how prior knowledge, computational capacity, tools, and goals determine extraction — it is fundamentally about different observers extracting different value from the same data. The bounded-context note uses this to support a claim about the *same* observer (the sub-agent) extracting different value from different *framings* of the same data. This is a valid application but a narrower one than the source develops: the source is about observer variance, while the note uses it to ground framing variance for a fixed observer. The attribution is reasonable but readers might infer broader grounding than the source provides for this specific claim.

INFO:
- [Completeness — boundary case] The note claims "Hierarchical decomposition is therefore not a separate mechanism but a pattern of use" — that planning calls returning plans into K, which the scheduler then reads deterministically, subsume hierarchy. Boundary case: what about systems where the LLM's planning call triggers *nested* select/call loops that themselves have sub-schedulers with their own K? The note's formalism handles this (the inner loop is just another sequence of select/call steps operating on a scoped K), but the claim that this is "not a separate mechanism" elides the coordination problem between nested loops — who owns K, how do sub-loop results propagate back? The note-selection example (step 4b) shows one level of nesting; deeper nesting may strain the "pattern of use" framing.

- [Completeness — boundary case] The model accommodates "architectures where the LLM emits a symbolic control program rather than a direct natural-language answer" but the formal pseudocode shows `r = call(P)` returning a single result. When the LLM emits a program, execution of that program may produce multiple results, side effects, or failures that need error handling. The model handles this implicitly (execution results get appended to K), but the simple `K = K + r` notation understates the complexity of the state update when r is a program execution trace rather than a single judgment.

- [Internal consistency] The note says `K` contains "unbounded exact state" (The model section) while the cost measure `||P||_t <= M` constrains what can be loaded from K into a prompt. The note is consistent — K is unbounded but each prompt P drawn from K is bounded — but the phrasing "unbounded exact state" might mislead readers into thinking the system has no capacity constraints at all. The note does clarify this in context, but the juxtaposition of "unbounded" (for K) and "bounded" (for P) could benefit from a single sentence making the distinction explicit where "unbounded" is first introduced.

- [Grounding] The note cites "effective context is task-relative and complexity-relative not a fixed model constant" to justify writing `||P||_t` with task-dependent subscript. The source note says "effective context is relational" and interprets it as "a task-shaped cost measure ||P||_t <= M — the cost norm depends on what you're asking the model to do." This is actually circular: the source note attributes the `||P||_t` notation to the bounded-context orchestration model itself. Both notes reference each other as the origin of this formalism. This is not a grounding problem per se — the formalism is internally motivated — but readers following the citation expecting independent evidence for task-dependent cost measures will find the same framework restated rather than independently validated.

- [Completeness — boundary case] The "satisfied(K)" termination condition is acknowledged as "itself a judgment call" in the Scope section, but the pseudocode presents it as a simple boolean check. For systems where satisfaction is genuinely hard to assess (open-ended research, creative tasks), the satisfied check may itself require an LLM call — making it another instance of select/call rather than a loop guard. The note's scope section acknowledges this limitation honestly, so this is not a gap in the argument, but the clean pseudocode somewhat understates the difficulty.
