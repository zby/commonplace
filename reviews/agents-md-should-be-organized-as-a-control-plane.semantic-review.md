=== SEMANTIC REVIEW: agents-md-should-be-organized-as-a-control-plane.md ===

Claims identified: 15

WARN:
- [completeness] The core model claims "two variables determine placement: loading frequency and failure cost," but the three-way placement rule also smuggles in a third variable — specificity (task-specific vs. universal). A high-failure-cost instruction that is task-specific does not clearly map: the 2x2 of frequency x cost would place it in AGENTS.md (high cost), but the three-way rule routes it to "skill/instruction body or task doc" (task-specific). The note resolves this by implicitly treating "always loaded" as a proxy for "universal," but this conflation is not stated. Boundary case: a pre-commit safety check that only applies to one directory — high failure cost, low loading frequency. The placement rule gives contradictory guidance depending on which variable you weight.

- [completeness] The three AGENTS.md layers (Invariants, Routing, Escalation boundaries) claim to be a complete internal structure for AGENTS.md, but the note's own quality tests section introduces content that does not fit any of the three layers. "Does it point to a deeper artifact instead of duplicating it?" is meta-guidance about how to write AGENTS.md entries — it is neither an invariant, a route, nor an escalation boundary. The layers describe what goes in; they do not account for the meta-instructions about how to compose the file itself. This is a minor gap — meta-guidance might reasonably live in a methodology note rather than in AGENTS.md — but the note does not draw that distinction.

- [grounding] The link annotation for the OSS study says it "validates" the control-plane model because Table 1 shows "14 content categories practitioners actually put in context files, with no established structure." The ingest file confirms the paper finds "no established content structure yet" and lists content categories. However, the absence of structure in the wild is evidence for a gap, not validation of a specific proposed structure. The paper does not evaluate the invariants/routing/escalation layering or any normative model. The note's claim that the empirical data validates this particular three-layer model is a scope upgrade — the data validates that *some* structure is needed, not that *this* structure is correct.

INFO:
- [completeness] The exclusion rules list six categories that "should generally stay out of AGENTS.md." Boundary case: error-handling conventions. These are high failure cost (agents that handle errors wrong can cause damage), needed across sessions (universal), and concisely expressible — meeting all the quality tests for inclusion. Yet "conventions" is not one of the three layers (Invariants, Routing, Escalation). The note's Layer 1 ("repository-wide conventions that prevent silent damage") would cover them, but the label "Invariants" is a stretch for conventions that are best practices rather than hard rules. The note could clarify whether Layer 1 is strictly invariants (things that must never be violated) or also includes strong conventions.

- [completeness] The lifecycle section describes five stages of guidance maturation (methodology note -> AGENTS.md pointer -> maintenance staging -> instructions -> scripts/hooks). Boundary case: guidance that should skip stages. A well-understood deterministic check (e.g., "never commit .env files") does not need to start as a methodology note — it can go directly to a hook. The lifecycle as written implies a sequential trajectory, but the note's own linked source (methodology-enforcement-is-constraining) explicitly says "not everything should complete the trajectory." The lifecycle section does not acknowledge that stages can be skipped.

- [grounding] The link annotation for the Harness Engineering source says the 100-line AGENTS.md "is the control-plane model in production at 1M LOC scale." The ingest confirms Lopopolo describes a "short AGENTS.md as a map with pointers." However, the ingest characterizes the AGENTS.md as "a map with pointers to deeper sources of truth" — this aligns with the routing layer specifically, but the ingest does not confirm that the file uses the invariants/routing/escalation three-layer decomposition. The source exemplifies the slim-router principle, not necessarily the full control-plane model with all three layers.

- [internal consistency] The note defines two failure modes (token pressure and attention dilution) and claims "the same placement also controls both cost modes." But the Exclusion rules section introduces six exclusion categories without mapping them back to the two failure modes. For example, "high-rationale background theory" is excluded presumably because of attention dilution (low signal-to-noise), while "volatile project state" is excluded presumably because of staleness risk — which is neither token pressure nor attention dilution but a third failure mode not named in the core model.

PASS:
- [internal consistency] The three layers (Invariants, Routing, Escalation) are internally consistent with the placement rule. Each layer serves "always loaded + high failure cost" content. The layers do not contradict each other — invariants are rules, routing is navigation, escalation is exit conditions. No pairwise contradiction found.
- [internal consistency] The escalation boundary examples (lines 68-71) are consistent with the routing table in the project's actual CLAUDE.md, which uses nearly identical language. The note practices what it prescribes.
- [grounding] The link to instruction-specificity-should-match-loading-frequency correctly attributes the "loading frequency" variable as a foundation. The linked note does describe loading frequency as the primary placement axis, and the control-plane note builds on this by adding failure cost as a second variable. The attribution is accurate and the extension is clearly marked as the note's own contribution.
- [grounding] The link to methodology-enforcement-is-constraining accurately describes the lifecycle trajectory. The linked note's maturation trajectory (instruction -> skill -> hook -> script) aligns with this note's lifecycle section, and the relationship label ("lifecycle framing") is appropriate.
- [internal consistency] The nested topology section (child files add/tighten, never contradict parent) is consistent with the layer model — invariants at root, local routing at subtree. No definition drift detected between the layers section and the topology section.

Overall: 3 warnings, 4 info
