WARN:
- [Completeness / boundary case] The note claims "the error channel is an instruction channel" and that teaching errors are effective because "the agent's only knowledge of what went wrong and how to fix it is what appears in its context window." This implicitly claims the error message is the sole source of corrective information. Boundary case: an agent that encounters a linter error, then uses a tool to search for documentation or examples in the codebase, derives the fix from those secondary sources rather than the error message itself. In that scenario, the error message is a signal (like it is for humans), not "the training for that interaction." The note's claim holds for agents that cannot or do not use tools to search, but many agent systems do have tool access. The claim would be more precise as "the error channel is a primary instruction channel" or with an explicit scope qualifier about tool-free error-handling loops.

- [Grounding alignment / scope mismatch] The Lopopolo source (ingest) says "Linter error messages double as remediation instructions — every failure message teaches the agent the fix." The note uses this as evidence for the broader thesis that the error channel IS an instruction channel and that the enforcement/guidance distinction collapses. But Lopopolo's claim is specifically about linter messages in a CI pipeline — a narrow, structured verification context. The note promotes this to all verification output ("linter messages, test failures, hook warnings, CI output"). Test failures and CI output often produce complex, multi-page logs where the "teaching" property does not naturally hold — a failing integration test that dumps a stack trace is not straightforwardly an instruction channel. The source evidence grounds the linter case well but the generalization to all verification output is the note's own move.

INFO:
- [Completeness / boundary case] The dual-function table shows constrain+inform at five layers, but the "Script" row's constrain+inform variant is "corrects and logs what it changed and why." A script that silently corrects (constrain-only) already produces the correct output — the agent does not need to learn the fix because the script applied it. The "inform" axis for scripts serves a different purpose (auditability, human understanding) than it does for the other layers (agent self-correction). The table treats all five layers as symmetrical, but the script layer's inform function targets a different audience and serves a different goal. This is acknowledged implicitly ("logs what it changed and why" is not the same as "teaches the fix") but the table's visual symmetry may overstate the uniformity.

- [Completeness / boundary case] The note claims "moving right on this axis is cheap constraining." Boundary case: a hook that must explain a complex architectural violation (e.g., "this import creates a circular dependency between module A and module B; restructure by extracting the shared interface into module C"). Writing error messages that teach complex fixes requires significant upfront design effort — understanding the space of possible violations, crafting messages that generalize across cases, and maintaining them as the codebase changes. For simple, rule-based checks (missing license header), the cost is indeed negligible. For violations that require contextual judgment to explain, the "cheap" claim understates the authoring and maintenance cost. The note's own example (license header) sits at the easy end of this spectrum.

- [Grounding alignment] The note claims teaching errors are "a form of frontloading" and links to the frontloading note. The frontloading note defines frontloading as "pre-computing static parts of LLM instructions whose inputs are known before the LLM runs." Teaching error messages are not pre-computed before the LLM runs — they fire at runtime, in response to the agent's output. The connection is that both replace derivation with insertion (the agent gets the answer instead of deriving it), but the temporal property that defines frontloading (static inputs known before runtime) does not hold for error messages, which are triggered by runtime behavior. The note's framing is "teaching error messages are a form of frontloading" — the relationship is more accurately "teaching error messages share the mechanism of frontloading (replacing derivation with insertion) but apply it reactively rather than proactively."

- [Internal consistency] The intro states the gradient "treats enforcement and instruction as separate concerns," which the note then challenges. However, the linked methodology-enforcement-is-constraining note already describes a gradient where each layer has both a trigger and a response, and the response at higher layers (skills, warning hooks) is explicitly "indeterministic (LLM interprets)" — meaning the gradient already acknowledges that the LLM receives information from enforcement output and acts on it. The "separate concerns" framing slightly overstates the gap the note fills: the enforcement gradient does not treat enforcement as purely blocking; it already models the agent interpreting enforcement output. The note's actual contribution (the orthogonal inform axis, the table of constrain+inform variants) is real, but the motivating framing ("the gradient treats these as separate") is somewhat stronger than the gap warrants.
