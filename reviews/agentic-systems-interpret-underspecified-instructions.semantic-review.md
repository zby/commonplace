Claims identified: 17

WARN:
- [Grounding alignment] Ma et al. cited as "strongest empirical evidence for the two-phenomena separation" but the paper does not frame itself in terms of separating underspecification from indeterminism. Ma et al. study *prompt stability* — whether semantically equivalent but differently phrased prompts produce different outputs. The note maps this onto its own framework ("isolating underspecification (which interpretation?) from indeterminism (which run?)"), but the paper's actual finding is that emotion/personality variations cause performance changes, which could equally be read as interpreter failure (the sibling note interpretation-errors-are-failures-of-the-interpreter.md already cites the same paper as evidence for *interpretation errors*, not underspecification). The note's two-phenomena framing treats prompt sensitivity as evidence for underspecification, but a competing reading is that the spec is fully specified (same functional requirements) and the LLM is simply responding incorrectly to irrelevant stylistic cues — a failure of the interpreter, not ambiguity in the spec. The paper itself does not adjudicate between these readings.
- [Completeness] The two-phenomena model claims to identify the ways "LLM-based systems differ from traditional programs." Boundary case: a traditional program given an ambiguous configuration file (e.g., a YAML config with a field that could be interpreted multiple ways) also faces semantic underspecification, and a traditional program using randomized algorithms also exhibits execution indeterminism. The note's implicit claim is that these phenomena are *characteristic* of LLM systems in a way they are not of traditional ones, but the note defines both phenomena in terms that are not exclusive to LLMs. The note partially addresses this by saying underspecification is "a property of the specification language itself" — but traditional systems also accept natural language specs (requirements documents) and face the same issue. What is distinctive is that LLMs operate *at runtime* on underspecified specs, whereas traditional systems resolve ambiguity at design time. The note gestures at this but never makes the temporal distinction explicit.
- [Grounding alignment] The note claims ABC "contracts resolve semantic underspecification with formal YAML DSL; probabilistic compliance model (p,delta,k) quantifies how tightly a contract narrows the interpretation space." The ABC ingest file confirms contracts address underspecification and indeterminism, but the ABC paper's (p,delta,k) model is about *compliance probability* and *recovery from violations*, not about narrowing the interpretation space per se. The note reinterprets compliance-as-narrowing, which is a reasonable inference but not what the source directly claims. The ABC paper frames contracts as specifying behavioral boundaries and measuring how often the agent stays within them — this is enforcement, not interpretation-space narrowing in the sense the note uses.

INFO:
- [Completeness] The seven design implications (section "Design Implications") are presented as what the framework "suggests" rather than as a closed enumeration, so there is no strong completeness claim. However, a notable omission is any implication about *monitoring and observability*. Given that the note emphasizes boundaries as "natural checkpoints" and discusses debugging at length, a design implication about instrumenting boundaries for observability seems implied by the framework but absent from the list.
- [Completeness] The note's boundary model ("LLM components have both [phenomena], code has neither") is clean but boundary case: structured output with JSON schemas. When an LLM generates JSON constrained by a schema, the output is partially determined by the schema (reducing underspecification) and partially by the LLM (retaining both phenomena for unconstrained fields). This is a boundary that is neither fully LLM nor fully code — it sits between the two poles. The note's binary model (LLM = both, code = neither) does not account for this intermediate regime, though the "Narrowing the Interpretation Space" section implicitly acknowledges schemas as a narrowing mechanism without connecting this back to the boundary model.
- [Internal consistency] The note says indeterminism is "largely eliminable via temperature=0" in the Two Distinct Phenomena section, then later says "implementation details (floating-point non-determinism, batching, infrastructure changes) mean true determinism is hard to guarantee in practice." These are not contradictory but create tension — the first framing suggests temperature=0 mostly solves it, the second walks it back significantly. The note then says "All deployed systems exhibit indeterminism and often benefit from it," which further complicates the "largely eliminable" claim. The progression is defensible as a rhetorical narrowing (conceptually eliminable, practically not), but a reader could be confused about whether the note considers indeterminism practically eliminable or not.
- [Grounding alignment] The compiler analogy ("A compiler performs a semantics-preserving translation... An LLM performs a projection") is crisp but may overstate the precision of compilation. Real compilers make optimization choices that are not uniquely determined by the source — different compilers or optimization levels produce different executables from the same source. The analogy holds at the level of *semantic preservation* (the compiled program has the same behavior), but could mislead readers into thinking compilation is a uniquely determined mapping, which it is not.
