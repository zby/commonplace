WARN:
- [Completeness] The enumeration of "Practices we apply" lists five items (Typing, Progressive compilation, Testing, Version control, Design for testability), but the note's title and opening claim frame this as a general transfer — "programming practices apply to prompting." Several standard programming practices are absent from the enumeration yet arguably transfer: abstraction/modularity (composing prompts from reusable parts), code review (reviewing prompts for ambiguity), dependency management (managing which context loads which context), refactoring (restructuring prompts without changing behavior). The note's own Open Questions section acknowledges "Code review for prompts? Dependency injection for context? Refactoring patterns?" — which suggests the author knows the list is incomplete. However, the note presents its five practices as "Practices we apply" without qualifying "these are representative, not exhaustive." A reader could mistake this for a claimed complete enumeration, especially since the section heading uses the definite form. Boundary case: abstraction/modularity is arguably the most fundamental programming practice and its omission is notable given the KB itself relies heavily on composable notes.

- [Completeness] The claim "Indeterminism doubles the test runs" and "Underspecification doubles the test targets" frames the testing burden as a clean 2x multiplier along two orthogonal axes. But the note also says "The two phenomena compound," which implies the combined effect is multiplicative (4x), not additive (2x). The "doubles" framing in the sub-headings is individually correct for each axis, but the compounding claim means the total testing burden is not just "doubled" in any single sense — it is quadrupled if you take both dimensions seriously. The note states this compounding but does not reconcile it with the "doubles" framing, which could mislead a reader who reads the headings without the body.

- [Grounding — domain coverage] The note claims "constraining is projection from an underspecified spec — the natural-language specification admits multiple valid implementations, and committing to code means choosing one interpretation." The linked source (agentic-systems-interpret-underspecified-instructions.md) carefully distinguishes projection (spec-to-program) from compilation (semantics-preserving translation) and notes "An LLM performs a projection: it collapses a space of valid interpretations to one concrete program." But the source's projection model describes what the LLM does when interpreting a spec, not what the human/system does when constraining. When a developer observes a stable LLM behavior pattern and extracts it into code, they are not performing a projection from an underspecified spec — they are codifying an observed interpretation. The note collapses two different acts (LLM projecting from a spec, and developer extracting an observed pattern) under the single term "projection." The source does not endorse this collapse.

INFO:
- [Completeness] The claim "In deterministic code there's no gap between what the code says and what it does, so you only test outputs" is a simplification. Deterministic code can have bugs where the code does not match the programmer's intent — the gap between specification and implementation exists in traditional programming too (it is just that the specification is usually informal/mental rather than written). The note's point holds at the level of execution (same input always gives same output), but the broader framing that "you only test outputs" elides specification testing in traditional software (e.g., requirements validation, acceptance testing). The claim is approximately right for unit testing but overstated for the full software testing practice.

- [Grounding — vocabulary] The note says "the verifiability gradient maps the spectrum from prompt tweaks through evals to deterministic modules" and links to deploy-time-learning-the-missing-middle.md. The linked source calls this "The Verifiability Gradient" and frames it as grades of hardening (restructured prompts, structured output schemas, prompt tests/evals, deterministic modules). The note's phrasing "from prompt tweaks through evals to deterministic modules" compresses four grades into three and replaces "restructured prompts" with "prompt tweaks" — a minor vocabulary mismatch that slightly shifts the meaning (a "tweak" suggests a small change; "restructuring" suggests a significant reorganization). The inference is reasonable but the attribution is not exact.

- [Internal consistency] The note says "Execution indeterminism is genuinely novel; underspecification is not" in the Hard Cases section. But the Typing and Version Control sections treat underspecification as the more interesting and difficult phenomenon — the one that requires novel practices (testing the spec itself, not just the outputs; versioning specs because regeneration is a new projection). If underspecification "is not novel" because legal drafting has centuries of methodology for it, then the practices addressing it should also be familiar — yet the note presents them as insights. The tension is resolvable (the practices are familiar in legal drafting but novel when applied to programming/prompting), but the note does not make this resolution explicit, leaving an apparent contradiction between "underspecification is not novel" and the note's own treatment of underspecification-driven practices as the more interesting contribution.

The source makes a more nuanced claim: "Persistent memory across sessions would relax the statelessness assumption, but routing scales with the knowledge base itself. Even an agent that remembers where it found things last time faces a navigation problem proportional to KB size." The source acknowledges that statelessness could be partially relaxed by persistent memory; the note's "permanently stateless" is stronger than what the source asserts.
