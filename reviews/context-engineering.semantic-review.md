WARN:
- [Completeness] The four-component decomposition (routing, loading, scoping, maintenance) claims to be the "operational core" of context engineering "within a single bounded call." A boundary case that strains this framework is **prioritization / attention management** -- deciding not just *what* to include (routing) or *how* to assemble it (loading), but *what emphasis or ordering to give the included material*. The note hints at this in the Loading section ("how to frame it"), but framing is treated as a sub-concern of loading rather than a first-class operation. The linked bounded-context orchestration model note treats framing as a distinct concern: "the same knowledge, presented differently, has different value to a bounded observer" appears under "What makes selection hard" as its own named problem ("Framing matters, not just selection"), separate from the selection/loading problem. If framing is important enough to be called out separately in the orchestration model, folding it into "loading" in this definitional note may undercount the operational components. A reader could reasonably expect a fifth component ("framing" or "arrangement") to sit between loading and scoping.

- [Completeness] The architectural scope section enumerates five concerns beyond a single call: storage format, knowledge lifecycle, session boundaries, inter-agent communication, and tool/interface design. A boundary case is **evaluation and feedback** -- how the system determines whether its context engineering decisions were good. The note's linked sources address this: the bounded-context orchestration model includes a `satisfied(K)` check in its loop, and the context-efficiency note mentions that "context efficiency should be evaluated at design time." But the architectural scope section here is silent on evaluation as a concern. A system that does routing, loading, scoping, maintenance, and all five architectural concerns but has no feedback on whether its context choices worked is arguably incomplete as a context engineering system. This may be intentional scoping (the note is a definition, not a complete theory), but the scope claim "the scope is wider than prompt assembly" invites the question of where evaluation falls.

- [Grounding alignment / domain coverage] The note claims its treatment is "consistent with [Anthropic's] operational view but broader in architectural scope, because context efficiency is the central design concern." The linked note (context-efficiency-is-the-central-design-concern-in-agent-systems.md) does argue that context scarcity makes the discipline architectural. But the justification chain has a gap: the Anthropic definition ("strategies for curating and maintaining the optimal set of tokens during LLM inference") is about **token curation during inference**, while the architectural extensions in this note include storage format, knowledge lifecycle, and session boundaries -- concerns that operate *between* inferences, not during them. The note presents the Anthropic definition as the narrow view it extends, but does not explicitly acknowledge that the extension moves from an inference-time discipline to a system-lifetime discipline. This is the note's own move, not something the cited Anthropic source or the context-efficiency note claims. Readers could mistake the Anthropic citation as endorsing the full architectural scope.

INFO:
- [Completeness] The simplest possible instance of context engineering is a single prompt with no routing, no scoping, and no maintenance -- just a human writing one prompt for one LLM call. Under the note's framework, this is "loading" alone. The framework handles this gracefully (three of four components are null), but it raises the question: at what point does assembling a prompt become context *engineering*? The note's title claim ("architectural discipline") implies there must be architecture, which a single prompt lacks. This is a terminological boundary, not a framework failure, but the note does not draw the line.

- [Completeness] The maintenance component is described as "keeping loaded context healthy over time." The examples given (compaction, observation masking, holistic-rewrite) all target long-running sessions or accumulated state. A boundary case is **proactive invalidation** -- detecting that previously loaded context has become stale or contradicted by new information mid-session. This could be classified as maintenance, but the note's examples all describe post-hoc cleanup rather than real-time monitoring. The distinction may matter for systems where the environment changes during a session.

- [Grounding alignment] The note says loading includes "the same knowledge under different framing has different extractable value" and links to information-value-is-observer-relative.md. That note discusses "extractable structure" and why the same content in different arrangements "can teach more." However, the information-value note frames this in terms of "observer-relativity" -- the value depends on the observer's priors, capacity, and goals. The context-engineering note simplifies this to "different framing has different extractable value," dropping the observer-relativity framing. This is a reasonable simplification for a definitional note, but the source's central point is that value is observer-relative (not just framing-relative), which is a stronger and more specific claim than what the context-engineering note attributes.

- [Grounding alignment] The note claims distillation is "the main operation these components perform, but not the only one." The linked distillation note says the same thing verbatim: "Distillation is the main operation that architecture performs, though not the only one (routing, scoping, and maintenance are also context engineering operations)." The attribution is accurate, but both notes leave vague *what else* the components do besides distillation. The distillation note's parenthetical lists routing, scoping, and maintenance as non-distillation operations, but these are components of context engineering, not operations those components perform. There is a category slip: components (routing, scoping, maintenance) are being listed as alternative operations to distillation, but routing is not an operation in the same sense as distillation -- routing is a component that *performs* operations. Both notes share this ambiguity.

- [Internal consistency] The note says the four-component decomposition is "within a single bounded call," and the architectural scope section covers concerns "beyond a single call." This two-level structure is clean. However, the maintenance component blurs the boundary. Maintenance is described as "keeping loaded context healthy over time" with examples including "compaction" and "observation masking" -- both of which operate across the lifetime of a conversation, not within a single call. If maintenance is within-call, it would mean managing context within one prompt assembly. If it is across-calls, it belongs in the architectural scope section. The note places it in both locations implicitly (within the four components, and implicitly under "knowledge lifecycle" in the architectural section), which creates ambiguity about whether maintenance is a single-call or multi-call concern.

That note does argue that "sub-agents are the one place where real isolation is achievable" and describes them as "lexically scoped frames," directly supporting the context-engineering note's claim that sub-agents provide scoping.

The degradation-at-low-token-counts claim is supported by the linked note's argument about complexity being a separate dimension from volume.

The two levels are presented as complementary, and no claim from one level conflicts with a claim from the other.
