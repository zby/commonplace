=== SEMANTIC REVIEW: stale-indexes-are-worse-than-no-indexes.md ===

Claims identified: 9

1. "When an index exists and is incomplete, the agent reads it, feels oriented, and stops looking." (Opening paragraph)
2. "Absence of an index degrades to search; presence of a stale index suppresses search entirely." (Core asymmetry)
3. "The mechanism generalises beyond indexes to any authoritative artifact — specs, documentation, plans, curated lists." (Generalization paragraph)
4. "The most common staleness is a new note that doesn't get added to a relevant index." (Critical moment section)
5. "The `areas:` frontmatter field is the first defense — it declares index membership at creation time." (Defenses section)
6. "`docs/indexes.md` serves this purpose — a single file listing all indexes with descriptions." (Defenses section)
7. "The /connect skill's Phase 5 reads `docs/indexes.md` and checks each index against the note." (Defenses section)
8. "For every note with `areas: [X]`, verify it appears in the X index. This check is automatable." (Defenses section — deterministic check)
9. "This observation draws on arscontexta methodology research ('stale navigation actively misleads because agents trust curated maps completely')." (Sourcing section)

WARN:
- [Completeness] The note claims the mechanism "generalises beyond indexes to any authoritative artifact — specs, documentation, plans, curated lists." The grounding definition is implicit: any artifact an agent treats as exhaustive will suppress fallback discovery when stale. Boundary case: **a non-exhaustive but authoritative artifact** — e.g., a style guide that is clearly partial ("here are some conventions") rather than exhaustive. Does a stale style guide suppress search in the same way? The note's mechanism depends on the agent treating the artifact as exhaustive, but "authoritative" and "exhaustive" are conflated. A spec can be authoritative without being treated as a complete enumeration. The note says "any artifact that an agent treats as exhaustive" but then lists "specs, documentation, plans, curated lists" — not all of which are typically treated as exhaustive enumerations. A plan that goes stale might mislead about direction without suppressing search for additional tasks. The suppression mechanism is strongest for navigational artifacts (indexes, curated lists) and weakest for directional artifacts (plans, specs). The generalization may overextend.

- [Grounding] Claim 6: "`docs/indexes.md` serves this purpose — a single file listing all indexes with descriptions." No `docs/indexes.md` file exists in the repository. The note presents this as a current, operational defense mechanism using present tense ("serves this purpose"), but the file has never existed at this path (or has been removed without trace). The /connect skill (SKILL.md) does not reference `docs/indexes.md` in its Phase 5 — Phase 5 is a reflection step that logs observations to `kb/log.md`, not an index-checking step. Claim 7 ("The /connect skill's Phase 5 reads `docs/indexes.md`") is therefore doubly incorrect: Phase 5 does not check indexes, and the file it supposedly reads does not exist.

- [Grounding] Claim 5: "The `areas:` frontmatter field is the first defense." The `areas:` field no longer exists anywhere in the KB. ADR 004 (status: proposed) replaced `areas:` with `tags:`. Every note in `kb/notes/` now uses `tags:`, not `areas:`. The defense mechanism described here is a historical artifact. Additionally, claim 8's deterministic check ("For every note with `areas: [X]`, verify it appears in the X index") references the same superseded field. The entire Defenses section describes infrastructure that does not match the current system.

- [Grounding] Claim 9 attributes the core observation to "arscontexta methodology research" with the quoted phrase "stale navigation actively misleads because agents trust curated maps completely." The arscontexta directory no longer exists in the repository (acknowledged as stale in `kb/notes/related-systems/arscontexta.md`). The arscontexta related-system note does not contain the quoted phrase or a paraphrase of it. The closest related concept found in the KB is the "confidence trap" mechanism identified in `kb/sources/what-spec-driven-development-gets-wrong-2025993446633492725.ingest.md`, which describes "authoritative artifacts suppress fallback discovery when stale" — but that ingest report was written *after* this note and cites this note, not the other way around. The attribution to arscontexta methodology research is unverifiable from within the current KB. The quoted claim may be accurate to a source that once existed locally but is no longer available.

INFO:
- [Completeness] The note claims "the most common staleness is a new note that doesn't get added to a relevant index." This is a scope claim ("most common") without evidence. Other staleness modes include: (a) an existing index entry whose linked note has been substantially revised so the context phrase no longer applies, (b) an index whose organizational groupings no longer reflect the current state of the topic, (c) an index that includes notes that have been deleted or renamed (broken links). The note focuses exclusively on the omission case (missing entry) and does not acknowledge these other modes. Whether omission is truly the "most common" case is an empirical claim that could go either way depending on the KB's lifecycle stage.

- [Completeness] Boundary case: **a generated index** (as ADR 004 proposes with its "All notes" generated section). A generated index is never stale by definition — it is rebuilt from tags. The note's core asymmetry holds only for manually curated indexes. ADR 004's generated sections are a direct architectural response to this note's problem, yet the note does not distinguish curated from generated indexes. This is worth noting because ADR 004 explicitly cites this note as motivation, meaning the system has already evolved a defense the note doesn't describe.

- [Internal consistency] The note's Sourcing section says it "draws on arscontexta methodology research... adapted to our specific system where `areas:` frontmatter and `docs/indexes.md` provide the defense mechanisms." This frames the note as an adaptation of external research to local infrastructure. But the core claim (the asymmetry between absent and stale indexes) is a general observation about agent behavior that does not depend on the local system's specific mechanisms. The Sourcing section undervalues the note's own contribution by framing the general insight as merely an adaptation, while simultaneously overspecifying the local defenses that no longer exist.

PASS:
- [Internal consistency] The core argument is internally consistent. The opening paragraph establishes the mechanism (stale index satisfies navigation need, suppressing search). The "core asymmetry" sentence correctly summarizes this. The generalization to authoritative artifacts follows logically from the identified mechanism. The "critical moment is note creation" section correctly identifies the point where staleness is introduced. No pairwise contradictions between sections.

- [Internal consistency] The title claim ("stale indexes are worse than no indexes") is faithfully represented throughout the body. The note does not silently weaken or strengthen this claim — it maintains the comparison consistently and the generalization paragraph extends it without contradicting it.

- [Grounding] The link-graph-plus-timestamps note (`kb/notes/link-graph-plus-timestamps-enables-make-like-staleness-detection.md`) correctly cites this note as "foundation: the high cost of stale navigation that motivates detection." That note's characterization accurately reflects this note's core claim. The kb-maintenance-index also accurately summarizes this note: "a missing index entry suppresses search entirely; absence of an index degrades to search, presence of a stale index prevents it." Both downstream consumers use this note faithfully.

- [Completeness] The core asymmetry claim — absence degrades to search, presence suppresses search — holds under boundary testing. Simplest case: an index with one missing entry out of fifty. The agent reads the index, gets 49 entries, and has no reason to search for the 50th. The mechanism holds. Most extreme case: an index that is entirely out of date with all entries stale. The agent reads it, gets wrong orientation, and operates on entirely incorrect information. The mechanism holds and is even stronger — not just suppressed search but actively misdirected navigation.

Overall: 4 warnings, 3 info
