WARN:
- [Completeness] The note claims agile's "implied end state is fully codified software" and that "agile implicitly assumes everything can be codified if you iterate enough" (claims 6, 8). This overstates the agile position to sharpen the contrast with deploy-time learning. Agile's own literature (e.g. the Agile Manifesto's emphasis on "working software over comprehensive documentation" and "responding to change") does not claim that all specs eventually become code. Agile accepts permanent backlogs, evolving acceptance criteria, and stories that never get implemented. The "code wins eventually" framing is the note's own construction, not a claim grounded in agile sources. The contrast with deploy-time learning would still hold without this overstatement — the difference is that deploy-time learning explicitly theorizes which parts should stay in prose and why, while agile is largely silent on the question.

- [Completeness] The five-step learning loop (claim 4) omits a step present in both agile and deploy-time learning: **evaluate/validate** between "codify" and "repeat." In agile, the sprint review and retrospective check whether the codified work actually resolved the ambiguity it was meant to resolve. In deploy-time learning, the verifiability gradient (described in the parent note deploy-time-learning-the-missing-middle.md) is precisely about checking codified artifacts against reality. Without an explicit evaluation step, the loop implies that codification is always correct on the first pass, which both frameworks explicitly deny (agile has rework; the parent note lists "premature codification" as a failure mode).

- [Grounding — domain coverage] The note's central analogy (agile ↔ deploy-time learning) is the note's own synthesis. None of the three linked sources ground this analogy — they each take it as given and build on it. The parent note (deploy-time-learning-the-missing-middle.md) briefly mentions the agile parallel in its "Related Work" section with a back-link to this note, but does not independently argue for the analogy. The changing-requirements note takes the analogy as a premise ("Deploy-time learning extends agile's loop"). The specification-strategy note references it as context. This means the analogy is self-grounding: the note asserts it, linked notes cite it, and no external source independently validates the structural mapping. This is not necessarily wrong — the analogy may be original to this KB — but a reader following the links expecting independent validation will find circular reference instead.

INFO:
- [Completeness — boundary case] The note frames the media pair for agile as "specs and code" and for deploy-time learning as "prompts and code." But in practice, deploy-time learning systems contain more than two media: there are also structured schemas, eval suites, configuration files, and documentation (as the parent note's verifiability gradient table shows). The binary framing (prose vs code) is clean but potentially masks a spectrum. The note's own data-report example partly acknowledges this (statistics move to Python, narrative stays with LLM), but the framework vocabulary stays binary.

- [Completeness — boundary case] The three-stage progression (waterfall → agile → deploy-time learning, claim 10) presents a linear evolution. A boundary case: what about formal verification applied to AI systems? The note mentions formal methods only as a separate dead end ("tried to solve it by abandoning natural language for mathematics"), but formal methods applied to LLM systems (e.g. behavioral contracts, structured output schemas) are alive and well within deploy-time learning itself. The progression might be better understood as branching rather than linear, with formal methods contributing tools that re-enter at the deploy-time learning stage.

- [Grounding alignment] The link to changing-requirements-conflate-genuine-change-with-disambiguation-failure.md is accurately attributed. The note says agile's "changing requirements" "partly conflates genuine change with late-surfacing disambiguation failures," and the linked note does argue exactly this. The hedging word "partly" is the note's own addition (the linked note uses "conflates" more directly), but this is a reasonable softening, not a misattribution.

- [Grounding alignment] The link to specification-strategy-should-follow-where-understanding-lives.md is described as "extends: splits prose-code co-evolution into spec-first, bidirectional, and behavior-extracted phases." This is accurate — the linked note does exactly this. Notably, the linked note also pushes back on the binary framing: "Co-evolution is real, but the right prose strategy depends on when the system can say something trustworthy." This tension is not reflected back into the current note, which stays with the simpler binary model.

The note itself contains no data report example — this refers to an example in the parent note (deploy-time-learning-the-missing-middle.md). The reference is accurate but assumes the reader has read the parent note. This is a minor composability concern rather than an inconsistency.
