=== SEMANTIC REVIEW: rlm-has-the-model-write-ephemeral-orchestrators-over-sub-agents.md ===

Claims identified: 12

WARN:
- [Completeness] The mapping table claims four correspondences between model components and RLM implementation, but omits `call()` — the bounded-context orchestration model's loop is `select(K) -> call(P) -> K+r`. The table maps K, select, and the inner scheduler, but there is no row for `call`. In RLM, `call` would be the REPL execution engine (or the LLM API invocation that `recursive_llm()` wraps). The omission is notable because the table's implicit claim is a complete mapping ("maps directly onto the symbolic scheduler model"), yet a core loop component is absent.

- [Completeness] The note claims "RLM has two layers of symbolic orchestration" — the outer tool loop and the inner REPL-authored orchestrators. But the source ingest identifies at least five distinct RLM mechanisms: (1) persistent REPL with `context` variable indirection, (2) programmatic exploration via print/regex/slicing, (3) scaffold-level output truncation, (4) subagent results returned as Python variables, and (5) variable return bypassing autoregressive generation. The note discusses (1) and (4) but does not mention (2), (3), or (5). The "two layers" framing accounts for the architectural shape but not for several mechanisms that make RLM work in practice — mechanisms the note's own cited source describes in detail. A reader relying solely on this note would miss that RLM's effectiveness depends on concrete REPL affordances beyond just "the model writes code."

- [Grounding — domain coverage] The note's central claim is that RLM "maps directly onto the symbolic scheduler model." The bounded-context orchestration model note does accommodate this: "The model also accommodates architectures where the LLM emits a symbolic control program rather than a direct natural-language answer." However, the orchestration model assumes the scheduler has "unbounded exact state," while in RLM the scheduler is ephemeral code that the model writes fresh each time — the scheduler itself is stochastic (different model calls may produce different orchestrators for the same problem). The mapping is structural (the REPL namespace is exact state during execution) but the note's "maps directly" language elides the fact that the scheduler's *construction* is stochastic, which the clean model does not address. The multi-dimensional design space note correctly identifies this as a "boundary case," but the note under review uses "maps directly" rather than acknowledging the boundary.

INFO:
- [Completeness] The note's boundary-case probe: what about an RLM orchestrator that has side effects? The note says "If the REPL is restricted to pure computation with no side effects, the approval problem becomes much simpler." This is presented as a conditional, which is accurate. But the note does not explore what happens when the restriction is lifted — the source ingest notes that RLM implementations use sandboxed REPLs (Pyodide inside Deno.js), but sandboxing is an implementation detail, not a guarantee. A reader might want to know whether RLM architectures in practice enforce the pure-computation restriction or merely encourage it.

- [Grounding — vocabulary] The note says RLM "avoids the degraded scheduler failure mode where bounded context is wasted on bookkeeping." The degraded-scheduler note defines three recovery strategies (compaction, externalisation, factoring into code). RLM's inner orchestrator most closely matches "factoring into code" (strategy 3) — the LLM emits a program that runs outside the conversation. But the note does not position RLM on this recovery spectrum. This is not inaccurate, but a reader familiar with the degraded-scheduler note might expect the connection to be made explicit.

- [Internal consistency] The Ephemerality section says "it's possible that accumulation will come through other paths" and lists three speculative mechanisms: improved model capabilities, strategies learned in weights, and mining orchestrators from execution logs. The third mechanism (mining from logs) is described as "an out-of-band process could gather the generated orchestrators together with their prompts and results, and distill recurring patterns into reusable knowledge." This would effectively convert ephemeral computation into accumulated artifacts — which is precisely what the note says RLM "opts out of." The note frames this as a possible future path rather than a contradiction, which is fair, but the tension could be made more explicit: log-mining would mean RLM is not inherently ephemeral but contingently so (because current implementations discard the artifacts, not because the architecture requires it).

- [Completeness] The note claims "The model-authored orchestrators are structurally the same thing a programmer would write by hand." This equivalence claim is plausible for simple cases (list comprehensions over `recursive_llm()`), but breaks down at the boundary: a programmer writes orchestrators with error handling, retries, timeouts, and logging; model-authored REPL code typically does not. The source ingest flags this gap: "what happens when the model writes buggy code? How does error recovery work?" The structural equivalence holds for the happy path but not for the full engineering surface.

- [Grounding — scope] The note says the repo-as-learning-substrate approach carries "real costs (approval complexity, maintenance burden, the risk of codifying vision features)" and cites the codification note. The codification note does discuss the risk of codifying things that should remain flexible, but "approval complexity" is the note's own addition — neither the codification note nor the deploy-time-learning note uses that specific phrase. The inference is reasonable (persistent artifacts do require approval) but is not directly sourced.

The first section praises the architectural elegance; the second identifies the cost. The framing "This is a genuine trade-off, not a deficiency" explicitly bridges the two, preventing the common failure where a note's positive section contradicts its limitations section.
