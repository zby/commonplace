=== SEMANTIC REVIEW: mcp-bundles-stateless-tools-with-stateful-runtime.md ===

2. "Most tool operations are pure functions" (opening paragraph)
3. "every tool pays the complexity cost of state management whether it needs state or not" (opening paragraph)
4. Four-item enumeration of state costs: lifecycle management, test isolation, connection management, concurrency ("The state tax")
5. "The difficulty of writing isolated tests is a signal that the architecture is mixing concerns" ("The state tax")
6. "The typical MCP tool surface -- read a file, search for content, run a command, fetch a URL -- is stateless" ("Most tools don't need state")

11. Generalisation: "when an architecture bundles a simple common case with a complex rare case under a single abstraction, the common case pays complexity rent" ("The economic argument")
12. "The economic alternative: stateless tool invocation as the default, where the tool is a function call with no server process" ("The economic argument")

WARN:
- [Completeness] The four-item "state tax" enumeration (lifecycle, test isolation, connection management, concurrency) omits **configuration and deployment complexity** -- a stateful server requires configuration (ports, transport, environment), deployment artifacts, and operational monitoring that a bare function call does not. This is arguably the most visible cost to practitioners and sits outside the four listed items. It is not reducible to "lifecycle management," which the note defines narrowly as "start, stay alive, handle crashes, and clean up."
- [Completeness] The enumeration of legitimate stateful cases ("authorisation... connection pooling... caches... streaming results") omits **shared conversational/session context** -- MCP's resources and prompts features are designed to let a server expose context that persists across tool calls within a session (e.g., a database server tracking which database is selected, or an IDE server tracking the open file). This is a genuinely stateful capability that many MCP servers use and that is distinct from authorisation or caching. The note's own framing -- "No session context needed between calls" -- implicitly defines the space that should include this case.
- [Grounding alignment] The note claims "This is the files-not-database argument applied to the tool layer," but the analogy is strained. The files-not-database note argues that files earn simplicity specifically because they provide versioning, browsing, agent access, and deferred schema commitment -- none of which map to stateless function calls. The database side of the analogy (transactions, concurrent access, indexes) also does not map to what MCP servers provide (discovery, transport, session state). The structural parallel is "simple default vs. complex default," but the note presents it as the *same argument* rather than an analogous one. The source supports the abstract pattern but not the specific claim of identity.

INFO:
- [Completeness] The note's characterisation of "the typical MCP tool surface" as "read a file, search for content, run a command, fetch a URL" is a reasonable sample but may undercount the proportion of tools that benefit from state. MCP servers for databases, APIs with pagination, multi-step workflows, and IDE integration commonly maintain session state. The note acknowledges legitimate stateful cases, but the framing "most tools don't need state" is an empirical claim presented without evidence. Whether "most" is accurate depends on the distribution of deployed MCP servers, which is not cited.
- [Grounding alignment] The link to "ephemeral computation prevents accumulation" is annotated: "stateless tool invocation is ephemeral by design; MCP's persistent server pushes toward accumulating state whether or not the tools need it." But the ephemeral computation note is about whether *generated artifacts* are kept or discarded across runs, not about whether the runtime process is persistent. An MCP server can be stateful (persistent process) while its tools are ephemeral (no artifacts retained). The link annotation conflates process persistence with artifact accumulation -- a vocabulary mismatch, though not a severe one since the analogy is loose rather than claiming direct grounding.
- [Grounding alignment] The link to "indirection is costly in LLM instructions" is annotated: "same principle: indirection through a server process costs overhead on every call, unlike code where indirection is nearly free." But the indirection note is specifically about *interpretive overhead in the LLM's context window* -- variables, abstraction layers, and conditional paths that cost tokens. MCP server indirection is a runtime architectural cost (latency, failure modes, process management), not a context-window interpretation cost. These are different kinds of overhead. The note's annotation implies they are the "same principle," which overstates the connection.
- [Internal consistency] The note states Claude Code's tools are "direct function calls with no intermediate server, no session, no lifecycle." This is offered as the economic alternative to MCP. However, Claude Code's tools run within a host process that manages their invocation, handles permissions, maintains conversation state, and orchestrates tool calls -- which is itself a form of runtime with lifecycle and session management. The distinction is that the *tools themselves* are stateless, but they still operate within a stateful harness. The note's framing could be read as claiming the tools exist in isolation, which would be slightly misleading. The point about tool-level statelessness is valid; the characterisation of "no lifecycle" is an overstatement of the actual architecture.

PASS:
- [Internal consistency] The note's three sections build a coherent argument: (1) state imposes costs, (2) most tools don't need state, (3) therefore bundle the simple case separately. No section contradicts another, and the definitions are used consistently throughout.
- [Internal consistency] The term "stateless" is used consistently to mean "takes input, returns output, no session context between calls." There is no definition drift.
- [Completeness] The generalisation in the final paragraph -- "when an architecture bundles a simple common case with a complex rare case under a single abstraction, the common case pays complexity rent" -- is well-supported by the preceding argument and correctly identified as a pattern rather than a fact specific to MCP.
- [Grounding alignment] The link to "files beat a database for agent-operated knowledge bases" accurately identifies the structural parallel at a pattern level (simple default vs. complex default, with the complex case earning its keep only for specific capabilities). The source note does make this argument, and the connection is legitimate despite the analogy strain noted above.
- [Grounding alignment] The link to "context efficiency is the central design concern" is annotated: "unnecessary intermediate layers add latency and failure modes without improving the agent's context." The context-efficiency note does argue that unnecessary layers degrade agent performance. This connection is accurate and appropriately scoped.

Overall: 3 warnings, 4 info
