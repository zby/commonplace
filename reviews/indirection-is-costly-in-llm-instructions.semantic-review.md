=== SEMANTIC REVIEW: indirection-is-costly-in-llm-instructions.md ===

Claims identified: 13

WARN:
- [Completeness] The note lists three costly indirection patterns (config variables, abstraction layers, conditional paths) as "indirection patterns that are free in code become expensive in prompts." This is presented with a colon and bullet list that implies coverage of the space, yet several common indirection patterns in instructions are missing: **reference indirection** (e.g., "follow the procedure in WRITING.md" rather than inlining the procedure) and **naming indirection** (defining a term in one place and using it elsewhere, requiring the LLM to maintain the definition). Reference indirection is arguably the most prevalent form in this KB's own CLAUDE.md, which routes rather than inlines. It is not clearly a subcase of "config variables" or "abstraction layers." The note's framing implies a catalog but delivers an incomplete one.

- [Completeness] The "Where the boundary is" section gives three conditions for when indirection should be eliminated: (1) value known at setup time, (2) instruction read frequently, (3) LLM must act on the value. These are presented conjunctively ("The principle applies when:") but the note does not clarify whether all three must hold or any subset suffices. The third condition -- "The LLM must act on the value in commands or paths, not just understand it conceptually" -- introduces a distinction between acting and understanding that the rest of the note does not develop. Is an abstraction layer like "use the appropriate search command" (from the bullet list) something the LLM "acts on" or "understands conceptually"? The abstraction-layer example from the bullet list seems to violate condition 3 (it is not a path or command value) yet the note clearly marks it as costly. This creates a tension between the bullet list and the boundary conditions.

INFO:
- [Completeness] The note's central analogy -- "In traditional programming, a variable lookup or config reference costs essentially nothing at runtime" -- is true for the single-lookup case but elides the well-known costs of indirection in traditional programming: cache misses, pointer chasing, virtual dispatch overhead, and the architectural complexity the note itself acknowledges in the same sentence. The note does note "architectural complexity" as a cost, but presenting runtime cost as "essentially nothing" is a simplification. This is a minor point since the note is making a relative comparison, not an absolute claim, and the simplification serves the argument. Still, a reader familiar with systems programming might find the claim too strong.

- [Grounding] The link to frontloading-spares-execution-context claims "indirection elimination is a specific case of frontloading." The frontloading note does list "Variable resolution -- paths, project names, configuration values known at setup time (the indirection elimination case)" as one of its frontloadable items, and explicitly links back to this note. The attribution is accurate. However, the relationship claim is slightly asymmetric: this note says frontloading "generalises" indirection elimination, implying indirection elimination is strictly a subset. The frontloading note frames it more carefully as one of several cases. The relationship holds but the "generalises" framing could overstate the subsumption -- indirection cost includes the interpretation-overhead argument (LLM must "maintain the mapping in working memory") which is not purely about context saving (frontloading's focus) but also about reliability. The frontloading note focuses on context saving; this note also emphasizes error rates ("occasionally gets it wrong"), which is a distinct concern.

- [Grounding] The link to programming-practices-apply-to-prompting claims that "indirection cost is another case where a programming practice (abstraction via variables) transfers to prompting but with a different cost model." The programming-practices note does not specifically discuss indirection or variable costs. It discusses typing, progressive compilation, testing, version control, and design for testability. The claim is a reasonable inference -- indirection as a programming practice that transfers with a changed cost model fits the note's thesis -- but the specific case is not attributed in the source. This is the note's own inference labeled as a link relationship rather than a citation, which is fine, but worth noting.

- [Internal consistency] The note's "test" for build-time resolution -- "if you could do a find-and-replace before the LLM ever sees the text, and the result would always be the same, then do the find-and-replace" -- is a clean heuristic. However, the second bullet in the indirection pattern list -- "Abstraction layers ('use the appropriate search command' vs 'run `rg term kb/notes/`')" -- describes a case where a find-and-replace would NOT always produce the same result (the "appropriate" command depends on the search need). This is consistent with the note's own framework (the abstraction layer is costly but not eliminable by find-and-replace), but it means the "fix" (resolve at build time) does not apply to all three listed patterns. The conditional-paths bullet ("if X, do A; if Y, do B where X is always true") also has a nuance: the note adds "where X is always true," making it build-time resolvable, but this qualifier is doing significant work that could be missed.

PASS:

The methodology-enforcement note explicitly describes a gradient from instructions to scripts, and the generate-instructions-at-build-time note (linked from the example section) describes build-time generation as "constraining applied to configuration" and links to methodology-enforcement-is-constraining. The attribution checks out.

The claim that indirection makes this worse is this note's own contribution, properly presented as an extension rather than an attribution. Sound.
- [Grounding] The example section references generate-instructions-at-build-time. That note's description reads "KB skills should be generated from templates at setup time, not parameterised with runtime variables -- applying the general principle that indirection is costly in LLM instructions." It explicitly cites this note as its foundation. The cross-reference is accurate and mutual.
- [Internal consistency] The note maintains consistent definitions throughout. "Indirection" is used consistently to mean any layer of reference that requires the LLM to resolve a mapping before acting. The cost model (context + interpretation overhead + error risk) is introduced in paragraph 2 and applied uniformly to all three examples. No definition drift detected.
- [Internal consistency] The note's structure -- general principle, cost patterns, boundary conditions, concrete example -- builds without contradiction. The boundary conditions narrow the principle appropriately, and the example illustrates a case that satisfies all three conditions. The description field ("In code, indirection is nearly free at runtime -- in LLM instructions, every layer of indirection costs context and interpretation overhead on every read") faithfully compresses the body's argument.
