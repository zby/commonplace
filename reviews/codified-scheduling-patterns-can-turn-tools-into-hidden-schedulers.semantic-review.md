=== SEMANTIC REVIEW: codified-scheduling-patterns-can-turn-tools-into-hidden-schedulers.md ===

Claims identified: 9

1. "As agent applications mature, some repeated decisions stop needing open-ended judgment and codify into deterministic procedure." (opening paragraph)
2. "Many codified patterns fit naturally inside ordinary tools -- formatting, parsing, validation, data transformation." (paragraph 2)
3. These patterns describe "how to perform a single capability." (paragraph 2)
4. "A deterministic multi-step implementation can still be an ordinary tool when those steps remain internal to one coherent capability rather than choosing what broader task step should happen next." (paragraph 3)
5. Scheduling patterns are different -- they describe "what should happen next," illustrated by four examples. (paragraph 4)
6. "When a tool embeds next-step policy across subtasks, retries, recursion, or tool-surface selection, it stops acting like an ordinary capability and starts acting like a hidden scheduler." (paragraph 5)
7. The `run_feature_workflow` example "is not a tool -- it is an orchestration runtime behind a tool-shaped facade." (paragraph 5)
8. "Once next-step policy has stabilized enough to become code, the application needs to replace model-chosen transitions with explicit control logic." (paragraph 6)
9. "A framework that exposes tools but not progression forces that control logic to masquerade as tool implementation." (paragraph 6)

WARN:
- [Completeness] The note draws a binary between "ordinary tool" (internal steps within one coherent capability) and "hidden scheduler" (embeds next-step policy). The distinguishing criterion is whether steps "remain internal to one coherent capability rather than choosing what broader task step should happen next." But the four scheduling examples span a wide spectrum of ambiguity. "After collecting search results, deduplicate and re-rank before presenting" looks like post-processing internal to a retrieval capability, not cross-task scheduling -- it does not choose which broader task step comes next, it refines the output of one step. If that counts as scheduling, the boundary offered in paragraph 3 is weaker than stated, because re-ranking search results fits naturally as "internal to one coherent capability." The note would benefit from clarifying why this example crosses the line.

- [Completeness] The note's criterion ("choosing what broader task step should happen next") is tested by a boundary case: a tool that runs a deterministic retry loop with exponential backoff on its own API calls (e.g., a web-fetch tool retrying transient HTTP errors). This is multi-step, deterministic, involves retries and stopping conditions -- all listed as scheduling signals in claim 6 -- yet it is clearly internal to one capability. The note lists "retries" as a scheduling indicator without distinguishing capability-internal retries (retry my own HTTP call) from cross-task retries (retry with narrower context then escalate). This conflation weakens the boundary.

INFO:
- [Completeness] The four scheduling examples (paragraph 4) are presented as an illustrative list, not an exhaustive enumeration, so there is no strict completeness obligation. However, they all share one trait: they are all sequential/linear patterns (do X then Y). Branching patterns -- where next-step policy depends on inspecting intermediate results and choosing among divergent paths (not just retry-or-escalate but genuinely distinct continuations) -- are absent. This is a natural boundary case for the framework: branching schedulers are arguably harder to hide in a tool than sequential pipelines, and mentioning them would strengthen the argument.

- [Grounding alignment] The note links to codification.md with the annotation "this note identifies a class of patterns that do not codify cleanly into ordinary tool implementations." Codification.md itself does not discuss this class -- it defines codification as the far end of constraining, discusses when to codify, and lists examples (slug generators, CSV statistics, frontmatter checks). The claim that scheduling patterns resist clean tool codification is the reviewed note's own contribution, not something found in codification.md. The link annotation slightly overstates what codification.md provides: it provides background on what codification is, not on which patterns resist it. The inference is reasonable -- the reviewed note is applying the codification concept to a new domain -- but readers could mistake the link as grounding for the "does not codify cleanly" claim.

- [Grounding alignment] The link to stateful-tools-recover-control-by-becoming-hidden-schedulers.md is annotated "codified scheduling hidden in a tool is another way a tool becomes a covert runtime." The linked note discusses a different mechanism: a singleton stateful runtime behind the tool boundary, where the tool holds retries, checkpoints, branch records, and recursion state. The reviewed note's mechanism is different: scheduling patterns that were previously model-chosen transitions, once codified, end up in tools because the framework offers no other place. These are related but distinct causal paths (stateful accumulation vs. codified policy migration). The "another way" annotation is accurate, but the two mechanisms could be confused as identical when they differ in cause and character.

- [Internal consistency] The note says scheduling patterns "describe what should happen next" and then says codified scheduling should be replaced with "explicit control logic." This implies the note advocates moving codified scheduling out of tools into application-level orchestration. But the note never names what that target looks like -- it says what the solution is not (tool implementation) without saying what it is (sub-agent spawning? a workflow DSL? explicit state machines?). The linked notes (subtasks-that-need-different-tools, semantic-sub-goals) describe sub-agents as the canonical response, but this note stops short. This is not a contradiction but a gap in the argument's resolution.

The linked note reciprocates with the exact same framing ("parallel case: scheduling forced by codified experience rather than structural overflow; same architectural consequence, different cause").
