=== SEMANTIC REVIEW: writing-styles-are-strategies-for-managing-underspecification.md ===

Claims identified: 12

1. [Section: intro] "An empirical study of AI context files across 466 open-source projects identified five writing styles in convention sections: descriptive, prescriptive, prohibitive, explanatory, and conditional."
2. [Section: intro] "The authors treat these as stylistic variation, but viewed through the underspecification lens, each style is a distinct strategy for narrowing the interpretation space."
3. [Section: intro] "the choice encodes how much autonomy the author grants the agent."
4. [Section: five styles — prescriptive] "Direct imperatives that select a single interpretation. The agent's job is compliance, not judgment."
5. [Section: five styles — prohibitive] "Eliminates specific interpretations without prescribing alternatives. [...] Cheaper than prescriptions: forbidding a few bad outcomes costs less than enumerating all acceptable ones."
6. [Section: five styles — descriptive] "Documents what exists without directing action. The agent must infer correct behavior from described state — narrowing through context rather than directive."
7. [Section: five styles — explanatory] "Attaches a warrant to a rule. The warrant adds no direct constraint, but enables generalisation: an agent that understands *why* can extend the principle to unanticipated situations."
8. [Section: five styles — conditional] "Encodes branching logic [...] This partitions the interpretation space rather than uniformly narrowing it. Each branch is precise, but selecting which branch applies is itself an interpretation act."
9. [Section: autonomy allocation] "The styles form a spectrum from tight constraint to loose context: prescriptive → prohibitive → conditional → explanatory → descriptive"
10. [Section: autonomy allocation] "Moving left buys reliability at the cost of coverage [...] Moving right buys coverage at the cost of reliability"
11. [Section: loading tier] "Always-loaded instructions (CLAUDE.md) occupy context every session and should favour prescriptive and prohibitive styles — tight constraint, few tokens."
12. [Section: loading tier] "On-demand instructions (skill bodies, WRITING.md) load in task context where the agent needs judgment, so explanatory and descriptive styles earn their token cost through better generalisation."

WARN:
- [Completeness] The note claims prescriptive style "select[s] a single interpretation" and that "the agent's job is compliance, not judgment." But prescriptive instructions can still be semantically underspecified. "Follow the existing code style and conventions" (the note's own example, taken from the source) is an imperative, but what counts as "the existing code style" requires substantial interpretation — especially in a codebase with inconsistent style. Prescriptive form does not guarantee a single interpretation; it signals the author's *intent* to constrain to one, which is a different claim. The note conflates the illocutionary force of the instruction (imperative mood) with the semantic precision of the instruction (how many valid readings it admits).
- [Completeness] The ordering "prescriptive → prohibitive → conditional → explanatory → descriptive" is presented as a spectrum but the placement of conditional between prohibitive and explanatory is asserted without justification. A conditional instruction like "If you need to use reflection, use ReflectionUtils APIs" contains a prescriptive branch — within its active branch it is as constraining as a pure prescriptive instruction. What makes it less constraining overall is the branching, but one could argue conditional instructions are *more* constraining than prohibitive ones when the branch condition is met, since they prescribe a specific action rather than merely excluding options. The spectrum conflates "average constraint across situations" with "constraint within a given situation," and the note does not acknowledge that the ordering depends on which framing you choose.
- [Grounding — scope mismatch] The source paper identified the five writing styles by analyzing only the 50 "Conventions" sections of AGENTS.md files (Section 4.2: "All 50 sections labeled Conventions were analyzed, the most common category in the dataset"). The note generalises these styles to all context-file writing without qualification — the intro says "five writing styles in convention sections" but the rest of the note drops "convention sections" and treats the taxonomy as universal ("The five styles as constraint strategies," "Style choice encodes autonomy allocation," "Interaction with loading tier"). The ingest file flags this same limitation: "Other content categories (architecture, build commands, test strategy) may exhibit different style distributions." The note should acknowledge that the empirical basis is convention sections specifically, and the generalisation to all context-file writing is the note's own move.

INFO:
- [Completeness — boundary case] Hybrid instructions — a single sentence combining multiple styles — are not addressed. The source's own example "Avoid hard-coded waits to prevent timing issues in CI environments" is coded as explanatory, but it contains both a prohibitive component ("avoid hard-coded waits") and an explanatory component ("to prevent timing issues"). Real context-file instructions frequently combine styles within a single sentence. The note's framework treats them as discrete categories along a spectrum, which may not cleanly handle hybrid cases. This is a potential gap in the taxonomy rather than a clear error.
- [Completeness — boundary case] The note does not address what happens when styles conflict within the same context file — e.g., a descriptive statement documenting a convention that conflicts with a prescriptive instruction elsewhere in the file. Since the note argues each style narrows the interpretation space differently, conflicting styles would send contradictory signals about how much autonomy the agent has. This interaction effect is unexplored.
- [Grounding — inference validity] The claim that explanatory style's "warrant adds no direct constraint" relies on a specific reading of the Toulmin source. The Purdue OWL source defines a warrant as "the assumption — stated or implied — that connects the grounds to the claim." In Toulmin's model, the warrant is load-bearing for the argument's validity, not an optional extra. The note's framing — that the warrant "adds no direct constraint, but enables generalisation" — is a reasonable adaptation of the concept to the instruction domain, but calling it "no direct constraint" understates the warrant's role: an explanatory justification can actively constrain interpretation by ruling out readings that are inconsistent with the stated reason. "Avoid hard-coded waits to prevent timing issues in CI environments" constrains interpretation more than "Avoid hard-coded waits" alone — the reason eliminates the reading that hard-coded waits are forbidden for aesthetic reasons, which would lead to different generalisation patterns.
- [Internal consistency] The loading-tier section claims always-loaded instructions "should favour prescriptive and prohibitive styles — tight constraint, few tokens." But tight constraint and few tokens are independent properties. A prescriptive instruction can be verbose ("Follow the existing code style and conventions, including but not limited to indentation, naming, bracket placement..."), and a descriptive instruction can be terse ("Uses ESLint"). The note conflates conciseness with constraint strength. The correlation may hold on average but is not inherent in the style definitions the note itself provides.
