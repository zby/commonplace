WARN:
- [Completeness] "The calculator surface grows monotonically" (Concrete workflow, step 5) overstates the claim. The note's own Risks section acknowledges that mined specs can encode biases or accidents, and the linked relaxing-signals note describes scenarios where mined specs must be relaxed back. If a mined spec is relaxed (removed or loosened because it encoded a vision feature), the calculator surface shrinks. The monotonicity claim holds only if no mined spec is ever reverted, which contradicts the note's own constrain/relax framing. A more precise claim would be "the calculator surface grows monotonically net of relaxing."
- [Grounding alignment — domain coverage] The note attributes a specific workflow to the Lopopolo source: "20% of engineering time (Fridays) went to manually cleaning 'AI slop' — observing failure patterns. The team then codified those observations into structural tests and linter rules whose error messages teach the fix, and finally automated the observation step itself with background cleanup agents that scan for drift and open refactoring PRs." The source does describe these elements (entropy management, structural tests, linter error messages as remediation, background cleanup agents). However, the source does not frame this as "spec mining" or as an extraction-from-observed-behavior loop. The source frames it as entropy management and harness engineering — designing constraints, enforcing architecture, and automating cleanup. The note interprets the Codex team's practice through the spec-mining lens, which is a reasonable inference but not an attribution. A reader could mistake this for the source's own framing. The note says "documents this workflow" which implies the source describes the spec-mining pattern; in fact, the source describes practices that the note reinterprets as spec mining.
- [Completeness — boundary case] The four-step pattern (watch, identify micro-actions, extract, re-run) and the five-step concrete workflow both assume the system produces observable behavior from which regularities can be extracted. Boundary case: what about greenfield systems or novel task domains where there is no production history to mine? The note implicitly assumes an existing behavioral corpus. Step 1 of "The pattern" says "Watch the system do tasks (or watch humans do tasks the system will do)" — the parenthetical partially addresses this, but the mechanism for spec mining from human observation is different from mining from system logs, and the note does not distinguish the two paths or their different failure modes.

INFO:
- [Grounding alignment — vocabulary] The note says the cybernetics thread calls the design philosophy "externalizing system-specific judgment." Checking the ingest: the source summary says "the hard part is not giving agents more capability but externalizing system-specific judgment so the harness can evaluate and correct them," and the extractable value section says "out-evaluate, not out-implement." The note combines these two phrases into a single attribution, which is accurate but slightly compresses two distinct moves (externalized judgment as bottleneck, and out-evaluate as the human role) into one.
- [Internal consistency — definition drift] The note uses "codification" in two slightly different senses. In the opening and in "The pattern," codification means extracting deterministic artifacts from stochastic behavior (the compilation metaphor). In the concrete workflow step 4, "partial codification" is used for cases where no deterministic rule exists but you get a regression test. A regression test that detects a known failure is closer to oracle manufacturing than to codification-as-compilation (it doesn't replace the stochastic process with a deterministic one; it adds a check). This is a mild stretch of the term rather than a contradiction, but worth noting.
- [Completeness — boundary case] The note claims spec mining moves components "from soft/delayed oracle toward hard oracle." Boundary case: a mined spec that is itself a soft oracle (e.g., a heuristic rule that catches 80% of failures but has false positives). This is partially addressed by the error-correction link ("even a weak spec is useful"), but the note's own framing suggests mined specs produce hard oracles ("deterministic verifiers," "deterministic artifacts," "deterministic code"), while the reality may often be soft oracles that are harder than what existed before but not fully hard. The oracle-strength note's manufacture/amplify distinction covers this, but the spec-mining note's language leans harder toward "deterministic" than the full picture warrants.

PASS:
- [Grounding alignment — codification] The note's claim that codification "says knowledge hardens into repo artifacts" accurately reflects the codification note, which defines codification as constraining that crosses a medium boundary from natural language to executable code, producing "tests, specs, conventions." The attribution is faithful.
- [Grounding alignment — bitter lesson boundary] The note's claim that "the bitter lesson boundary says calculators survive scaling because the spec is the problem" faithfully represents the bitter-lesson-boundary note's core distinction between arithmetic (spec IS the problem) and vision features (spec approximates the problem).
- [Grounding alignment — oracle strength spectrum] The note's claim about moving "from soft/delayed oracle toward hard oracle" correctly references the oracle-strength-spectrum note's gradient from hard oracle through soft, interactive, delayed, to no oracle.
- [Grounding alignment — inspectable substrate] The note's claim that mined specs produce "an inspectable substrate — reviewable, testable, revertable artifacts" aligns with the inspectable-substrate note's core argument that repo artifacts are inherently inspectable and can be diffed, tested, and reverted.
- [Grounding alignment — cybernetics source] The note's attribution of "out-evaluate, not out-implement" to the cybernetics thread is confirmed by the ingest, which lists "Human role as oracle design, not implementation: 'out-evaluate, not out-implement'" as extractable value.
- [Grounding alignment — error correction] The note's claim that "error correction can amplify through decorrelated checks" with the requirement of "TPR > FPR" is consistent with the error-correction note's framework as referenced through the oracle-strength-spectrum note.
- [Internal consistency — risks vs. pattern] The note's Risks section ("mined specs should be falsifiable... candidates for relaxing, not permanent codification") is internally consistent with the main claim pattern. The note does not claim spec mining is infallible; it acknowledges the constrain/relax cycle. (The monotonicity claim in the concrete workflow is the exception, flagged as WARN above.)
- [Internal consistency — open questions] The open questions section correctly identifies the threshold problem (when to codify) and the automation question, both of which follow naturally from the note's framework without contradicting it.

Overall: 3 warnings, 3 info
