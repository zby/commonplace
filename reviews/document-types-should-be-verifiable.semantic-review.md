WARN:
- [Completeness] The base types table lists four types: `note`, `spec`, `review`, `index`. But the companion spec (document-classification.md) lists seven: `text`, `note`, `spec`, `review`, `index`, `adr`, `structured-claim`. The note itself later mentions `structured-claim` in the body ("What it called 'insight' becomes `structured-claim`...") but never adds it to the base types table. A reader relying on the table as the definitive enumeration would miss `text`, `adr`, and `structured-claim` as base types. Since the note presents the table as the solution to the flat-types problem ("Instead of a flat enum, use a base type plus traits"), this looks like an incomplete enumeration rather than a deliberate simplification.
- [Completeness] The verifiability gradient (Section: The verifiability gradient) references a link to `deploy-time-learning-the-missing-middle.md` with the anchor text "verifiability gradient," suggesting the concept originates there. However, the deploy-time-learning note's verifiability gradient is about grades of artifact hardening (restructured prompts -> structured output schemas -> prompt tests/evals -> deterministic modules). The note's verifiability gradient is about document type maturation (untyped note -> traits accumulate -> base type promoted). These are different gradients applied to different domains. The note frames its gradient as continuous with the deploy-time-learning gradient ("This connects to the verifiability gradient"), but the mapping is loose: document type promotion does not correspond to any specific grade in the deploy-time-learning table. The link implies tighter alignment than the sources support.
- [Grounding — domain coverage] The note claims "Type assignment is itself a case of storing an LLM output as constraining" and links to `storing-llm-outputs-is-constraining.md`. That source is about storing generated artifacts (code, documents, configuration) — the output of an LLM generation task. Type assignment is a classification decision, not a generation task; the "output" being stored is a label, not a document. The source's framework (resolving semantic underspecification + freezing against indeterminism) does apply, but the source develops its argument entirely around generated artifacts, not metadata labels. The note is extending the source's domain from "storing generated content" to "storing classification decisions," which is a reasonable inference but not directly grounded by the source.

INFO:
- [Completeness] The traits table lists three traits: `has-comparison`, `has-external-sources`, `has-implementation`. The note presents these as the trait system, but the document-classification spec may define additional traits. Boundary case: a document with external sources that also has a comparison section and implementation sketches — would three traits be sufficient to capture its properties? The note's own programming-language parallel to "protocols / structural typing" suggests traits should be open-ended ("A document satisfies `has-external-sources` if it references external material, regardless of whether someone labeled it"), but the table reads as a closed enumeration. A note clarifying that the trait list is extensible (or not) would resolve this ambiguity.
- [Completeness] The programming language parallels section lists four PL concepts: gradual typing, protocols/structural typing, refinement types, soft typing. Boundary case: **sum types / union types** — the note's base types are effectively a sum type (a document is exactly one of note | spec | review | index), yet this parallel goes unmentioned while the more exotic "refinement types" is included. This is a minor gap — the note is illustrating, not enumerating — but the omission of the most obvious parallel is notable.
- [Internal consistency] The note says "`note` is the base type that makes no structural claim — like `Any` in a gradually typed language." But earlier, the "What verifiable means" section argues that types should "assert a structural property you can check" and that types failing this test are useless. If `note` makes no structural claim, it fails the note's own verifiability test. The note doesn't explicitly acknowledge this tension. The gradual-typing analogy suggests this is intentional (you need `Any` to bootstrap the system), but a reader could see `note` as contradicting the title claim. The tolerance-of-misclassification section partially addresses this by framing types as optimization rather than correctness, but the tension between "types should be verifiable" and "the default type is unverifiable" is never directly stated.
- [Grounding — vocabulary] The note says types help agents make "informed rather than blind" navigation decisions and links to `agents-navigate-by-deciding-what-to-read-next.md`. That source discusses pointers (links, index entries, search results, skill descriptions) and the context they carry. It never mentions document types or the `type` frontmatter field as a navigation mechanism. The note's inference — that types serve the same role as context-carrying pointers — is reasonable but is the note's own contribution, not something the source argues.
