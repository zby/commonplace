WARN:
- [Grounding / domain coverage — Finzi et al.] The note claims "High-epiplexity regularities are genuinely structural — codifying them is safer because they reflect real patterns, not artefacts of the observer's computational budget." The epiplexity paper defines epiplexity as learnable structure extractable by a computationally bounded observer. But it does not address codification, safety of encoding patterns, or the distinction between arithmetic-regime and vision-feature specs. The note's inference chain — high epiplexity means genuinely structural, therefore safer to codify — bridges two domains that the source does not connect. Epiplexity measures extractable structure in data for a given computational budget; it does not predict whether extracted structure will survive scaling by a different, more powerful observer. A high-epiplexity pattern (learnable at current compute) could still be a vision feature that a larger-compute observer would supersede with a better decomposition. The note presents this as if the source grounds the codification-safety claim, but the source grounds only the "genuinely structural vs artefact" distinction within a fixed computational bound.

- [Grounding / scope mismatch — Ebrahimi et al.] The note claims that the induction bias results show "codification bets in the arithmetic regime are not merely safe-for-now; the step-by-step structure that codification encodes is the kind of regularity that persists under scaling." The source studies data efficiency of architectural induction bias (recurrent vs transformer) on synthetic algebraic state-tracking tasks (modular addition, permutation composition) using small fixed-size models. It does not test whether the advantage persists under model scaling (more layers, larger dimensions), and the authors explicitly acknowledge this limitation. The note's upgrade from "architectural induction bias is efficient for algebraic tasks at tested scales" to "codification bets persist under scaling" is the note's own inference, not an attribution. The ingest file's limitations section specifically flags this: "this leaves open the possibility that the data-efficiency gap narrows with scale — which would be exactly the bitter lesson's prediction."

- [Completeness / boundary case — relaxing] The note defines relaxing as "replacing a codified component with a learned or general-purpose one when scale makes that viable." The examples given (edge detection relaxed to learned features, FlashAttention as re-codification) all involve replacement at the same level of abstraction. But a common real-world pattern is partial relaxing: loosening a strict spec into a softer constraint rather than wholesale replacement. For example, a rigid three-stage pipeline might not be entirely replaced by a learned system but instead have its stage boundaries made fuzzy or optional. The note frames codify/relax as a binary toggle ("replacing a codified component"), but the constraining note it links to describes a spectrum. The binary framing obscures the gradual relaxation that is probably more common in practice than wholesale replacement.

INFO:
- [Completeness / boundary case — "every codification is a bet"] The note asserts "In the arithmetic regime, the bet is safe: the spec IS the problem, and the solution is algorithmically determined." But even arithmetic-regime codification carries implementation bets. A codified sort algorithm bets on the comparison function being total and consistent; a codified validator bets that the schema captures all legitimate inputs. These are not vision-feature bets (the spec is not a theory about the problem), but they are still bets that can fail — the spec can be incomplete even when it fully captures the subproblem it addresses. The note could distinguish "the bet that the spec is right" (safe in arithmetic) from "the bet that the spec is complete" (not guaranteed even in arithmetic).

- [Completeness / boundary case — the simplest possible instance] The simplest case of the codify/relax dynamic would be a single function that computes a well-defined output. The note's framework handles this cleanly (arithmetic regime, pure codification gain). But consider a configuration file: it encodes choices (a form of codification) but the choices themselves may be vision features (e.g., "always use batch size 32"). Configuration sits at the boundary between codification and parameterization, and the note's framework does not clearly distinguish codification of logic from codification of parameters. This is a soft gap — the note's scope is about the codify/relax trajectory, not a taxonomy of codifiable artifacts — but it could mislead readers into thinking all codification has the same risk profile.

- [Internal consistency — "trajectory, not a law" vs. "persist under scaling"] The note states "The bitter lesson describes a trajectory, not a law of nature" in the first section, which implies contingency and reversibility. But the final paragraph of "Every codification is a bet" claims the Ebrahimi results show "the step-by-step structure that codification encodes is the kind of regularity that persists under scaling" — which sounds like a law-like claim for the arithmetic regime. These are not contradictory (the trajectory claim is about the vision-feature regime; the persistence claim is about the arithmetic regime), but the juxtaposition without explicit scoping could read as tension. A reader might ask: if the bitter lesson is "not a law of nature," why does the note end by citing evidence that certain structures persist permanently?

The source uses "time-bounded entropy" for the irreducible-randomness component and "epiplexity" for the extractable-structure component. The source's framing is neutral: low epiplexity means "not extractable at this compute level," not "accidental."
