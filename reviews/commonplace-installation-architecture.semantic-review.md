The note says "everything in `kb/` is theirs" and methodology stays in `commonplace/`. But if a user disagrees with a methodology conclusion and writes their own note in `kb/notes/`, the agent's search across `kb/` may find the user's override, while the methodology in `commonplace/` remains unchanged. - [Completeness] The "Inclusion mechanism" section lists two approaches (submodule and clone+gitignore) but both assume a single-machine, git-based workflow. A boundary case: a team using a monorepo or a CI system that builds the project in ephemeral containers. The install step and upgrade mechanism described assume `commonplace/` persists on disk as a git-managed directory. The note's enumeration isn't wrong, but its implicit scope claim ("Two supported approaches") could be read as exhaustive when it only covers two git-based variants of the same pattern.

- [Grounding alignment] The note claims: "skills derive from methodology through distillation and handle the common cases. But distillation is not lossless: when the agent hits an edge case the skill doesn't cover, she needs the full reasoning." The linked source (skills-derive-from-methodology-through-distillation.md) does say this, with one nuance: the source note's Caveats section adds "Not all skills are distilled from methodology" and describes Cramer's skill synthesis from raw domain artifacts. The installation architecture note elides this caveat, presenting distillation-from-methodology as the universal case. The elision is reasonable in context (the note is about commonplace, where skills *are* methodology-derived) but worth flagging since the universal framing could mislead in the installed-project context where a user might distill skills from their own domain artifacts.

- [Grounding alignment] The note references agent-statelessness-makes-routing-architectural-not-learned.md with the claim "methodology is permanent infrastructure the agent returns to, not a learning aid she graduates from." The source note's core argument is about routing infrastructure being permanent, and it uses "source vs. compiled" as its central metaphor. The installation architecture note accurately captures the spirit but shifts the vocabulary from "routing infrastructure" to "methodology as permanent infrastructure." The source note treats methodology notes as "source code" and skills/routing as "compiled artifacts" — it's the compiled artifacts (routing, skills) that are permanent infrastructure the agent needs every session, while methodology is the source consulted during maintenance. The installation note slightly repurposes the claim: in the source, permanence applies to the *routing artifacts*; here, it's applied to the *methodology itself*. The inference is reasonable (if routing derives from methodology, methodology must persist too) but the direction is subtly different from what the source argues.

- [Internal consistency] The note uses "she" pronouns for the agent (lines 37, 53, 55, 63, 87) and also the impersonal "the agent" throughout. This is a stylistic choice, not a consistency issue per se, but the pronoun usage is inconsistent within the note — some sections anthropomorphize ("she decides," "she needs") while others don't. No semantic conflict arises, but definition drift in how the agent is conceptualized could matter in a note that distinguishes between human navigation intuition and agent statelessness.
