WARN:
- [Completeness] The note claims unified calling makes "each codification step" no longer a breaking change and that "each link can be independently refactored without disturbing the rest." This treats the calling convention as sufficient for local refactoring, but semantic compatibility is a separate concern. If an agent returns free-form text and the codified replacement returns structured JSON, the call site may not change but the caller's downstream handling breaks. The calling convention unifies dispatch, not the contract on return values. The note's own "typed callables" section implicitly acknowledges this (typed inputs and outputs), but the body's claims about locality don't qualify on output-type compatibility.
- [Completeness] The four-step progression (start neural, observe patterns, codify, extend via relaxing) is presented as the smooth path enabled by unified calling. Boundary case: what about the reverse entry point — starting with a symbolic tool and relaxing it to an agent? The note mentions relaxing only as "extend via relaxing" (step 4), which is adding new LLM capability alongside existing code. The case of wholesale replacement of a tool with an agent (the symmetric counterpart to step 3) is absent. The title claims "bidirectional refactoring," but the worked example is unidirectional (neural-to-symbolic) with relaxing only as extension, not as full reversal. The underspecified-instructions note explicitly describes replacement relaxing ("rigid code is drowning in edge cases, so you swap it for an LLM call"), but this note does not demonstrate that path through the calling convention.
- [Grounding] The note states the underspecified-instructions framing "doesn't say how to make the boundary movable in practice." But that note's "Design Implications" section explicitly includes item 2: "Enable bidirectional refactoring — design interfaces so components can move across the boundary without rewriting call sites." The source note does say what to do; it just doesn't provide a concrete implementation. The claim that it "doesn't say how" is defensible if read as "doesn't provide a mechanism," but as written it could be read as the stronger claim that the source note is silent on the topic, which is inaccurate.

INFO:
- [Completeness] The three properties attributed to name-based dispatch (dynamic resolution, late binding, implementation-agnostic interfaces) are presented as a list without a grounding definition of what name-based dispatch must provide. The first two — dynamic resolution and late binding — are closely related (late binding is a consequence of dynamic resolution in most runtime models). Whether these are genuinely distinct properties or two framings of one property is ambiguous.
- [Completeness] The scheduler comparison table contrasts graph DSLs with the imperative scheduler across four dimensions. Boundary case: a graph DSL that compiles to imperative code (e.g., LangGraph compiles a graph to a runnable that executes imperatively). The table treats "Graph DSLs" as a uniform category, but the refactoring row's claim ("Redraw edges, update graph" vs. "Change code — extract functions, inline agents") may not hold for frameworks where the graph is defined in code and refactored with normal code tools. The contrast is real for visual/declarative graph builders but strained for code-defined graphs.
- [Grounding] The note says llm-do "operationalises" the typed-callables view. The typed-callables note (status: speculative) discusses skill signatures and type checking on document operations. llm-do's .agent files have YAML frontmatter and tools have Python type annotations, which is a form of typed interface — but the typed-callables note is specifically about document-type signatures on KB operations (e.g., `/ingest: source -> source-review`), not about agent/tool dispatch. The "operationalises" link is reasonable but covers adjacent territory rather than the same concept.
- [Internal consistency] The note's description says "name-based dispatch over a hybrid VM," but the body never defines "hybrid VM" beyond the implicit meaning of "a runtime that hosts both agents and tools." The term "VM" (virtual machine) implies an execution model with defined semantics (instruction set, state model). If the runtime is just a dispatcher that routes string names to either an LLM call or a Python function, calling it a VM may overstate the computational formality. This is a framing choice rather than a contradiction, but "dispatcher" and "VM" carry different connotations.
