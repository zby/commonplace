=== SEMANTIC REVIEW: ad-hoc-prompts-extend-the-system-without-schema-changes.md ===

Claims identified: 15

WARN:
- [Completeness] The two-strata model ("a deterministic base...and a prompt layer on top") omits a boundary case: what about requirements absorbed by changing configuration files (e.g., YAML, JSON) that are neither code nor prompts? A new environment variable or a feature flag in a config file is not a "schema change" or a "code change," but it is also not a natural language prompt. The two-strata framing presents a binary (deterministic base vs. prompt layer) that elides this middle ground of declarative configuration, which is neither deterministic code nor LLM-interpreted prose. The note's central claim still holds for the LLM-specific case, but the universalizing framing ("Any system with an LLM agent layer has two strata") implies exhaustive coverage that the binary does not deliver.

- [Completeness] The claim "zero validation and zero reuse" for ad hoc prompts is stated as a characterization of the loosest end of the gradient. However, the methodology-enforcement note places "ad hoc prompts" as a distinct row *below* persistent instructions, with its own reliability profile. The ad-hoc-prompts note itself acknowledges that ad hoc prompts can become skills (reuse through extraction), and that prompts can reference other prompts or skills (the open question "Can ad hoc prompts reference skills?"). This means "zero reuse" is a simplification — ad hoc prompts have *low* reuse, not literally zero. The note treats this as a bright-line property when the source it cites treats it as a gradient.

- [Grounding] The note claims "The prompt defines what's visible in the sub-agent's lexically scoped frame, and the sub-agent inherits nothing beyond what the caller explicitly passed." The linked source (llm-context-is-composed-without-scoping.md) says something more nuanced: sub-agents get "a fresh context — its own system prompt, its own input, no inherited conversation history," but also acknowledges dynamically scoped bindings ("User preferences, safety policies, global constraints, project-level conventions") that persist across frames. The sub-agent does inherit things beyond what the caller explicitly passed — it inherits the dynamic environment. The note's claim that "the sub-agent inherits nothing beyond what the caller explicitly passed" is stronger than what the source supports.

INFO:
- [Completeness] The note's examples of ad hoc prompt extension (CI pipeline, codebase review, deployment) are all developer-tool contexts. A boundary case from a different domain: could a customer support system absorb a new escalation rule via an ad hoc prompt? The pattern likely applies, but the note presents only software-engineering examples, which may make the "applies everywhere agents operate" claim seem narrower than intended — or, alternatively, may be masking domain-specific limitations (e.g., regulated environments where prompt-based policy changes would be unacceptable without audit trails).

- [Completeness] The KB collections example presents three formal alternatives (new type, file-listing format, directory glob) and then the ad hoc prompt solution. A boundary case between the enumerated items: what about a structured comment or annotation within an existing document (e.g., a YAML list inside a markdown note)? This sits between a formal type and an ad hoc prompt — it has some structure but no schema enforcement. The note's framing of "formal options vs. ad hoc prompt" is a useful contrast but presents it as a binary when there is a spectrum of semi-structured options in between.

- [Internal consistency] The note says ad hoc prompts have "zero reuse" in the constraining spectrum section, but the maturation trajectory section describes exactly a reuse pattern: "notice when you're writing the same kind repeatedly, extract a skill." The trajectory exists *because* ad hoc prompts do get reused (as patterns, if not as literal artifacts). The note is not contradicting itself — "zero reuse" likely means "no built-in reuse mechanism" rather than "never reused" — but the phrasing creates a tension that could confuse a reader.

- [Grounding] The note claims the extraction from ad hoc prompt to skill "is distillation — the ad hoc prompt carries reasoning about what to do and why; the extracted skill keeps the procedure and drops the justification." The linked source (skills-derive-from-methodology-through-distillation.md) defines distillation as the methodology-to-skill relationship where "discursive, multi-perspective, argumentative reasoning becomes procedural, step-sequenced instruction." An ad hoc prompt is typically already procedural (it tells an agent what to do), not discursive or argumentative. The note may be using "distillation" more loosely than the source defines it. The inference is plausible — ad hoc prompts do carry context that skills drop — but the fit is not exact.

- [Grounding] The note claims homoiconicity is the property that makes Lisp, Emacs, and Smalltalk "extensible from within." The linked homoiconicity note supports this for Lisp and Emacs but describes Smalltalk's property as "live image" extensibility — modifying the running system from inside — which is more accurately characterized as reflectivity or liveness than homoiconicity per se. Smalltalk's code is not represented as Smalltalk data structures the way Lisp code is represented as lists. The analogy holds loosely but the homoiconicity source itself frames Smalltalk differently from Lisp.

PASS:
- [Grounding] The enforcement gradient claim ("instructions -> skills -> hooks -> scripts") aligns accurately with the source (methodology-enforcement-is-constraining.md), which presents this exact gradient with ad hoc prompts as a row below instructions. The source even back-links to this note, confirming mutual consistency.

- [Grounding] The homoiconicity claim — that ad hoc extension works because instructions and content share the same representation — is well-supported by the linked source (llm-context-is-a-homoiconic-medium.md), which explicitly states "instructions and data share the same representation — natural language tokens" and cites ad hoc prompts as the primary exemplification of homoiconicity's extensibility benefit.

- [Grounding] The claim that typed callables sit "at the other end" of the spectrum from ad hoc prompts is consistent with the linked source (instructions-are-typed-callables.md), which describes skills with declared signatures, validated inputs, and composable operations — the structured counterpart to ad hoc prompts.

- [Internal consistency] The note's sections maintain a coherent progression: the intro states the mechanism, the KB example grounds it, the constraining spectrum places it on a gradient, the expressiveness section explains why it resists formalization, and the homoiconicity section provides the architectural explanation. No section contradicts another at the claim level.

- [Internal consistency] The description ("Any system with an LLM agent layer can absorb new requirements through natural language prompts without changing the deterministic base") faithfully represents the body's argument without eliding tensions the body acknowledges.

Overall: 3 warnings, 5 info
