WARN:
- [Completeness] The three-stage decomposition (cue match, priority arbitration, commitment) has no stage for *recognition that the activated knowledge is relevant to the current task*. Consider a case where context cues successfully retrieve a piece of knowledge (cue match succeeds), and it wins the competition for reasoning budget (priority arbitration succeeds), but the model misapplies it — e.g., it surfaces a concern about thundering herds in a context where there is no retry logic, or it raises a concurrency warning for a purely single-threaded script. This is a relevance-assessment failure that sits between stages 2 and 3: the knowledge is activated and prioritized, but the mapping from knowledge to current situation is incorrect. The note hedges the decomposition as "proposed, not established," but the gap is worth flagging because the note also suggests the stages are "independently useful for reasoning about where interventions can help" — and misdirected activation (false positive volunteering) would need a different intervention than any of the three listed stages.

- [Grounding — domain coverage] The link to "towards-a-science-of-ai-agent-reliability" (Rabanser et al.) is tagged "context: reliability dimensions motivate separating stored capability from operationally activated behavior." But the Rabanser paper's four dimensions (consistency, robustness, predictability, safety) are about *deployment reliability* — repeatable outcomes, graceful degradation, calibrated confidence, bounded harm. None of these dimensions address the storage-vs-activation distinction that is this note's central claim. The paper separates reliability from capability (Section 3.5: "reliability should be disentangled from capability"), but that is a different separation: capability = does the agent succeed, reliability = how predictably/robustly/safely. The note's distinction — the agent possesses knowledge but fails to surface it proactively — is not the same dimension the paper formalizes. The paper provides relevant *background* on evaluation gaps but does not directly motivate the storage/activation separation the link annotation claims.

- [Completeness — initiative gradient] The initiative gradient (artifact > operational > systemic) is presented as a consistent empirical finding, but the note's own evidence source (the-bug-that-shipped) tests only coding/deployment scenarios. The note claims "this gradient appears consistent across domains, though the specific failure families differ" but provides no evidence or argument for cross-domain consistency. The open questions section asks whether it holds in non-technical domains, which is appropriate, but the body text's "appears consistent" asserts more than the evidence supports. The gradient is plausible as a hypothesis about cue proximity, but it could invert in domains where systemic concerns are more salient than artifact-level ones — e.g., in legal or medical contexts where practitioners are trained to think systemically first.

INFO:
- [Grounding — vocabulary] The link to "agentic-systems-interpret-underspecified-instructions" is tagged "foundation: prompt context determines which interpretations are activated." The linked note's framework is about semantic underspecification and interpretation selection — the LLM projects from a space of valid interpretations. The activation note's framing is about knowledge retrieval — stored knowledge failing to surface. These are related but distinct: interpretation selection is about which *behavior* the model adopts given an instruction, while activation failure is about which *knowledge* the model brings to bear during execution. The link is reasonable as conceptual kin, but "foundation" overstates the relationship. The interpretation-selection framework does not directly ground the activation-failure framework; it addresses a different mechanism (spec ambiguity vs. knowledge retrieval failure).

- [Internal consistency] The note says "LLM systems make the control surface unusually explicit: prompt context is where cue match succeeds or fails, which makes the gap both more visible and more tractable than its human counterpart." The tractability claim sits in tension with the expertise-gap section, which argues "neither party can independently close the gap" and the fix requires "maintained question-generation systems." If the control surface is unusually explicit and tractable, why does the gap require organizational infrastructure to close? The note doesn't contradict itself — tractability of the mechanism doesn't mean tractability for the end user — but the rhetorical progression creates an expectation of a more actionable conclusion than the one delivered.

- [Completeness — expertise gap] The claim "the person who most needs activation scaffolds is usually the person least able to construct them" is structurally sound but admits an important boundary case: the expert user who delegates routine work. A senior engineer using an LLM to write boilerplate retry logic has the expertise to ask about thundering herds but delegates precisely because the task feels routine and not worth their deep attention. In this case, the expertise exists but the *attention allocation* fails — the user has the knowledge but does not activate *their own* relevant expertise because they have mentally categorized the task as trivial. This is activation failure in the human, not an expertise gap, and it is arguably more common in practice than the pure novice case the note emphasizes.

- [Grounding — the-bug-that-shipped] The source is accurately represented. The note's expert-witness metaphor, the initiative gradient, and the expertise-gap argument all map cleanly to claims in the source. One nuance: the source reports specific quantitative data (13% average self-review catch rate for deployment concerns, 0-8% for cron double-execution) that the note abstracts away. The abstraction is fair, but the source's quantitative specificity (particular models scoring 0% on particular scenarios) is stronger than the note's qualitative gradient framing suggests.
