=== SEMANTIC REVIEW: three-space-memory-separation-predicts-measurable-failure-modes.md ===

Claims identified: 11

WARN:
- [Completeness] The note frames the test as binary — three-space separation helps vs. "single flat memory" — but the actual design space is larger. A system could use two spaces (e.g., persistent knowledge + transient working files, with no distinct "self" space), or N spaces split along a different axis (e.g., by access pattern, as AgeMem does per the companion note's own link to memory-management-policy-is-learnable-but-oracle-dependent.md). The observation protocol cannot distinguish "three spaces are the right decomposition" from "any non-flat decomposition helps." The note's own "boring explanation" section gestures at this ("it's file hygiene, not cognitive architecture") but doesn't account for it in the protocol design — there is no observation that would differentiate two-space hygiene from three-space cognitive architecture.
- [Completeness] Failure mode 2 ("identity scatters across ephemeral logs") conflates two distinct problems: (a) self-knowledge is not consolidated, and (b) self-knowledge is not findable. The observation questions for FM2 ask about both ("Is self-knowledge findable?" and "Does the agent re-derive operational wisdom?"), but these could have different causes. An agent could have consolidated self-knowledge that is simply not loaded into context — a routing failure, not a memory-space failure. The protocol does not distinguish memory-space architecture failures from context-loading failures.
- [Grounding — domain coverage] The link to claw-learning-is-broader-than-retrieval.md claims these failure modes are "symptoms of forcing action-oriented knowledge into a retrieval-oriented structure." The source note does say this (in its Open Questions section), but frames it as a speculative question ("Does the three-space memory model help here?"), not a conclusion. The link annotation in the reviewed note upgrades a speculation to an argument: "argues these failure modes are symptoms of..." The source note raises the possibility; it does not argue for it.

INFO:
- [Completeness] The "most extreme instance" boundary case: at very large scale (thousands of notes), the three predicted failure modes may be dominated by a fourth failure mode the note does not discuss — cross-contamination within a single space. If the knowledge space itself becomes large enough, operational-style debris can accumulate there (e.g., stale notes, superseded claims, speculative notes that were never validated). The note implicitly assumes the three spaces are internally clean, but the observation protocol could surface intra-space pollution that mimics inter-space pollution.
- [Completeness] The simplest-instance boundary case: a system with very few notes (say, under 10) will likely show none of the failure modes regardless of architecture. The note acknowledges this in Open Questions ("we may need 50+ notes before search pollution is measurable") but the observation protocol has no minimum-scale gate. Observations collected before that threshold would be noise, not evidence, yet the protocol treats all observations equally.
- [Grounding — vocabulary] The note says the source article is "Cornelius, Agentic Note-Taking #19" but the source is not ingested into the KB. The companion note (three-space-agent-memory-maps-to-tulving-taxonomy.md) links to a tweet URL. Without the source text, it is not possible to verify whether the three failure modes listed here are faithful to the article's claims or are the note author's reconstruction. The companion note uses slightly different language for FM2 ("identity scattering across ephemeral logs") compared to this note's "self-knowledge ends up embedded in session transcripts rather than consolidated in a persistent location" — the elaboration may be accurate but cannot be verified against the source.
- [Internal consistency] The note says "we already run a system with partial three-space separation (docs/notes/, arscontexta/self/, arscontexta/ops/)" — these paths reference a different system (arscontexta), not the current commonplace KB. The current KB's structure (kb/notes/, kb/instructions/, kb/sources/, kb/tasks/) does not map cleanly to the three-space model. This does not contradict the note's claims (it was likely written when the system used arscontexta paths), but it means the observation protocol's premise — that the current system has partial three-space separation to observe against — may no longer hold for the repo where this note now lives.

PASS:
- [Internal consistency] The three failure modes are logically distinct and non-overlapping. FM1 is about search quality, FM2 is about identity persistence, FM3 is about knowledge graduation. No pairwise contradiction found.
- [Internal consistency] The "What counts as evidence" section is well-structured — it specifies evidence for the claim, against the claim, and a deflationary interpretation. The three categories are exhaustive for the binary hypothesis.
- [Completeness] The observation protocol questions map faithfully to the three failure modes. Each failure mode has 3 diagnostic questions that operationalize it into observable behavior.
- [Grounding] The link to storing-llm-outputs-is-constraining.md is accurately described: "graduation from operational space to knowledge graph is a form of constraining." The source note does discuss how storing/keeping specific outputs is a constraining move, and the concept of graduation from working state to durable artifact is consistent with this framing.
- [Grounding] The link to deploy-time-learning-the-missing-middle.md is accurately described: the source note discusses a "verifiability gradient" that motivates durable knowledge, which is a coherent reason to care about knowledge persistence — the concern that motivates distinguishing durable knowledge from transient operational artifacts.
