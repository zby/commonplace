=== SEMANTIC REVIEW: unit-testing-llm-instructions-requires-mocking-the-tool-boundary.md ===

Claims identified: 13

WARN:
- [completeness] The three-part test fixture ("Mock KB state, Mock tool layer, Assertions") omits a fourth element the note itself uses: the **instructions under test** (the skill being exercised). A fixture without instructions to execute is inert. The note's connect-skill example implicitly supplies this ("For the connect skill, the mock surface is..."), but the enumeration claims three parts, not four. A reader building a test harness from the enumeration alone would have no slot for the skill text.
- [completeness] The assertion taxonomy ("behavioral assertions (hard oracle)" vs "output assertions (soft oracle)") maps neatly to oracle-strength-spectrum but misses a third kind the note's own examples hint at: **absence assertions** — "No Edit calls made," "No false connections to unrelated fixture notes." These are neither behavioral-sequence checks (they assert non-occurrence, not sequencing) nor output-quality checks. They could be force-fit into "behavioral," but the fit is strained: asserting that something *didn't happen* is a different verification operation from asserting that a sequence *did happen*. The binary split may be incomplete.
- [grounding] The note claims instruction tests "sit between skills and hooks" on the methodology enforcement gradient. Checking methodology-enforcement-is-constraining.md: that note's gradient is instruction -> skill -> hook (warn) -> hook (block) -> script, ordered by reliability. Instruction tests are not an enforcement layer in that gradient at all — they are a *verification* mechanism applied to skills, not a layer that triggers methodology. The note is inserting instruction tests into a gradient they were not part of, which readers could mistake for an established positioning from the source note.

INFO:
- [completeness] The boundary case of **stateless skills** (skills that produce output purely from their instructions without reading any KB state) is not addressed. The three-part fixture assumes mock KB state is always needed, but a skill like "reformat this text" might need only mock tools and assertions. This is a minor gap — the fixture still works with an empty mock KB — but the "five to ten notes suffice" guidance implies KB state is always present.
- [completeness] The note frames tool mocking as the *only* mechanism for instruction testing ("This makes the tool layer the natural seam for dependency injection"). An alternative approach — prompt-level testing where you provide real tools but constrained inputs (e.g., a real but tiny KB directory) — is not considered. This is an integration-test analogue that sits between the note's unit-test proposal and production use. The note's framing is defensible (it is specifically about unit testing), but the scope claim ("the natural seam") could be read as excluding other valid seams.
- [grounding] The note says "This is the instruction-testing half of the doubled testing surface" and cites programming-practices-apply-to-prompting.md. That source does identify a "doubled testing surface" — but it frames the doubling as *test runs* (indeterminism) and *test targets* (underspecification), not as instruction testing vs. artifact testing. The source's "Underspecification doubles the test targets" passage says you need to "test the instructions themselves (are they consistent? unambiguous? sufficiently constraining?) as well as the outputs" — which is about inspecting the spec for ambiguity, not about mocking tool boundaries and running skills. The note's concrete mechanism (tool mocking) is a reasonable operationalization of the source's abstract claim, but it is the note's own contribution, not a direct attribution.
- [internal-consistency] The "Open questions" section asks "What granularity? Test full skill execution, or test individual phases?" — but the Architecture section already commits to full-skill execution (the fixture runs the whole skill against mocks). The open question reads as if the note hasn't taken a position, when it has implicitly taken one. Minor tension, not a contradiction.

PASS:
- [grounding] The note's claim that "a test suite for instructions is a constraining mechanism" aligns well with constraining.md's definition ("narrows the interpretation space"). The note's formulation — "each test constrains the interpretation space by asserting 'this instruction, given these inputs, must produce behavior in this range'" — is a faithful application of the constraining concept.
- [grounding] The two-kind assertion split (behavioral = hard oracle, output = soft oracle) maps correctly to oracle-strength-spectrum.md. Behavioral assertions (tool call patterns) are deterministic and cheap to check, matching the hard-oracle definition. Output assertions (connection quality) require judgment or careful fixture design, matching the soft-oracle definition.
- [grounding] The note's claim that "both are needed because underspecification and indeterminism create failures at both levels" correctly reflects agentic-systems-interpret-underspecified-instructions.md, which treats these as two distinct phenomena requiring different testing approaches.
- [grounding] The note's framing of skills as "programs written in natural language and executed by an LLM" with "dependencies that flow through tool calls" is consistent with the broader KB's treatment of skills in methodology-enforcement-is-constraining.md (skills have deterministic triggers with indeterministic LLM execution).
- [internal-consistency] The note is internally consistent across its sections. The Architecture section's three-part fixture, the assertion taxonomy, the constraining framing, and the open questions all cohere. No definition drift detected — "instruction testing," "tool boundary," "constraining," and "oracle strength" are used consistently throughout.
