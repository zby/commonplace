=== SEMANTIC REVIEW: specification-level-separation-recovers-scoping-before-it-recovers-error-correction.md ===

Claims identified: 10

WARN:
- [Completeness] The three-level taxonomy (flat prompting / specification-level separation / architectural separation) is presented as a covering partition of the space between no-structure and full-structure, but the note itself identifies tool-use frameworks as "a different route to partial reliability that does not pass through the intermediate regime described here." This means the taxonomy does not cover the space — it describes one axis (naming structure within the LLM's execution) while acknowledging a second axis (interface-boundary codification) that produces a different intermediate regime. The note frames this as a contrast, but a reader could mistake the three levels for an exhaustive ordering. A system that combines both paths (e.g., a DSL with typed tool outputs validated by JSON schema) has no clear placement in the taxonomy.

- [Grounding alignment — scope of "scoping benefits"] The note claims OpenProse-style systems "recover some of the scoping benefits described in [LLM context is composed without scoping]." The scoping note identifies several distinct benefits: isolation via sub-agents (lexical scoping), prevention of spooky action at a distance, avoidance of name collision, and the ability to reason locally. Specification-level separation as described here — naming control flow, marking discretion, externalising state into files — plausibly addresses externalisation (moving state out of the conversation) and some frame-boundary clarity, but does not provide the sub-agent isolation that the scoping note identifies as "the one place where real isolation is achievable." The note does not distinguish which scoping benefits are recovered and which are not, risking an overstatement of what specification-level separation delivers relative to its cited source.

INFO:
- [Completeness — boundary case: resumability] The note lists "scoping, resumability, and some orchestration discipline" as gains of the intermediate regime. Resumability is attributed without argument. Whether a specification-level system achieves resumability depends on how reliably the LLM interpreter can reconstruct execution state from externalised files after an interruption. If the parser and state loader are themselves LLM-mediated (as the note acknowledges), resumability is probabilistic rather than guaranteed — the system might misinterpret the saved state on reload. The note could distinguish between having a resumability *protocol* (which specification-level separation provides) and having reliable resumability (which may still require architectural separation).

- [Grounding alignment — "named but not hardened"] The note says the boundary between symbolic bookkeeping and semantic judgment is "named, but not yet hardened." The scheduler-LLM separation note frames the asymmetry in terms of three phenomena (underspecification, indeterminism, bias) and argues that symbolic substrates eliminate all three for bookkeeping. The current note's "named but not hardened" formulation is a reasonable compression, but it elides the specific mechanism: what "hardened" means is that discrete-state restoration eliminates the three phenomena. A reader unfamiliar with the source note might read "hardened" as meaning merely "enforced by convention" rather than "moved to a substrate with fundamentally different error properties." The inference is valid but the compression loses precision.

- [Grounding alignment — oracle-strength-spectrum] The note cites oracle-strength-spectrum to ground the claim that "compile-time validation and `**...**` conditions remain soft-oracle operations." The oracle-strength note defines soft oracles as "proxy score that correlates but isn't the real thing" with examples like BLEU, helpfulness rubrics, heuristic checks. LLM-mediated compile validation is arguably weaker than a soft oracle in the spectrum's terms — it is an LLM judgment call, not even a proxy score. The attribution is directionally correct (not hard oracle), but the specific placement on the spectrum could be more precise.

- [Internal consistency — "feel substantially better"] The note claims specification-level separation "can feel substantially better than raw prompting." This phenomenological claim is unsupported within the note and sits oddly alongside the more analytical claims. The note's own framework would predict improved scoping and reduced ambiguity, but whether this translates to a subjective experience of "substantially better" is an empirical claim the note cannot ground. Minor, since the sentence is clearly functioning as a motivating intuition rather than a formal claim.

PASS:

The characterisation of each level in the bulleted list matches how the body uses the terms. Flat prompting = no explicit scheduler vocabulary; specification-level = named structure but LLM-mediated execution; architectural = bookkeeping in code. No definition drift detected.

- [Internal consistency — relationship to scheduler note] The note positions itself as identifying "a meaningful middle ground" that the scheduler note should not be read as excluding. The scheduler note (read in full) does in fact contain a paragraph acknowledging this intermediate regime and linking back to this note, confirming the relationship is bidirectional and consistent. No contradiction.

- [Grounding alignment — codification contrast] The note claims OpenProse "constrains orchestration practice but does not codify the runtime semantics," citing the codification note. The codification note defines codification as "constraining that crosses a medium boundary from natural language to a symbolic medium (code), where the consumer changes (LLM -> interpreter)." Since OpenProse's runtime consumer remains the LLM, the contrast is accurately drawn.

- [Grounding alignment — tool-use as different path] The note distinguishes tool-use frameworks from specification-level separation, calling them "mini-codification at the boundary." This aligns with the codification note's definition (medium boundary crossed, consumer changes to interpreter for the typed parameters) and the oracle-strength note's framing (JSON schema validation is a hard oracle). The distinction is well-grounded.

- [Internal consistency — cost claim] The final paragraph's claim that "bookkeeping is still paid for on the stochastic substrate" is consistent with the body's characterisation that the parser, validator, and scheduler remain LLM-mediated. No contradiction with earlier claims about what the intermediate regime gains.
