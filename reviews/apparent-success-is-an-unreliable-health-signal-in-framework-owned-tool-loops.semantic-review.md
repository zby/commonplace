WARN:
- [Completeness] The three-outcome enumeration ("primary-path success," "fallback success," "hard failure") omits a fourth plausible outcome: **partial success with acknowledged degradation** — a run where the framework explicitly signals that it used a fallback but still completed. The note later proposes exactly this ("degraded execution" as a first-class notion) but does not register it as a missing item in the enumeration that motivates the argument. The enumeration describes the current state, and the proposed fix implies a fourth state that the enumeration does not account for. This is not a logical error — the note's point is that frameworks currently compress to a binary — but the gap between the enumeration and the later proposal could confuse a reader who expects the enumeration to be exhaustive over the design space rather than descriptive of current practice.

- [Completeness] The two observability strategies (synchronous reporting and asynchronous observation) are presented as covering the recovery space: "There are two ways to recover the missing signal." A boundary case falls between them: **instrumented fallback with structured metadata** — a mechanism where the runtime annotates the artifact itself (e.g., provenance headers, fidelity tags on the output) rather than reporting to the user or logging for later analysis. This is neither "the run tells the user" (synchronous reporting) nor "a later process scans logs" (asynchronous observation); it is the artifact carrying its own degradation record. The note's two-item enumeration may undercount the design space.

- [Grounding] The note attributes to the tool-loop index: "This is one reason tool loop argues for keeping orchestration visible." The tool-loop index does argue for exposing the loop, but its reasons are about enabling different tool surfaces, handling multi-window tasks, and avoiding hidden schedulers — not primarily about observability of fallback paths. The index's "Downstream consequences" section does list this note as a consequence, but does not itself argue that hidden orchestration hides the fallback/success distinction. The attribution reverses the direction: the tool-loop index links to this note as a downstream consequence, not as an argument it makes. The note reads as though the tool-loop index independently motivates the observability point, when the tool-loop index actually points back to this note for that claim.

This is consistent with the bounded-context orchestration model, which indeed says nothing about how individual calls are executed or whether fallbacks occur. However, the scheduler in that model includes a `satisfied(K)` check. If fallback success corrupts the signal reaching `satisfied`, the scheduler's ability to function correctly is affected — so the issue may not be purely "one layer down." The note's boundary claim is defensible but may understate the degree to which degraded execution propagates upward.

These are compatible but not identical terms — one refers to a path, the other to an execution status. The note does not define either term precisely; the conceptual referent is clear from context, but readers looking for a crisp definition will find two slightly different phrasings.
