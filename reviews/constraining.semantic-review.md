WARN:
- [Completeness] The constraining spectrum table lists five rows (store output, write description, create convention, add structured sections, extract deterministic function) but the note says "constraining is a gradient, not a single operation." The table presents discrete steps without acknowledging cases that sit between them. Specifically: **renaming a file to follow a naming convention** is both "creating a convention" and "storing an output" (the name is a committed interpretation). **Adding an eval or test harness** constrains behavior significantly (the linked note on methodology-enforcement-is-constraining.md covers hooks, which occupy exactly this middle ground) but does not appear in the spectrum table. The table reads as illustrative rather than exhaustive, but the absence of any qualifier ("e.g.," "such as," "key steps include") makes it look like a complete enumeration. The linked underspecification note's own constraining discussion includes prompt restructuring and evals, which this note's spectrum table omits.
- [Completeness] The note claims constraining and distillation are "the two co-equal learning mechanisms" (opening line). This is a scope claim ("two" implies completeness). Boundary case: **pruning** (removing outdated or wrong artifacts from the KB) is a learning operation — the system's behavior changes because knowledge is removed. Pruning is neither distillation (no extraction/compression) nor constraining (no narrowing of interpretation space on a spec). It is closer to "forgetting" or "unlearning." The methodology-enforcement note itself observes that "removals (23) and section deletions (2)" occur, changing system behavior. If pruning is a third mechanism, the "two co-equal" framing is incomplete. If pruning is subsumed under one of the two, the note should explain how.
- [Grounding] The note says "When scale makes a general approach good enough on reliability+speed+cost, the bitter lesson boundary tells you to relax." The bitter-lesson-boundary note actually argues something more specific: it distinguishes cases where the spec IS the problem (arithmetic regime — constraining survives) from cases where the spec approximates the problem (vision-feature regime — scale eats the constrained solution). The bitter lesson boundary doesn't simply say "when scale is good enough, relax" — it says constraining is permanent when the spec fully captures the problem, and vulnerable when it doesn't. The note's attribution flattens this distinction into a simpler trigger ("scale makes a general approach good enough"), which loses the mechanism the source note provides.

The note may be using "constraining" in two senses: (1) narrowing how an agent interprets a spec, and (2) committing to any specific interpretation/characterization. The opening defines sense (1); the table uses sense (2) in at least one row.

But some constrain/relax cycles ARE maintenance — e.g., relaxing a brittle convention that stopped working after an environment change is repair, not learning. The note's claim that the cycle is "not maintenance" is defensible as a reframing but overstates the distinction.
