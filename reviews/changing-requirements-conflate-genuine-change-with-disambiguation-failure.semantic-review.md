WARN:
- [Completeness] The two-kind enumeration (genuine change vs. disambiguation failure) may miss a third category: **emergent understanding** — cases where neither the world changed nor the spec was ambiguous, but execution itself produces new understanding that makes the original spec inadequate. Example: a team builds a data pipeline per a clear spec, but seeing it run reveals that the data itself has properties nobody anticipated (e.g., temporal correlations that make batch processing inappropriate). The spec was not ambiguous — batch processing was clearly specified and correctly interpreted — but contact with reality revealed the spec was asking for the wrong thing. This is neither "the market moved" (genuine change) nor "someone read the spec wrong" (disambiguation failure); it is learning through execution that the problem was mis-framed. The note's open question about retrospectives hints at this ("we learned the world changed" vs. "we discovered our reading of the spec was wrong") but does not acknowledge a third possibility: "we discovered our understanding of the problem was incomplete." This case can be forced into "genuine change" only by stretching "the world moved" to include "our understanding of the world moved," which dilutes the category.

- [Completeness] The note claims disambiguation failure means "going back to the original spec and choosing a different interpretation — or narrowing the spec." But some disambiguation failures reveal that the original spec itself was insufficient — no available interpretation of the upstream spec was correct because the spec failed to mention a relevant constraint. This is distinct from ambiguity (multiple valid readings) — it is incompleteness (no reading covers the actual requirement). The note's framing conflates underspecification-as-ambiguity with underspecification-as-omission.

INFO:
- [Completeness] The note's boundary between the two categories may be blurrier than presented. Consider: a stakeholder reviews a deployed feature, sees it does X, and says "no, I wanted Y." Was the requirement always Y (disambiguation failure — the spec admitted both X and Y, and the team picked wrong)? Or did seeing X crystallize the stakeholder's preference for Y (genuine change — they didn't know they wanted Y until they saw not-Y)? This is the well-known phenomenon where deployment itself changes requirements by making preferences concrete. The note does not address cases where the categories are genuinely entangled rather than merely conflated.

- [Grounding alignment] The note claims the cascading interpretation problem is "the same one-to-many mapping that the spec-to-program projection model describes for LLM systems." The source note (agentic-systems-interpret-underspecified-instructions.md) carefully distinguishes two phenomena: semantic underspecification and execution indeterminism. The reviewed note maps only the first phenomenon (underspecification) to traditional development, which is accurate. However, the source note's projection model is specifically about LLMs collapsing interpretation at runtime via sampling — in traditional development, the "projection" happens in a human developer's head over days/weeks, with social review processes (code review, QA) that can catch it before deployment. The note acknowledges this difference ("implicit — buried in implementation decisions") but still calls it "the same" mapping, which slightly overstates the structural similarity.

- [Grounding alignment] The note states constraining is "the direct mechanism for resolving disambiguation." The constraining note defines constraining as "narrowing the interpretation space" and includes a spectrum from conventions to deterministic code. This is accurate — constraining does resolve disambiguation. But the word "direct" implies constraining is the primary or exclusive mechanism. The deploy-time learning note describes a broader loop (deploy, observe, learn, codify) where observation and learning are also mechanisms for resolving disambiguation; constraining is the step where the resolution is committed, not the step where the correct interpretation is identified. This is a minor emphasis issue rather than an error.
