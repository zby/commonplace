=== SEMANTIC REVIEW: ephemeral-computation-prevents-accumulation.md ===

Claims identified: 17

WARN:
- [Completeness] The binary fork ("keep the artifacts or discard them") omits a significant intermediate: systems that generate artifacts, use them, and then selectively persist some while discarding others within a single run. The note acknowledges this later ("Most practical systems sit somewhere between") but the opening framing in "The fork" section presents it as a clean binary: "Systems that generate artifacts face a fork: keep the artifacts or discard them. The choice determines whether the system can learn across runs." The spectrum acknowledgment in "Ephemerality as anti-codification" partially rescues this, but the title claim and opening paragraph both assert a binary that the body then softens. A reader who stops at "The fork" section gets a stronger claim than the note actually defends.

- [Completeness] The three benefits of ephemerality ("No approval problem," "No state management," "No maintenance burden") are introduced under a precondition that is easy to miss: "restricting them to pure computation (no side effects)." The benefits enumeration conflates two independent choices: (1) ephemeral vs. persistent artifacts and (2) pure computation vs. side-effecting computation. A system could persist artifacts that are pure computations (getting the maintenance burden but not the approval problem), or run ephemeral side-effecting computations (getting the approval problem but not the maintenance burden). The note bundles these as though ephemerality alone buys all three, but the precondition "combined with restricting them to pure computation" is doing significant load-bearing work that the enumeration items don't individually acknowledge.

- [Grounding] The note attributes to Kirsch: "What has to be rediscovered is not just code shape but production memory: deployment-discovered edge cases, migrations and integration constraints, interface expectations, and audit-relevant interpretations." The ingest file confirms Kirsch names these four barriers. However, the note's framing "production memory" is the note's own vocabulary, not Kirsch's. Kirsch's essay (per the ingest) argues about "state, integration surfaces, interface stability, and auditability" — he frames these as structural barriers to disposability, not as "production memory." The repackaging as "memory" imports a learning-theory connotation that aligns with this KB's framework but is not present in the source. This is a reasonable inference but a vocabulary mismatch worth flagging.

INFO:
- [Completeness] The four costs of ephemerality ("No learning across runs," "No testing," "No review," "No reuse") omit a boundary case: systems that achieve partial accumulation through indirect channels without persisting the artifact itself. For example, an ephemeral system whose outputs are logged, reviewed post-hoc, and used to improve the prompt or system instructions for the next run achieves a form of learning across runs without persisting the generated artifact. The note's framing treats accumulation as artifact-mediated only, but prompt/instruction improvement is a real accumulation channel that bypasses artifact persistence. This is a strained fit rather than a clear gap — one could argue the improved prompt is itself a persisted artifact — but the note's "generate -> execute -> discard" model does not account for it.

- [Completeness] The RLM example ("Recursive Language Models work this way — the LLM writes code, the REPL executes it, the result feeds back into the conversation, and the code disappears") is stated as the canonical ephemeral system, but conversation history itself is a form of accumulation within a session. The code disappears, but the result persists in the conversation context and can influence later steps. The note's claim that the code "disappears" is true at the artifact level but arguably false at the information level within a session. This is a minor boundary case — the note's point is about cross-run learning, and RLMs do lose everything between sessions.

- [Internal consistency] The note defines ephemeral computation as "the deliberate refusal to codify" and says "It stays permanently in the stochastic regime, re-deriving solutions each time." But the earlier section says ephemeral systems "generate an artifact, use it, and discard it." The generated artifact (e.g., code executed by a REPL) is deterministic at execution time — a Python script that computes a sum is not stochastic when it runs. The note conflates "stochastic generation process" with "stochastic regime" as a property of the system. What stays stochastic is the generation, not the execution. The phrasing "stays permanently in the stochastic regime" slightly overstates: the system cycles through stochastic generation and deterministic execution, never crossing into persistent deterministic artifacts.

- [Grounding] The note links to codification.md and claims "Codification converts stochastic LLM behavior into deterministic, testable artifacts." The codification note actually says codification is "the point where constraining the interpretation space crosses a medium boundary" and describes the transition from "stochastic -> deterministic" as one of several changes. The note's one-line summary is a reasonable compression but emphasizes the stochastic/deterministic axis while dropping the medium-boundary and consumer-change aspects that the codification note treats as co-equal. This is a lossy but not misleading compression.

PASS:

The ephemeral side maps to "no state management, no maintenance burden" and the accumulating side maps to "each run potentially makes the next run easier." No contradiction.

The linked note's title and content confirm this is its central claim. Attribution is accurate.

That note's content confirms it discusses exactly this: codification and its complement (relaxing) as bidirectional movements. Attribution is accurate.

That note's description and content confirm it explicitly builds on this note's fork by adding reach as the discriminating variable. The relationship semantics are accurate.
