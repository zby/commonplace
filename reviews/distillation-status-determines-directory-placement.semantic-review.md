=== SEMANTIC REVIEW: distillation-status-determines-directory-placement.md ===

Claims identified: 10

1. [Title/Opening] The criterion for directory placement is "distilled into an executable procedure" -- not "compressed" or "frequently loaded."
2. [Opening] WRITING.md "sits at the top of the KB directory as a writing guide" and is a procedure that belongs in kb/instructions/.
3. [Maintenance] If all procedural distillations live in one place, you can audit them as a set.
4. [Legibility] "Is this a procedure an agent executes, or reasoning an agent deliberates with?" is a question agents and humans can answer quickly -- a clean binary.
5. [Legibility] The boundary maps directly to reading mode: follow steps vs. build understanding.
6. [Loading hierarchy] The loading-frequency note describes a hierarchy from always-loaded to on-demand; procedures are loaded when an agent needs to act, discursive notes when an agent needs to reason.
7. [Loading hierarchy] Grouping by reading mode makes the hierarchy concrete in the filesystem.
8. [Link: skills-derive] That note "defines the distillation relationship this note proposes using as a directory boundary."
9. [Link: areas-exist] "areas organize notes for comparative reading; distillation status organizes artifacts for maintenance operations" -- a parallel.
10. [Disconfirming evidence] Some artifacts are genuinely mixed -- partly distilled procedure, partly discursive reasoning -- and the boundary forces an awkward split.

WARN:
- [Completeness] The note proposes a binary boundary (procedure -> instructions/, reasoning -> notes/) but the codebase contains type templates in kb/notes/types/ (structured-claim.md, adr.md, index.md, related-system.md) that are procedural artifacts -- skeleton templates with imperative guidance on how to fill sections -- living inside kb/notes/. These are not discursive reasoning; they are execution-oriented artifacts that tell an agent "put X here." The note's framework does not account for them. They are a clear case that falls between the two categories: procedural in function but serving as structural templates rather than standalone procedures. The note's own disconfirming-evidence section anticipates "genuinely mixed" artifacts, but type templates are not mixed -- they are unambiguously procedural, just a different kind of procedure (structural template vs. step-sequenced workflow). This is a gap in the framework, not a known limitation.
- [Grounding] The note states WRITING.md "sits at the top of the KB directory as a writing guide." In the current filesystem, WRITING.md already lives at kb/instructions/WRITING.md, not at kb/. The note's "immediate trigger" -- that WRITING.md is misplaced and should move to instructions/ -- appears to describe a state that no longer holds (or never held in the current tree). This weakens the motivating example, since the artifact the note uses to illustrate the principle is already where the principle says it should be.

The note might argue CLAUDE.md is a special case (control-plane router), but its existence as a prominent mixed artifact is worth noting as a boundary case the framework should address.

These are neither procedures nor discursive reasoning -- they are reference material. The note's binary does not need to cover sources (they have their own directory), but the omission means the note is not proposing a complete organizational principle for the KB, only a sharper boundary between two of its directories. The linked note (instruction-specificity-should-match-loading-frequency.md) describes a hierarchy organized by specificity and loading frequency (CLAUDE.md -> skill descriptions -> skill bodies -> task-specific docs), not by reading mode. The alignment the note claims is a reasonable inference, but the source note does not frame it in terms of "reading mode" or "procedure vs. reasoning." The note is extending the source's framework rather than attributing it.
- [Internal consistency] The note's third open question asks: "How does this interact with the constraining gradient? Distillation and constraining are different operations -- does the directory boundary conflate them?" This is an important self-identified tension. The linked note (methodology-enforcement-is-constraining.md) places instructions on a constraining gradient (ad hoc prompt -> instruction -> skill -> hook -> script). In that framework, instructions are defined by their position on the enforcement-reliability axis, not by their distillation status. The two organizational principles (distillation status vs. enforcement position) could produce conflicting placement recommendations for artifacts that are highly constrained but not procedurally distilled, or procedurally distilled but loosely enforced. The note flags this honestly as an open question, which is appropriate.
