=== SEMANTIC REVIEW: error-correction-works-above-chance-oracles-with-decorrelated-checks.md ===

Claims identified: 14

1. [Amplification condition] "The oracle has discriminative power when TPR > FPR" — this is the condition for amplification to work (section: The amplification condition)
2. [Condorcet] Condorcet's jury theorem (1785): if each voter independently chooses correctly with probability p > 1/2, majority vote converges to the correct answer (section: The amplification condition)
3. [Aggregate accuracy misleading] "accuracy > 50%" is misleading when classes are imbalanced; an always-accept oracle has 90% accuracy but no discriminative power (section: The amplification condition)
4. [Boosting equivalence] The boosting formulation handles imbalanced classes directly: a weak learner must achieve > 50% accuracy on any reweighted distribution (section: The amplification condition)
5. [Scaling law] The number of repetitions needed grows as 1/(TPR - FPR)^2 (section: How discriminative power determines cost)
6. [Decorrelation binding] "Signal strength alone is not enough. Repetitions must produce independent errors." Correlated noise does not average out (section: Decorrelation)
7. [LLM systematic bias] LLMs have systematic biases, so naive repetition produces highly correlated checks (section: Decorrelation)
8. [Four decorrelation strategies] Vary prompt, vary model, metamorphic transformations, orthogonal check dimensions (section: Decorrelation)
9. [Content bias shared across models] Content effects are shared across architecturally different models (Chinchilla, PaLM 2, GPT-3.5) and survive scaling and instruction tuning (section: Content bias)
10. [Model diversity insufficient] "Model diversity alone is insufficient for decorrelating reasoning errors" driven by content bias (section: Content bias)
11. [CoT as decorrelation] Chain-of-thought prompting partially achieves decorrelation by improving performance on abstract/unfamiliar conditions (section: Content bias)
12. [Five oracle construction methods] Structural checks, self-consistency, metamorphic checks, LLM-as-judge, cross-document consistency (section: Ways to construct)
13. [Structured types as error correction] Structured document types already function as error correction (section: Implications)
14. [Codification not prerequisite] Codification moves toward harder oracles but is not a prerequisite for error correction (section: Implications)

WARN:
- [Completeness — decorrelation strategies] The four decorrelation strategies (vary prompt, vary model, metamorphic transformations, orthogonal check dimensions) are presented as a list without an explicit claim of exhaustiveness, but the framing ("Decorrelation strategies must actively disrupt the sources of correlation") implies coverage of the strategy space. A missing strategy: **varying the decision threshold or acceptance criterion**. If the oracle's accept/reject boundary is shifted (e.g., stricter criteria on some checks, looser on others), different checks will make errors on different instances even with the same model and prompt. This is a standard technique in ensemble methods (diversity via decision boundary perturbation). It sits between "vary the prompt" and "orthogonal check dimensions" but is not clearly covered by either. The note's list is reasonable as a practical guide but may mislead readers into thinking decorrelation requires changing the input or the model, when changing the criterion also works.

- [Grounding — CoT as content-bias decorrelation] The note claims "Chain-of-thought prompting partially achieves this [decorrelation] by improving performance on abstract/unfamiliar conditions without degrading familiar ones, suggesting that process structure can serve as a content-bias decorrelation mechanism." The Lampinen et al. source does show CoT improves abstract/unfamiliar performance, but it does not test or discuss CoT as a *decorrelation* mechanism across multiple checks. The source shows CoT reduces content bias *within a single pass* (improving accuracy on hard conditions). The note's inference — that this implies decorrelation *between repeated checks* — is a substantial leap. Reducing bias in a single oracle and decorrelating errors across multiple oracle instances are different properties. A less biased oracle is better (higher TPR-FPR gap), but it is not necessarily more decorrelated from other oracles using the same CoT prompt. The note labels the link as "speculative" in the Relevant Notes section, but the body text presents it without that qualifier.

- [Grounding — domain coverage] The note's central framework (TPR > FPR + decorrelation = amplification) is grounded in Condorcet's jury theorem and boosting theory, both of which assume *binary* classification (correct/incorrect). The note applies this to LLM output checking, where the decision is often not binary — an output can be partially correct, correct in structure but wrong in content, etc. The MAKER source sidesteps this because Towers of Hanoi moves are strictly right or wrong. The note generalizes beyond MAKER to soft oracles and semantic checks (section: "Ways to construct above-chance oracles") where the binary assumption is strained. The note never acknowledges this gap between the binary-decision grounding and the richer decision space of real LLM output evaluation.

INFO:
- [Completeness — oracle construction methods] The five methods for constructing above-chance oracles are presented as a practical list. A boundary case: **external tool verification** (running code, calling an API, checking a database) is not clearly covered. It could be forced into "structural checks" but the fit is strained — structural checks as described ("required sections, valid formatting, correct length") are about document structure, while running generated code against test cases is a functional check that goes beyond structure. The MAKER source's own oracle (checking whether a Hanoi move is legal) is closer to functional verification than to any of the five listed methods. Given that MAKER is the note's primary empirical grounding, this omission is notable.

- [Completeness — scaling law boundary] The 1/(TPR-FPR)^2 scaling law is presented with a clean numerical table (gap 0.5 -> ~4 repetitions, gap 0.1 -> ~100, etc.). The simplest possible instance — gap approaching 1.0 (a nearly perfect oracle) — implies ~1 repetition, which is correct and unremarkable. The most extreme instance — gap of exactly 0 — is covered ("no number of repetitions helps"). An intermediate boundary worth noting: the scaling law assumes the gap is *known*. In practice, estimating TPR and FPR requires labeled data, which is precisely what you lack when you need an oracle. The note's practical question "how cheaply can you get TPR > FPR?" implicitly assumes you can *measure* the gap, but measurement cost is not discussed.

- [Internal consistency — "structured types as error correction"] The Implications section claims "Structured document types already function as error correction: they constrain the output space (structural checks) and steer the model toward higher-quality training distributions." This conflates two different mechanisms under "error correction": (1) constraining the output space is a *prevention* mechanism (reducing the chance of error on a single pass), and (2) error correction as developed in the body of the note is an *amplification* mechanism (detecting and correcting errors across multiple passes via voting). The body carefully distinguishes signal strength from amplification, but the Implications section blurs this by calling output-space constraint "error correction." The note could mean that structured types increase the TPR-FPR gap of downstream checks (making amplification cheaper), which would be consistent — but as written, it reads as equating prevention with correction.

- [Grounding — MAKER attribution] The note says "MAKER uses hard oracles (deterministic verification)." The MAKER ingest confirms this — Towers of Hanoi has deterministic per-step correctness. The note then claims to generalize "beyond hard oracles." This generalization is the note's own theoretical contribution, clearly signposted. However, the note says "voting across independent samples achieves zero errors over a million steps" without noting that the MAKER ingest's Limitations section flags "correlated errors are mitigated, not solved" — the paper acknowledges steps with "substantially higher inherent error rates" handled by brute-force additional sampling, not by the theoretical i.i.d. framework. The note's phrasing makes MAKER's result sound cleaner than the source presents it.

PASS:
- [Amplification condition — TPR > FPR] The core claim that TPR > FPR is the correct condition for amplification (not aggregate accuracy > 50%) is well-argued. The always-accept oracle counterexample is valid and clearly demonstrates why aggregate accuracy is misleading under class imbalance. The connection to Condorcet (equiprobable classes) and boosting (reweighted distributions) correctly identifies the two classical framings and explains why they converge. No boundary cases break this argument.
- [Scaling law derivation] The 1/(TPR-FPR)^2 scaling follows from standard binomial separation arguments. The intuition ("separating two binomial distributions whose means differ by TPR-FPR, standard deviation scales as 1/sqrt(N)") is correct and the numerical examples check out.
- [Content bias grounding] The attribution to Lampinen et al. is accurate on the core finding: content effects are shared across model families and survive scaling and instruction tuning. The note correctly identifies models tested (Chinchilla, PaLM 2, GPT-3.5) and the survival through Flan-PaLM 2. The conclusion that "model diversity alone is insufficient for decorrelating reasoning errors" follows validly from the source.
- [MAKER as motivating example] The note correctly positions MAKER as using hard oracles and correctly identifies the O(s ln s) scaling and first-to-ahead-by-k voting mechanism. The generalization to soft oracles is clearly marked as the note's own contribution, not attributed to MAKER.
- [Internal consistency — TPR/FPR framework] The TPR/FPR framework is used consistently throughout. The definition in the Amplification section (TPR = P(accept|correct), FPR = P(accept|incorrect)) is maintained in the table, the scaling law, and the oracle construction section. No definition drift detected.

Overall: 3 warnings, 4 info
