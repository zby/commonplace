WARN:
- [Completeness] The note claims semantic recovery is "qualitatively different from exception handling" and that "the recovery path is not pre-specified." This framing omits a middle ground: engineered-but-flexible recovery strategies that exist in traditional software. Chaos engineering, self-healing infrastructure (e.g. Kubernetes pod restarts, circuit breakers with adaptive routing), and rule-based expert systems all synthesize recovery paths that are not fully pre-specified yet are not LLM-based semantic reasoning either. The note's binary (pre-specified vs. semantically synthesized) skips this intermediate zone. A programmer familiar with resilience engineering patterns would not find the old intuition as cleanly broken as the note implies, because they already expect non-trivial recovery without final-failure states.

- [Completeness] The enumeration of what the model can do upon failure — "reinterpret the error, call different tools, browse directly instead of using the intended helper, or reformulate the task so the broken path is no longer needed" — is presented as an illustrative list, not a closed taxonomy, so it is not strictly incomplete. However, a notable omission is **partial execution on the intended path**: the agent might successfully complete most steps on the prescribed route, fail on one, patch that single step with improvisation, and continue on the original path. This hybrid case (mostly-intended path with one improvised link) is arguably the hardest to detect and the most misleading for debugging, yet it does not map cleanly to any of the listed recovery moves. The note would be stronger if it acknowledged that blended paths are the common case, not full rerouting.
