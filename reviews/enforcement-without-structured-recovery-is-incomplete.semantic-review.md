=== SEMANTIC REVIEW: enforcement-without-structured-recovery-is-incomplete.md ===

Claims identified: 17

WARN:
- [Completeness] The oracle-strength-to-recovery mapping omits "Delayed oracle" entirely. The oracle-strength-spectrum note defines five levels (Hard, Soft, Interactive, Delayed, No oracle), but this note's mapping in "Oracle strength constrains recovery automation" uses only four (Hard, Soft, Interactive, No oracle). Delayed oracles — "you only know later" (user churn, bug surfacing, decision payoff) — are a real oracle type with no obvious recovery strategy. A violation detected only after the fact cannot use corrective action, fallback, or escalation in the moment. This is a gap in the framework, not just an omission of a label.
- [Completeness] The three recovery strategies (corrective, fallback, escalation) do not cover temporal retry — retrying the same approach later without changing strategy or involving a human. A warning hook that fires because an external service is unavailable does not need a "simpler, safer strategy" (fallback) or human intervention (escalation); it needs a retry with delay. This sits between fallback and escalation in the cost ordering but maps to neither. The ABC source may not discuss this case, but the note's own framing ("recovery as a typed strategy") claims to cover what happens after enforcement fires, and temporal retry is a common real-world recovery pattern.
- [Grounding] The note presents ABC's recovery framework (corrective -> fallback -> escalation) as providing concrete recovery strategies, but the ABC ingest's Limitations section states: "Recovery is monitoring, not intervention. The authors acknowledge this: recovery mechanisms 'operate as monitoring by default.'" The note does not surface this caveat. Readers may infer that ABC demonstrates working recovery strategies when the paper itself acknowledges that recovery is aspirational. The note's claim that "ABC provides vocabulary for this missing layer" is accurate — but "vocabulary" understates the gap between what ABC describes and what ABC demonstrates.

INFO:
- [Completeness] The note does not address prevention as an alternative to recovery. Structured output schemas, type constraints, and other mechanisms that make violations impossible by construction occupy the same problem space (what happens after enforcement fires) but solve it by eliminating the need for recovery. The note's framing — "detection and blocking are systematic; recovery is ad hoc" — implicitly assumes violations will occur. For some enforcement layers (especially scripts operating on structured data), prevention may be more appropriate than recovery. This is an adjacent strategy, not a missing enumeration item, but the note's scope claim ("what happens *after* enforcement fires") could acknowledge that sometimes the answer is "make sure it doesn't fire."
- [Consistency] The note defines corrective action as requiring "a hard oracle for the corrected state" in the three-strategy enumeration, but the soft-oracle row in the oracle-strength section says "corrective action requires the agent to interpret the signal" and "the correction remains probabilistic." This uses "corrective action" at two different oracle-strength levels — once as requiring hard oracles, once as functioning (probabilistically) with soft oracles. The term drifts from "deterministic fix" to "attempted fix with uncertain outcome." The note partially manages this by distinguishing "fully automated" from "requires interpretation," but the reuse of the same term for both creates ambiguity.
- [Grounding] The note claims "Without structured recovery, gamma approaches zero and drift is unbounded." The D*=alpha/gamma formula supports this mathematically, but gamma=0 (zero recovery rate) is an extreme assumption. Even unstructured recovery — an agent improvising a fix after a warning — has some non-zero recovery rate. The note's actual argument is that *structured* recovery increases gamma relative to *unstructured* recovery, which is a weaker (and more defensible) claim than "gamma approaches zero." The Drift Bounds Theorem supports the directional argument well; the limit case is stronger than necessary.
- [Completeness] Recovery loops and infinite regress are not addressed. If corrective action introduces a new violation that triggers another hook, the recovery framework offers no termination guarantee. ABC's (p,delta,k)-satisfaction model includes a recovery window parameter k that bounds recovery steps, but the note does not mention this bound. For a framework that claims to structure recovery, the absence of a stopping condition is notable.

- [Grounding] The Drift Bounds Theorem attribution (D*=alpha/gamma) matches the ABC ingest summary exactly: "Drift Bounds Theorem using Lyapunov stability analysis showing behavioral drift converges to D*=alpha/gamma (natural drift rate divided by recovery rate)." Faithful.
- [Grounding] The claim that "Teaching error messages narrow the interpretation space" accurately reflects the error-messages-that-teach note, which describes teaching messages as constraining + informing, "frontloading the answer" while leaving application to the agent.
- [Consistency] The note's central thesis — enforcement without structured recovery is incomplete — is consistently supported across all sections. The gap identification, the ABC vocabulary, the oracle-strength mapping, and the open questions all reinforce the same argument without internal contradiction.
- [Consistency] The distinction between detection oracles and recovery oracles for blocking hooks ("Blocking hooks are hard oracles for *detection* but when the violation requires judgment to resolve, the *recovery* is escalation") is internally consistent and adds genuine nuance to the oracle-strength mapping.

The characterization of each layer's current recovery state ("agent improvises," "skill prescribes no failure strategy," etc.) is consistent with the source note's descriptions.
