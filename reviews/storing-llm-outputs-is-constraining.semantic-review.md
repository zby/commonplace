=== SEMANTIC REVIEW: storing-llm-outputs-is-constraining.md ===

Claims identified: 19

WARN:
- [Completeness] The "applies broadly" enumeration (generated code, generated documents, configuration, accumulated logs) is presented as a broad illustration but omits cases that probe the boundary of the title claim. **Caching and memoization** — storing an LLM output purely for performance (avoiding re-generation cost) without any human selection of quality — is a common case of "storing LLM outputs" that does not clearly involve "resolving semantic underspecification." The agent is not choosing one interpretation as preferable; it is saving a result to avoid re-computation. This case sits between items: it is constraining in the freezing-against-indeterminism sense (it locks in one run) but the "semantic commitment" the note calls "the deeper operation" is absent or incidental. The note's framing implies every act of storing is a semantic commitment; caching suggests some storing is purely economic.
- [Completeness] The two-strategy framework ("constrain the generator" vs. "filter the samples") claims to be exhaustive ("There are two strategies for getting reliable output from a generator whose outputs vary"). A third strategy is missing: **iterative refinement / self-correction**, where the same generator revises its own output across turns (chain-of-thought self-critique, multi-turn editing). This is neither constraining the generator (the prompt stays the same) nor filtering samples (no independent verifier rejects/accepts); it is a feedback loop within the generator. Evans' own critic/judge pipeline, cited in the note via the ingest, is closer to this third pattern than to either of the two enumerated strategies.
- [Grounding] The note claims "You get consistently mediocre results" as a consequence of constraining the generator (Strategy 1 — "tighter prompts, more examples, lower temperature"). This is stated as fact but is not grounded in any cited source. The Evans ingest actually contradicts this: Evans' "freeze taxonomy then classify" approach — cited in the very next sentence as "a specific instance" of Strategy 1 — produced *better* results (consistent, high-confidence classifications), not "consistently mediocre" ones. The note uses Evans as an exemplar of the strategy it just characterized as producing mediocre results, creating a tension between the characterization and the cited evidence.

INFO:
- [Completeness] The "applies broadly" enumeration includes "accumulated logs" with append-only formats, which introduces a structural enforcement mechanism (append-only) that is conceptually different from the other three cases (where the constraining is a human or agent *choice* to keep an output). Append-only constraining is a format property, not a selection decision. The note acknowledges this ("enforce constraining structurally") but lists it alongside selection-based cases without flagging the category difference. This is not wrong — both narrow the space — but a reader could mistake structural enforcement for selection-based constraining or vice versa.
- [Grounding] The Koylanai data-loss incident is cited as "concrete evidence that without append-only constraints, agents will accidentally destroy constrained artifacts." The ingest source characterizes this incident as self-reported and not independently verifiable (Limitations section: "Claims like... the 3-month data-loss incident are plausible and useful, but they are not backed here by artifacts we can inspect"). The note's use of "concrete evidence" is slightly stronger than the source's evidentiary basis warrants. The claim is directionally correct but "anecdotal practitioner evidence" (the ingest's own language) would be more accurate than "concrete evidence."
- [Internal consistency] The note says "The semantic commitment is the deeper operation" (opening paragraph) and later says "Constraining the prompt is pushing reliability into the generator instead — a different tradeoff, not a strictly better one" (end of generator/verifier section). These are not contradictory, but there is a mild tension: the first claims semantic commitment is deeper/more important; the second says constraining (which produces that semantic commitment) is merely a different tradeoff, not superior. A reader might ask: if constraining is the deeper operation, why is it not the better strategy?
- [Completeness] The testing section claims "two distinct testing targets" — prompt testing and artifact testing. A boundary case: **regression testing after model updates** — re-running a prompt after the underlying model changes to check if stored artifacts remain the best choice — is partially covered by prompt testing but introduces a temporal dimension (drift) that neither testing target cleanly addresses. The note's parent (deploy-time learning) does mention model drift as a failure mode; this note's testing framework does not account for it.
- [Internal consistency] The verbatim risk section says comparing output against source "is a judgment call with low oracle strength." But the oracle-strength note defines oracle strength as a spectrum (hard, soft, interactive, delayed, no oracle). "Low oracle strength" is vague — it could mean soft oracle (proxy heuristics exist but are imperfect) or no oracle (vibes only). Since the note acknowledges that structural checks exist but are insufficient and that semantic comparison is needed, this seems to sit at the soft-oracle level rather than truly "low." The imprecision does not mislead, but sharper placement on the spectrum would strengthen the claim.

PASS:
- [Grounding] The central claim — that storing an LLM output resolves semantic underspecification and freezes against indeterminism — aligns accurately with the parent note (agentic-systems-interpret-underspecified-instructions.md), which defines constraining as resolving "semantic underspecification by choosing one interpretation from the space the spec admits and committing to it" and "removing execution indeterminism by eliminating sampling noise." The note correctly characterizes this as "the same constraining move... applied to the artifact rather than the implementation."
- [Grounding] The link to constraining.md is accurate. The constraining note explicitly lists "Store an LLM output — Commit to one interpretation" as the first row of the constraining spectrum table, and links back to this note as "the simplest instance."
- [Grounding] The reference to oracle-strength-spectrum.md is accurately used. The oracle-strength note explicitly states "The generator/verifier pattern... only works when oracle strength is sufficient" and links to this note in its "generator/verifier depends on this" section. The bidirectional grounding is consistent.
- [Grounding] The Evans citation is accurately attributed regarding the modeling/classification separation. The ingest confirms Evans argued for separating taxonomy creation from classification application, and this note correctly identifies it as "a specific instance" of constraining the generator (freezing the interpretation space before operating within it).
- [Internal consistency] The note's core terminology — semantic underspecification, execution indeterminism, constraining — is used consistently throughout. The definitions in the opening paragraph carry through to the testing section, generator/verifier section, and verbatim risk section without drift.
- [Internal consistency] The two-part structure (storing as constraining + generator/verifier as alternative) is internally coherent. The second half (generator/verifier) follows logically from the first: if storing is constraining, then an alternative to constraining the prompt is filtering the stored samples.
