=== SEMANTIC REVIEW: instructions-are-skills-without-automatic-routing.md ===

Claims identified: 12

WARN:
- [Completeness] The six properties of instructions ("imperative voice," "step-sequenced," "behaviourally complete," "minimal reasoning," "explicit scope boundaries," "frontloaded") are presented as what distinguishes instructions from notes. But these same properties apply equally to skills — the note's own claim is that instructions are "same distillation quality as skills." If the properties are shared with skills, they don't explain why instructions need a separate directory from skills; they only explain why instructions aren't notes. The enumeration is doing double duty (distinguishing from notes AND justifying the separate directory) but only the first job is covered. What property distinguishes an instruction file from a skill file besides its location and the absence of routing plumbing? The note is silent on whether any structural or content difference exists.

- [Grounding] The note claims methodology notes and instructions have "the same source vs. compiled split that holds between methodology and skills" and links to agent-statelessness-makes-routing-architectural-not-learned.md. That note develops the source/compiled distinction specifically for methodology (source) vs. routing artifacts like skills, routing tables, and type templates (compiled). It emphasizes that compiled artifacts "must be imperative, complete, unambiguous — they're for execution" while source "can be exploratory, tentative, argumentative — it's for reasoning." The target note applies this to methodology notes (source) vs. instructions (compiled). This is reasonable, but the linked source never mentions instructions as a category distinct from skills in its source/compiled framework. The note is extending the linked source's framework to a new category (instructions) without flagging that this is its own move rather than something the source establishes.

INFO:
- [Completeness] The note claims instructions "work with any agent that can read a file and follow it," making the KB "portable across agent platforms." Boundary case: an instruction that depends on tool availability (e.g., an instruction that says "run script X" or "use grep to search"). Such an instruction is portable at the file-reading level but not at the execution level. The platform independence claim is about discoverability mechanism independence, not execution independence, but the note's phrasing ("work with any agent") could be read more broadly than intended.

- [Completeness] The promotion/demotion claim ("Instructions that prove themselves through repeated use get promoted to skills... Skills that turn out to be too niche can be demoted to instructions rather than deleted") implies a clean bidirectional movement. Boundary case: a skill that has accumulated platform-specific routing entries (symlinks, CLAUDE.md entries, slash-command triggers across multiple platforms). Demotion requires removing all these touchpoints — it is not symmetric with promotion (which the note describes as "move it into a dedicated skill subdirectory, add the symlink, and add the routing table entry"). The note presents promotion as frictionless but does not acknowledge that demotion may involve cleanup across multiple files. This is a minor asymmetry, not a flaw.

- [Grounding] The note says ad hoc instructions are "one-off — written for a specific situation, not expected to be reused" and links to ad-hoc-prompts-extend-the-system-without-schema-changes.md. That source describes ad hoc prompts more richly: they "carry judgment that type signatures can't express" and serve as "clean context boundaries" for sub-agent handoff. The target note's characterization is accurate but reductive — it captures the reuse dimension while omitting the expressiveness dimension that the source note develops as its central point (the "Prompts carry what types can't" section). This omission is fine for a brief comparison, but a reader following the link might find the source says something different in emphasis than what the attribution suggests.

- [Internal consistency] The note says instructions should be "frontloaded — self-contained enough for an agent with no prior context. Terms must be defined inline, not by reference to other documents." But the note also says instructions "include a description in frontmatter, same as skills. This helps a human scanning the directory decide which instruction to invoke." These two statements serve different audiences (the agent executing the instruction vs. the human deciding which instruction to invoke) and are compatible, but the tension between "self-contained, no references" and "description helps a human pick the right one" raises a question: does "frontloaded" mean the instruction should not assume the agent has read the description? Or is the description part of the frontloading? The note does not resolve this.

PASS:
- [Internal consistency] The core definition is stable throughout: instructions are defined as "same distillation quality as skills, but without the activation trigger plumbing" in the introduction, "instructions plus routing" (with skills being the "plus routing" side) in the comparison section, and "same process that produces skills" in the creation section. No definition drift detected — the note maintains a consistent position that the procedure is identical, only discoverability differs.

- [Grounding] The note's claim that skills have "two aspects: the procedure itself... and the routing machinery" aligns well with skills-derive-from-methodology-through-distillation.md, which develops the procedure aspect (distillation from methodology) extensively, and with methodology-enforcement-is-constraining.md, which places skills on the enforcement gradient by their trigger mechanism (deterministic user invocation). The two-aspect decomposition is a clean synthesis of what those sources develop separately.

- [Grounding] The note's three-step creation process (do the task manually, notice the stable core, extract the procedure) is consistent with the distillation process described in skills-derive-from-methodology-through-distillation.md ("Distillation requires judgment... What to extract, what to leave behind, what sequence to impose on ideas that aren't inherently sequential — these are design decisions") and with the maturation trajectory in methodology-enforcement-is-constraining.md (instruction -> skill -> hook/script). The note faithfully represents both sources.

- [Grounding] The link to instructions-are-typed-callables.md is used with the relationship label "extends: instructions, like skills, have implicit type signatures." The linked note does develop type signatures for skills and argues instructions should declare which document types they accept. The attribution is accurate — the typed-callables note does extend the instruction concept in the direction claimed.

- [Internal consistency] The note's argument structure is coherent: it defines instructions (intro), lists their properties (section 2), differentiates from adjacent concepts (section 3), describes their creation (section 4), and claims platform independence (section 5). No section contradicts another. The "optimized for execution" properties in section 2 are consistent with the distillation framing in section 4.

Overall: 2 warnings, 4 info
