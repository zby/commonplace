WARN:
- [Completeness / boundary cases] The note claims two limits where hidden schedulers "start to buckle" — changing the action alphabet and sub-goals exceeding one context window — and calls these "the strongest cases." But there is a plausible third case the note does not address: **observability and debuggability**. A hidden scheduler masquerading as tool state is opaque to the application developer. When orchestration logic fails inside the covert runtime, the framework has no visibility into what went wrong or how to recover. This is not an expressivity limit (the note's framing) but an operational limit that is arguably as strong as the two listed. Because the note uses "strongest cases" (scope claim), this omission could mislead readers into thinking the space of limits is exhausted by the two cited. The note's own framing — "forces it to masquerade as tool implementation" — implicitly acknowledges that masquerading has costs, but does not count those costs among the limits.

- [Completeness / boundary cases] The enumeration "retries, checkpoints, branch records, and recursion state" (claim 3) is presented as illustrative but pairs with "enough orchestration logic that the model never has to invoke a visible scheduler." Consider the boundary case of **inter-agent coordination state** — a hidden scheduler that must synchronize with other agents or external systems (e.g., waiting for a human approval, coordinating parallel agent branches with shared mutable state). This state does not map cleanly to any of the four items and may exceed what a singleton runtime behind the tool boundary can manage without becoming visible. The "never has to invoke a visible scheduler" phrasing is strong; the boundary case suggests it holds only for single-agent, single-stream orchestration.
