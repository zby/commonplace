2. [Opening paragraph] "When a spec leaves an implementation-significant branch unresolved and the agent silently picks one reading, success hides that the contract didn't determine the path."

4. [Second paragraph] This is NOT interpreter failure (spec was clear, model violated it).
5. [Second paragraph] This excludes specs that explicitly delegate discretion.

WARN:
- [Completeness — boundary between silent disambiguation and delegated discretion] The note draws a clean boundary: silent disambiguation excludes "specs that explicitly delegate discretion." But many real specs occupy the gray zone between explicit delegation and unacknowledged ambiguity. Consider: "Refactor this for readability" — the spec acknowledges that multiple approaches exist (implicit delegation), yet the user may still have a strong preference they failed to articulate. The agent picks "extract helper functions" when the user wanted "rename variables." Is this silent disambiguation (the spec didn't acknowledge the branch) or delegated discretion (the spec said "readability," implicitly allowing agent choice)? The note's two-way partition (disambiguation vs. delegation) may need a third category: specs that delegate some discretion but underspecify the bounds of that delegation. The current framing could force ambiguous cases into one bin or the other without principled criteria for the classification.

- [Completeness — the "identical" observability claim is stronger than the analogy supports] The note claims the observability failure is "identical" between tool fallback and semantic disambiguation. But the tool-fallback note (apparent-success-is-an-unreliable-health-signal) describes a three-way outcome compression (primary-path success / fallback success / hard failure) and proposes two concrete remedies (synchronous reporting and asynchronous log observation). Semantic disambiguation has a structurally different detection problem: tool failures leave traces in logs (error messages, changed tool calls), while semantic disambiguation leaves no operational trace at all — the agent simply proceeds as if the spec were unambiguous. "Identical" observability failure would mean the same detection mechanisms apply, but the tool-fallback note's remedies (log scanning, degraded-execution reporting) presuppose a detectable event. Silent semantic choices may require different mechanisms (e.g., requiring the agent to narrate interpretation choices, as the Augment bidirectional spec does). The analogy holds at the structural level (success hides a gap), but "identical" overstates the operational similarity.

INFO:
- [Completeness — boundary between silent disambiguation and interpretation error is porous in practice] The note distinguishes silent disambiguation (spec was insufficient, agent repaired) from interpreter failure (spec was clear, model violated it). This is a clean conceptual boundary, but in practice the two interact: a spec might be sufficient for some interpretations but the LLM misreads it as ambiguous and "resolves" an ambiguity that wasn't there. The interpretation-errors note describes cases where "the spec rules [outputs] out, but the LLM fails to comply." A subtler case: the LLM treats an unambiguous spec as ambiguous and makes a silent choice. This would be an interpretation error (the spec was clear) that presents as silent disambiguation (the agent appeared to resolve ambiguity). The note's clean boundary is useful for analysis but may undercount interpretation errors that masquerade as disambiguation.

- [Grounding — "changing requirements" link claim is slightly stronger than the source supports] The link to changing-requirements-conflate-genuine-change-with-disambiguation-failure carries the annotation "sharpens: the first silent choice is already a degraded run state, not only a future maintenance problem." The changing-requirements note focuses on cascading interpretation errors across specs over time — it does not frame the first silent choice as a "degraded run state." That framing (degraded execution) comes from the tool-fallback note. The link annotation synthesizes across two source notes, which is reasonable but means the "sharpens" relationship is the current note's own inference, not something the linked note independently establishes.

- [Completeness — no treatment of severity or frequency] The note treats all silent disambiguation equally. But some disambiguation choices are inconsequential (formatting decisions, variable naming) while others are architecture-altering (choosing an auth pattern, selecting a data model). The analogy to tool fallback could be sharpened by acknowledging that not all semantic gaps are equally problematic — the observability requirement should perhaps be proportional to the stakes of the disambiguated choice. The tool-fallback note at least distinguishes "changes guarantees the user is relying on" from cases where "the goal is infrastructure repair," suggesting a severity axis that the current note does not carry over.

The current note extends this pattern to the semantic layer faithfully: the spec-determined path and the agent-disambiguated path are compressed into the same "task completed" outcome. The structural parallel is sound.
- [Grounding — interpreter failure distinction] The note correctly distinguishes its phenomenon from interpretation errors as defined in interpretation-errors-are-failures-of-the-interpreter. That note defines interpreter failure as "outputs outside the space of valid interpretations" where "the spec rules them out." The current note's subject — disambiguation of genuinely ambiguous specs — is a different phenomenon, and the note draws the boundary accurately.
- [Grounding — underspecification foundation] The note's premise that natural-language specs admit multiple valid interpretations is well-grounded in agentic-systems-interpret-underspecified-instructions, which establishes the spec-to-program projection model and the one-to-many mapping. The current note applies this framework correctly.

The ingest note confirms this: Augment's design has agents surface interpretation choices back to the spec ("found an existing auth context, wired into that"). This is a concrete mechanism for making silent disambiguation visible, and the attribution is accurate.
- [Internal consistency] The note is internally consistent. The two-paragraph structure makes one claim (the analogy) and one boundary (what it is not). No definition drift — "silent disambiguation" is used consistently to mean the agent resolving an unacknowledged spec gap without reporting it. The description in frontmatter ("final success hides that the contract failed to determine the path") faithfully represents the body.
