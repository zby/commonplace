WARN:
- [Completeness] The enumeration of frontloadable items (claim 4) lists four categories — variable resolution, file listings, aggregations, template expansion — but the note's own framing ("can this be computed without the LLM's runtime state?") admits cases that do not clearly map to any of the four. For example: pre-computed type-checking or validation results ("this file passes schema X"), pre-resolved conditional branches (the "dead branches" case mentioned in the indirection note but not here), and pre-fetched API or tool outputs (e.g., embedding a git log rather than asking the LLM to run it). "Aggregations" could be stretched to cover some of these, but the fit is strained for validation results or resolved conditionals. The note does not claim this is an exhaustive taxonomy (no "complete set" language), so this is a soft warning — the list reads as illustrative but could mislead a reader into treating it as the full space.

- [Grounding alignment — domain coverage] The note claims "frontloading addresses the complexity dimension of that scarcity" (claim 2), citing the context-efficiency note. The context-efficiency note does list "Frontloading and partial evaluation" as "(primarily complexity)" in its architectural responses table. However, the frontloading note's own body describes savings that are also volume savings: "A pre-computed listing costs only the bytes of the listing itself; the same listing produced at runtime costs the instruction, the tool call, the full output, and the LLM's interpretation of it." The tool call output and full listing are volume costs, not complexity costs. The note's evidence straddles both dimensions, but its framing claims only the complexity dimension. This is an overstatement of specificity — frontloading addresses both dimensions, with complexity as the primary but not exclusive target.

INFO:
- [Completeness — boundary case] The note's static/dynamic boundary ("can this be computed without the LLM's runtime state?") is acknowledged as sometimes blurry: "some sub-procedures depend partially on known and partially on runtime information." A boundary case that probes this further: a search over a stable corpus with a runtime query. The corpus listing is static, the relevance filtering is dynamic. The note advises "frontload the known parts," but the most valuable frontloading here might be pre-computing an index structure (embeddings, keyword indexes) rather than inserting raw listings. This kind of structural pre-computation — producing a different representation rather than inserting a result — sits between the note's categories. The note's framework handles it via "aggregations" but the mechanism is subtly different (changing representation vs. inserting a result).

- [Grounding alignment — vocabulary] The note says "you've specialised a program with respect to known inputs, producing a residual program in the same medium. That's partial evaluation, not just preprocessing." The homoiconicity note confirms that instructions and data share the same representation (natural language tokens). The PE framing is the note's own theoretical contribution, not something the homoiconicity note claims. This is intellectually honest — the note builds on the homoiconicity observation rather than attributing the PE framing to it — but a reader following the link might expect the homoiconicity note to discuss partial evaluation, which it does not.

- [Internal consistency — claim 7 vs claim 6] Claim 7 says "the context saving comes from replacing derivation with insertion, whether the result is deterministic or still underspecified." This is consistent with claim 6 ("frontloading does not require determinism"). However, the note's examples of non-deterministic frontloading are absent. Every concrete example in the note (variable resolution, file listings, aggregations, template expansion) produces a deterministic result. The claim that frontloading works for non-deterministic cases is asserted but not illustrated, making it harder to evaluate. A concrete example of frontloading a non-deterministic result would strengthen this section.

- [Grounding alignment] The note maps frontloading to the scheduling model: "the single-step case of its separation between symbolic computation and bounded LLM calls." The bounded-context-orchestration-model note confirms this relationship in its Relevant Notes: "frontloading spares execution context — mechanism: the single-step mechanism this note extends to an iterative loop." The attribution is accurate and bidirectional. However, the scheduling model note describes the single-step case as symbolic computation happening outside the LLM call, while the frontloading note frames it as partial evaluation of the instruction itself. These are compatible framings but emphasize different things — the scheduling model sees it as "do computation in code," while this note sees it as "specialize the instruction." Neither note acknowledges the framing difference.

The body consistently frames the benefit as sparing execution context, and the mechanism sections (PE, homoiconicity) support rather than contradict the title. No definition drift detected — "frontloading" means the same thing in every section.
