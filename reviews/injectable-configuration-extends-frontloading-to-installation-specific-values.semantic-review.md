=== SEMANTIC REVIEW: injectable-configuration-extends-frontloading-to-installation-specific-values.md ===

Claims identified: 13

WARN:
- [Completeness] The note claims build-time generation and hardcoded CLAUDE.md values "only handle values that are the same across all installations," positioning injectable configuration as "a third category." But the frontloading note's list of frontloadable items ("paths, project names, configuration values known at setup time") does not restrict itself to cross-installation constants — "configuration values known at setup time" already covers installation-specific values. The note is correct that current *channels* (build-time generation, hardcoded values) don't handle them, but the framing as a new *category of frontloadable thing* overstates the novelty. It is more precisely a new delivery mechanism for a category the frontloading note already recognized. The distinction between category-of-value and delivery-channel is blurred throughout.

- [Completeness] The note claims context savings occur "only when the orchestrator dispatches tasks to sub-agents" with isolated frames. Boundary case: an orchestrator that uses injectable configuration to *pre-filter* or *select* which instructions to load (e.g., "if arscontexta is configured, include the cross-repo review skill; otherwise omit it") would spare context in the main window by preventing irrelevant instructions from being loaded at all — not through frame isolation but through conditional inclusion. This is a context saving from injection that does not require sub-agent dispatch. The "only" claim is too strong.

- [Grounding — domain coverage] The note's comparison table claims auto-injection's trigger is "Document reference encountered during reading" while injectable configuration's trigger is "Task/skill loaded by orchestrator." But the auto-injection source note explicitly discusses triggers beyond document references: "term detection in the document text could also serve as triggers" and lists multiple trigger candidates (type-based, link-semantic, metadata-driven). The table simplifies auto-injection's trigger model to sharpen the contrast with injectable configuration, making the two channels appear more distinct than the source supports.

INFO:
- [Completeness — boundary case] The note's two defining properties ("static at runtime" and "variable at install time") are tested against three examples (sibling repo paths, local tool paths, environment-specific endpoints). Boundary case: API keys listed under "environment-specific endpoints" may not be fully static at runtime — keys rotate, tokens expire mid-session. The note groups them with truly static paths, but their lifetime may be shorter than a session. This is acknowledged implicitly by the "Lifetime: Installation" row in the comparison table, but the API key example strains the "static at runtime" criterion.

- [Completeness — boundary case] The implementation spectrum lists three options (instructions, code templates, runtime resolution) but does not consider environment variables — the most common mechanism for installation-specific configuration in software systems. Environment variables read by the orchestrator at startup would be a fourth delivery mechanism that fits the note's own criteria (local, not committed, read before agent sees the task). Their omission is notable given how standard they are.

- [Grounding — inference validity] The note says "The orchestrator composes them when constructing the sub-agent frame" for recurring tasks, and cites the recurring task template type. The recurring task template (`kb/tasks/types/task-recurring.md`) is a minimal runbook template with no mention of configuration injection, orchestrator composition, or sub-agent frames. The note uses the recurring task pattern as a motivating example, but the cited template does not support the claims made about how orchestration works — the note is projecting a design onto an artifact that does not yet embody it. This is design intent presented as existing mechanism.

- [Internal consistency] The final section says injectable configuration is "specifically for values consumed by specific tasks in sub-agent frames" and that global preferences "belong in CLAUDE.md or `.claude/settings.json`." But the note's own example of "environment-specific endpoints" and "API keys that differ per machine" could plausibly be needed by many tasks globally, not just specific sub-agent frames. The boundary between "installation-specific value needed by many tasks" and "global setting" is not clearly drawn. A value like a staging URL might be needed by every task that hits an API — is it still "not a general settings system"?

PASS:
- [Grounding — frontloading alignment] The note correctly identifies that the frontloading note's "variable resolution — paths, project names, configuration values known at setup time" is the parent category. The partial evaluation framing ("the orchestrator is the specialiser, configuration values are the static inputs, and the residual is a task with concrete paths") is consistent with the frontloading note's PE formalism (program P specialized with static inputs s producing residual Ps). The analogy is correctly applied.

- [Grounding — scoping note alignment] The claim that "context savings come from frame isolation, not from the injection itself" is well-grounded in the scoping note, which describes sub-agents as "the one place where real isolation is achievable" with "a fresh context — its own system prompt, its own input, no inherited conversation history." The note correctly identifies that without sub-agent isolation, injected values just add to the flat window.

- [Grounding — auto-injection alignment] The note correctly characterizes auto-injection as resolving KB content based on document references, consistent with the source note's description. The "parallel but distinct" framing is reasonable — both are runtime-side resolution channels, differing in what they resolve and when.

- [Grounding — typed-callables alignment] The note's claim that "the orchestrator should inject only what the task declares it needs — the typed-callable pattern applied to configuration dependencies" is a reasonable extension of the typed-callables note, which proposes that skills declare type signatures for their inputs. Extending from document-type signatures to configuration-dependency declarations is a modest and clearly flagged inference.

- [Internal consistency — core argument] The note's core argument is internally consistent: (1) installation-specific values exist that cannot be committed, (2) they are static enough to frontload, (3) frontloading them requires a local config file plus orchestrator injection, (4) the context savings depend on sub-agent isolation. Each step follows from the previous without contradiction.

- [Internal consistency — comparison table] The four dimensions in the auto-injection comparison table (Source, Trigger, Varies by, Lifetime) are internally consistent and the "Both spare execution context" synthesis correctly qualifies the shared property with the sub-agent isolation dependency stated earlier in the note.

Overall: 3 warnings, 3 info
