=== SEMANTIC REVIEW: link-graph-plus-timestamps-enables-make-like-staleness-detection.md ===

Claims identified: 12

WARN:
- [Completeness] The note claims "No new annotation is needed" because "The link graph already encodes the dependency; git provides the timestamps." However, the link graph encodes that note A references note B, not *what* about B the note relies on. A link to WRITING.md might reference its title-as-claim convention, its template structure, or its frontmatter requirements. All three could change independently, but the timestamp mechanism treats any change to WRITING.md as equally relevant. The `make` analogy holds at the file level, but `make` targets have typed dependencies (header includes, library links) while KB links are untyped dependency pointers. The note partially acknowledges this in the filters section ("only flag notes that link to targets with specific relationship types"), but the central claim that no new annotation is needed is in tension with the observation that relationship type matters for filtering. Some form of annotation (at minimum, relationship type on links) would meaningfully improve precision. This is the gap between "technically works without annotation" and "works well without annotation."

- [Completeness] The three-step mechanism (walk link graph, compare timestamps, flag if newer) omits a boundary case: notes that link to *themselves* indirectly through a chain. If A links to B and B links to C, and C changes, A is not flagged even though A's reasoning may depend on C's content (mediated through B). The note's `make` analogy actually highlights this gap: `make` handles transitive dependencies natively (changing a header file triggers recompilation of all files that include files that include it). The note's mechanism is one-hop only, but the analogy it invokes is transitive. This is distinct from the "accumulated context changes" defeat condition the note identifies -- that defeat condition is about *unlinkable* drift, while transitive dependency is about *linked but multi-hop* drift that the mechanism could in principle detect but doesn't as described.

INFO:
- [Completeness] The three proposed noise filters (substantive change detection, relationship-type filtering, centrality weighting) are presented as a flat list, but they operate at different stages and have different implementation costs. Substantive change detection requires diff analysis at flag time; relationship-type filtering requires link annotations that the note elsewhere claims are unnecessary; centrality weighting requires precomputed graph metrics. The note would benefit from noting that these filters have different prerequisites, particularly since the relationship-type filter reintroduces the annotation the note claims to avoid.

- [Completeness] The note's scope section identifies the boundary at external references but does not address another boundary: links to non-note files. KB notes may link to scripts, configuration files, or instruction documents. The `make` analogy applies equally to these, but the staleness semantics differ -- a script changing may not invalidate a note that describes its purpose, while a configuration file changing may invalidate a note that documents its values. The mechanism as described would treat all link targets identically.

- [Grounding] The note claims it "adds a dependency-aware mechanism that is more precise than age-based heuristics" and cites quality-signals-for-kb-evaluation.md's "note-age-vs-connection-count" signal. The source note lists this signal as one item in a long catalogue under "Structural health" with the gloss "old notes with few connections may be stale or orphaned by topic drift." The characterization as an "age-based heuristic" is a reasonable shorthand but slightly inflates what the source proposes -- the source's signal is age *combined with* connection count (a compound signal), not age alone. The claim of superior precision still holds, but the comparison is against a compound signal, not a simple one.

- [Internal consistency] The note states "The cost of a false positive is 'agent re-reads a note unnecessarily.' That's cheap." But the note's own mechanism flags notes for review, and the linked note traversal-improves-the-graph.md argues that loading writing methodology to properly review a note is expensive enough to justify deferring improvements. If the "review prompt" triggered by this mechanism requires the same kind of careful re-reading that traversal-improves-the-graph.md deems expensive, then false positives may not be as cheap as claimed. The note may be conflating "re-reading" (cheap) with "evaluating whether the note's claims still hold given target changes" (potentially expensive, requiring loading both the note and its changed target in context).

PASS:
- [Grounding] The link to stale-indexes-are-worse-than-no-indexes.md is accurate. That note's core claim -- "presence of a stale index suppresses search entirely" -- is correctly characterized as identifying "the high cost of stale navigation that motivates detection." The staleness detection note positions itself as a response to this problem, and the motivation chain is sound.

- [Grounding] The defeat condition citing brainstorming-how-reach-informs-kb-design.md is well-grounded. That note explicitly states "High-reach changes (revising a foundational principle) may touch few files directly, but the downstream reasoning across many notes may silently break." The staleness detection note correctly identifies this as the specific failure mode where pairwise timestamp comparison breaks down, and the reach note even links back with a "contradicts/qualifies" relationship, confirming mutual awareness.

- [Grounding] The link to mechanistic-constraints-make-popperian-kb-recommendations-actionable.md with the relationship "its proposal-pruning pattern is a concrete use case for this detection" is reasonable. That note's falsifier blocks and contradiction-first connection passes would benefit from knowing which notes have become stale relative to their targets, making staleness detection a useful input to the Popperian review workflow.

- [Internal consistency] The note's core analogy (make's file-based rebuild logic) is used consistently throughout. The mechanism section, false positives section, and scope section all maintain the same mapping: links = dependencies, timestamps = modification times, flagging = triggering a rebuild recipe. The analogy is not stretched beyond its stated scope.

- [Internal consistency] The defeat condition section is internally consistent with the main claim. The note does not claim the mechanism catches all staleness -- it explicitly carves out the failure mode where "most staleness comes from notes whose claims drift due to accumulated context changes across many files." This self-limiting move prevents the note from overclaiming.

Overall: 2 warnings, 4 info
