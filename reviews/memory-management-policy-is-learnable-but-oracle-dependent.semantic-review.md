=== SEMANTIC REVIEW: memory-management-policy-is-learnable-but-oracle-dependent.md ===

Claims identified: 18

WARN:
- [Completeness] The six-operation decomposition ("Add, Update, Delete (LTM) and Retrieve, Summary, Filter (STM)") is presented as the complete set of memory operations AgeMem uses, and the note treats them as collectively exhaustive of memory management. But the note's own framing -- "knowing when to use them is hard" -- implies the hard part is composition policy, not operations. The ingest file's limitations section flags that the fixed tool set may miss needed operations ("linking two memories, tagging with metadata, or creating hierarchical summaries"). The note itself never raises this limitation from within its own analysis -- a boundary case like "link this fact to that fact" (a relational memory operation) falls outside all six items but is central to what the KB does. The note treats the six as given without questioning whether the policy it praises could be learning a suboptimal strategy because the operation set is incomplete.

- [Grounding] The note claims "AgeMem's STM operations (Retrieve, Summary, Filter) are distillation -- extracting and compressing from a larger body of information into the working context, shaped by the current task." The distillation note defines distillation as "targeted extraction from a larger body of knowledge under context-budget constraints." Summary and Filter fit this definition, but Retrieve is a lookup operation -- it fetches stored content, it does not extract or compress. Calling all three "distillation" stretches the term. Retrieve is more accurately a selection or recall operation; it becomes distillation only if the retrieval mechanism itself filters and compresses, which the AgeMem paper does not describe it doing. The individual Retrieve action returns a stored memory item.

- [Grounding -- domain coverage] The note uses the bitter-lesson boundary to frame AgeMem's architecture: "the operations are the calculator part (specifiable, deterministic), while the composition policy is the vision-feature part (learnable, benefits from scale)." The bitter-lesson note's core distinction is whether "the specification fully captures the problem." The note applies this by saying memory operations are fully specified (calculator) while the policy is not (vision feature). But AgeMem's operations are not fully specified in the bitter-lesson sense -- they are hand-crafted tools with designed interfaces, which is different from specifications that ARE the problem (like multiplication). "Add a key-value pair" is a human-designed interface to memory, not a formal specification of what memory storage IS. The operations are closer to the vision-feature side than the note acknowledges -- they are a theory of what memory operations look like, not a definition. The bitter-lesson note's own chess example is instructive: the rules of chess are the arithmetic part because they define legal play, not because they are hand-crafted tools. AgeMem's six tools define an API, not the problem.

INFO:
- [Completeness] The "What AgeMem teaches us" section lists four lessons. Lesson 3 ("Curriculum structure matters") claims that if KB learning is automated, "a similar curriculum -- learn to extract before learning to connect, learn to connect before learning to synthesise -- would likely be needed." This is a plausible inference but the evidence base is thin: AgeMem's curriculum success in a task-completion setting with clear rewards does not straightforwardly transfer to a domain the note itself identifies as lacking oracles. The curriculum worked because each stage had a measurable training signal. The KB's stages (extract, connect, synthesise) don't have equivalent signals, which is precisely the oracle gap the note documents. The lesson may still be right, but the analogy is weaker than it appears.

- [Completeness] The note frames a clean binary between "facts without reach" and "theories with reach," claiming AgeMem "never moves up the reach gradient from facts to theories." Boundary case: AgeMem's STM Summary operation produces compressed representations of multiple observations -- these are not raw facts but abstractions over facts. A summary like "keys are usually found in the kitchen" derived from multiple episodes would have modest reach (it transfers to new kitchen-related tasks). The note's reach gradient should have a middle zone for empirical generalizations that are above single facts but below theories. The learning-is-not-only-about-generality note does describe a spectrum, but this note collapses it to a binary.

- [Internal consistency] The note says "Hand-crafted heuristics of systems like A-MEM are plausible theories about memory management, not definitions of it" (section 2). But section 1 also treats AgeMem's six operations as fixed specifications ("their specs fully capture what they do"). If heuristic memory policies are "theories not definitions," the same argument could apply to the operation specifications themselves -- "Add a key-value pair" is a theory about what storing memory looks like, not a definition of memory storage. The note treats the operations as settled specifications while treating policies as unsettled theories, but the justification for this asymmetry is not made explicit.

- [Grounding] The note states "the step-wise GRPO mechanism broadcasts advantages across all timesteps specifically because memory operations (storing a fact early) are temporally distant from their payoff (using it later)." The ingest file describes the step-wise GRPO as broadcasting "advantages across all trajectory timesteps" but notes it is a mechanism for addressing "sparse rewards," not specifically temporal distance. Temporal distance is one reason rewards are sparse, but the note presents it as the specific causal reason for the mechanism's design, which slightly overstates the attribution.

- [Internal consistency] The final section acknowledges "Theoretical advantages in substrate design don't count for much while the loop is open" -- a strong concession that the KB's symbolic-substrate advantages are currently unrealized. But the "How it stores what it learns" section builds an extended argument for why the unified substrate is superior ("policies participate in the same accumulation loop as everything else"). The concession at the end partially undermines the authority of the substrate comparison earlier, though the note does flag this tension explicitly ("the comparison cuts both ways"). The tension is acknowledged but the weighting feels uneven -- the substrate comparison gets ~250 words of argument while the concession gets ~80 words.

PASS:
- [Grounding] The core claim that AgeMem's improvement (23-49% over no-memory baselines, 8-9 percentage points from RL) constitutes "genuine learning in Simon's sense" checks out against the learning-is-not-only-about-generality note, which defines Simon's criterion as "any change in a system that produces a more or less permanent change in its capacity for adapting to its environment." The RL-trained policy is durable and improves adaptation. The attribution is accurate.
- [Grounding] The claim that the KB's learning problem "lacks" the task-completion oracle AgeMem uses is well-grounded in the automating-kb-learning note, which identifies this gap as the core open problem ("we need more usage before we can design the learning loop properly") and explicitly names evaluation as the bottleneck.
- [Grounding] The learning-substrates note accurately supports the claim that "AgeMem puts the governing policy in a subsymbolic substrate, while commonplace tries to keep both policies and facts in a symbolic artifact substrate." The taxonomy table in that note directly confirms this framing.
- [Internal consistency] The note maintains a consistent definition of "oracle" throughout -- always meaning a verifiable evaluation signal for whether a learning step succeeded. This term does not drift across sections.
- [Internal consistency] The note's compressed description ("AgeMem stores facts in memory but learns the governing policy in weights; it is a clean subsymbolic case of durable learning, but one that depends on task-completion oracles the KB lacks") faithfully represents the body. No elided tensions found between frontmatter description and body content.
- [Completeness] The decomposition of AgeMem's operations into LTM (storage-side) and STM (context-side) is consistent with the source paper as represented in the ingest file. The three-stage curriculum mapping (LTM first, STM second, coordination third) matches the ingest's description.

Overall: 3 warnings, 4 info
