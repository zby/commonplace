WARN:
- [Completeness] The central framework claims verification cost is "the boundary" and "the primary structural determinant" of automation. Boundary case: **tasks with cheap verification that still resist automation for non-verification reasons**. The note's own Caveats section acknowledges "regulatory constraints, trust requirements, liability concerns" as alternative blockers, and the title uses "the" boundary despite the body defending "the primary structural" boundary. But the framework offers no mechanism for weighting verification cost against these other factors — it merely acknowledges them as exceptions. A task like automated medical diagnosis may have reasonably cheap verification (compare to expert panel) yet resist automation due to liability and trust requirements. The caveat exists but the framework has no way to predict when these "other factors" dominate, which limits the claimed generality.

- [Grounding alignment] The note states Rabanser et al. find "discrimination trends are mixed — improving on some benchmarks, worsening on others." The source (Section 4.2, Predictability) says: "discrimination trends diverge across benchmarks: on tau-bench it has generally improved in recent models, whereas on GAIA it has in fact mostly worsened." The note's characterization as "mixed" is accurate but the note then infers this "suggest[s] self-assessment is not reliably scaling, which favors external oracles." This is the note's own inference — Rabanser et al. do not draw this conclusion. The source frames discrimination as a dimension "requiring immediate research focus" (Section 4.2) and recommends "optimizing agents directly for reliability dimensions" (Recommendation 2), which implies the authors see discrimination as improvable through training, not as a structural ceiling favoring external oracles. The note's inference is reasonable but could mislead readers into thinking the source itself endorses the external-oracle conclusion.

- [Completeness] Boundary case: **tasks where verification is expensive but automation proceeds anyway because errors are self-correcting or low-consequence**. The note's Caveats section (error-cost tolerance) acknowledges this: "Some tasks get automated despite poor verification because errors are cheap." But the caveat is listed as a separate variable rather than integrated into the framework. The title claim — "the boundary of automation IS the boundary of verification" — does not accommodate this. If error-cost tolerance can independently determine automation viability, then verification cost is a necessary-but-not-sufficient condition, not "the boundary." The caveat undermines the title claim more than the note acknowledges.

INFO:
- [Completeness] Boundary case: **tasks where verification is cheap but generation is genuinely hard**. The framework predicts these should be automatable, since the bottleneck is "oracle construction, not generation." Novel mathematical theorem proving has cheap verification (formal proof checkers) but extremely difficult generation. The framework would predict this is automatable — and indeed recent AI systems have made progress here — so this case arguably supports the framework. However, the note never discusses domains where cheap verification coexists with generation difficulty, which would strengthen the argument if addressed.

- [Grounding alignment] The note describes Amodei's confidence pattern — "strong optimism on coding and math... but acknowledged uncertainty on novel writing and scientific discovery" — and interprets this as "his confidence correlates with verification availability, not raw capability claims." The source (Section "On Verification and Uncertainty") says: "Amodei admits higher uncertainty on tasks lacking objective verification — novel writing, scientific discovery, mission planning. For verifiable domains (coding, mathematics), confidence extends to 'almost certain' timelines." The note accurately represents the pattern and transparently labels it as "our interpretive frame." However, an alternative interpretation is available: Amodei's confidence may correlate with benchmark availability (he has data on coding progress, not on novel-writing progress) rather than with verification as a structural determinant. The note flags this ("Amodei's confidence split could be mere selection bias") in the convergence section but only as a dismissible single-source concern, not as an interpretation that survives the convergence argument.

- [Internal consistency] The opening paragraph states the claim applies "regardless of raw model capability," which is a strong universality claim. The Caveats section then lists multiple exceptions: error-cost tolerance, oracle gaming, regulatory constraints. The body defends "the primary structural" boundary, softening "regardless" to "typically." The compressed summary in the opening is stronger than the body supports. This is acknowledged as a tension in Caveat 5 ("The title uses 'the' boundary as a claim title, but the argument defends verification as 'the primary structural' boundary, not the only one"), which is a good self-aware hedge, but the opening paragraph's "regardless" still overreaches relative to the caveated position.

- [Grounding alignment] The note characterizes the three sources as arriving "from different domains, using different vocabulary." It then acknowledges "the oracle-theory notes already cite Tam et al. and Rabanser et al." The note hedges this appropriately ("the shared citations between the sources temper the evidential weight"). However, the Tam et al. ingest was itself explicitly framed by the KB's oracle vocabulary — the ingest file's description reads "oracle availability (not capability) determines automation boundary." The Amodei source's description in the KB reads "verifiable domains get confident timelines, unverifiable ones get hedged, implicitly confirming oracle-strength thesis." Both source snapshots were captured and described through the lens of the framework they are being used to support, which means the "convergence" may partly reflect consistent interpretive framing rather than independent arrival at the same conclusion.

- [Completeness] Boundary case: **the simplest possible instance — a task where verification is trivially cheap**. Addition of two numbers: verification is free (just re-compute), generation is also trivial. The framework predicts automation, and this is trivially true. More interesting: a task where verification is trivially cheap but the domain is contentious — e.g., spell-checking. Verification is cheap (dictionary lookup), and indeed spell-checking was automated decades ago. The framework handles the simplest cases cleanly.
