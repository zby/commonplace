=== SEMANTIC REVIEW: llm-mediated-schedulers-are-a-degraded-variant-of-the-clean-model.md ===

Claims identified: 11

1. [Intro] "The symbolic scheduling model assumes the scheduler is a program with unbounded exact state."
2. [Intro] "many current systems (Claude Code, Codex, chat-based agent loops) carry orchestration state partly in an LLM conversation"
3. [Intro] "The LLM serves as both scheduler and executor"
4. [Intro] "it suffers the same attention dilution and compositional overhead as the sub-agent calls it is trying to orchestrate"
5. [Intro] "The clean separation between unbounded scheduler and bounded LLM calls collapses"
6. [Recovery] "Three responses restore the separation to increasing degrees" — enumeration of exactly three, claimed to be ordered
7. [Recovery/Compaction] "Keep summaries and conclusions rather than raw results in the conversation, applying distillation to the scheduler's own state. This reduces degradation but does not eliminate it."
8. [Recovery/Externalisation] "Write intermediate state to files and re-read selectively. This moves scheduler state out of the conversation and into exact symbolic state outside the LLM context — partially recovering the clean model."
9. [Recovery/Externalisation] The Ralph Loop "combines externalisation with extreme compaction: a hook intercepts the model's exit attempt and reinjects the original prompt in a clean context window, while the filesystem bridges iterations"
10. [Recovery/Factoring] "Encode the bookkeeping and recursion as a program that runs outside the LLM conversation entirely. This fully recovers the clean model."
11. [Closing] "the architectural direction is toward the third option" — claim of a single correct direction

WARN:
- [Completeness] The three recovery strategies are presented as exhaustive and ordered by degree of separation recovered, but a fourth strategy is plausible: **hybrid delegation**, where the LLM-mediated scheduler spawns sub-agents with clean context windows for specific sub-tasks while itself remaining conversation-mediated for top-level decisions. This is distinct from all three listed strategies — it is not compaction (no compression of the scheduler's own state), not externalisation (the scheduler's state stays in the conversation), and not factoring into code (the scheduler is still an LLM). Yet it partially recovers the clean model by isolating sub-task execution from accumulated scheduler state. The note's own link to [tool loop](./tool-loop-index.md) discusses sub-agent spawning as a forcing case, and the orchestration model note discusses "parallel calls" and "hierarchical decomposition," suggesting the KB itself recognizes this pattern — but this note's enumeration does not account for it. The boundary case sits between externalisation and factoring: the scheduler remains LLM-mediated but delegates bounded work to clean sub-agent calls.

- [Completeness] The claim that factoring into code "fully recovers the clean model" omits a significant caveat: even when the scheduler is code, the `select` function may itself require LLM judgment (the orchestration model note explicitly says "Some select decisions come from planning calls"). If `select` requires LLM planning calls, the scheduler is not purely symbolic — it contains bounded LLM calls for its own decision-making. The note says factoring "fully recovers the clean model," but the parent model it references acknowledges that planning calls are part of the scheduler. This is not a contradiction with the parent note (which accommodates planning calls returning to K as symbolic state), but it does mean "fully recovers" overstates the cleanliness — the scheduler still has stochastic components even in the factored case.

- [Grounding] The note claims the Ralph Loop "combines externalisation with extreme compaction" and characterizes the context reset as "compaction (taken to its logical extreme of full context reset)." But a full context reset is not compaction — compaction means keeping summaries and conclusions (as the note itself defines in strategy 1). A full context reset discards everything; the filesystem provides continuity, not a compressed version of prior context. Calling a full reset "extreme compaction" conflates two mechanisms: (a) compressing state into a smaller representation (compaction proper) and (b) discarding conversational state while persisting it externally (externalisation). The Ralph Loop is better described as pure externalisation with zero in-conversation state, not as a combination of externalisation and compaction. The ingest source itself describes it as "a hook intercepts the model's exit attempt and reinjects the original prompt in a clean context window" — the prompt is reinjected, not a compressed summary.

INFO:
- [Completeness] The note claims compaction "reduces degradation but does not eliminate it." This is plausible but stated without qualification about how much reduction is typical or when compaction is sufficient versus insufficient. The distillation note warns that "a distillate can look adequate while losing behavioral influence" — this risk applies directly to compaction of scheduler state (summaries of prior tool results may lose the specific details needed for correct next-step decisions). The note could acknowledge this failure mode rather than presenting compaction as a straightforward partial fix.

- [Completeness] The ordering claim ("to increasing degrees") implies a strict ordering of recovery quality: compaction < externalisation < factoring. But compaction and externalisation are not always strictly ordered. A system that aggressively compacts its conversation state (keeping high-quality summaries) may recover more clean-model properties than a system that externalises to files but reads them back poorly (loading too much, losing structure). The ordering holds as a tendency but is not as strict as the framing implies.

- [Internal consistency] The note's description field says "three recovery strategies — compaction, externalisation, factoring into code — restore the clean separation to increasing degrees." The body confirms this structure. However, the closing sentence says "the architectural direction is toward the third option," which is an evaluative claim that goes beyond the descriptive framework of "here are three strategies ordered by recovery degree." The note shifts from descriptive taxonomy to normative recommendation in the final sentence without arguing for why systems should prefer maximum recovery over context-appropriate recovery. A system where the scheduling task is simple enough for compaction to suffice would not benefit from factoring into code.

- [Grounding] The note links to [context efficiency](./context-efficiency-is-the-central-design-concern-in-agent-systems.md) with the relationship "cost model: the degradation is a context-efficiency problem within the scheduler itself." The context-efficiency note discusses context scarcity for agent execution, not specifically for schedulers. The note's inference — that a scheduler running inside a conversation faces the same context-efficiency pressures — is reasonable but is the note's own extension, not something the context-efficiency note explicitly addresses. The link label "cost model" slightly overstates the directness of the grounding.

PASS:
- [Grounding] The note's central claim — that carrying orchestration state in an LLM conversation makes the scheduler bounded — is well-grounded in the linked orchestration model note, which explicitly states: "A system that keeps bookkeeping inside an LLM conversation is a degraded variant that spends bounded context on work the symbolic scheduler handles for free." The attribution is accurate and the relationship (degraded variant of the clean model) is faithfully represented.
- [Grounding] The link to distillation for strategy 1 (compaction) is accurate. The distillation note defines distillation as "compressing knowledge so that a consumer can act on it within bounded context." Compaction of scheduler state — keeping summaries rather than raw results — fits this definition. The note correctly identifies compaction as "distillation applied to the scheduler's own state."
- [Grounding] The Ralph Loop description aligns with the ingest source, which describes the pattern as: "a hook intercepts the model's exit attempt and reinjects the original prompt in a clean context window, forcing continuation against a completion goal. The filesystem bridges iterations." The note's characterization is faithful to this source.
- [Grounding] The link to [tool loop](./tool-loop-index.md) is appropriate. The tool-loop index explicitly lists this note as a consequence ("workarounds for hidden loops push scheduling into the conversational medium") and the note's framing — that framework design should expose control surfaces — aligns with the tool-loop argument about when the framework-owned loop becomes insufficient.
- [Internal consistency] The three strategies are internally consistent with each other and with the ordering claim. Each genuinely moves more state out of the conversation than the previous one: compaction reduces in-conversation state, externalisation moves it out, factoring eliminates the LLM-as-scheduler entirely. No section contradicts another.

Overall: 3 warnings, 4 info
