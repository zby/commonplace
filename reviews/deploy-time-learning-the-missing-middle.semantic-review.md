WARN:
- [Completeness — Three Timescales] The note claims exactly three timescales of adaptation. Boundary case: **retrieval-augmented generation (RAG) with a continuously updated knowledge store**. A system with a vector database that is updated between sessions but not by the agent itself (e.g., by an ETL pipeline) adapts durably — the behavior changes across sessions — but not through "symbolic artifacts" in the sense the note intends (inspectable, diffable, testable). The note's substrate description for deploy-time learning ("symbolic artifacts ... durable, inspectable, and verifiable") does not clearly cover opaque embedding stores or continuously updated retrieval corpora where the adaptation is durable but not fully inspectable or diffable. The linked note [learning-substrates-backends-and-artifact-forms](kb/notes/learning-substrates-backends-and-artifact-forms.md) acknowledges vector stores as a backend, but the three-timescale table in this note ties deploy-time learning specifically to properties (inspectable, verifiable) that vector-store-backed adaptation may not fully satisfy. This boundary case sits uncomfortably between "training" (opaque, durable) and the note's "deploy-time learning" (inspectable, durable).

- [Completeness — Two Mechanisms] The note claims deploy-time learning operates through exactly "two mechanisms — constraining and distillation." Boundary case: **curation and deletion** — removing a note, deprecating a prompt, or pruning a skill library. The constraining note defines constraining as "narrowing the interpretation space," and distillation is "compressing knowledge so a consumer can act on it within bounded context." Deleting an obsolete artifact changes future system behavior durably but is neither narrowing an interpretation space nor compressing knowledge. It is closer to forgetting or pruning. One could stretch "constraining" to cover deletion (the space of loaded artifacts narrows), but that strains the definition, which is about narrowing *interpretations of a spec*, not narrowing the artifact inventory. Similarly, reorganization (moving artifacts between directories, changing routing) changes behavior without obviously being either mechanism.

- [Grounding — SECI Mapping] The note claims "Constraining and distillation map to externalization and combination phases" of Nonaka & Takeuchi's SECI spiral. The SECI spiral has four phases: Socialization (tacit-to-tacit), Externalization (tacit-to-explicit), Combination (explicit-to-explicit), and Internalization (explicit-to-tacit). The note maps constraining and distillation to only two of four phases, which is fine — but the mapping itself is questionable. Distillation (compressing explicit knowledge for a bounded consumer) maps more naturally to *Combination* (recombining explicit knowledge into new forms). Constraining (narrowing interpretation space by adding structure) is arguably *Externalization* (making tacit understanding of "what we actually mean" explicit in schemas, tests, code). However, the note says "constraining and distillation map to externalization and combination" without specifying which maps to which, and the unstated implication that the mapping is clean deserves scrutiny. Additionally, the note self-flags this citation as unsystematic (the TODO on line 92), which is appropriate.

- [Grounding — Double-Loop Learning] The note claims "Deploy-time learning is double-loop learning for agent systems — constraining revises the rules, not just the outputs." Argyris & Schon's double-loop learning specifically involves questioning and modifying the governing variables (norms, policies, objectives) that frame decision-making, as opposed to single-loop learning which adjusts actions within existing frames. Much of the verifiability gradient — structured output schemas, prompt tests, deterministic modules — looks more like single-loop learning: the system gets better at executing within established frames. Only some constraining acts (e.g., revising the fundamental structure of a prompt, changing what the system is *trying to do*) qualify as double-loop. The blanket equation "deploy-time learning is double-loop learning" overstates the mapping; deploy-time learning *includes* double-loop learning but also includes plenty of single-loop hardening.

INFO:
- [Completeness — Verifiability Table] The four-grade verifiability table (restructured prompts, structured output schemas, prompt tests, deterministic modules) is presented as a gradient, not an exhaustive enumeration, so gaps are less concerning. Still, **type-checked prompt templates** (e.g., Jinja templates with typed variables, or prompt pipelines with schema-validated intermediate outputs) sit between "restructured prompts" and "structured output schemas" in a way that is not clearly covered. They constrain inputs rather than outputs. The table focuses on output-side verification; input-side verification (constraining what goes *into* the prompt) is a related but distinct grade not represented.

- [Grounding — Kirsch Attribution] The note says Kirsch's "framing lacks the verifiability gradient (different parts of the stack harden to different degrees)." The Kirsch ingest confirms this is accurate — Kirsch argues for durable artifact stacks but does not introduce a gradient of hardening levels. The note's characterization is fair. However, the note also says Kirsch provides "useful public vocabulary for the durable-artifact claim." The ingest's own assessment is that Kirsch's "code remains the source of truth" is actually *weaker* than the KB's "durable inspectable artifacts remain central." The note's framing of Kirsch as providing vocabulary for *the note's own claim* slightly overstates the alignment — Kirsch's vocabulary is for a code-centric version of the claim, not the broader artifact-stack version.

- [Consistency — "Not a Training Paradigm" vs. OpenClaw-RL] The note says "Deploy-time learning is not a new training paradigm — the model weights don't change" and then immediately acknowledges OpenClaw-RL as a counterexample where "Weight updates during deployment are possible in principle." The note handles this tension explicitly ("training infrastructure is too heavy for most deployment contexts"), so it is not a contradiction. However, the three-timescale table draws a clean boundary where training happens "before deployment" and deploy-time learning happens "across sessions, during deployment" — the OpenClaw-RL case blurs this boundary since it performs training *during* deployment. The table's clean separation is a simplification that the prose acknowledges but the table does not.

- [Completeness — Failure Modes] The four failure modes (premature codification, Goodharting on evals, model drift, bad assumptions codified) are all about over-constraining or mis-constraining. A complementary failure mode — **under-constraining** or failure to learn (the system accumulates artifacts but never hardens them, so behavior remains as variable as session one) — is not listed. This is the "knowledge base entropy" problem: artifact volume grows but system capability does not improve because nothing is verified or pruned.

PASS:
- [Internal Consistency — Constraining/Distillation] The definitions of constraining and distillation in the note ("narrowing the interpretation space" and "extracting procedures from reasoning") are consistent with the dedicated constraining.md and distillation.md notes. The constraining note defines it as "narrows the interpretation space" and distillation.md defines it as "compressing knowledge so that a consumer can act on it within bounded context." The deploy-time note's shorthand "extracting procedures from reasoning" is a slight narrowing of distillation's full definition (which is about compression, not just procedure extraction), but this is acceptable compression for a non-definitional reference.

- [Grounding — Karpathy Verifiability] The note attributes three properties (resettable, efficient, rewardable) to Karpathy and links to the source. The application of these properties to the verifiability gradient table is the note's own contribution, not attributed to Karpathy, and this distinction is clear in the text ("Deploy-time learning applies this insight to symbolic artifacts").

- [Internal Consistency — Body vs. Summary] The closing summary (line 94) says deploy-time learning is "a taxonomy (three timescales) and a verifiability gradient (from prompt tweaks to deterministic code)." This faithfully represents the body's two main sections. No elision of tensions detected.

- [Grounding — Agile Comparison] The claim that "agile assumes code wins eventually; deploy-time learning treats the hybrid as the end state" is checked against deploy-time-learning-is-agile-for-human-ai-systems.md, which elaborates: "Agile treats natural-language specs as temporary — stories waiting to become code. The implied end state is fully codified software. Deploy-time learning recognises that some parts should stay in prose." The parent note's compressed version is consistent with the elaborated argument.

- [Internal Consistency — Verifiability Table] The four grades in the table are internally ordered correctly along all three Karpathy dimensions. Each row increases in at least one dimension relative to the row above. The prose description ("verification gets cheaper and sharper") matches the table contents.

Overall: 4 warnings, 4 info
