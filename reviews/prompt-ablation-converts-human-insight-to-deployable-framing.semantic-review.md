=== SEMANTIC REVIEW: prompt-ablation-converts-human-insight-to-deployable-framing.md ===

Claims identified: 15

WARN:
- [Completeness] The 8-step pattern (The pattern, steps 1-8) claims to be a complete methodology, but has no step for validating that the deployed framing generalizes beyond the original test case. Step 7 says "this is what transfers beyond the specific test case," but the pattern itself contains no validation of that transfer. You learn which cognitive move worked on one target finding, then deploy it as a general instruction. The boundary case: a framing that reliably surfaces one class of finding (e.g., pointless complexity) but actively suppresses another class (e.g., subtle correctness errors) would score well in this pattern and deploy as a general step, creating a blind spot. The pattern has no mechanism for detecting this — "bonus findings" (step 6) partially address it but are not scored or weighted.

- [Grounding — domain coverage] The note claims "The deployed framing trades depth for reliability — good enough, not equivalent" as a general property of the conversion. But this is grounded in exactly one experiment (curiosity-prompts) with one target finding, on one system (Decapod), using one model. The experiment report itself notes cost/benefit scored 2/2, which is the minimum run count the note recommends. The note presents a single data point's outcome as a general characterization of the pattern's output. The note's own open questions section acknowledges "2 is minimum for detecting consistency; it may be insufficient for close calls" — yet the "Why it works" section draws confident conclusions from exactly that minimum.

- [Grounding — scope mismatch] The note states "Four of six converged on 'does it work?'" (Design constraints section), but the experiment report is more precise: it says impossibility, implications, adversarial, and mechanistic converged on that question — however, adversarial scored Partial/Partial, meaning it did NOT simply verify and move on. The adversarial framing found the OVERRIDE.md escape hatch and proof self-attestation, which are genuine findings beyond "does it work?" Collapsing adversarial into the "converged on does it work?" group mischaracterizes the experiment's own findings to strengthen the rhetorical point about needing different cognitive moves.

INFO:
- [Completeness — boundary case] The pattern assumes a binary outcome: there is a winning framing, or there isn't. Boundary case: what happens when two framings score equally well but via different mechanisms (e.g., both hit Yes but one through cost/benefit reasoning and the other through mechanistic tracing)? Step 7 says "analyze mechanisms, not just scores," which partially addresses this, but step 8 says "the winning framing (or combination) becomes a step." The parenthetical "(or combination)" does significant work but gets no methodological guidance. The experiment report's actual recommendation was indeed a combination (curiosity + cost/benefit + oracle-strength), suggesting this case is common, not exceptional.

- [Grounding — vocabulary] The note uses "ablation" in its title and introduction: "'Ablation' because only the framing varies — input, tools, and model are held constant." In experimental methodology, ablation typically means *removing* components to test their contribution (ablation study = what breaks when you remove X?). This pattern is closer to a controlled variation or A/B test — it adds/substitutes different framings, it does not remove components from a working system. The analogy is suggestive but technically imprecise. The note does explain its usage, so this is naming, not a substantive error.

- [Grounding — deployment claims] The note says the curiosity-prompts experiment produced the "Curiosity Pass" in the related-system template and links to `kb/notes/types/related-system.md`. The template does contain a "Curiosity Pass" section with the three-layer structure described. However, the note also says both deployments "include the oracle-strength question ('what could this achieve even if it works perfectly?') despite inconsistent agent performance." The experiment report's conclusion #5 states this question is "the most powerful but hardest to prompt for" and that "whether agents can be prompted to ask this systematically remains an open question." Including a question that agents inconsistently execute in a deployed template is a defensible design choice (the note explains the rationale), but the word "despite" underplays the tension — the experiment's evidence for this question's value comes from the *human* reviewer, not from agent runs where it mostly did not fire.

- [Internal consistency] The note characterizes the human finding as "oracle-strength but not reusable" (opening paragraph). The "Why it works" section then says "The experiment uses the human's finding as a hard oracle." These use "oracle" in different senses: the first means "the human's reasoning quality is oracle-level" (a competence claim), the second means "the finding serves as a ground truth for scoring" (a methodological role). Both uses are defensible and the oracle-strength-spectrum note does support the distinction between oracle as verification quality and oracle as scoring criterion. But the two senses are close enough to cause confusion on a quick read.

- [Completeness — boundary case] The open questions section asks whether synthetic targets can substitute for human-discovered findings and warns they "risk testing for the wrong thing — a planted flaw is obvious by construction." Boundary case: what about targets discovered by a *different* agent or model, rather than by a human? This sits between "human-discovered" and "synthetic" — it is a genuine miss found by a capable system but not by the specific system being tested. The note's framing assumes the oracle source is either human or synthetic, but the more relevant dimension may be whether the target represents a *natural* miss versus a *constructed* one.

PASS:
- [Internal consistency] The 8-step pattern is internally consistent. Each step follows logically from the previous one, and the design constraints section reinforces rather than contradicts the pattern. The "Why it works" section provides a coherent rationale that aligns with the pattern's structure.

- [Grounding — oracle-strength-spectrum] The note's central framing — using a known finding as a hard oracle to evaluate which soft question (framing) works — accurately reflects the oracle-strength-spectrum note's definitions. Hard oracle = exact, cheap, deterministic check; the known finding serves exactly this role. Soft oracle = proxy that correlates but is not the real thing; "which framing works" is indeed a soft question. The attribution is sound.

- [Grounding — experiment report] The note's empirical claims about the curiosity-prompts experiment (cost/benefit scored 2/2, curiosity scored 1.5/2, six framings tested, mechanism analysis showing "does it work?" vs "what can it achieve?" distinction) accurately reflect the experiment report's data and analysis. The scores, framing names, and structural conclusions match.

- [Grounding — unit testing link] The note's claim that prompt ablation and unit testing of LLM instructions "both test instructions against known expectations, using different mechanisms" accurately characterizes the relationship. The unit-testing note tests skill behavior via mock tool boundaries; this note tests prompt framings via known findings. Both use known-correct targets as oracles. The "complements" relationship is well-grounded.

- [Internal consistency — open questions] The open questions section honestly acknowledges the pattern's limitations (run count sufficiency, synthetic targets, extension beyond review framings) without contradicting the body's claims. The body presents the pattern as useful; the open questions identify where it is untested. No tension between these.

Overall: 3 warnings, 4 info
